Por que isso importa?

Até agora, todo o seu Git viveu dentro da sua máquina. Isso é suficiente para você trabalhar sozinho — mas programar profissionalmente é um esporte coletivo. Times de dezenas (ou centenas) de engenheiros precisam trabalhar no mesmo código ao mesmo tempo, sem pisarem uns nos pés dos outros, sem perderem trabalho, sem introduzirem bugs silenciosos no código de outra pessoa.

O GitHub é a infraestrutura social que torna isso possível. Ele hospeda repositórios remotos na nuvem, mas o que o torna especial é o conjunto de ferramentas de colaboração: Pull Requests para propor e revisar mudanças de forma estruturada, Issues para rastrear problemas e ideias, Projects para organizar o trabalho visualmente. É aqui que o código ganha contexto, conversa, justificativa e história permanente.

Não saber usar Pull Requests é, no mercado atual, como saber escrever mas não saber usar e-mail. É o mecanismo de entrega de código em praticamente 100% das empresas de tecnologia do mundo. Vamos dominar isso de forma profunda.

💼 Como times sênior usam isso no dia a dia

Em uma empresa como Nubank, Stripe ou Google, nenhum código vai para produção sem um Pull Request aprovado por pelo menos um colega. O PR não é burocracia — é o ponto de encontro entre quem escreveu e quem vai manter. O revisor pergunta "por que você fez assim?", o autor explica, ambos aprendem. É a forma mais eficiente de disseminar conhecimento de arquitetura no time. Times maduros têm regras como: PR com mais de 400 linhas não passa (deve ser quebrado em partes menores), revisão deve acontecer em menos de 24h, e todo PR deve ter testes passando. Essas convenções são o que separa engenharia séria de amadorismo.

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Imagine uma grande editora de livros. Existe um único manuscrito oficial guardado no cofre central da editora — essa é a versão definitiva que vai para impressão e distribuição. Nenhum editor mexe diretamente nesse original.

Quando um autor quer contribuir com um capítulo novo, ele solicita uma cópia do manuscrito para levar para casa e trabalhar. Em casa, com suas canetas e anotações, ele trabalha em seus próprios rascunhos. Quando termina, ele não simplesmente substitui o original — ele entrega à editora uma proposta formal e documentada: "aqui estão as alterações que fiz, no capítulo 3, pelas razões X e Y, e aqui está como testar se está correto."

A editora então convoca uma reunião de revisão com outros autores e editores. Eles leem a proposta, fazem perguntas, sugerem ajustes, e só depois de um consenso coletivo, as alterações são incorporadas ao manuscrito oficial. Se alguém identificar um problema no original antes disso, deixa um bilhete descritivo na mesa da editora descrevendo o problema com todos os detalhes. O quadro na parede da editora que mostra o status de cada capítulo? Isso é o GitHub Projects.

Parte 1 — Repositório Remoto: local vs nuvem

Nas aulas anteriores, seu repositório Git existia apenas no seu computador — isso é chamado de repositório local. O GitHub hospeda uma segunda cópia desse mesmo repositório em servidores na nuvem. Essa cópia na nuvem é chamada de remote (repositório remoto).

Você pode ter quantos remotes quiser apontando para lugares diferentes. Por convenção universal, o remote principal (o do GitHub) recebe o apelido origin. Quando você trabalha em contribuições open source, costuma ter dois: origin (seu fork) e upstream (o repositório original). Veremos isso em detalhes.

Push, Pull e Fetch — a diferença exata

Estes três comandos são a base da sincronização. A confusão entre eles é a causa de 80% dos problemas que iniciantes enfrentam com o Git remoto. Veja a tabela abaixo e guarde para sempre:

Comando Direção O que faz exatamente Quando usar
git push Local → Remote Envia seus commits locais para o GitHub. O remote fica atualizado com o que você tem localmente. Após fazer commits que deseja compartilhar com o time
git fetch Remote → Local (seguro) Baixa as mudanças do GitHub sem tocar no seu código. Atualiza apenas o rastreamento interno (origin/main) do que existe no remote. Quando quer ver o que o time fez sem arriscar conflitos agora
git pull Remote → Local (direto) Equivale a git fetch + git merge em um único comando. Baixa E integra ao seu branch atual. Para atualizar seu código com o trabalho mais recente do time
git clone Remote → Local (novo) Cria uma cópia completa de um repositório remoto na sua máquina, incluindo todo o histórico de commits desde o início. Primeira vez que você baixa um projeto
⚠️ Por que git pull pode criar problemas (e como evitar)

git pull faz fetch + merge. Se você tem commits locais que divergem do remote (ou seja, tanto você quanto seu colega fizeram commits desde a última sincronização), o merge pode criar um merge commit desnecessário — um nó extra no histórico que não representa trabalho real, só uma sincronização técnica. Em um projeto ativo isso polui o histórico rapidamente.

A alternativa mais limpa é git pull --rebase, que coloca seus commits locais sobre o que veio do remote, mantendo o histórico como uma linha reta. Times experientes configuram isso como padrão global: 

git config --global pull.rebase true

O que é origin afinal?

Quando você conecta seu repositório local ao GitHub, o Git precisa de um apelido (alias) para referenciar esse endereço remoto. Por convenção universal, esse apelido é chamado origin. Não tem nada de mágico — você poderia chamar de github, nuvem, ou qualquer palavra. Mas origin é o padrão que todo repositório do planeta usa, então seguir essa convenção evita confusão para qualquer pessoa que leia seu código.

Você pode ver todos os remotes configurados a qualquer momento:

terminalbash
git remote -v
# origin  https://github.com/seu-usuario/seu-repo.git (fetch)
# origin  https://github.com/seu-usuario/seu-repo.git (push)

# Em um fork com upstream configurado:
# origin    https://github.com/SEU-USUARIO/repo.git (fetch)
# origin    https://github.com/SEU-USUARIO/repo.git (push)
# upstream  https://github.com/DONO-ORIGINAL/repo.git (fetch)
# upstream  https://github.com/DONO-ORIGINAL/repo.git (push)

Parte 2 — Clone vs Fork: qual a diferença real?

Esta é umas das confusões mais frequentes entre quem está começando. Clone e Fork têm nomes parecidos e produzem resultados parecidos — uma cópia do repositório — mas funcionam em contextos completamente diferentes e têm implicações muito distintas.

📥
git clone
Copia um repositório do GitHub (ou de qualquer URL Git) para a sua máquina local. O "dono" do repositório no GitHub não muda — você só cria uma cópia local para trabalhar. Usado quando você tem permissão de escrita no repositório (você mesmo criou, ou é membro do time com acesso).
🍴
Fork (botão no GitHub)
Cria uma cópia independente e completa do repositório dentro da sua conta GitHub, na nuvem. A cópia é inteiramente sua — você pode fazer o que quiser nela sem afetar o original. Usado quando quer contribuir para projetos aos quais não tem acesso de escrita (projetos open source, repos de outras pessoas).

💡 O fluxo canônico de open source

Fork → Clone do fork → Criar branch → Commits → Push para o fork → Pull Request para o original.

Esse é o caminho exato que qualquer contribuição para projetos como React, Vue, Python, Linux, VS Code ou Django percorre. Quando você entender e dominar esse fluxo, o ecossistema inteiro de open source se abre para você. Você pode contribuir com qualquer projeto do mundo, mesmo sem ser convidado.

Parte 3 — Pull Requests: o coração da colaboração

Um Pull Request (PR) é uma proposta formal e documentada de mudança no código. Ao abrir um PR, você está dizendo: "Eu fiz estas alterações, neste branch, por estas razões — por favor, revisem, comentem, e se concordarem, incorporem ao main." O nome vem historicamente da ideia de pedir para o dono do repositório fazer um pull das suas mudanças para dentro do projeto dele.

Anatomia de um Pull Request de qualidade

📝
1. Título preciso
Curto, ativo e descritivo. "feat: adiciona validação de CPF no cadastro" — não "atualização" ou "fix stuff". O título é o que aparece no histórico para sempre.
📋
2. Descrição com contexto
O porquê da mudança: qual problema resolve, qual abordagem foi escolhida, quais alternativas foram consideradas e descartadas. Sem esse contexto, o revisor vai perguntar tudo isso.
🔗
3. Issue relacionada
Closes #42 na descrição liga o PR a uma Issue. Quando o PR for mergeado, a Issue fecha automaticamente. Cria trilha auditável: "esse bug existia → esse PR resolveu."
🧪
4. Como testar
Passo a passo para o revisor verificar que a mudança funciona. "Acesse /checkout com carrinho cheio → espere carregar → veja que não trava mais."
📸
5. Screenshots (quando visual)
Para mudanças de interface, um antes/depois visual comunica em segundos o que parágrafos não conseguem. Parte obrigatória em times de produto.
✅
6. Checklist de PR
Times profissionais têm templates com perguntas: testes passam? documentação atualizada? não há breaking changes? variáveis de ambiente documentadas?

O ciclo de vida completo de um PR

Como fazer um code review de qualidade

Revisar código é uma habilidade que a maioria dos tutoriais ignora completamente — mas é tão importante quanto escrever código. Uma boa revisão pode prevenir bugs em produção, melhorar a arquitetura e distribuir conhecimento no time. Uma revisão ruim é pior que não revisar, porque gera atrito sem valor.

❌ Comentário que não agrega ✅ Comentário que agrega valor Por que a diferença importa
"Isso está errado" "Essa abordagem pode causar N+1 queries no banco. Sugiro usar select_related('user') aqui para buscar tudo em uma query só. Referência: Django docs seção QuerySet." Explica o problema, propõe solução, indica onde aprender mais
"Não gostei desse nome de variável" "O nome d não comunica intenção. Considerando o contexto, user_data ou response_payload seria mais legível para quem mantiver isso daqui a 6 meses." Específico, contextualizado e orientado ao futuro
"Código feio" "Nit: poderia usar list comprehension aqui — [x*2 for x in items]. Fica mais idiomático em Python. (Não bloqueante, muda só se concordar)" Marca como "nit" (não bloqueante), sugere concretamente, respeita a autonomia do autor
"Por que você fez assim???" "Curiosidade técnica: você considerou usar useCallback aqui para evitar re-renders? Ou tem algum motivo específico para não usar? Pode ser que eu esteja perdendo contexto." Pergunta com curiosidade genuína, abre diálogo, reconhece que pode estar errado
"Não aprovado" "Não vou aprovar ainda por conta do item 2 acima (N+1 query). Os outros pontos são nits e não bloqueantes. Quando esse for resolvido, aprovo." Deixa claro o que bloqueia vs o que é sugestão, facilita a comunicação
✅ A regra de ouro do code review

Você está revisando o código, não avaliando a pessoa. "Esse trecho pode ser simplificado porque..." é completamente diferente de "você fez errado". Times de alta performance tratam PRs como conversas técnicas entre pares com o mesmo objetivo: código melhor. Não como avaliação de performance ou prova de superioridade técnica.

Os três tipos de merge: qual escolher?

Quando um PR é aprovado, o GitHub oferece três formas de integrar as mudanças ao main. Cada uma tem consequências no histórico do repositório:

Tipo O que faz ao histórico Quando usar
Merge commit Preserva TODOS os commits do branch + adiciona um commit de merge. O histórico mostra que houve um branch e quando ele foi integrado. Features grandes onde a narrativa de desenvolvimento tem valor histórico. Times que querem rastreabilidade total.
Squash and merge ✅ Comprime TODOS os commits do branch em um único commit no main. O histórico do main fica limpo e cada PR aparece como um único commit bem descrito. Recomendado para a maioria dos casos. Ideal quando o branch tem commits "wip", "fix typo", "teste" que poluiriam o histórico principal.
Rebase and merge Coloca os commits do branch linearmente no main sem criar merge commit. Histórico linear mas preserva commits individuais. Quando todos os commits do branch são bem escritos e têm valor individual. Exige disciplina no processo de desenvolvimento.

Parte 4 — Issues & GitHub Projects

O que é uma Issue?

Uma Issue é o sistema de rastreamento de trabalho nativo do GitHub. Qualquer pessoa (com as permissões certas) pode abrir uma Issue para reportar três tipos de situações:

  • Bug report: "A página de checkout trava quando o usuário tem mais de 100 produtos no carrinho"
  • Feature request: "Seria muito útil ter filtro por data de criação na listagem de pedidos"
  • Tarefa técnica: "Migrar autenticação de JWT para OAuth 2.0 — débito técnico identificado"

Issues têm Labels para categorizar (bug, enhancement, documentation, wontfix), Assignees para atribuir responsáveis, Milestones para associar a metas do produto, e integração com PRs via palavras-chave mágicas.

Quando um PR usa Closes #42, Fixes #42 ou Resolves #42 na descrição e é mergeado no main, a Issue #42 fecha automaticamente. Isso cria um rastro auditável e permanente: "esse problema foi reportado na Issue → esse PR específico o resolveu → esses commits foram as mudanças".

GitHub Projects — quadro kanban integrado ao código

O GitHub Projects é uma ferramenta de gestão visual integrada diretamente ao repositório. Permite organizar Issues e PRs em colunas customizáveis: Backlog → Em Andamento → Em Revisão → Bloqueado → Concluído. É o equivalente gratuito e nativo do Trello ou Jira para times de desenvolvimento.

💡
Por que usar Projects em vez de Trello ou Notion? Porque Issues do GitHub têm contexto de código embutido — você pode clicar em uma tarefa no kanban e ver exatamente qual PR a resolveu, quais commits foram feitos, quem revisou, quando foi mergeado, e qual código mudou. Essa rastreabilidade bidirecional (tarefa ↔ código) é impossível quando o gerenciamento fica em ferramentas separadas do repositório.

Parte 5 — README.md: a porta de entrada do seu projeto

O README.md é o arquivo exibido automaticamente na página inicial do repositório no GitHub. É a primeira coisa que qualquer pessoa vê: um recrutador avaliando seu portfólio, um colega recém-chegado ao time, um usuário querendo instalar a ferramenta, um potencial contribuidor open source. Um README ausente ou mal escrito faz projetos bons parecerem ruins ou abandonados.

A extensão .md significa Markdown — uma linguagem de marcação simples onde # vira título, **texto** vira negrito, `` `código` `` vira código inline, e blocos com três backticks viram blocos de código coloridos. GitHub renderiza Markdown automaticamente.

Mãos na massa — Fluxo completo de colaboração

Vamos percorrer o fluxo real de trabalho em um time. Você vai criar um repositório, conectar ao GitHub, trabalhar em um branch de feature, abrir um Pull Request, incorporar feedback e fazer o merge — exatamente como é feito em empresas reais.

1
Criar o repositório no GitHub e conectar ao local
No GitHub.com: botão "+" → "New repository" → nome: projeto-colaborativo → NÃO marque "Initialize with README" (você já tem o repo local com commits) → Create repository. O GitHub vai mostrar exatamente os comandos abaixo.
terminalbash 
# Estando dentro do diretório do projeto com commits existentes:

# Registra o GitHub como remote com apelido "origin"
git remote add origin https://github.com/SEU-USUARIO/projeto-colaborativo.git

# Renomeia o branch atual para "main" (padrão moderno — antes era "master")
git branch -M main

# Primeiro push: envia o histórico local + configura rastreamento (-u)
git push -u origin main

# A flag -u (--set-upstream) faz com que daqui em diante
# "git push" e "git pull" sem argumentos já saibam que
# devem usar "origin main". Só precisa do -u na PRIMEIRA vez.

# Confirma que funcionou:
git remote -v
# origin  https://github.com/SEU-USUARIO/projeto-colaborativo.git (fetch)
# origin  https://github.com/SEU-USUARIO/projeto-colaborativo.git (push)
2
Criar branch de feature e trabalhar nele
Regra fundamental de times profissionais: nunca commite diretamente no main. Sempre crie um branch dedicado para cada feature, bugfix ou tarefa. Isso mantém o main sempre estável e pronto para deploy.
terminalbash 
# Cria e muda para o branch de feature (sintaxe moderna)
git switch -c feature/pagina-sobre-nos

# Equivalente mais antigo (ainda funciona):
# git checkout -b feature/pagina-sobre-nos

# Confirme que mudou de branch:
git branch
# * feature/pagina-sobre-nos    ← o asterisco indica o branch atual
#   main

# Faça as mudanças nos arquivos...
# (crie/edite arquivos normalmente)

# Commita com mensagens descritivas no padrão Conventional Commits:
git add sobre.html
git commit -m "feat: cria estrutura HTML da página Sobre Nós"

git add sobre.css
git commit -m "style: adiciona estilos da página Sobre Nós"

git add sobre.js
git commit -m "feat: adiciona animação de entrada na seção de equipe"

# Publica o branch no GitHub pela primeira vez:
git push -u origin feature/pagina-sobre-nos

# ✅ Agora o branch existe tanto local quanto no GitHub
# O GitHub vai mostrar um banner sugerindo abrir um PR
💡 Convenção de nomenclatura de branches

Times profissionais usam prefixos para organizar e categorizar branches de forma visual:

  • feature/nome-descritivo — nova funcionalidade
  • fix/descricao-do-bug — correção de bug em desenvolvimento
  • hotfix/problema-critico — correção urgente que afeta produção agora
  • chore/o-que-faz — manutenção técnica: deps, configs, refactoring sem mudança de comportamento
  • docs/o-que-documenta — apenas documentação, sem mudança de código
  • test/o-que-testa — adição ou correção de testes

Fluxo open source: fork + PR para repositório de outra pessoa
Este é o fluxo para contribuir com qualquer projeto público do qual você não é membro. Vamos simular uma contribuição real.
terminal — fluxo completo open sourcebash
# ── ETAPA 1: Fork ─────────────────────────────────────────────
# No GitHub: acesse o repositório → clique "Fork" (canto superior direito)
# Isso cria: github.com/SEU-USUARIO/repo-original
# (cópia completa e independente na sua conta)

# ── ETAPA 2: Clone do SEU fork (não do original!) ─────────────
git clone https://github.com/SEU-USUARIO/repo-original.git
cd repo-original

# ── ETAPA 3: Configurar upstream ──────────────────────────────
# Adiciona o repositório ORIGINAL como remote "upstream"
# Convenção: origin = seu fork | upstream = repositório original
git remote add upstream https://github.com/DONO-ORIGINAL/repo-original.git

# Confirma:
git remote -v
# origin    https://github.com/SEU-USUARIO/repo-original.git (fetch)
# origin    https://github.com/SEU-USUARIO/repo-original.git (push)
# upstream  https://github.com/DONO-ORIGINAL/repo-original.git (fetch)
# upstream  https://github.com/DONO-ORIGINAL/repo-original.git (push)

# ── ETAPA 4: Branch, commits, push para o fork ────────────────
git switch -c fix/corrige-typo-documentacao

# ... edita o arquivo com o typo ...

git add docs/instalacao.md
git commit -m "docs: corrige typo na seção de instalação do Linux"
git push -u origin fix/corrige-typo-documentacao

# ── ETAPA 5: Abrir PR no GitHub ───────────────────────────────
# GitHub detecta o push e sugere abrir PR
# Base: DONO-ORIGINAL/repo-original:main  (destino)
# Compare: SEU-USUARIO/repo-original:fix/corrige-typo-documentacao

# ── ETAPA 6: Manter fork sincronizado com o original ──────────
# Faça isso regularmente para não ficar desatualizado:
git fetch upstream
git switch main
git merge upstream/main  # integra o que o projeto original avançou
git push origin main     # atualiza seu fork no GitHub também

Parte 6 — Conflitos de Merge: quando dois branches mudam a mesma linha

Um conflito de merge acontece quando dois branches diferentes modificaram a mesma linha do mesmo arquivo. O Git coloca os dois numa situação e diz: "não sei qual versão você quer — preciso que um humano decida." Isso não é um erro do Git, nem do desenvolvedor — é uma situação esperada e completamente normal em qualquer time que trabalha em paralelo.

O GitHub exibe "This branch has conflicts that must be resolved" quando um PR tem conflito. Você pode resolver pelo editor visual do GitHub (conveniente para conflitos simples de 1-2 linhas) ou localmente no seu editor de código (recomendado para conflitos mais complexos que envolvem lógica).

Resolvendo conflitos localmente — o processo completo

terminal — passos 1 e 2bash
# Situação: seu branch de feature está atrasado em relação ao main
# que avançou com commits de outros PRs

# Atualiza o rastreamento do main sem tocar no seu código
git fetch origin

# Rebasa seu branch SOBRE o main atualizado
# (coloca seus commits depois dos novos commits do main)
git rebase origin/main

# O Git vai processar commit a commit e parar onde há conflito:
# CONFLICT (content): Merge conflict in src/usuarios.js
# error: could not apply a1b2c3d... feat: adiciona validação de CPF
# hint: Resolve all conflicts manually, mark them as resolved with
# hint: "git add/rm ", then run "git rebase --continue".

Abra o arquivo conflitado no seu editor. O Git inseriu marcações especiais que mostram as duas versões:

src/usuarios.js — arquivo em estado de conflito
function validarCadastro(dados) { if (!dados.email) return false; <<<<<<< HEAD ← o que existe no main (após rebase) if (!dados.telefone) return false; return dados.email.includes('@'); ======= ← separador if (!dados.cpf || dados.cpf.length !== 11) return false; const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/; return regex.test(dados.email); >>>>>>> a1b2c3d ← seus commits (a feature que você está fazendo) }

Você precisa decidir qual versão fica (ou criar uma terceira que combina o melhor dos dois) e remover completamente as marcações <<<, === e >>>. O arquivo deve ficar como código válido:

src/usuarios.js — após resolução manualjavascript
function validarCadastro(dados) {
  if (!dados.email) return false;

  // Mantém a validação de telefone do main (necessária para o formulário)
  if (!dados.telefone) return false;

  // Incorpora a validação de CPF da feature (era o objetivo do PR)
  if (!dados.cpf || dados.cpf.length !== 11) return false;

  // Usa a regex mais robusta da feature em vez da verificação simples do main
  const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  return regex.test(dados.email);
}
terminal — continuando após resolverbash
# Marca o arquivo como resolvido (staging)
git add src/usuarios.js

# Continua o rebase — processa o próximo commit
git rebase --continue
# Se pedir mensagem de commit: pode manter a original ou editar

# Se houver mais conflitos em outros arquivos: repita o processo
# Se quiser CANCELAR e voltar ao estado anterior:
# git rebase --abort

# Quando o rebase terminar sem mais conflitos:
# Force-push — necessário porque o rebase reescreveu o histórico do branch
git push --force-with-lease origin feature/sua-feature

# --force-with-lease é mais seguro que --force puro:
# ele FALHA se alguém fizer push no branch entre o seu fetch e o seu push
# prevenindo que você sobrescreva acidentalmente o trabalho de outra pessoa
🚨 NUNCA use --force em branches compartilhados ou em main

git push --force reescreve o histórico remoto. Se outra pessoa estiver trabalhando no mesmo branch e você fizer force-push, você destrói os commits dela sem aviso. Use --force somente em branches que são exclusivamente seus (seu branch de feature onde você é o único desenvolvedor). E mesmo assim, prefira sempre --force-with-lease.

🏁 Checkpoint — valide seu aprendizado

Você vai executar o fluxo completo de colaboração profissional: criar um repositório público no GitHub com documentação de qualidade, trabalhar em branches de feature, abrir um Pull Request bem documentado, e fazer o merge corretamente. Se tiver um amigo aprendendo também, façam PRs um para o repositório do outro — o aprendizado é exponencial.

  • ✓ Repositório público criado no GitHub com README.md profissional contendo: título com badge de status, descrição clara do projeto fictício, seção "Como Rodar" com pelo menos 3 comandos comentados, lista de tecnologias e seção de contribuição
  • ✓ Clone local realizado — git remote -v mostra origin apontando para o GitHub
  • ✓ Issue criada com título descritivo, descrição com contexto (bug report ou feature request) e ao menos uma label aplicada
  • ✓ Branch feature/adiciona-secao-faq criado com pelo menos 3 commits de trabalho — mensagens seguindo Conventional Commits (feat:, style:, docs: etc.)
  • ✓ Pull Request aberto com: título no padrão Conventional Commits, descrição explicando o quê e o porquê, seção "Como testar", referência Closes #N para a Issue criada, e pelo menos uma label
  • ✓ Commit adicional feito no branch (simule um ajuste de feedback), publicado com git push, e o PR mostra o commit automaticamente
  • ✓ Merge realizado com "Squash and merge" no GitHub, branch deletado no GitHub, e localmente: git switch main && git pull && git branch -d feature/adiciona-secao-faq
  • ✓ Fork realizado em qualquer repositório público de um colega ou projeto real, clone do fork feito localmente, e upstream configurado com git remote add upstream URL-do-original
💬 Dica: Para o item do Fork, escolha um repositório real de código aberto que você usa ou admira — pode ser um tema de CSS, uma biblioteca JavaScript pequena, ou um projeto de um colega. Tente encontrar genuinamente um typo no README ou documentação para corrigir. Abrir seu primeiro PR para um projeto real (mesmo que seja um typo) é um marco que nenhum tutorial substitui. Muitos engenheiros sênior ainda lembram do seu primeiro PR aceito em um projeto de outro.
