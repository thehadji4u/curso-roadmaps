Git — Do Zero ao Primeiro Commit

⏱ ~30 min de leitura 🎯 Iniciante absoluto 🔧 Hands-on incluído 💻 Terminal necessário
git init git add git commit git log git status SSH .gitignore
// 01 — motivação

Por que isso importa?

Você provavelmente já teve uma pasta chamada projeto_final_v3_ESSE_MESMO_agora_vai.zip. Isso é controle de versão manual — e é um desastre esperando para acontecer. O Git resolve isso de forma elegante, rápida e profissional.

Sem Git, você não consegue colaborar em equipe, contribuir para open source, usar GitHub Actions, fazer deploy automático ou trabalhar em qualquer empresa de tecnologia do planeta. É o pré-requisito de tudo neste roadmap.

💼 Por que um engenheiro sênior usa isso hoje
Todo time de engenharia opera com Git como infraestrutura fundamental — não é opcional. Um sênior usa Git para: revisar o histórico de decisões técnicas (git blame), reverter bugs em produção em segundos (git revert), criar branches isolados para experimentos sem quebrar a main, e conectar cada commit a uma issue/PR rastreável. É a base da comunicação técnica entre engenheiros.
// 02 — analogia

Antes do código: a analogia do videogame

🎮 Mundo Real → Git
Imagine que você está jogando um RPG de mundo aberto. A cada hora você aperta Salvar — criando um checkpoint com o estado exato do jogo: sua vida, posição, inventário, missões completas. Se você morrer ou tomar uma decisão ruim, você carrega o checkpoint e volta exatamente para aquele estado. Você pode ter vários saves, comparar dois saves diferentes, e até jogar numa cópia do save sem arriscar o original.

O Git é exatamente isso para o seu código. Cada git commit é um checkpoint. Você pode voltar para qualquer ponto da história, criar ramificações paralelas, e nunca mais perder trabalho.
Apertar "Salvar" no jogo → git commit Carregar um save antigo → git checkout <hash> Jogar numa cópia do save → git branch Ver todos os seus saves → git log Nuvem de sincronização → GitHub (remote)
// 03 — conceito

Como o Git pensa sobre arquivos

O Git não rastreia arquivos inteiros a cada commit — ele rastreia mudanças (diffs). Todo arquivo no seu projeto existe em um de três estados. Entender esses estados é a chave para nunca mais ficar confuso com Git.

📁
Working Directory
Seus arquivos como você os vê no editor. Modificações ainda não registradas.
git status (vermelho)
→
📋
Staging Area
Arquivos marcados para entrar no próximo commit. "Sala de espera" do commit.
git status (verde)
→
🔒
Repository
Histórico permanente e imutável. Cada commit tem um hash SHA único.
git log

// 04 — setup

Instalação & configuração

Sistema Comando Verificar
macOS brew install git ou instalar Xcode CLI tools git --version
Ubuntu/Debian sudo apt install git git --version
Windows Baixar em git-scm.com · inclui Git Bash git --version no Git Bash

Após instalar, configure sua identidade — essa informação vai em todo commit que você fizer:

terminal bash 
# Configuração global (feita uma única vez)
git config --global user.name "Seu Nome"
git config --global user.email "seu@email.com"

# Editor padrão (VS Code — recomendado)
git config --global core.editor "code --wait"

# Branch padrão como 'main' (padrão atual da indústria)
git config --global init.defaultBranch main

# Verificar tudo que foi configurado
git config --list
⚠️ Use o mesmo email do GitHub
O email que você configura aqui precisa ser o mesmo da sua conta GitHub para que seus commits apareçam corretamente no seu perfil e no histórico de contribuições (o famoso gráfico verde).
// 05 — mãos na massa

Seu primeiro repositório — passo a passo

Abra o terminal e execute exatamente estes comandos. Vamos criar um projeto real do zero.

1
Crie a pasta do projeto
Todo projeto Git começa com uma pasta. O Git vai rastrear tudo dentro dela.
terminalbash 
mkdir meu-primeiro-repo
cd meu-primeiro-repo
2
Inicialize o repositório Git
Este comando cria uma pasta oculta .git/ que guarda todo o histórico. Você só faz isso uma vez por projeto.
terminalbash 
git init
# Saída esperada: Initialized empty Git repository in .../meu-primeiro-repo/.git/

# Veja o que foi criado (pasta oculta)
ls -la #linux
dir -Force #windows
3
Crie arquivos e veja o status
O git status é seu melhor amigo — use-o constantemente. Ele mostra exatamente o que está acontecendo no repositório.
terminalbash 
# Crie um arquivo README
echo "# Meu Primeiro Repositório" > README.md
echo "Aprendendo Git do zero." >> README.md

# Veja o status — arquivo aparece em vermelho (untracked)
git status
4
Adicione à staging area
git add não salva nada ainda — ele apenas marca o arquivo como "quero incluir isso no próximo commit". É a staging area — a sala de espera do commit.
terminalbash 
# Adicionar um arquivo específico
git add README.md

# OU adicionar tudo de uma vez (cuidado: adiciona TUDO)
git add .

# Status agora — arquivo aparece em verde
git status
5
Faça o primeiro commit
O commit é o checkpoint. A mensagem deve descrever o que foi feito, não como. Use o padrão Conventional Commits desde o início: feat:, fix:, docs:, chore:
terminalbash 
# Commit com mensagem descritiva
git commit -m "docs: add README with project description"

# Saída esperada:
# [main (root-commit) a3f9c21] docs: add README with project description
#  1 file changed, 2 insertions(+)
6
Explore o histórico
O git log mostra toda a linha do tempo do projeto. Cada commit tem um hash SHA único — 40 caracteres que identificam aquele estado exato do código.
terminalbash 
# Log completo
git log

# Log compacto e visual (use sempre)
git log --oneline --graph --all

# Ver o que mudou num commit específico
git show a3f9c21

# Ver diferença entre working dir e último commit
git diff
7
Crie um .gitignore
O .gitignore lista arquivos que o Git deve ignorar completamente. Nunca commite senhas, chaves de API, node_modules/ ou arquivos de build.
.gitignoregitignore
# Dependências
node_modules/
__pycache__/
*.pyc

# Variáveis de ambiente (NUNCA commite isso)
.env
.env.local
*.env

# Builds e compilados
dist/
build/
*.o
*.exe

# Editor e sistema
.DS_Store
.vscode/
*.swp
Thumbs.db
terminalbash 
# Crie e commite o .gitignore
git add .gitignore
git commit -m "chore: add .gitignore"
// 06 — referência

O fluxo diário de um desenvolvedor

Este é o ciclo que você vai repetir centenas de vezes. Memorize-o.

fluxo-diario.sh bash 
### ─── INÍCIO DO DIA ──────────────────────────────────
# 1. Verificar o estado atual do repositório
git status

# 2. Ver histórico recente
git log --oneline -5

### ─── DURANTE O TRABALHO ─────────────────────────────
# 3. Editar arquivos no editor...

# 4. Ver o que mudou (linha por linha)
git diff

# 5. Adicionar arquivos modificados à staging
git add src/minha-feature.js    # arquivo específico
git add src/                     # toda uma pasta
git add -p                       # interativo: escolhe quais partes adicionar

# 6. Revisar o que vai no commit
git diff --staged

# 7. Commitar com mensagem clara
git commit -m "feat: add user authentication flow"

### ─── INSPEÇÃO E DIAGNÓSTICO ──────────────────────────
# Ver quem escreveu cada linha (investigar bugs)
git blame src/minha-feature.js

# Ver detalhes de um commit específico
git show abc1234

# Buscar commits por mensagem
git log --grep="authentication"

# Ver histórico de um arquivo específico
git log --follow -p src/minha-feature.js
// 07 — arquitetura

Anatomia de um commit

Cada commit é um objeto imutável no banco de dados do Git, com estrutura bem definida:

// estrutura interna de um commit Git

📦 COMMIT

O objeto principal. Contém metadados (autor, data, mensagem) e um ponteiro para a tree que representa o estado dos arquivos naquele momento. Também aponta para o commit anterior (parent).

🌳 TREE

Um diretório. Lista todos os arquivos e subpastas daquele commit, apontando para blobs (arquivos) ou outras trees (subpastas).

📄 BLOB

O conteúdo real de um arquivo. O Git armazena cada versão como um blob único identificado por um hash SHA. Se dois arquivos forem idênticos, compartilham o mesmo blob.

🔗 Como ler o diagrama
Quando você faz git commit, o Git cria esse grafo: o COMMIT aponta para uma TREE (a "foto" da pasta raiz), que por sua vez aponta para os BLOBs (o conteúdo de cada arquivo). A seta tracejada na parte inferior mostra que cada commit aponta para seu commit anterior — formando a linha do tempo do projeto.
// 08 — armadilhas

Erros que todo iniciante comete

⚠️ Commitar a pasta .git ou arquivos .env

A pasta .git/ nunca deve ser commitada — ela JÁ É o repositório. Arquivos .env com senhas e chaves de API nunca devem entrar no Git. Sempre crie o .gitignore antes do primeiro commit.

⚠️ Mensagens de commit vazias ou inúteis

"fix", "wip", "atualizei", "commit" — estas mensagens são inúteis no futuro. Quando você procurar "quando esse bug foi introduzido?" em 3 meses, vai agradecer por ter escrito "fix: resolve null pointer on user login when session expires".

⚠️ Usar git add . sem revisar

git add . adiciona TUDO, incluindo arquivos temporários, logs e segredos. Prefira git add -p (modo interativo) ou adicione por arquivo. Ao menos faça git status antes de commitar.

⚠️ Confundir Git com GitHub

Git é local, GitHub é remoto. Você pode usar Git sem GitHub. git commit não envia nada para a internet — para isso você precisa de git push (que veremos na aula 03).

// 10 — desafio

🏁 Checkpoint — valide seu aprendizado

Execute este desafio do zero, sem copiar comandos desta aula. Se conseguir, você dominou o conteúdo.

  • Crie um repositório chamado checkpoint-git do zero
  • Configure sua identidade Git corretamente (git config)
  • Crie pelo menos 3 arquivos diferentes (ex: index.html, style.css, README.md)
  • Faça 3 commits separados com mensagens no padrão Conventional Commits
  • Crie um .gitignore que ignore arquivos .env e node_modules/
  • Execute git log --oneline --graph e confirme que vê os 3 commits no histórico
  • Use git diff HEAD~1 HEAD para ver o que mudou entre o penúltimo e último commit
💬 Dica: Se travar em qualquer passo, volte para a seção "Mãos na Massa" desta aula. O objetivo não é decorar os comandos — é entender o que cada um faz. Use git status depois de cada ação para ver o que mudou.
