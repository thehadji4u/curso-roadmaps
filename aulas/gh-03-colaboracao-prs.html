// MOTIVAÇÃO

Por que isso importa?

Git no computador local é poderoso. Git + GitHub é onde a mágica da colaboração acontece. Praticamente todo projeto de software profissional hoje — startups, big tech, projetos open source — usa o modelo de Pull Requests para revisar, discutir e integrar código. Saber navegar nesse fluxo é o que separa alguém que "usa Git" de alguém que trabalha em equipe de verdade.

🚫

Sem esse conhecimento

Você edita código diretamente na main, quebra o projeto de todos, e não tem como revisar mudanças antes de integrar.

✅

Com esse conhecimento

Você propõe mudanças isoladas em branches, recebe feedback via PR, e contribui para qualquer projeto open source do mundo.

💡 Contexto Sênior
Um engenheiro sênior hoje nunca faz commit diretamente na branch principal de um projeto compartilhado. O fluxo de Pull Request + Code Review é o padrão da indústria — é assim que Google, Meta, Netflix e qualquer startup séria trabalha.
// ANALOGIA

O Manuscrito da Editora

🎭 Analogia do Mundo Real
Imagine que você é um autor contribuindo para uma enciclopédia colaborativa impressa. Você não pode simplesmente arrancar páginas do livro e reescrever. Em vez disso: você faz uma cópia do capítulo (branch), escreve suas alterações, e envia o rascunho para o editor-chefe (Pull Request). O editor revisa linha por linha, faz comentários, pede ajustes. Quando todos aprovam, o texto é incorporado ao livro oficial (merge). Issues são os bilhetinhos de "Alguém deveria atualizar essa seção". O README é a capa e sumário do livro — a primeira coisa que qualquer leitor vê.

Remote, Push & Pull

Git é um sistema distribuído: cada desenvolvedor tem uma cópia completa do repositório. O GitHub é apenas um repositório remoto — um servidor central que serve como ponto de encontro entre todos. A comunicação acontece por dois comandos fundamentais:

⬆️

git push

Envia seus commits locais para o repositório remoto. Você "empurra" seu trabalho para o servidor.

⬇️

git pull

Busca os commits do remoto e integra na sua branch local. Você "puxa" o trabalho dos colegas.

📊 Diagrama — Fluxo Remote + Branch + PR
flowchart LR subgraph Local["💻 Máquina Local"] A[main local] -->|git checkout -b| B[feature/login] B -->|commits| B end subgraph Remote["☁️ GitHub"] C[main remota] D[feature/login remota] D -->|Pull Request| E{Code Review} E -->|Aprovado| C end B -->|git push origin| D C -->|git pull| A

Configurando um Remote

Quando você cria um repo no GitHub e quer conectar sua pasta local, os passos são:

shell
# Ver os remotes configurados
git remote -v

# Adicionar o GitHub como remote "origin"
git remote add origin https://github.com/seu-usuario/meu-projeto.git

# Enviar sua branch main pela primeira vez
git push -u origin main
# A flag -u define o "upstream" — próximos pushes serão apenas: git push

# Clonar um repositório existente (já configura o remote automaticamente)
git clone https://github.com/outro-usuario/projeto.git
💡 origin é apenas um nome
"origin" é a convenção — é o apelido padrão para o seu remote principal. Você poderia chamar de "github" ou "servidor". Projetos open source geralmente têm "origin" como seu fork e "upstream" como o repo original.
// PULL REQUESTS

O Coração da Colaboração

Um Pull Request (PR) é uma proposta formal de integrar mudanças de uma branch para outra. Mais do que um mecanismo técnico, é uma conversa: o autor explica o que fez e por quê, e os revisores analisam, questionam e aprovam. É onde o conhecimento técnico da equipe é compartilhado e a qualidade do código é mantida.

1
Criar uma branch para a feature
Nunca trabalhe diretamente na main. Toda mudança vive em sua própria branch.
git checkout -b feature/adicionar-login
# ou em versões mais recentes:
git switch -c feature/adicionar-login
2
Desenvolver e commitar
Faça seus commits com mensagens claras e descritivas.
git add .
git commit -m "feat: adicionar formulário de login com validação"
# Convenção: feat, fix, docs, style, refactor, test, chore
3
Enviar a branch para o GitHub
O GitHub vai sugerir criar um PR assim que você fizer push de uma branch nova.
git push origin feature/adicionar-login
4
Abrir o Pull Request no GitHub
Preencha um título claro, descrição do que foi feito, screenshots se tiver UI, e adicione revisores.
5
Code Review & Ajustes
Revisores comentam linha por linha. Você responde, faz ajustes e faz novos commits na mesma branch — o PR atualiza automaticamente.
6
Merge & limpeza
Após aprovação, o PR é mesclado. A branch pode ser deletada no GitHub e localmente. 
# Atualizar sua main local
git checkout main
git pull origin main

# Deletar a branch local que já foi mergeada
git branch -d feature/adicionar-login

Tipos de Merge

Tipo Comportamento Quando usar Histórico
Merge Commit Cria um commit de merge explícito Features grandes, manter contexto completo Preserva tudo
Squash & Merge Comprime todos commits da branch em 1 Muitos commits "WIP", histórico limpo na main Comprimido
Rebase & Merge Reaplica commits linearmente, sem commit de merge Projetos que exigem histórico linear Linear
// ISSUES & PROJECTS

Organização & Gestão de Tarefas

GitHub não é só versionamento de código — é uma plataforma completa de gestão de projetos. Issues e Projects formam o sistema de rastreamento de tarefas, bugs e features que mantém equipes alinhadas.

Issues — O Rastreador de Tudo

Uma Issue é a unidade básica de trabalho no GitHub. Pode ser um bug report, uma feature request, uma discussão técnica, ou uma tarefa. Qualquer pessoa pode abrir uma Issue em um projeto público.

📋 Anatomia de uma boa Issue
## Bug: Botão de login não funciona no Firefox

**Descrição**
Ao clicar em "Entrar", nada acontece. O console mostra um erro de CORS.

**Passos para reproduzir**
1. Abrir o site no Firefox 120
2. Preencher email e senha válidos
3. Clicar em "Entrar"

**Comportamento esperado**
Usuário deve ser redirecionado para /dashboard

**Comportamento atual**
Nada acontece. Console: "CORS policy blocked"

**Screenshots**
[imagem do console aqui]

**Ambiente**
- OS: Windows 11
- Browser: Firefox 120.0
- Node: 18.17.0

Labels, Milestones & Assignments

Recurso Para que serve Exemplo
Labels Categorizar Issues e PRs bug, enhancement, good first issue, help wanted
Milestones Agrupar Issues por objetivo/versão "v1.0 Launch", "Sprint 3", "Q1 2024"
Assignees Responsável pela Issue/PR @usuario
Linked PRs Conectar PR à Issue (fecha automaticamente) Closes #42 na descrição do PR

GitHub Projects — Kanban Nativo

GitHub Projects é um quadro kanban integrado ao repositório. Issues e PRs viram cards que você move entre colunas. É mais simples que Jira, mas suficiente para projetos pequenos e médios.

📊 Fluxo típico de um GitHub Project
kanban Backlog id1[Issue: Criar tela de perfil] id2[Issue: Integrar API de pagamento] Em Progresso id3[Issue: Bug no login Firefox] Em Review id4[PR: Adicionar autenticação OAuth] Concluído id5[PR: Setup inicial do projeto]
🔥 Pro tip — Fechando Issues com commits
Você pode fechar Issues automaticamente referenciando-as em commit messages ou na descrição do PR: Fixes #42, Closes #15, Resolves #8. Quando o PR for mergeado na main, a Issue fecha sozinha.

Fluxos de Trabalho com Git

Existem diferentes estratégias de branching que equipes adotam. Cada uma tem prós e contras:

Workflow Ideal para Vantagem Desvantagem Complexidade
GitHub Flow Deploy contínuo, SaaS Simples: branch → PR → merge → deploy Não gerencia múltiplas versões Baixa
Git Flow Apps com releases versionados Separa develop, release, hotfix Muitas branches, complexo Alta
Trunk Based Times grandes, feature flags Integração contínua real, poucos conflitos Exige disciplina e feature flags Média
Forking Open source, contribuições externas Qualquer um pode contribuir sem acesso direto Sincronização com upstream é manual Média
🔥 Pro tip
Para 90% dos projetos pessoais e startups, GitHub Flow é perfeito. É o que você vai usar. Git Flow ficou popular nos anos 2010 mas é considerado excessivamente complexo para a maioria dos casos hoje.

Seu Primeiro PR: Passo a Passo

Vamos simular o fluxo completo de colaboração: você vai criar um repo, fazer uma branch, abrir um PR e mergear.

1
Criar um repositório no GitHub
Acesse github.com → "New repository" → Nome: meu-primeiro-pr → Marque "Add a README file" → Create repository.
2
Clonar o repositório localmente
git clone https://github.com/SEU-USUARIO/meu-primeiro-pr.git
cd meu-primeiro-pr
3
Criar uma branch de feature
git checkout -b feature/adicionar-sobre
4
Editar o README e fazer commit
Abra o README.md no VS Code e adicione uma seção "## Sobre mim". Salve o arquivo. 
git add README.md
git commit -m "docs: adicionar seção sobre mim no README"
5
Enviar a branch para o GitHub
git push origin feature/adicionar-sobre
6
Abrir o Pull Request
O GitHub mostrará um banner amarelo "Compare & pull request". Clique nele. Preencha o título e descrição. Clique em "Create pull request".
7
Mergear o PR
Como é seu próprio repo, você pode aprovar e mergear. Clique em "Merge pull request" → "Confirm merge". Pronto!
8
Sincronizar localmente e limpar
git checkout main
git pull origin main
git branch -d feature/adicionar-sobre

Fluxo Completo com Fork (Open Source)

Para contribuir com projetos de terceiros (onde você não tem acesso de escrita), o fluxo é levemente diferente: você faz um fork (cópia do repo na sua conta), trabalha nele, e abre um PR de volta para o original.

shell — fluxo completo de fork contribution
# 1. Fork pelo GitHub (botão "Fork" no repo original)
# O repo aparece em: github.com/SEU-USUARIO/repo-original

# 2. Clonar seu fork
git clone https://github.com/SEU-USUARIO/repo-original.git
cd repo-original

# 3. Adicionar o repo original como "upstream"
# (para poder sincronizar atualizações futuras)
git remote add upstream https://github.com/DONO-ORIGINAL/repo-original.git

# Verificar: deve mostrar origin (seu fork) e upstream (original)
git remote -v

# 4. Criar branch de feature
git checkout -b fix/corrigir-typo-readme

# 5. Fazer as mudanças...
# 6. Commit
git add .
git commit -m "fix: corrigir typo na seção de instalação"

# 7. Push para SEU fork (não para o upstream!)
git push origin fix/corrigir-typo-readme

# 8. Abrir PR no GitHub: do seu fork → repo original
# GitHub detecta automaticamente e sugere o PR cross-repo

# ----- MANUTENÇÃO DO FORK -----
# Manter seu fork atualizado com o original:
git fetch upstream          # busca atualizações do repo original
git checkout main
git merge upstream/main     # integra na sua main local
git push origin main        # atualiza seu fork no GitHub
⚠️ Nunca commite direto na main do seu fork
Mesmo sendo "seu" repo, trabalhar em branches facilita muito quando você quiser sincronizar com o upstream e abrir múltiplos PRs. Commits diretos na main do fork criam conflitos difíceis de resolver.
// PITFALLS

⚠️ Erros Comuns de Iniciantes

💥

Trabalhar direto na main e fazer push forçado

git push --force na main de um repo compartilhado é capaz de apagar o trabalho de toda a equipe. Nunca force-push em branches compartilhadas. Configure "Branch protection rules" no GitHub para bloquear isso.

😵

PR gigante com 50 arquivos alterados

PRs grandes são difíceis de revisar. O ideal é PRs pequenos e focados. Se sua feature é grande, quebre em PRs menores e sequenciais. A regra de ouro: se o revisor precisar de mais de 20 minutos para entender, o PR é grande demais.

🙈

Commitar credenciais e secrets no repositório

API keys, senhas, tokens nunca devem ir para o Git. Use um arquivo .env (e adicione ao .gitignore) para variáveis sensíveis. Se você commitou acidentalmente, a chave está comprometida — rote imediatamente.

🔀

Branch desatualizada por semanas

Branches que ficam muito tempo sem sincronizar com a main acumulam conflitos. Faça git pull origin main ou git rebase main na sua feature branch regularmente (pelo menos a cada 1-2 dias em projetos ativos).

📝

Mensagens de commit sem significado

"fix", "update", "mudanças" não ajudam ninguém no futuro. Use o padrão Conventional Commits: feat:, fix:, docs:, refactor:. Boas mensagens de commit são documentação.

// GLOSSÁRIO

Termos da Aula

remote
Repositório Git hospedado em um servidor (GitHub, GitLab, etc.). É o ponto de sincronização entre colaboradores.
origin
Apelido padrão para o remote principal. Configurado automaticamente quando você clona um repo.
upstream
Convenção para o remote do repositório original do qual você fez fork.
Pull Request (PR)
Proposta formal de integrar mudanças de uma branch para outra. Inclui discussão, code review e aprovação.
fork
Cópia completa de um repositório na sua conta GitHub. Permite contribuir sem ter acesso de escrita ao original.
merge
Integração das mudanças de uma branch em outra. Pode ser feito via Merge Commit, Squash ou Rebase.
Issue
Unidade de trabalho no GitHub: bug report, feature request ou discussão técnica rastreável.
Conventional Commits
Padrão de mensagens de commit: tipo: descrição. Exemplos: feat:, fix:, docs:.

🏁 Checkpoint — Colaboração Real

Vamos simular um fluxo de colaboração completo. Você vai criar um projeto com Issues, trabalhar em branches e abrir PRs como faria em um time real.

  • Crie um repositório público chamado portfolio-site com README
  • Abra 3 Issues: "Criar página inicial", "Adicionar seção de projetos", "Escrever seção de contato"
  • Crie Labels customizados: feature e content
  • Para cada Issue, crie uma branch (feature/home, etc.), faça pelo menos 1 commit, e abra um PR com a mensagem Closes #[número da issue]
  • Mergee os 3 PRs e verifique que as Issues fecharam automaticamente
  • Atualize o README com uma seção "Como contribuir" seguindo o padrão da aula
💬 Dica: No GitHub, vá em Settings → Branches → Add branch protection rule para a main. Ative "Require a pull request before merging" — isso simula um ambiente de equipe real onde ninguém pode commitar direto na main.
