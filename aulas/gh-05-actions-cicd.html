GitHub Actions — CI/CD Completo

⏱ ~50 min de leitura 🎯 Intermediário–Avançado 🔧 Hands-on incluído
workflows YAML CI — testes automáticos CD — deploy automático secrets runners jobs & steps matrix strategy marketplace actions cache de dependências
// MOTIVAÇÃO

Por que isso importa?

Todo time de engenharia que opera com velocidade e confiança tem uma coisa em comum: ninguém faz deploy manual. Cada push de código dispara automaticamente uma bateria de testes, uma verificação de qualidade, uma build da imagem Docker e, se tudo passar, o deploy para o servidor — sem intervenção humana, sem esquecimentos, sem "funciona no meu computador".

O GitHub Actions é a ferramenta que viabiliza esse fluxo diretamente no GitHub, sem precisar de servidores externos como Jenkins ou CircleCI. É gratuito para repositórios públicos e tem planos generosos para privados. Mais importante: os workflows vivem no próprio repositório, versionados em Git, revisados em pull requests como qualquer outro código.

💼 Por que um engenheiro sênior domina GitHub Actions
CI/CD não é "nice to have" — é a fundação de qualidade de software. Sem ele, bugs que teriam sido pegos por testes automatizados chegam a produção. Deploys demoram horas e são estressantes. Com um pipeline bem construído, você faz dezenas de deploys por dia com confiança. Um dev sênior sabe configurar o pipeline inteiro: testes em paralelo, cache de dependências para builds rápidos, deploy condicional por branch e gestão segura de segredos.
// ANALOGIA

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense numa linha de montagem de automóveis. Quando um novo componente chega à fábrica, ele não vai direto para o carro do cliente. Ele passa por uma esteira automática com estações de inspeção: verificação de dimensões, teste de resistência, pintura, controle de qualidade final — e só então, se aprovado em cada etapa, é instalado no veículo e entregue.

O GitHub Actions é essa esteira de montagem do seu código. Cada push é o componente que entra na fábrica. Os jobs são as estações de inspeção. Os steps são as operações de cada estação. Os runners são os robôs que executam cada etapa. E o deploy é a entrega do produto final aprovado — automaticamente, sem precisar de um operário humano em cada etapa.
🏭 Linha de montagem completa → Workflow (.yml) 📦 Componente novo chegando → Push / Pull Request (trigger) 🔧 Estação de inspeção → Job (ex: test, build, deploy) ⚙️ Operação dentro da estação → Step (ex: run tests, docker push) 🤖 Robô que executa a operação → Runner (ubuntu-latest) ✅ Aprovado em todos os controles → Deploy disparado automaticamente

Anatomia de um Workflow

Um workflow é um arquivo YAML localizado em .github/workflows/nome.yml no seu repositório. Ele define quando rodar (trigger), onde rodar (runner) e o que fazer (jobs e steps).

Hierarquia: Workflow → Jobs → Steps

Os 5 blocos fundamentais

BlocoO que éExemplo

// TRIGGERS

Triggers — quando o workflow roda

O bloco on: define os eventos que disparam o workflow. Você pode combinar múltiplos triggers e filtrar por branches, tags ou caminhos de arquivos.

Principais triggers — referência completa
on:
  # ── Push de código ────────────────────────────────────────────
  push:
    branches:
      - main          # Só na branch main
      - 'release/**'  # Qualquer branch com prefixo release/
    paths:
      - 'src/**'      # Só quando arquivos em src/ mudam
      - '**.py'       # Só quando arquivos .py mudam
    tags:
      - 'v*'          # Qualquer tag começando com v (ex: v1.2.3)

  # ── Pull Requests ─────────────────────────────────────────────
  pull_request:
    branches:
      - main          # PRs apontando para main
    types:
      - opened        # PR criado
      - synchronize   # Novo commit no PR
      - reopened      # PR reaberto

  # ── Agendamento (cron) ────────────────────────────────────────
  schedule:
    - cron: '0 2 * * *'    # Toda noite às 2h UTC (ex: backup, scan de segurança)
    - cron: '0 9 * * 1-5'  # Dias úteis às 9h UTC

  # ── Disparo manual pela interface do GitHub ───────────────────
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment to deploy'
        required: true
        default: 'staging'
        type: choice
        options: [staging, production]

  # ── Disparado por outro workflow ──────────────────────────────
  workflow_call:
    inputs:
      version:
        type: string
        required: true
🔥 Pro tip — paths-ignore para evitar loops
Use paths-ignore para não disparar o workflow quando apenas documentação ou arquivos de config mudam: paths-ignore: ['**.md', '.github/**', 'docs/**']. Evita gastar minutos gratuitos em pushes que não afetam o código.

.github/workflows/ci.yml — pipeline CI completo com cache
name: CI — Testes & Qualidade

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

# Cancela runs anteriores do mesmo PR para economizar minutos
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:

  # ─── Job 1: Lint & Formatação ───────────────────────────────────────────────
  lint:
    name: 🎨 Lint & Format
    runs-on: ubuntu-latest

    steps:
      - name: Checkout código
        uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Cache pip
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements*.txt') }}
          restore-keys: |
            ${{ runner.os }}-pip-

      - name: Instalar ferramentas de lint
        run: pip install ruff black isort

      - name: Verificar formatação com Black
        run: black --check .

      - name: Verificar imports com isort
        run: isort --check-only .

      - name: Lint com Ruff
        run: ruff check .

  # ─── Job 2: Testes ─────────────────────────────────────────────────────────
  test:
    name: 🧪 Testes
    runs-on: ubuntu-latest
    needs: lint   # Só roda se lint passar

    # Matrix: testa em múltiplas versões do Python simultaneamente
    strategy:
      fail-fast: false   # Continua testando outras versões se uma falhar
      matrix:
        python-version: ['3.11', '3.12']

    # Serviços auxiliares (sobem como containers no runner)
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_USER: testuser
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

      redis:
        image: redis:7-alpine
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s

    steps:
      - name: Checkout código
        uses: actions/checkout@v4

      - name: Setup Python ${{ matrix.python-version }}
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}

      - name: Cache pip
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-${{ matrix.python-version }}-pip-${{ hashFiles('**/requirements*.txt') }}

      - name: Instalar dependências
        run: |
          pip install -r requirements.txt
          pip install -r requirements-dev.txt

      - name: Rodar testes com cobertura
        env:
          DATABASE_URL: postgresql://testuser:testpass@localhost:5432/testdb
          REDIS_URL: redis://localhost:6379/0
          SECRET_KEY: ci-test-secret-key
        run: |
          pytest \
            --cov=app \
            --cov-report=xml \
            --cov-report=term-missing \
            --cov-fail-under=80 \
            -v \
            --tb=short

      - name: Upload coverage para Codecov
        uses: codecov/codecov-action@v4
        if: matrix.python-version == '3.12'   # Sobe cobertura apenas uma vez
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          file: ./coverage.xml
          fail_ci_if_error: false

Pipeline de CD — build Docker e deploy automático

O CD só roda após o CI passar e apenas na branch main. Ele builda a imagem Docker, publica no registry e faz o deploy no servidor. O segredo está no uso de needs e condicionais if para garantir que só código validado chega a produção.

.github/workflows/cd.yml — build Docker + deploy via SSH
name: CD — Build & Deploy

on:
  push:
    branches: [main]
  workflow_dispatch:   # Permite disparo manual pela interface do GitHub

# Garante que apenas um deploy roda por vez
concurrency:
  group: deploy-production
  cancel-in-progress: false   # NÃO cancela deploy em andamento

jobs:

  # ─── Job: Build & Push Docker Image ─────────────────────────────────────────
  build:
    name: 🐳 Build & Push
    runs-on: ubuntu-latest

    # Exporta outputs para o job de deploy
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
      image-digest: ${{ steps.build-push.outputs.digest }}

    steps:
      - name: Checkout código
        uses: actions/checkout@v4

      # Login no GitHub Container Registry (GHCR)
      - name: Login no GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}   # Automático — sem configuração!

      # Gera tags automáticas: latest + SHA do commit + versão semântica
      - name: Extrair metadata (tags, labels)
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=ref,event=branch
            type=sha,prefix=sha-
            type=semver,pattern={{version}}
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}

      # Habilita BuildKit com cache do GitHub Actions
      - name: Setup Docker Buildx
        uses: docker/setup-buildx-action@v3

      # Build e push com cache entre runs (builds muito mais rápidos!)
      - name: Build e Push
        id: build-push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha        # Usa cache do GitHub Actions
          cache-to: type=gha,mode=max # Salva cache para próximo run

  # ─── Job: Deploy no servidor ─────────────────────────────────────────────────
  deploy:
    name: 🚀 Deploy
    runs-on: ubuntu-latest
    needs: build   # Só roda se build passar
    environment: production   # Requer aprovação manual se configurado

    steps:
      - name: Checkout código (para docker-compose.yml)
        uses: actions/checkout@v4

      # Deploy via SSH: conecta no servidor, puxa nova imagem e reinicia
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            # Navegar para o diretório do projeto
            cd /opt/myapp

            # Login no GHCR para puxar a imagem privada
            echo ${{ secrets.GITHUB_TOKEN }} | docker login ghcr.io -u ${{ github.actor }} --password-stdin

            # Baixar nova imagem e restartar apenas o serviço da API
            docker compose pull api
            docker compose up -d --no-deps api

            # Verificar se subiu corretamente
            docker compose ps api

            # Limpar imagens antigas (economiza disco)
            docker image prune -f

      # Notifica o PR/commit com link para o deploy
      - name: Comentar no commit
        uses: actions/github-script@v7
        if: always()
        with:
          script: |
            const status = '${{ job.status }}' === 'success' ? '✅' : '❌';
            github.rest.repos.createCommitComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              commit_sha: context.sha,
              body: `${status} Deploy para produção: **${{ job.status }}**`
            })

Secrets — gerenciando credenciais com segurança

Secrets são variáveis criptografadas armazenadas no GitHub — nunca expostas em logs, nunca visíveis em texto plano, mascaradas automaticamente se acidentalmente impressas. São acessadas no workflow via ${{ secrets.NOME_DO_SECRET }}.

🔐
Acesse via: Settings → Secrets and variables → Actions → New repository secret
EscopoOnde configurarAcessoQuando usar
Repository secret Settings do repositório Todos os workflows do repo Credenciais específicas do projeto
Environment secret Settings → Environments Só jobs com environment: definido Recomendado para staging/produção — permite aprovação manual
Organization secret Org Settings → Secrets Vários repositórios da org Credenciais compartilhadas entre projetos
GITHUB_TOKEN Automático — sempre disponível Qualquer workflow Push para GHCR, comentar em PRs, criar releases
⚠️ Secrets NÃO são passados para PRs de forks
Por segurança, workflows disparados por Pull Requests de forks externos não têm acesso aos secrets do repositório. Isso evita que contribuidores maliciosos criem PRs para exfiltrar suas credenciais. Use pull_request_target com extrema cautela se precisar desse caso de uso — e nunca execute código do fork com acesso a secrets.
Usando secrets no workflow
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production   # Habilita environment secrets + aprovação manual

    steps:
      - name: Login no Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}    # Token de acesso, não senha!

      - name: Deploy com variáveis de ambiente
        env:
          # Injetar secret como variável de ambiente do step
          DATABASE_URL: ${{ secrets.PROD_DATABASE_URL }}
          API_KEY: ${{ secrets.EXTERNAL_API_KEY }}
        run: |
          echo "Iniciando deploy..."
          # secrets.DATABASE_URL não aparece no log — mascarado automaticamente!
          ./scripts/deploy.sh

      # ⚠️ NUNCA faça isso — expõe o secret no log:
      # - run: echo ${{ secrets.MY_SECRET }}
      # Use variável de ambiente como no step acima!

Marketplace de Actions — não reinvente a roda

O GitHub Marketplace tem mais de 20.000 Actions prontas para as tarefas mais comuns. Em vez de escrever scripts para instalar Python, fazer login no Docker ou enviar notificação no Slack, você usa Actions testadas pela comunidade com uma única linha.

ActionO que fazUso típico
actions/checkout@v4 Clona o repositório no runner Primeiro step de quase todo workflow
actions/setup-python@v5 Instala e configura Python (com cache de pip) Projetos Python
actions/setup-node@v4 Instala e configura Node.js Projetos JavaScript/TypeScript
actions/cache@v4 Cacheia pastas entre runs (pip, npm, gradle) Acelerar builds em 80%+
docker/build-push-action@v5 Build e push de imagem Docker com cache BuildKit Pipelines com Docker
docker/login-action@v3 Login em qualquer registry (GHCR, Docker Hub, ECR) Antes do push de imagem
appleboy/ssh-action@v1 Executa comandos via SSH num servidor remoto Deploy em VPS / servidor próprio
codecov/codecov-action@v4 Envia relatório de cobertura para o Codecov Após rodar testes com cobertura
softprops/action-gh-release@v2 Cria GitHub Release com changelog automático Automatizar releases por tag
⚠️ Sempre fixe a versão da Action com SHA ou tag de versão maior
uses: actions/checkout@v4 é seguro (tag de versão major). Evite @main ou @latest — o conteúdo pode mudar a qualquer momento e comprometer a segurança do seu pipeline. Para máxima segurança em repositórios corporativos, fixe o SHA completo: uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683.

Técnicas Avançadas

Matrix Strategy — testes em múltiplos ambientes em paralelo

Matrix permite rodar o mesmo job com diferentes combinações de parâmetros simultaneamente — múltiplas versões de Python, múltiplos SOs, múltiplas configurações de banco. O que seria um teste sequencial de 15 minutos vira 3 minutos em paralelo.

Matrix: Python 3.11 e 3.12 × Ubuntu e macOS
jobs:
  test:
    strategy:
      fail-fast: false    # Não cancela as outras combinações se uma falhar
      matrix:
        python-version: ['3.11', '3.12']
        os: [ubuntu-latest, macos-latest]
        # Excluir combinação específica:
        exclude:
          - os: macos-latest
            python-version: '3.11'
        # Adicionar combinação extra com variáveis extras:
        include:
          - os: ubuntu-latest
            python-version: '3.12'
            experimental: true

    runs-on: ${{ matrix.os }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pytest -v

Reusable Workflows — workflows como funções

Workflows reutilizáveis permitem extrair pipelines comuns (build Docker, deploy, etc.) em arquivos separados e chamá-los de outros workflows — exatamente como funções em programação.

❌ Duplicação em cada workflow
# ci.yml — duplica o build
jobs:
  build:
    steps:
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        # ... 20 linhas repetidas em cada workflow
✅ Reusable workflow centralizado
# ci.yml — chama o workflow reutilizável
jobs:
  build:
    uses: ./.github/workflows/docker-build.yml
    with:
      image-name: myapp
      push: true
    secrets: inherit   # Passa os secrets automaticamente

Environments com aprovação manual

Para deploy em produção, você pode exigir aprovação de um revisor antes do job de deploy rodar. Configure em Settings → Environments → production → Required reviewers. 

jobs:
  deploy-prod:
    runs-on: ubuntu-latest
    environment: production   # ← Pausa aqui e aguarda aprovação dos revisores
    needs: [test, build]

    steps:
      - name: Deploy para produção
        run: ./scripts/deploy-prod.sh

app = FastAPI()

@app.get("/")
def root():

def health():

from fastapi.testclient import TestClient
from app.main import app, soma

client = TestClient(app)

def test_root():
    response = client.get("/")
    assert response.status_code == 200
    assert response.json()["status"] == "ok"

def test_health():
    response = client.get("/health")
    assert response.status_code == 200
    assert response.json()["status"] == "healthy"

def test_soma():
    assert soma(2, 3) == 5
    assert soma(-1, 1) == 0
    assert soma(0, 0) == 0

Criar o workflow de CI
Crie o arquivo .github/workflows/ci.yml:
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  test:
    name: 🧪 Testes Python ${{ matrix.python-version }}
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.11', '3.12']

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}

      - name: Cache pip
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-${{ matrix.python-version }}-${{ hashFiles('**/requirements*.txt') }}

      - name: Instalar dependências
        run: |
          pip install -r requirements.txt
          pip install -r requirements-dev.txt

      - name: Lint com Ruff
        run: ruff check .

      - name: Verificar formatação com Black
        run: black --check .

      - name: Rodar testes com cobertura
        run: |
          pytest tests/ \
            --cov=app \
            --cov-report=term-missing \
            --cov-fail-under=80 \
            -v

  build-check:
    name: 🐳 Verificar Docker Build
    runs-on: ubuntu-latest
    needs: test

    steps:
      - uses: actions/checkout@v4

      - name: Setup Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build (sem push — apenas verifica)
        uses: docker/build-push-action@v5
        with:
          context: .
          push: false
          cache-from: type=gha

// PITFALLS

⚠️ Armadilhas Comuns

1. Gastar minutos com workflows desnecessários

Cada push dispara workflows que consomem os minutos gratuitos (2.000/mês no plano Free). Use paths-ignore para não rodar CI em mudanças de documentação, use concurrency para cancelar runs anteriores do mesmo PR, e use if: github.ref == 'refs/heads/main' para limitar jobs pesados à branch main.

2. Usar @latest ou @main em actions de terceiros

Actions com @latest podem ser atualizadas com código malicioso (supply chain attack). Sempre use tags de versão major (@v4) como mínimo, ou SHA completo para ambientes de alta segurança. Verifique se a action é oficial (actions/) ou tem boa reputação na comunidade.

3. Secrets visíveis em logs

Nunca faça echo ${{ secrets.MINHA_SENHA }} diretamente — mesmo que o GitHub mascare o valor, é um hábito perigoso. Injete sempre via variável de ambiente no bloco env: do step. E revise os logs de runs anteriores se suspeitar que um secret vazou.

4. Jobs de deploy sem concurrency — deploys sobrepostos

Se dois pushes acontecem em sequência rápida, dois jobs de deploy podem rodar simultaneamente, gerando estado inconsistente em produção. Sempre adicione concurrency: group: deploy-production com cancel-in-progress: false (não cancele um deploy já em andamento — espere terminar).

5. Usar run: sem shell explícito para scripts longos

O bloco run: usa bash -e por padrão no Linux — qualquer comando que retorna código de erro diferente de zero aborta o step. Isso é bom! Mas para scripts complexos com condicionais, prefira criar um arquivo .sh e chamar com run: bash scripts/deploy.sh para melhor legibilidade e testabilidade.

Termos desta Aula

CI — Integração Contínua
Prática de integrar mudanças de código frequentemente, com testes automáticos rodando a cada integração para detectar problemas rapidamente.
CD — Entrega / Deployment Contínuo
Extensão do CI que automatiza também a etapa de deploy: código aprovado vai automaticamente para staging (Entrega Contínua) ou produção (Deployment Contínuo).
Workflow
Arquivo YAML em .github/workflows/ que define todo o pipeline — triggers, jobs, steps. Versionado no Git como qualquer outro código.
Runner
Máquina virtual (hospedada pelo GitHub ou self-hosted) que executa os jobs. GitHub oferece Ubuntu, Windows e macOS. Cada job roda numa instância limpa e descartável.
Job
Unidade de trabalho que roda num runner. Jobs rodam em paralelo por padrão; use needs: para sequenciamento. Cada job começa do zero — sem estado do job anterior.
Step
Menor unidade do pipeline — um comando (run:) ou uma Action (uses:). Steps dentro de um job compartilham o mesmo ambiente e filesystem.
Action
Pacote reutilizável de automação publicado no Marketplace. Referenciado com uses: owner/repo@version. Pode ser de JavaScript, Docker ou composite steps.
GITHUB_TOKEN
Secret automático gerado pelo GitHub para cada run de workflow. Tem permissões para operações no repositório (push, comentar em PRs, criar releases, push no GHCR) sem configuração manual.
Matrix Strategy
Recurso que multiplica um job com diferentes combinações de parâmetros (versões, SOs, configs) rodando em paralelo. Transforma N testes sequenciais em 1 run paralelo.
Environment
Configuração de deployment com nome (staging, production) que pode ter secrets próprios, URL de deployment e revisores obrigatórios que aprovam antes do job rodar.

🏁 Checkpoint — Desafio Final

Configure um pipeline CI/CD completo em um repositório público no GitHub com pelo menos dois workflows:

  • Workflow de CI com matrix strategy (Python 3.11 e 3.12) rodando em push e pull_request
  • Testes com pytest e verificação de cobertura mínima de 80% (--cov-fail-under=80)
  • Lint e verificação de formatação (ruff + black) bloqueando o merge se falharem
  • Cache de pip usando actions/cache@v4 — meça a diferença de tempo entre 1º e 2º run
  • Workflow de CD que builda a imagem Docker e publica no GHCR (GitHub Container Registry)
  • O CD deve rodar apenas na branch main e somente após o CI passar (needs)
  • Bônus: Configure Branch Protection Rules na main exigindo status checks obrigatórios
💬 Dica: use workflow_dispatch no seu workflow de CD para poder disparar manualmente durante o desenvolvimento sem precisar fazer push real. E explore a aba Actions no GitHub — ela mostra logs em tempo real, tempos de cada step e permite re-rodar jobs específicos sem novo push.
