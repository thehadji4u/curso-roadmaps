// motivação

Por que embeddings são a espinha dorsal da IA moderna?

Embeddings são a tecnologia subjacente de praticamente toda aplicação de IA que vai além de uma simples chamada de chat. Busca semântica, RAG pipelines, sistemas de recomendação, detecção de similaridade, clustering de documentos — tudo depende de representar texto (ou imagens, áudio, código) como vetores numéricos comparáveis.

Sem entender embeddings, você não consegue construir um RAG pipeline funcional. Você não sabe por que sua busca retorna resultados ruins. Você não consegue diagnosticar por que o modelo "confunde" conceitos semanticamente próximos. Embeddings são a camada de fundação invisível que sustenta tudo.

💼 O que um engenheiro sênior faz com embeddings

A diferença entre busca por palavra-chave e busca semântica é a diferença entre um produto mediano e um produto excelente:

  • Busca que entende intenção — "como eu configuro autenticação?" retorna documentos sobre OAuth, JWT, tokens de sessão — mesmo sem essas palavras na query.
  • RAG sem embeddings não existe — para recuperar os chunks certos para o LLM responder, você precisa comparar a query do usuário com milhares de documentos em milissegundos. Isso é o trabalho do vector store.
  • Recomendação personalizada — "usuários que gostaram de X também gostam de Y" é calculado via distância vetorial, não por regras manuais.
  • Deduplicação e clustering — identificar documentos duplicados ou agrupar tickets de suporte por tema sem precisar de labels manuais.
🔍

Busca semântica real

Encontra documentos por significado, não por coincidência de palavras. Query em inglês pode retornar resultado em português se o conteúdo for semanticamente equivalente.

🧱

Fundação do RAG

Todo pipeline de Retrieval-Augmented Generation usa embeddings para recuperar o contexto certo antes de enviar ao LLM. Aula 43 é inteiramente construída sobre esta.

📊

Clustering sem labels

Agrupe milhares de documentos por tema usando K-means sobre embeddings. Zero anotação manual, resultados surpreendentemente coerentes.

⚡

Latência de milissegundos

Vector stores com índices ANN (HNSW, IVF) retornam os N vizinhos mais próximos entre milhões de vetores em menos de 10ms.

// conceito técnico

O que são Embeddings — por dentro

Um embedding é um vetor numérico de alta dimensionalidade que captura o significado semântico de um texto. O modelo de embedding transforma uma string de comprimento arbitrário num vetor de tamanho fixo, onde textos semanticamente similares ficam próximos no espaço vetorial.

1536 dimensões · text-embedding-3-small
3072 dimensões · text-embedding-3-large
384 dimensões · all-MiniLM-L6-v2 (local)
768 dimensões · all-mpnet-base-v2 (local)

Cada dimensão captura uma "feature" semântica abstrata — não interpretável por humanos, mas matematicamente poderosa. A combinação de 1536 números representa a "localização" do texto num espaço semântico aprendido por bilhões de exemplos de linguagem.

🔢 Visualizando um embedding

"gato" → [0.2, -0.5, 0.8, 0.1, -0.3, ..., 0.4]  ·  1536 números

"felino" → [0.19, -0.48, 0.79, 0.12, -0.31, ..., 0.38]  ·  muito próximo!

"avião"  → [-0.7, 0.3, -0.2, 0.9, 0.5, ..., -0.1]  ·  bem distante

⚠️ Embedding não é conhecimento factual

Embeddings capturam similaridade semântica, não verdade. "A Terra é plana" e "A Terra é redonda" podem ter embeddings próximos porque falam do mesmo conceito (forma da Terra), mas o embedding não sabe qual é verdadeiro. Nunca use embeddings para verificação factual — use LLMs com RAG para isso.

Gerando Embeddings com Python

OpenAI Embeddings (cloud, alta qualidade)

O
OpenAI text-embedding-3
Melhor qualidade geral · Multilíngue nativo · Pay-per-use
python · openai_embeddings.py
import os
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

def gerar_embedding(texto: str) -> list[float]:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=texto
    )
    return response.data[0].embedding

# Exemplo
vetor = gerar_embedding("Como instalar Python no Ubuntu?")
print(f"Dimensões: {len(vetor)}")  # 1536
print(f"Primeiros 5 valores: {vetor[:5]}")

# Batch embedding (mais eficiente — uma única chamada de API)
textos = ["Python básico", "Machine Learning", "Docker containers"]
response = client.embeddings.create(
    model="text-embedding-3-small",
    input=textos
)
vetores = [item.embedding for item in response.data]
print(f"Total de vetores gerados: {len(vetores)}")  # 3

sentence-transformers

from sentence_transformers import SentenceTransformer

# Modelo local — roda sem API, sem custo por chamada
modelo = SentenceTransformer("sentence-transformers/all-MiniLM-L6-v2")

textos = [
    "Como instalar Python?",
    "Guia de instalação do Python",
    "Receita de bolo de chocolate"
]

vetores = modelo.encode(textos, normalize_embeddings=True)
print(f"Shape: {vetores.shape}")  # (3, 384)

# normalize_embeddings=True → vetores unitários (norma = 1)
# Com vetores normalizados, distância euclidiana == 2*(1 - similaridade_cosseno)
# Isso torna cosseno e dot product matematicamente equivalentes
💡 Quando usar cada abordagem

Use OpenAI para produção com dados públicos onde qualidade máxima importa. Use sentence-transformers local para dados sensíveis (código proprietário, PII, dados médicos), prototipagem sem custo de API, ou quando você precisa rodar offline. Os modelos locais são surpreendentemente bons — all-MiniLM-L6-v2 com 384 dimensões bate muitas APIs pagas em benchmarks de recuperação.

Similaridade Cosseno — Como funciona

A similaridade cosseno mede o ângulo entre dois vetores, não a distância entre eles. Isso é fundamental: dois textos curtos e dois textos longos sobre o mesmo tema terão vetores de magnitudes diferentes, mas apontando na mesma direção — e o cosseno ignora a magnitude, focando apenas na direção.

-1.0 Oposto semântico
"bom" vs "ruim"
0.0 Sem relação
"cachorro" vs "foguete"
0.7 Relacionado
"Python" vs "programação"
1.0 Idêntico
mesma frase, mesmo embedding

Por que cosseno e não distância euclidiana?

A distância euclidiana mede o comprimento do segmento entre dois pontos. Mas um documento de 10 palavras e um de 1000 palavras sobre o mesmo tema terão vetores de magnitudes muito diferentes — a distância euclidiana os colocaria distantes mesmo sendo semanticamente próximos. O cosseno resolve isso ao normalizar pela magnitude, comparando apenas a direção dos vetores.

python · cosine_similarity.py
import numpy as np

def similaridade_cosseno(vetor_a: list[float], vetor_b: list[float]) -> float:
    a = np.array(vetor_a)
    b = np.array(vetor_b)
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))

# Teste com embeddings reais
v1 = gerar_embedding("Python é uma linguagem de programação")
v2 = gerar_embedding("Python é usado para desenvolver software")
v3 = gerar_embedding("Futebol é o esporte mais popular do Brasil")

sim_12 = similaridade_cosseno(v1, v2)
sim_13 = similaridade_cosseno(v1, v3)

print(f"v1 vs v2 (ambos sobre Python): {sim_12:.4f}")  # ~0.92
print(f"v1 vs v3 (Python vs Futebol): {sim_13:.4f}")   # ~0.31

# Interpretação:
# > 0.9  → quase idêntico em significado
# 0.7–0.9 → altamente relacionado
# 0.5–0.7 → relacionado mas distinto
# < 0.5  → pouco ou nenhuma relação semântica
✅ Boa prática: vetores normalizados

Quando você usa normalize_embeddings=True no sentence-transformers (ou normaliza manualmente com v / np.linalg.norm(v)), o dot product simples equivale ao cosseno. Isso é mais rápido computacionalmente e é o formato esperado pela maioria dos vector stores. O pgvector com o operador <=> (cosine distance) faz isso internamente.

Chroma — Local, ideal para desenvolvimento

Chroma é o vector store mais fácil de começar: pip install chromadb, zero configuração de servidor, persiste em arquivo local. Perfeito para desenvolvimento, prototipagem e projetos pequenos/médios. Para produção em larga escala, migre para Pinecone ou Qdrant.

C
ChromaDB
pip install chromadb · SQLite local · Sem servidor · Gratuito
python · chroma_example.py
import chromadb
import os
from chromadb.utils import embedding_functions

# Cliente local com persistência em arquivo
client = chromadb.PersistentClient(path="./chroma_db")

# Função de embedding (OpenAI) — Chroma embute automaticamente
openai_ef = embedding_functions.OpenAIEmbeddingFunction(
    api_key=os.environ["OPENAI_API_KEY"],
    model_name="text-embedding-3-small"
)

# Criar ou recuperar coleção
colecao = client.get_or_create_collection(
    name="documentos_curso",
    embedding_function=openai_ef
)

# Adicionar documentos (Chroma gera os embeddings automaticamente)
colecao.add(
    documents=[
        "Python é uma linguagem de alto nível",
        "Docker containeriza aplicações",
        "Git é um sistema de controle de versão"
    ],
    metadatas=[
        {"fonte": "aula-01", "topico": "python"},
        {"fonte": "aula-17", "topico": "docker"},
        {"fonte": "aula-01", "topico": "git"}
    ],
    ids=["doc1", "doc2", "doc3"]
)

# Busca semântica por linguagem natural
resultados = colecao.query(
    query_texts=["Como usar containers?"],
    n_results=2,
    where={"topico": "docker"}  # filtro por metadado (opcional)
)

print(resultados["documents"])
print(resultados["distances"])  # menor = mais similar (distância, não similaridade)

# Atualizar documento existente
colecao.update(
    ids=["doc1"],
    documents=["Python é uma linguagem interpretada de alto nível"],
    metadatas=[{"fonte": "aula-01-v2", "topico": "python"}]
)

# Total de documentos na coleção
print(f"Total: {colecao.count()}")

Pinecone — Cloud gerenciado para produção

Pinecone é o vector store mais usado em produção: escalabilidade automática, SLA garantido, filtragem avançada de metadados, suporte a bilhões de vetores. O custo é por uso, mas o free tier permite prototipagem e aplicações pequenas.

P
Pinecone
Cloud managed · Serverless tier gratuito · Bilhões de vetores
python · pinecone_example.py
from pinecone import Pinecone, ServerlessSpec
import os

pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])

# Criar índice (executar apenas uma vez)
# dimension deve ser igual ao modelo de embedding usado
if "curso-docs" not in pc.list_indexes().names():
    pc.create_index(
        name="curso-docs",
        dimension=1536,       # text-embedding-3-small
        metric="cosine",
        spec=ServerlessSpec(cloud="aws", region="us-east-1")
    )

index = pc.Index("curso-docs")

# Upsert: inserir ou atualizar vetores com metadados
vetores_para_upsert = [
    {
        "id": "doc-001",
        "values": gerar_embedding("Python para iniciantes"),
        "metadata": {"titulo": "Aula 1", "topico": "python", "aula_id": 1}
    },
    {
        "id": "doc-002",
        "values": gerar_embedding("Docker e containers"),
        "metadata": {"titulo": "Aula 17", "topico": "docker", "aula_id": 17}
    }
]

index.upsert(vectors=vetores_para_upsert)

# Query com filtro de metadados
resultado = index.query(
    vector=gerar_embedding("Como rodar aplicação em container?"),
    top_k=3,
    include_metadata=True,
    filter={"topico": {"$in": ["docker", "python"]}}
)

for match in resultado.matches:
    print(f"Score: {match.score:.3f} | ID: {match.id} | {match.metadata['titulo']}")

# Stats do índice
stats = index.describe_index_stats()
print(f"Total de vetores: {stats['total_vector_count']}")

pgvector — PostgreSQL com superpowers

Se sua stack já usa PostgreSQL, pgvector é a opção mais pragmática: você mantém tudo num único banco, aproveita SQL completo para filtragem complexa, transações ACID, joins com outras tabelas. Perfeito para quando você quer busca semântica integrada ao seu modelo relacional existente.

PG
pgvector
PostgreSQL extension · SQL completo · HNSW index · Self-hosted
sql · setup_pgvector.sql
-- Instalar extensão (uma vez por banco)
CREATE EXTENSION IF NOT EXISTS vector;

-- Tabela com coluna vetorial
CREATE TABLE documentos (
    id          SERIAL PRIMARY KEY,
    conteudo    TEXT NOT NULL,
    embedding   vector(1536),        -- dimensão deve bater com o modelo
    metadados   JSONB DEFAULT '{}',
    criado_em   TIMESTAMP DEFAULT NOW()
);

-- Índice HNSW para busca aproximada rápida (recomendado para > 100k docs)
-- m: número de conexões por nó | ef_construction: qualidade do índice
CREATE INDEX ON documentos
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- Para datasets menores, IVFFlat é mais simples:
-- CREATE INDEX ON documentos USING ivfflat (embedding vector_cosine_ops)
-- WITH (lists = 100);
python · pgvector_example.py
import psycopg2
import json
import os

conn = psycopg2.connect(os.environ["DATABASE_URL"])
cur = conn.cursor()

# Inserir documento com embedding
vetor = gerar_embedding("Python é excelente para data science")
cur.execute(
    "INSERT INTO documentos (conteudo, embedding, metadados) VALUES (%s, %s, %s)",
    (
        "Python é excelente para data science",
        vetor,              # psycopg2 aceita lista de floats diretamente com pgvector
        json.dumps({"topico": "python"})
    )
)
conn.commit()

# Busca por similaridade cosseno
# Operador <=> = distância cosseno (0 = idêntico, 2 = oposto)
# 1 - (embedding <=> query) = similaridade cosseno
query_vetor = gerar_embedding("linguagem para análise de dados")
cur.execute("""
    SELECT
        conteudo,
        1 - (embedding <=> %s::vector) AS similaridade,
        metadados
    FROM documentos
    ORDER BY embedding <=> %s::vector
    LIMIT 5
""", (query_vetor, query_vetor))

for row in cur.fetchall():
    print(f"Similaridade: {row[1]:.3f} | {row[0][:60]}")

# Busca com filtro SQL (vantagem única do pgvector!)
cur.execute("""
    SELECT conteudo, 1 - (embedding <=> %s::vector) AS sim
    FROM documentos
    WHERE metadados->>'topico' = 'python'   -- filtro SQL normal
      AND criado_em > NOW() - INTERVAL '7 days'
    ORDER BY embedding <=> %s::vector
    LIMIT 3
""", (query_vetor, query_vetor))

cur.close()
conn.close()

Qdrant é um vector store de código aberto escrito em Rust, com foco em performance e recursos avançados: payloads ricos, filtragem sofisticada, quantização de vetores para reduzir memória, e tanto cloud gerenciado quanto Docker self-hosted gratuito.

python · qdrant_example.py
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct, Filter, FieldCondition, MatchValue

client = QdrantClient(url="http://localhost:6333")

# Criar coleção
client.create_collection(
    collection_name="curso_docs",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE)
)

# Inserir pontos com payload (metadados ricos)
client.upsert(
    collection_name="curso_docs",
    points=[
        PointStruct(
            id=1,
            vector=gerar_embedding("FastAPI para APIs REST"),
            payload={
                "titulo": "FastAPI",
                "dificuldade": "intermediário",
                "aula_id": 22,
                "tags": ["python", "api", "web"]
            }
        ),
        PointStruct(
            id=2,
            vector=gerar_embedding("Docker containers e microserviços"),
            payload={
                "titulo": "Docker",
                "dificuldade": "intermediário",
                "aula_id": 17,
                "tags": ["docker", "containers", "devops"]
            }
        )
    ]
)

# Busca com filtro estruturado
resultados = client.search(
    collection_name="curso_docs",
    query_vector=gerar_embedding("criar API web em Python"),
    query_filter=Filter(
        must=[
            FieldCondition(
                key="dificuldade",
                match=MatchValue(value="intermediário")
            )
        ]
    ),
    limit=3
)

for r in resultados:
    print(f"Score: {r.score:.3f} | ID: {r.id} | {r.payload['titulo']}")

# Scroll para listar todos os documentos
pontos, _ = client.scroll(
    collection_name="curso_docs",
    limit=100,
    with_payload=True
)
print(f"Total de pontos: {len(pontos)}")

Chunking — Preparando textos para embeddings

Modelos de embedding têm um limite de tokens (geralmente 512 a 8192). Mas mais importante: chunks maiores produzem embeddings "médios" imprecisos. Um chunk de 3000 palavras tem seu embedding contaminado por muitos temas diferentes, tornando a recuperação vaga. O tamanho ideal de chunk depende do seu caso de uso — tipicamente 200 a 600 tokens, com overlap de 10-20%.

Visualização de chunking com overlap

1
Python é uma linguagem de alto nível criada por Guido van Rossum em 1991...
⟵ overlap 50 chars ⟶
2
...criada por Guido van Rossum em 1991. Python é amplamente usado em ciência de dados...
⟵ overlap 50 chars ⟶
3
...usado em ciência de dados, IA e desenvolvimento web. Sua sintaxe clara...

O overlap garante que informações no limite entre chunks não se percam — a continuidade de contexto é preservada.

Estratégias de chunking

Fixed-size

Divide por número fixo de caracteres/tokens. Simples, mas pode cortar sentenças no meio.

Sentence-based

Divide por pontuação (., !, ?). Preserva sentenças inteiras, mas tamanho irregular.

Recursive Character

Tenta dividir por parágrafos, depois por linhas, depois por sentenças. Melhor compromisso — padrão do LangChain.

Semantic Chunking

Agrupa sentenças por similaridade semântica. Mais lento mas produz chunks tematicamente coerentes.

python · chunking.py
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # tamanho máximo em caracteres
    chunk_overlap=50,     # overlap entre chunks consecutivos
    separators=["\n\n", "\n", ".", " ", ""]  # ordem de preferência de split
)

texto = """
Python é uma linguagem de programação de alto nível.
Criada por Guido van Rossum em 1991, tornou-se uma das mais populares do mundo.

Python é amplamente usado em ciência de dados, inteligência artificial e desenvolvimento web.
Sua sintaxe clara e legível facilita o aprendizado.
"""

chunks = splitter.create_documents([texto])
print(f"Total de chunks: {len(chunks)}")
for i, chunk in enumerate(chunks):
    print(f"Chunk {i+1}: {len(chunk.page_content)} chars | Preview: {chunk.page_content[:60]}...")

# Adicionar metadados a cada chunk (importante para rastreabilidade)
chunks_com_meta = splitter.create_documents(
    texts=[texto],
    metadatas=[{"fonte": "aula-01.txt", "autor": "professor", "topico": "python"}]
)
# Cada chunk herda o metadado do documento-pai
print(chunks_com_meta[0].metadata)
⚠️ A regra de ouro do chunk size

Para sistemas de QA/RAG, o tamanho de chunk deve refletir a granularidade da informação buscada. Se o usuário faz perguntas pontuais ("qual o comando para X?"), use chunks pequenos (100-300 tokens). Se o usuário pede análises contextuais ("explique a arquitetura de Y"), use chunks maiores (500-1000 tokens). Experimente e meça com seu dataset real — não existe tamanho universalmente ideal.

Erros comuns que custam horas de debugging

🪤 Misturar modelos de embedding na ingestão e na query

O erro mais comum e mais silencioso: você ingere documentos com text-embedding-3-small (1536 dims) e depois faz queries com text-embedding-3-large (3072 dims). O vector store retorna resultados aleatórios ou lança erro de dimensão. Regra: o mesmo modelo deve ser usado do início ao fim. Documente e fixe a versão do modelo nos metadados do índice.

🪤 Chunk muito grande → embedding impreciso

Um chunk de 3000 palavras discutindo Python, depois Docker, depois SQL tem um embedding que é a "média" de todos esses temas. Quando o usuário pergunta sobre Docker, esse chunk pode não ser recuperado porque o sinal de Docker foi diluído pelos outros temas. Mantenha chunks temáticos e coerentes — de preferência um tópico por chunk.

🪤 Chunk muito pequeno → perde contexto

O chunk "O método retorna True" sem contexto do que é o método, em qual classe, em qual contexto — é inútil quando recuperado. O LLM não consegue formular uma resposta com chunks sem contexto suficiente. Use overlap e mantenha chunks com pelo menos uma sentença completa de contexto.

🪤 Embeddings não capturam fatos, só similaridade

"A Terra é plana" e "A Terra é esférica" terão embeddings próximos porque falam do mesmo conceito. O embedding não sabe qual é verdadeiro. Nunca use similaridade de embedding para verificação factual — isso requer um LLM com raciocínio.

🪤 Não normalizar vetores antes do cosseno no pgvector

O operador <=> do pgvector calcula distância cosseno corretamente independente da normalização, mas alguns cálculos manuais com <#> (produto interno negativo) assumem vetores normalizados. Normalize sempre com v / np.linalg.norm(v) antes de inserir para ser seguro com qualquer operador.

🪤 Misturar idiomas sem normalização degrada qualidade

Documentos em PT-BR e queries em EN funcionam com modelos multilíngues (como os da OpenAI e all-MiniLM-L6-v2 multilingual), mas modelos só-inglês como alguns da SBERT vão produzir resultados degradados para PT-BR. Escolha um modelo adequado ao seu idioma ou normalize os textos para inglês antes de embedar.

Busca semântica para os documentos do curso

Coloque em prática todos os conceitos desta aula: gere embeddings reais, armazene num vector store e construa uma busca semântica funcional para as aulas do curso.

  • Crie uma lista com os títulos e descrições das 50 aulas do roadmap
  • Gere embeddings de cada aula com text-embedding-3-small ou all-MiniLM-L6-v2 (local)
  • Armazene no ChromaDB com metadados: aula_id, trilha, link
  • Crie uma função buscar_aulas(duvida: str, n=3) que retorna as aulas mais relevantes
  • Exiba título, score de similaridade e link de cada resultado
  • Bônus: adicione um fallback que usa similaridade de keywords como segundo critério quando o score semântico for menor que 0.6
Teste a busca com queries como: "como usar docker na nuvem", "autenticação segura", "criando APIs rápidas em Python". Os resultados devem fazer sentido semântico mesmo sem usar as palavras exatas dos títulos das aulas.
