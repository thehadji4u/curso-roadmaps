01 · Motivação

Por que unir n8n e IA importa agora?

Durante anos, automação e inteligência artificial viveram em mundos separados. O n8n conectava sistemas, APIs e bancos de dados — mas seguia regras rígidas e pré-definidas. Os LLMs raciocinam, interpretam linguagem natural, geram conteúdo — mas ficavam isolados em chatbots sem integração com o mundo real.

O AI Agent Node do n8n derrubou esse muro. Agora um workflow visual pode incorporar raciocínio autônomo: o agente decide quais ferramentas usar, em qual ordem, com base no contexto da tarefa. O resultado é uma nova classe de automações que não era possível antes.

💼 O que você consegue construir hoje com n8n + IA

Casos de uso reais que engenheiros estão entregando em produção:

  • Triagem inteligente de e-mails — classifica urgência, extrai ação requerida, roteia automaticamente para o canal certo.
  • Suporte automatizado — responde tickets de suporte consultando base de conhecimento via RAG, escalando apenas casos complexos.
  • Análise de documentos — recebe PDFs via webhook, extrai entidades, classifica, gera resumo estruturado e salva no banco.
  • Geração de relatórios — puxa dados de múltiplas APIs, interpreta tendências com LLM, monta relatório em Markdown e envia por e-mail.
  • Agente de pesquisa — dada uma pergunta, busca na web, Wikipedia e base interna, sintetiza e responde com fontes citadas.
🎯 A vantagem competitiva do engenheiro sênior

Um engenheiro que domina n8n + IA entrega automações que antes exigiriam um backend completo (API REST, banco, scheduler, filas) em horas, não semanas. A combinação de orquestração visual com raciocínio LLM elimina a necessidade de escrever boilerplate — você foca na lógica de negócio.

02 · Analogia

O maestro e a orquestra

🌍 Mundo Real → Tecnologia
Imagine uma orquestra sinfônica de classe mundial. O maestro não toca nenhum instrumento diretamente — ele não precisa ser o violinista mais técnico nem o pianista mais veloz. Seu papel é coordenar: ler a partitura, entender a intenção musical, dar entradas aos músicos certos no momento certo, garantir harmonia entre seções completamente diferentes.

Os músicos virtuosos são os especialistas: o violinista executa passagens rápidas com precisão, o pianista interpreta harmonias complexas, o percussionista mantém o ritmo. Cada um é excelente no seu instrumento — mas sozinhos, sem coordenação, criam cacofonia.

O n8n é o maestro. Os LLMs são os músicos virtuosos. O n8n não "pensa" por si só — ele orquestra: aciona o agente OpenAI para interpretar a solicitação, chama a ferramenta HTTP para buscar dados, passa o resultado ao agente Anthropic para síntese, formata e entrega. A inteligência está nos músicos; a coordenação está no maestro.

03 · Arquitetura

Arquitetura do AI Agent no n8n

O AI Agent Node é o coração da integração IA no n8n. Ele implementa o padrão ReAct (Reasoning + Acting): o LLM raciocina sobre a tarefa, decide uma ação, executa via tool, observa o resultado, e itera até a resposta estar completa.

Arquitetura completa — AI Agent Node no n8n
flowchart TD A([Webhook Trigger]) --> D B([Schedule Trigger]) --> D C([Chat Trigger]) --> D D[AI Agent Node] -->|"Consulta modelo"| E{LLM} E --> E1[OpenAI GPT-4o] E --> E2[Anthropic Claude] E --> E3[Google Gemini] D <-->|"Acessa tools"| F{Tools} F --> F1[HTTP Request Tool] F --> F2[Calculator Tool] F --> F3[Code Tool] F --> F4[Vector Store Tool] F --> F5[Workflow Tool] D <-->|"Lê/escreve contexto"| G{Memory} G --> G1[Window Buffer Memory] G --> G2[Redis Chat Memory] G --> G3[Postgres Chat Memory] D --> H{Output} H --> H1[Respond to Webhook] H --> H2[Send Email] H --> H3[Update Google Sheet] H --> H4[Telegram / Slack] style D fill:#10b981,color:#000,font-weight:bold style E fill:#1e1e2e,color:#e8e8f0 style F fill:#1e1e2e,color:#e8e8f0 style G fill:#1e1e2e,color:#e8e8f0 style H fill:#1e1e2e,color:#e8e8f0
🔄 Como o ciclo ReAct funciona internamente

A cada chamada ao AI Agent Node, o LLM executa este loop interno:

  1. Thought — O modelo raciocina: "Para responder esta pergunta, preciso buscar dados atualizados."
  2. Action — Seleciona a tool: http_request_tool com os parâmetros adequados.
  3. Observation — Recebe o resultado da tool e incorpora ao contexto.
  4. Repeat — Decide se precisa de mais ações ou se já pode formular a resposta final.
  5. Final Answer — Gera a resposta quando tem informação suficiente ou atinge maxIterations.
04 · Setup

Configurando OpenAI / Anthropic no n8n

Antes de usar o AI Agent Node, você precisa registrar as credenciais dos provedores de LLM no n8n. O processo é idêntico para OpenAI e Anthropic — apenas os campos específicos diferem.

Criando uma OpenAI API Credential no n8n

1
Acesse Settings → Credentials
No menu lateral do n8n, clique em "Settings" e depois em "Credentials". Clique em "+ Add Credential" no canto superior direito.
2
Busque "OpenAI" no modal de credenciais
Digite "OpenAI" no campo de busca. Selecione "OpenAI API" (não confunda com "OpenAI" que é outra integração).
3
Cole sua API Key
No campo "API Key", cole sua chave do OpenAI (começa com sk-). Opcionalmente, defina a Organization ID se usar conta corporativa.
4
Salve e teste a conexão
Clique em "Save". O n8n faz uma chamada de teste automaticamente. Se aparecer um checkmark verde, a credencial está ativa.

05 · AI Agent Node

AI Agent Node — Configuração completa

O AI Agent Node é o nó central da automação inteligente. Ele recebe uma mensagem ou tarefa, consulta o LLM configurado, usa as tools disponíveis de forma autônoma, e retorna o resultado. A escolha do Agent Type determina a estratégia de raciocínio.

Agent Types — escolhendo a estratégia certa

Conversational Agent
O mais simples e direto. Mantém histórico de conversa e usa as tools quando o LLM decide que são necessárias. Não há planning explícito — o modelo reage ao contexto imediato. Ideal para chatbots e assistentes interativos.
✦ Melhor para: suporte ao cliente, FAQ inteligente, assistente de chat
ReAct Agent
Implementa o padrão Reason + Act: o modelo explica o raciocínio antes de cada ação ("Preciso saber o preço atual, então vou usar a HTTP tool para consultar a API"). Mais transparente e auditável. Consome mais tokens mas entrega trace completo do raciocínio.
✦ Melhor para: tasks multi-step, quando você precisa auditar decisões do agente
OpenAI Functions Agent
Usa o mecanismo nativo de function calling/tool use da OpenAI. O modelo seleciona tools através do protocolo oficial da API, que é mais robusto para selecionar múltiplas tools em paralelo. Exclusivo para modelos OpenAI que suportam tool_calls.
✦ Melhor para: workflows com muitas tools, uso de modelos GPT-4o/GPT-4o-mini
Plan and Execute Agent
Divide a tarefa em duas fases: primeiro gera um plano completo ("1. Buscar dados. 2. Calcular média. 3. Gerar relatório."), depois executa cada passo. Muito mais robusto para tarefas complexas mas significativamente mais lento e caro — pode consumir 5x mais tokens.
✦ Melhor para: tarefas de análise complexas, pipelines de processamento de dados

Configurando System Message e Output

💡 System Message é o DNA do seu agente

O System Message define personalidade, restrições, formato de output e contexto do agente. É a instrução mais importante: um bom system message reduz ambiguidade, melhora consistência e elimina alucinações. Sempre inclua: idioma de resposta, tom, formato esperado e limitações.

Exemplo de configuração JSON de um agente de suporte técnico Python:

AI Agent Node · Configuração · JSON
{
  "agent": "conversationalAgent",
  "options": {
    "systemMessage": "Você é um agente de suporte técnico especializado em Python. Responda sempre em português, seja direto e forneça exemplos de código quando relevante.",
    "maxIterations": 10
  }
}

No UI do n8n, o System Message é um campo de texto livre com suporte a expressões dinâmicas usando {{ }}. Você pode injetar contexto do workflow:

System Message com contexto dinâmico
Você é um agente de suporte especializado em Python.

CONTEXTO DO USUÁRIO:
- Nome: {{ $('Webhook').item.json.userName }}
- Plano: {{ $('Webhook').item.json.planType }}
- Histórico de tickets: {{ $('Get Tickets').item.json.count }} tickets anteriores

INSTRUÇÕES:
- Responda SEMPRE em português brasileiro
- Seja conciso — máximo 3 parágrafos por resposta
- Forneça exemplos de código quando relevante (use blocos markdown)
- Se a pergunta estiver fora do escopo Python, diga que não pode ajudar
- Não revele informações sobre outros usuários
⚠️ Nunca use expressões para injetar API Keys no System Message

Um erro comum: usar {{ $env.OPENAI_KEY }} no System Message para "contexto". Isso expõe a chave no histórico de conversa e nos logs do LLM. Use sempre o mecanismo de Credentials do n8n para segredos.

06 · Tools

Tools disponíveis para o AI Agent

Tools são as "mãos" do agente — permitem que ele interaja com o mundo além do que está no contexto. O agente decide autonomamente quando e como usar cada tool com base na tarefa recebida.

🧮

Calculator Tool

Executa expressões matemáticas precisas. Essencial para evitar que o LLM "calcule" por conta própria (LLMs são notoriamente imprecisos em aritmética). Ex.: soma de valores de uma fatura, cálculo de percentual de crescimento.

🌐

HTTP Request Tool

Faz requisições HTTP para qualquer API externa. O agente define URL, método, headers e body com base na tarefa. Ex.: consultar preço em API de produtos, buscar status de pedido, chamar API de clima.

💻

Code Tool (JavaScript / Python)

Executa código JavaScript ou Python gerado pelo agente. Permite transformações de dados complexas, formatação customizada, lógica que seria difícil de expressar em prompt. Atenção: execute em ambiente sandboxed.

📖

Wikipedia Tool

Busca artigos na Wikipedia. Útil para agentes de pesquisa que precisam de informações factuais gerais. O agente recebe o resumo do artigo como contexto para formular a resposta.

🗄️

Vector Store Tool

Busca semântica em uma Vector Store (Pinecone, Qdrant, Weaviate, Supabase). É a base do padrão RAG no n8n: o agente faz uma query semântica e recebe os chunks de documento mais relevantes para fundamentar a resposta.

🔀

Workflow Tool

Chama outro workflow n8n como se fosse uma função. Permite criar agentes compostos: o agente principal delega subtarefas a agentes especializados em outros workflows — padrão de multi-agent.

07 · Memory

Memory no n8n — Mantendo contexto de conversação

Por padrão, cada execução do AI Agent Node é stateless — o agente não lembra de execuções anteriores. O módulo de Memory resolve isso, permitindo conversas com contexto persistente.

🪟
Window Buffer Memory
In-process
Armazena as últimas N mensagens da conversa na memória do processo n8n. Rápido, sem dependência externa. A desvantagem é que o histórico é perdido se o n8n reiniciar, e não suporta múltiplas sessões simultâneas de forma segura sem configuração de session ID. Ideal para protótipos, demos e fluxos de uma única conversa.
⚡
Redis Chat Memory
Persistente
Histórico de conversa armazenado no Redis com TTL configurável. Suporta múltiplas sessões simultâneas via session ID único por usuário. Ideal para produção: sobrevive a reinicializações do n8n, escala horizontalmente, e o TTL garante limpeza automática de sessões antigas.
🗃️
Postgres Chat Memory
Persistente
Histórico completo em banco de dados PostgreSQL. Permite queries SQL para análise de conversas, auditoria, compliance e histórico ilimitado. Mais lento que Redis para leitura/escrita, mas essencial quando você precisa consultar o histórico de conversas retroativamente ou gerar relatórios de uso.

Configuração de Redis Chat Memory

Redis Memory · Configuração JSON no n8n
{
  "sessionId": "={{ $('Webhook').item.json.sessionId }}",
  "sessionTTL": 3600,
  "maxWindowSize": 10
}
⚠️ Session ID fixo = todos os usuários compartilham memória

Um dos erros mais perigosos ao configurar memória no n8n: usar uma string fixa como session ID:

// ERRADO — todos os usuários vêem a conversa um do outro
{ "sessionId": "my-agent-session" }

// CORRETO — cada usuário/conversa tem seu próprio contexto
{ "sessionId": "={{ $('Webhook').item.json.userId }}-{{ $('Webhook').item.json.conversationId }}" }

Sempre use um identificador único derivado do contexto: user ID, session token, ou UUID gerado no início da conversa.

📐 Escolhendo maxWindowSize corretamente

O maxWindowSize define quantas mensagens (pares usuário/assistente) ficam no contexto. Cada mensagem adiciona tokens ao input do LLM — o que aumenta custo e latência. Para suporte técnico, 5-10 mensagens é suficiente. Para análise de documentos longos, 3-5 basta. Monitore o uso de tokens e ajuste conforme necessário.

08 · Exemplo Completo

Pipeline RAG no n8n — Q&A sobre documentos

RAG (Retrieval-Augmented Generation) é a técnica de enriquecer o contexto do LLM com informações buscadas dinamicamente em uma base de conhecimento. No n8n, isso se divide em dois workflows: ingestão (preparar os documentos) e consulta (responder perguntas).

Workflow 1: Ingestão de documentos

Schedule Trigger
→
Executa diariamente às 02:00 para ingestão incremental
Cron: 0 2 * * *
HTTP Request
→
Baixa PDFs de um bucket S3 ou endpoint interno
GET /api/documents?updated_since={{ $today.minus(1, 'day') }}
Extract from File
→
Extrai texto dos PDFs usando o nó nativo do n8n
Suporta PDF, DOCX, TXT, HTML, Markdown
Recursive Character Text Splitter
→
Divide texto em chunks de ~500 tokens com 50 tokens de overlap
Overlap garante que contexto não seja cortado no meio de uma frase
Embeddings OpenAI
→
Gera vetores de 1536 dimensões com text-embedding-3-small
Custo: ~$0.02 por 1M tokens (muito mais barato que GPT-4)
Pinecone Vector Store
→
Faz upsert dos vetores com metadados (doc_id, chunk_index, source)
Alternativas: Qdrant (self-hosted), Supabase, Weaviate

Workflow 2: Consulta com AI Agent + RAG

Webhook / Chat Trigger
→
Recebe pergunta do usuário com session ID
POST /webhook/qa com body: { "question": "...", "sessionId": "abc123" }
AI Agent Node
⇄
Orquestra busca e resposta com GPT-4o-mini
Tool 1: Vector Store Retriever → busca semântica (top-5 chunks)
Tool 2: HTTP Request → fonte externa para contexto adicional
Memory: Window Buffer (5 mensagens)
Respond to Webhook
→
Retorna resposta contextualizada com chunks fontes
JSON: { "answer": "...", "sources": [...], "confidence": 0.92 }

Roteamento por classificação

Baseado na resposta do AI Agent Node, o workflow divide em 3 branches:

System Message · Agente de Triagem de E-mails
Analise o e-mail abaixo e retorne um JSON válido com esta estrutura:

Critérios de classificação:

Para evitar que o LLM adicione texto antes ou depois do JSON, use o campo "Response Format" do OpenAI Chat Model Node e selecione json_object. Isso garante que a saída seja sempre JSON válido, eliminando a necessidade de parsing manual.

10 · Armadilhas

Erros comuns e como evitá-los

⏱️ Timeout muito curto para LLMs

LLMs podem demorar 20–60 segundos em respostas longas ou quando o servidor está sob alta carga. O timeout padrão do n8n é 5 segundos para nós HTTP — insuficiente. No OpenAI Chat Model Node, sempre defina timeout: 60000 (60 segundos) ou mais. Para agentes com múltiplas iterações, considere 120 segundos.

🔄 Loop infinito no agente (maxIterations muito alto)

Se o agente não consegue completar a tarefa (tool retorna erro, instrução ambígua), ele pode iterar indefinidamente, consumindo tokens e gerando custos sem retorno. Sempre defina maxIterations para um valor razoável (5–15). Para tasks simples, 5 é suficiente. Para pipelines complexos, 15 raramente é excedido. Monitore execuções com muitas iterações — geralmente indicam um problema no system message ou nas tools.

🔑 Expor API Keys no System Message via expressões

Usar {{ $env.OPENAI_KEY }} ou similar no System Message injeta segredos no contexto do LLM e nos logs de execução. Use EXCLUSIVAMENTE o sistema de Credentials do n8n para API keys. O n8n criptografa credenciais em repouso — expressões no System Message não recebem essa proteção.

❌ Não tratar erros do LLM (rate limit, context overflow)

APIs de LLM retornam erros específicos que precisam de tratamento diferente:

  • 429 Rate Limit — adicione um Wait Node com retry exponencial (2s, 4s, 8s)
  • 400 Context Length Exceeded — reduza o maxWindowSize da memory ou o tamanho dos chunks RAG
  • 500 Server Error — adicione Error Trigger workflow para alertar no Slack/Telegram

Use o nó "Error Trigger" do n8n para capturar todas as falhas e notificar automaticamente.

👥 Memory session ID fixo — todos os usuários compartilham memória

Se você hardcoda o session ID (ex: "sessionId": "agent-memory"), todos os usuários que interagem com o agente vêem o histórico de conversa uns dos outros. Sempre gere um session ID único por usuário/conversa: "sessionId": "={{ $('Webhook').item.json.userId }}" ou use um UUID gerado no início do fluxo.

📊 Não monitorar custo de tokens em produção

Agentes com memory e múltiplas tools podem consumir 10–50x mais tokens que uma chamada simples de completions. Uma execução de agente com 10 iterações, 5 mensagens de histórico e 3 tool calls pode facilmente consumir 20.000 tokens. Configure alertas de custo na OpenAI/Anthropic e registre o uso de tokens em cada execução no n8n para monitoramento.

✦ Checkpoint · Aula 41

Agente de suporte técnico com memória e registro

Construa do zero um agente de suporte técnico funcional no n8n que demonstre todos os conceitos desta aula em um workflow coeso e pronto para produção.

  • Crie um Webhook Trigger que recebe POST com { "question": "...", "userId": "...", "sessionId": "..." }
  • Configure um OpenAI Chat Model Node com GPT-4o-mini, temperature 0.3, timeout 60000ms
  • Adicione um AI Agent Node (Conversational Agent) com system message em português especializado em Python
  • Conecte um HTTP Request Tool apontando para uma API pública (ex: api.github.com para buscar issues)
  • Configure Window Buffer Memory com sessionId: {{ $json.sessionId }} e maxWindowSize: 5
  • Adicione um Google Sheets node ao final para registrar: timestamp, userId, pergunta, resposta, tokens usados
  • Responda via Respond to Webhook com JSON: { "answer": "...", "sessionId": "..." }
  • Teste com pelo menos 3 perguntas sequenciais para validar que a memória de conversa funciona
Dica de validação: faça a pergunta "Qual foi minha primeira pergunta desta sessão?" — o agente deve lembrar corretamente graças ao Window Buffer Memory configurado com o session ID correto.
