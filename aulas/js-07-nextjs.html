// motivação

Por que isso importa?

React resolve o problema de construir UIs interativas. Mas quando você precisa de SEO, performance de carregamento inicial, rotas, API, autenticação e deploy — você precisa de mais. Next.js é o framework que adiciona tudo isso em cima do React, de forma opinionada mas flexível.

Com o App Router (Next.js 13+), o modelo mental mudou completamente: por padrão, todos os componentes são Server Components que rodam no servidor — sem bundle JS no cliente, com acesso direto ao banco de dados, e com SSR automático. É a maior evolução do ecossistema React dos últimos anos.

💼 Por que um engenheiro sênior usa Next.js hoje
Sêniors usam Next.js porque ele resolve o problema mais difícil do front-end moderno: entregar páginas rápidas (para SEO e UX), com dados frescos (para conteúdo dinâmico), sem sobrecarregar o servidor (para escalar) e com uma DX excelente (para produtividade). O App Router com RSC redefine o que significa "full-stack em JavaScript".
// analogia

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense num restaurante com dois tipos de atendimento: balcão e mesa. No balcão (SSG), o prato já está pronto quando você chega — feito antes, servido imediatamente, sempre igual. Na mesa (SSR), você faz o pedido e a cozinha prepara na hora — mais personalizado, mas leva um instante. E tem ainda o modelo de "buffet atualizado" (ISR): a comida foi preparada antes, mas de tempos em tempos a cozinha renova o prato.

O Next.js deixa você escolher o modelo para cada "prato" (página) — e misturá-los no mesmo restaurante (app).
🍱 Prato pronto no balcão → SSG (Static Site Generation) 👨‍🍳 Cozinha prepara no pedido → SSR (Server-Side Rendering) 🔄 Buffet renovado periodicamente → ISR (Incremental Static Regeneration) 🧑‍💻 Cozinheiro na cozinha (invisível) → React Server Components (RSC)

App Router vs Pages Router

Next.js tem dois sistemas de roteamento. O Pages Router (legado, mas ainda suportado) usa convenções de arquivo em pages/. O App Router (Next.js 13+, atual) usa app/ e traz Server Components, Layouts aninhados e Server Actions.

CaracterísticaPages RouterApp Router (atual)
Diretório raizpages/app/
Componentes padrãoClient ComponentsServer Components
Data fetchinggetServerSideProps, getStaticPropsasync/await direto no componente
LayoutsManual (via _app.js)Automático via layout.tsx
API Backendpages/api/Route Handlers + Server Actions
Loading statesManualloading.tsx automático
Error handlingManualerror.tsx automático
💡 Qual usar?
Para novos projetos: sempre App Router. Para projetos existentes com Pages Router: migre gradualmente — os dois podem coexistir no mesmo projeto durante a transição.

Server Components — o coração do App Router

React Server Components (RSC) executam apenas no servidor. Eles não têm JavaScript no bundle do cliente, podem fazer async/await diretamente, acessar banco de dados, ler arquivos e variáveis de ambiente — sem expor nada ao browser.

Server Component

Padrão no App Router

Zero JS no cliente. Acesso direto ao banco/FS. Não pode usar hooks, eventos ou APIs do browser. Ideal para layouts, páginas, listas de dados.

"use client"

Client Component

Tem interatividade (hooks, eventos, useState). Recebe props de Server Components. Usar só quando necessário para manter bundle pequeno.

server component vs client component — quando usar cada um
// ✅ Server Component (padrão — sem diretiva)
// Busca dados direto do banco, sem fetch
import { db } from '@/lib/db';

async function ListaPosts() {
  // async/await direto — sem useEffect, sem loading state
  const posts = await db.post.findMany({
    orderBy: { criadoEm: 'desc' },
    take: 10,
  });

  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>
          <h3>{post.titulo}</h3>
          <p>{post.resumo}</p>
          {/* BotaoLike precisa de interatividade — é Client Component */}
          <BotaoLike postId={post.id} likes={post.likes} />
        </li>
      ))}
    </ul>
  );
}

// ✅ Client Component — apenas para o que precisa de interatividade
'use client';
import { useState } from 'react';

function BotaoLike({ postId, likes }: { postId: string; likes: number }) {
  const [count, setCount] = useState(likes);
  const [curtido, setCurtido] = useState(false);

  async function handleLike() {
    setCount(c => curtido ? c - 1 : c + 1);
    setCurtido(c => !c);
    await fetch(`/api/posts/${postId}/like`, { method: 'POST' });
  }

  return (
    <button onClick={handleLike}>
      {curtido ? '❤️' : '🤍'} {count}
    </button>
  );
}

Estratégias de Renderização

No App Router, você não declara explicitamente SSG ou SSR — o Next.js decide automaticamente com base em como você busca os dados. Você controla via opções de fetch e revalidate.

Static (SSG equivalente) — padrão
Quando: nenhuma configuração especial
Página gerada em build time. HTML estático, máxima performance, ideal para conteúdo que não muda frequentemente (blog, docs, landing pages).
Dynamic (SSR equivalente)
Quando: fetch(url, { cache: 'no-store' }) ou uso de cookies() / headers()
Página gerada a cada requisição. Dados sempre frescos. Ideal para dashboards autenticados, carrinho de compras, conteúdo personalizado.
Revalidation (ISR equivalente)
Quando: fetch(url, { next: { revalidate: 60 } })
Página estática que é regenerada após N segundos. Equilíbrio perfeito entre performance e dados frescos. Ideal para e-commerce, notícias.
as 3 estratégias no código
// 1. STATIC — gerado em build, nunca revalida
async function PaginaEstatica() {
  const dados = await fetch('https://api.exemplo.com/config');
  // sem opções = static por padrão
  return <div>...</div>;
}

// 2. DYNAMIC — novo a cada request
async function Dashboard() {
  const dados = await fetch('https://api.exemplo.com/user', {
    cache: 'no-store'  // força SSR
  });
  return <div>...</div>;
}

// 3. ISR — regenera a cada 60 segundos
async function PaginaProduto({ params }: { params: { id: string } }) {
  const produto = await fetch(`https://api.exemplo.com/produtos/${params.id}`, {
    next: { revalidate: 60 }  // revalida a cada 60s
  });
  return <div>...</div>;
}

// Configuração de revalidate no nível da rota (aplica a toda a página)
export const revalidate = 3600; // 1 hora

Layouts Aninhados & Rotas Dinâmicas

O App Router usa o sistema de arquivos como roteador. Cada pasta cria uma rota, e arquivos especiais (layout.tsx, page.tsx, loading.tsx, error.tsx) têm funções específicas.

app/layout.tsx — layout raiz (obrigatório)

import './globals.css';

export default function RootLayout({ children }: { children: React.ReactNode }) {

rotas dinâmicas e generateStaticParams
// app/blog/[slug]/page.tsx
import { notFound } from 'next/navigation';

interface Props { params: { slug: string } }

// Gerar todos os slugs em build time (para SSG)
export async function generateStaticParams() {
  const posts = await fetch('https://api.exemplo.com/posts').then(r => r.json());
  return posts.map((post: { slug: string }) => ({ slug: post.slug }));
}

// Metadata dinâmica por página
export async function generateMetadata({ params }: Props) {
  const post = await getPost(params.slug);
  if (!post) return {};
  return {
    title: post.titulo,
    description: post.resumo,
    openGraph: {
      images: [{ url: post.imagemCapa }],
    },
  };
}

async function getPost(slug: string) {
  const res = await fetch(`https://api.exemplo.com/posts/${slug}`, {
    next: { revalidate: 3600 }
  });
  if (!res.ok) return null;
  return res.json();
}

export default async function PaginaPost({ params }: Props) {
  const post = await getPost(params.slug);
  if (!post) notFound(); // Renderiza app/not-found.tsx automaticamente

  return (
    <article>
      <h1>{post.titulo}</h1>
      <time>{new Date(post.criadoEm).toLocaleDateString('pt-BR')}</time>
      <div dangerouslySetInnerHTML={{ __html: post.conteudoHTML }} />
    </article>
  );
}

Server Actions — backend sem API Routes

Server Actions são funções assíncronas que rodam no servidor, mas podem ser chamadas diretamente de componentes client — sem precisar criar uma API Route. Usadas principalmente em formulários.

server action — formulário sem uma linha de API
// app/contato/page.tsx
import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';

// Server Action — roda no servidor, chamada do form
async function enviarContato(formData: FormData) {
  'use server'; // diretiva obrigatória

  const nome = formData.get('nome') as string;
  const email = formData.get('email') as string;
  const mensagem = formData.get('mensagem') as string;

  // Validação
  if (!nome || !email || !mensagem) {
    throw new Error('Todos os campos são obrigatórios');
  }

  // Salvar no banco diretamente
  await db.contato.create({
    data: { nome, email, mensagem, criadoEm: new Date() }
  });

  // Enviar e-mail de confirmação
  await sendEmail({ to: email, subject: 'Recebemos seu contato!', ... });

  // Revalidar cache e redirecionar
  revalidatePath('/contato');
  redirect('/contato/obrigado');
}

export default function PaginaContato() {
  return (
    // action recebe a Server Action diretamente — sem fetch, sem useState
    <form action={enviarContato}>
      <input name="nome" placeholder="Seu nome" required />
      <input name="email" type="email" placeholder="Seu e-mail" required />
      <textarea name="mensagem" placeholder="Mensagem" required />
      <button type="submit">Enviar</button>
    </form>
  );
}

Route Handlers — API REST no Next.js

Para construir APIs REST (para mobile, integrações externas ou quando Server Actions não são suficientes), o App Router usa Route Handlers em arquivos route.ts.

app/api/posts/route.ts — CRUD completo
import { NextRequest, NextResponse } from 'next/server';
import { z } from 'zod';

// GET /api/posts — listar posts
export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url);
  const pagina = Number(searchParams.get('pagina') ?? '1');
  const limite = Number(searchParams.get('limite') ?? '10');

  const posts = await db.post.findMany({
    skip: (pagina - 1) * limite,
    take: limite,
    orderBy: { criadoEm: 'desc' },
  });

  return NextResponse.json({ posts, pagina, limite });
}

// POST /api/posts — criar post
const PostSchema = z.object({
  titulo: z.string().min(3).max(200),
  conteudo: z.string().min(10),
  tags: z.array(z.string()).optional(),
});

export async function POST(request: NextRequest) {
  // Verificar autenticação
  const session = await getServerSession();
  if (!session) return NextResponse.json({ error: 'Não autorizado' }, { status: 401 });

  const body = await request.json();
  const validado = PostSchema.safeParse(body);

  if (!validado.success) {
    return NextResponse.json({ error: validado.error.format() }, { status: 400 });
  }

  const post = await db.post.create({
    data: { ...validado.data, autorId: session.user.id }
  });

  return NextResponse.json(post, { status: 201 });
}

// app/api/posts/[id]/route.ts
export async function DELETE(request: NextRequest, { params }: { params: { id: string } }) {
  const session = await getServerSession();
  if (!session) return NextResponse.json({ error: 'Não autorizado' }, { status: 401 });

  await db.post.delete({ where: { id: params.id, autorId: session.user.id } });
  return new NextResponse(null, { status: 204 });
}

Erros comuns no Next.js

⚠️ Usar "use client" em tudo — perde os benefícios do RSC

O erro mais comum de devs que vêm do React puro é adicionar 'use client' em todo arquivo porque estão acostumados com hooks. Isso anula os benefícios dos Server Components.

✅ Regra: só adicione 'use client' quando o componente usa useState, useEffect, event handlers ou APIs do browser. Mantenha a lógica de dados em Server Components.

⚠️ Variáveis de ambiente expostas ao cliente

No Next.js, variáveis de ambiente sem o prefixo NEXT_PUBLIC_ ficam apenas no servidor. Tentar acessar process.env.DATABASE_URL num Client Component retornará undefined.

✅ Solução: use NEXT_PUBLIC_ apenas para variáveis que realmente precisam estar no browser (ex: NEXT_PUBLIC_API_URL). Nunca prefixe segredos — eles ficariam expostos no HTML.

⚠️ Instanciar Prisma sem singleton — too many connections

Em dev com hot reload, o Next.js pode criar múltiplas instâncias do PrismaClient, estourando o limite de conexões do banco.

// src/lib/prisma.ts — padrão singleton
import { PrismaClient } from '@prisma/client';
const globalForPrisma = globalThis as unknown as { prisma: PrismaClient };
export const prisma = globalForPrisma.prisma ?? new PrismaClient();
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;
