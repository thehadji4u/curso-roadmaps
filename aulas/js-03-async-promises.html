Async: Event Loop, Promises & Fetch

⏱ ~90 min de leitura 🎯 Intermediário 📁 Nenhum setup necessário

// POR QUE ISSO IMPORTA?

O segredo por trás de toda aplicação moderna

Toda aplicação web que você usa depende de código assíncrono. Quando o Twitter carrega seu feed, quando o Google Maps busca rotas, quando o Spotify faz streaming de músicas — tudo isso acontece sem travar a interface. Isso só é possível porque JavaScript tem um modelo de concorrência muito específico: o Event Loop.

Sem entender assincronicidade, você não consegue escrever código que consome APIs externas, que lida com banco de dados, que faz upload de arquivos ou que reage a eventos do usuário de forma eficiente. É literalmente a base de qualquer backend Node.js e de qualquer frontend que faz requisições HTTP.

🔥 Por que um engenheiro sênior se importa

Um senior JS dev pensa em termos de Promise chains, async/await, paralelismo com Promise.all() e gerenciamento cuidadoso de erros. Código síncrono bloqueante em um servidor Node.js é um bug de performance gravíssimo — pode derrubar o servidor inteiro sob carga.

// ANALOGIA DO MUNDO REAL

O Garçom do Restaurante

🍽️ Analogia
Imagine um restaurante com um único garçom. Se ele fosse síncrono, funcionaria assim: pega o pedido da mesa 1, vai à cozinha, fica parado esperando o prato ficar pronto (bloqueando), volta, entrega, aí pega o pedido da mesa 2. O restaurante inteiro ficaria travado esperando cada prato.

Na vida real, o garçom é assíncrono: pega o pedido da mesa 1, entrega o pedido à cozinha e volta para atender a mesa 2. Quando a cozinha avisa que o prato está pronto (um callback), ele vai buscar e entrega. Um único garçom atende todo o restaurante com eficiência máxima.

O garçom é o Event Loop. A cozinha são as Web APIs (timers, fetch, I/O). Os pedidos na fila são a Callback Queue.

📋 Pegar pedido
📝 Executar código síncrono
👨‍🍳 Cozinha processando
🌐 Web APIs (fetch, timer)
🔔 Cozinha avisa "pronto!"
✅ Promise resolve / callback
📋 Fila de pedidos prontos

O Event Loop em detalhe

JavaScript é single-threaded — executa uma coisa de cada vez em uma única Call Stack. Mas o ambiente (browser ou Node.js) tem Web APIs que rodam em threads separadas do OS. O Event Loop é o mecanismo que coordena tudo isso.

🗃 Call Stack
main()
fetchUser()
JSON.parse()
Execução síncrona
LIFO — last in, first out
🌐 Web APIs
setTimeout(3s)
fetch(url)
Processamento externo
Não bloqueia a thread
📬 Queues
Microtask: resolve(data)
Macrotask: callback()
Microtasks têm prioridade
sobre macrotasks
💡 Microtasks vs Macrotasks

Microtask Queue (alta prioridade): Promises (.then()), queueMicrotask(), MutationObserver.
Macrotask Queue (normal): setTimeout, setInterval, eventos do DOM, I/O callbacks. O Event Loop sempre esvazia toda a Microtask Queue antes de pegar a próxima Macrotask.

Diagrama — Fluxo do Event Loop

A ordem de execução na prática

javascript — ordem de execução
console.log('1 — síncrono');

setTimeout(() => console.log('4 — macrotask (setTimeout)'), 0);

Promise.resolve()
  .then(() => console.log('3 — microtask (Promise)'));

console.log('2 — síncrono');

// Output:
// 1 — síncrono
// 2 — síncrono
// 3 — microtask (Promise)    ← microtask antes do setTimeout!
// 4 — macrotask (setTimeout)
⚠️ Atenção ao "zero delay"

setTimeout(fn, 0) não significa "execute agora". Significa "execute assim que a Call Stack estiver vazia E todas as microtasks tiverem sido processadas". O delay mínimo real no browser é ~4ms.

// PROMISES

Promises: valores do futuro

Uma Promise é um objeto que representa a eventual conclusão (ou falha) de uma operação assíncrona. Em vez de passar um callback para dentro de uma função, você recebe um objeto Promise e registra o que fazer quando ele resolver.

⏳
Pending
Estado inicial. A operação ainda está em andamento.
✅
Fulfilled
A operação completou com sucesso. Tem um valor.
❌
Rejected
A operação falhou. Tem uma razão (Error).

Criando e consumindo Promises

javascript — promises básicas
// ─── Criando uma Promise ───────────────────────────────────────────────
const minhaPromise = new Promise((resolve, reject) => {
  const sucesso = Math.random() > 0.5;
  
  setTimeout(() => {
    if (sucesso) {
      resolve({ nome: 'Maria', idade: 30 });  // → fulfilled
    } else {
      reject(new Error('Algo deu errado'));     // → rejected
    }
  }, 1000);
});

// ─── Consumindo com .then() / .catch() / .finally() ───────────────────
minhaPromise
  .then(usuario => {
    console.log('Usuário:', usuario.nome);   // executa se fulfilled
    return usuario.nome.toUpperCase();        // pode retornar outro valor
  })
  .then(nomeUpper => {
    console.log('Maiúsculo:', nomeUpper);     // chaining!
  })
  .catch(erro => {
    console.error('Erro:', erro.message);    // executa se rejected
  })
  .finally(() => {
    console.log('Sempre executa — limpar loading spinner');
  });

Combinadores de Promises — poder real

javascript — Promise.all, allSettled, race, any
const p1 = fetch('/api/users');
const p2 = fetch('/api/products');
const p3 = fetch('/api/orders');

// ─── Promise.all ── falha fast: se UMA rejeitar, todas falham ─────────
const [users, products, orders] = await Promise.all([p1, p2, p3]);
// Paralelismo real! 3 requisições simultâneas, não sequenciais.

// ─── Promise.allSettled ── aguarda TODAS, independente do resultado ───
const results = await Promise.allSettled([p1, p2, p3]);
results.forEach(result => {
  if (result.status === 'fulfilled') {
    console.log('✅', result.value);
  } else {
    console.log('❌', result.reason);
  }
});

// ─── Promise.race ── vence a primeira que resolver ou rejeitar ────────
const winner = await Promise.race([
  fetch('/api/server-1'),
  fetch('/api/server-2'),  // fallback mais rápido ganha
]);

// ─── Promise.any ── vence a primeira que RESOLVER (ignora rejeições) ──
const firstSuccess = await Promise.any([
  fetch('/api/primary'),
  fetch('/api/backup-1'),
  fetch('/api/backup-2'),
]);
// ASYNC/AWAIT

Async/Await: Promises com cara de código síncrono

async/await é açúcar sintático sobre Promises — por baixo dos panos, é exatamente a mesma coisa. A vantagem é legibilidade: o código parece síncrono mas é completamente assíncrono.

❌ Com .then() — difícil de ler
function buscarDados() {
  return fetch('/api/users')
    .then(res => res.json())
    .then(users => {
      return fetch(`/api/orders/${users[0].id}`)
        .then(res => res.json())
        .then(orders => {
          return { users, orders };
        });
    })
    .catch(err => {
      console.error(err);
    });
}
✅ Com async/await — limpo!
async function buscarDados() {
  try {
    const res1 = await fetch('/api/users');
    const users = await res1.json();

    const res2 = await fetch(
      `/api/orders/${users[0].id}`
    );
    const orders = await res2.json();

    return { users, orders };
  } catch (err) {
    console.error(err);
  }
}
⚠️ Armadilha: await sequencial vs paralelo

await dentro de um loop for executa cada await sequencialmente — o próximo só começa quando o anterior terminar. Para paralelismo, use Promise.all().

javascript — sequencial vs paralelo
// ❌ Sequencial — demora 3s (1s + 1s + 1s)
async function lento() {
  const a = await buscar(1);  // espera 1s
  const b = await buscar(2);  // espera 1s
  const c = await buscar(3);  // espera 1s
  return [a, b, c];
}

// ✅ Paralelo — demora ~1s (todas simultâneas)
async function rapido() {
  const [a, b, c] = await Promise.all([
    buscar(1),
    buscar(2),
    buscar(3),
  ]);
  return [a, b, c];
}

// ✅ Inicializar promises antes, await depois
async function tambemRapido() {
  const pA = buscar(1);  // inicia sem await
  const pB = buscar(2);  // inicia sem await
  const pC = buscar(3);  // inicia sem await
  return [await pA, await pB, await pC];  // aguarda todas
}
// FETCH API

Fetch API: consumindo REST APIs

A Fetch API é a forma moderna de fazer requisições HTTP no JavaScript. Ela substitui o antigo XMLHttpRequest com uma interface baseada em Promises muito mais limpa.

⚠️ fetch() não rejeita em erros HTTP 4xx/5xx!

Isso é a armadilha mais comum. fetch() só rejeita em falhas de rede (sem conexão, CORS bloqueado). Para erros HTTP como 404 ou 500, você precisa checar response.ok ou response.status manualmente.

javascript — fetch completo e robusto
// ─── GET request ──────────────────────────────────────────────────────
async function buscarUsuarios() {
  const response = await fetch('https://jsonplaceholder.typicode.com/users');
  
  if (!response.ok) {
    throw new Error(`HTTP error! status: ${response.status}`);
  }
  
  const data = await response.json();
  return data;
}

// ─── POST request com body ────────────────────────────────────────────
async function criarPost(titulo, corpo) {
  const response = await fetch('https://jsonplaceholder.typicode.com/posts', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${TOKEN}`,
    },
    body: JSON.stringify({
      title: titulo,
      body: corpo,
      userId: 1,
    }),
  });

  if (!response.ok) {
    const errorData = await response.json().catch(() => null);
    throw new Error(errorData?.message ?? `Erro ${response.status}`);
  }

  return response.json();
}

// ─── Padrão wrapper reutilizável ──────────────────────────────────────
async function api(endpoint, options = {}) {
  const BASE_URL = 'https://api.meusite.com/v1';
  
  const response = await fetch(`${BASE_URL}${endpoint}`, {
    headers: {
      'Content-Type': 'application/json',
      ...options.headers,
    },
    ...options,
  });

  if (!response.ok) {
    throw new Error(`API Error: ${response.status} ${response.statusText}`);
  }

  // Lidar com responses sem body (204 No Content)
  const contentType = response.headers.get('content-type');
  if (contentType?.includes('application/json')) {
    return response.json();
  }
  return null;
}

// Uso:
const user = await api('/users/42');
await api('/users/42', { method: 'DELETE' });

AbortController — cancelando requisições

javascript — abortar fetch em andamento
// Útil para: timeout manual, cancelar ao desmontar componente React,
// cancelar busca quando usuário digita nova query

async function buscarComTimeout(url, timeoutMs = 5000) {
  const controller = new AbortController();
  
  // Cancela automaticamente após o timeout
  const timeoutId = setTimeout(
    () => controller.abort(),
    timeoutMs
  );

  try {
    const response = await fetch(url, {
      signal: controller.signal,
    });
    clearTimeout(timeoutId);
    return response.json();
  } catch (err) {
    if (err.name === 'AbortError') {
      throw new Error('Requisição cancelada por timeout');
    }
    throw err;
  }
}

// Cancelando manualmente (ex: React useEffect cleanup)
let controller;

function buscar(query) {
  if (controller) controller.abort();  // cancela busca anterior
  controller = new AbortController();
  
  return fetch(`/api/search?q=${query}`, {
    signal: controller.signal,
  });
}

Promise combinators: qual usar quando?

Combinador Quando usar Falha se Resultado
Promise.all() Todas devem ter sucesso Qualquer uma rejeita Array dos resultados
Promise.allSettled() Quer resultado de todas Nunca rejeita Array com status + valor/erro
Promise.race() Timeout ou múltiplas fontes A mais rápida rejeitar Valor da mais rápida
Promise.any() Fallback, múltiplos servidores Todas rejeitam (AggregateError) Primeiro que resolver

// Sem verificar response.ok — bug silencioso!

console.log('status:', res1.status);   // 404
console.log('ok:', res1.ok);           // false — mas não jogou erro!

// Com verificação correta
async function buscar(url) {
  const res = await fetch(url);
  if (!res.ok) throw new Error(`Erro HTTP: ${res.status}`);
  return res.json();

try {
  const post = await buscar('https://jsonplaceholder.typicode.com/posts/9999');
} catch (err) {
  console.error('Capturado:', err.message);  // Erro HTTP: 404
}
5
Crie seu mini cliente de API
Um wrapper reutilizável que você pode levar para qualquer projeto:
const JSONPlaceholder = {
  BASE: 'https://jsonplaceholder.typicode.com',
  
  async get(path) {
    const res = await fetch(this.BASE + path);
    if (!res.ok) throw new Error(`${res.status} ${res.statusText}`);
    return res.json();
  },

  async post(path, body) {
    const res = await fetch(this.BASE + path, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(body),
    });
    if (!res.ok) throw new Error(`${res.status} ${res.statusText}`);
    return res.json();
  }
};

// Uso
const user = await JSONPlaceholder.get('/users/1');
console.log(user.name);  // Leanne Graham

const newPost = await JSONPlaceholder.post('/posts', {
  title: 'Testando async/await',
  body: 'Funcionou!',
  userId: 1,
});
console.log(newPost.id);  // 101

Dashboard de dados assíncrono

Exemplo real: uma função que busca dados de múltiplas fontes com tratamento de erro robusto, loading state e cancelamento.

javascript — dashboard.js — exemplo completo e comentado
// ─── dashboard.js — Exemplo completo de async/await ──────────────────

const API = 'https://jsonplaceholder.typicode.com';

// ─── 1. Utilitário de fetch robusto ───────────────────────────────────
async function apiFetch(path, signal) {
  const response = await fetch(`${API}${path}`, { signal });
  
  if (!response.ok) {
    // Tenta parsear mensagem de erro da API, senão usa status
    const errorBody = await response.json().catch(() => null);
    throw new Error(
      errorBody?.message ?? `HTTP ${response.status}: ${path}`
    );
  }
  
  return response.json();
}

// ─── 2. Funções de domínio ────────────────────────────────────────────
async function buscarUsuario(id, signal) {
  return apiFetch(`/users/${id}`, signal);
}

async function buscarPostsDoUsuario(userId, signal) {
  return apiFetch(`/posts?userId=${userId}`, signal);
}

async function buscarTodosOsAlbuns(signal) {
  return apiFetch('/albums', signal);
}

// ─── 3. Função principal — paralelismo inteligente ────────────────────
async function carregarDashboard(userId) {
  // AbortController para cancelar tudo de uma vez se necessário
  const controller = new AbortController();
  const { signal } = controller;

  // Timeout de 10 segundos
  const timeoutId = setTimeout(() => controller.abort(), 10_000);

  try {
    // Busca usuário primeiro (precisamos do ID para os posts)
    const usuario = await buscarUsuario(userId, signal);
    
    // Agora busca posts e albums em paralelo!
    const [posts, albums] = await Promise.all([
      buscarPostsDoUsuario(usuario.id, signal),
      buscarTodosOsAlbuns(signal),
    ]);

    // Filtra apenas albums desse usuário
    const albumsDoUsuario = albums.filter(a => a.userId === usuario.id);

    clearTimeout(timeoutId);
    
    return {
      usuario,
      posts: posts.slice(0, 5),     // top 5 posts
      albums: albumsDoUsuario,
      carregadoEm: new Date().toISOString(),
    };

  } catch (err) {
    clearTimeout(timeoutId);
    
    if (err.name === 'AbortError') {
      throw new Error('Dashboard timeout: operação cancelada após 10s');
    }
    throw err; // re-throw para quem chamou lidar
  }
}

// ─── 4. Executando com tratamento de erro no ponto de chamada ─────────
async function main() {
  console.log('🔄 Carregando dashboard...');

  try {
    const dashboard = await carregarDashboard(1);
    
    console.log(`✅ Dashboard carregado!`);
    console.log(`👤 ${dashboard.usuario.name} (${dashboard.usuario.email})`);
    console.log(`📝 ${dashboard.posts.length} posts recentes`);
    console.log(`📷 ${dashboard.albums.length} albums`);
    console.log(`⏰ ${dashboard.carregadoEm}`);
    
  } catch (err) {
    console.error('❌ Falha ao carregar dashboard:', err.message);
    // Na UI: mostrar mensagem de erro amigável, botão "Tentar novamente"
  }
}

main();

⚠️ Erros que todo dev já cometeu

🪤 Esquecer de tratar erros HTTP do fetch()

fetch() só rejeita em falhas de rede. Um 404 ou 500 chega como resposta "normal". Sempre cheque response.ok ou response.status e jogue um erro manualmente.

🪤 await sequencial em loop — o "await em for of"

for (const id of ids) { const data = await fetch(...) } executa cada fetch um de cada vez. Use Promise.all(ids.map(id => fetch(...))) para paralelismo real. A diferença pode ser 10x no tempo de execução.

🪤 .forEach() com async — callbacks perdidas

array.forEach(async (item) => { ... }) não aguarda as promises! forEach ignora o retorno de cada iteração. Use for...of ou Promise.all(array.map(async ...)).

🪤 Promise não tratada — UnhandledPromiseRejection

Uma Promise rejeitada sem .catch() ou try/catch gera um warning (ou crash em Node.js). Sempre trate rejeições, especialmente em top-level async code.

🪤 Race condition — múltiplas requisições sem controle

Em um campo de busca, se o usuário digitar rápido e você fizer fetch a cada keypress, a última digitada pode chegar antes de uma intermediária, exibindo resultados errados. Sempre use AbortController para cancelar a requisição anterior.

🪤 async function em event handler sem try/catch

Um erro dentro de button.addEventListener('click', async () => { ... }) sem try/catch vai "sumir" silenciosamente. Sempre envolva em try/catch ou adicione .catch() na chamada.
