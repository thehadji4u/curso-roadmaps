01 · Motivação

Por que ir além dos workflows básicos?

Nas aulas anteriores você criou workflows, integrou APIs e automatizou processos reais. Mas conforme as automações crescem, surgem desafios de manutenção, reuso e escala:

🔁

Lógica repetida

A mesma sequência de nodes aparece em 10 workflows diferentes. Quando muda, você edita 10 vezes.

☁️

Dependência do cloud

n8n.cloud tem limites de execuções e dados passam por servidores de terceiros — problema para dados sensíveis.

🔌

Integração inexistente

A API que você precisa não tem node no n8n. Ou as opções disponíveis são limitadas demais.

💡 Por que um engenheiro sênior usa isso hoje?
Sub-workflows transformam n8n em uma plataforma de automação escalável — workflows reutilizáveis como funções. Self-hosting dá controle total sobre dados e elimina custos recorrentes. Custom nodes permitem integrar qualquer sistema, tornando o n8n extensível como código.
02 · Analogia do Mundo Real

Funções, Servidor Próprio e Plug-ins

🏗️ Analogia
Pense no n8n como uma empresa de construção. Os workflows básicos são como construir cada casa do zero — eficiente no início, mas caótico quando escala. Os sub-workflows são como pré-fabricados: você cria uma vez a estrutura de "banheiro" e reutiliza em todos os projetos. O self-hosting é ter sua própria sede em vez de alugar espaço no escritório compartilhado — mais controle, menos custo mensal. E os custom nodes são máquinas especializadas que você fabrica quando o catálogo do fornecedor não tem o que precisa.

03 · Sub-workflows

Modularizando com Sub-workflows

Um sub-workflow é um workflow n8n que pode ser chamado por outros workflows usando o Execute Workflow node. É o equivalente a funções em programação — encapsula lógica, recebe parâmetros e retorna resultados.

Como criar um Sub-workflow

1
Crie o workflow filho com trigger correto
No novo workflow, use o node "When called by another workflow" (ou "Execute Workflow Trigger" em versões mais antigas) como ponto de entrada. Configure os campos de entrada esperados.
🔥 Configuração do trigger
Em When called by another workflow → defina os campos de entrada: Ex: userId (Number), message (String). Esses campos serão validados automaticamente.
2
Construa a lógica do sub-workflow
Adicione os nodes necessários. No final, use um node Return (ou simplesmente deixe o último node retornar os dados) para devolver resultado ao workflow pai.
// Exemplo: Último node do sub-workflow (Code node)
// Retorna resultado formatado para o workflow pai
return [{
  json: {
    success: true,
    notificationId: $json.id,
    sentAt: new Date().toISOString(),
    recipient: $('When called').item.json.userId
  }
}]
3
Chame o sub-workflow com Execute Workflow
No workflow pai, adicione o node Execute Workflow. Configure:
CampoValor
SourceDatabase (seleciona workflow existente)
WorkflowSelecione o sub-workflow pelo nome/ID
Wait for completion✅ Ativado (para aguardar resultado)
Input dataCampos que você quer passar para o filho
4
Acesse o retorno no workflow pai
O resultado do sub-workflow fica disponível como $json no próximo node do workflow pai
// No workflow pai, após o Execute Workflow node:
// Acessar retorno do sub-workflow
const result = $json  // dados retornados pelo sub-workflow

// Se retornou { success: true, notificationId: "abc" }
console.log(result.success)        // true
console.log(result.notificationId) // "abc"

// Passar para próximo step
return [{ json: { ...result, processedBy: 'main-workflow' } }]

Padrões de uso de Sub-workflows

🔔 Notificações Centralizadas
Padrão: Fan-out

Um único sub-workflow cuida de Email + Slack + WhatsApp. Qualquer workflow que precisa notificar chama este sub-workflow com { channel, message, recipient }.

🔄 Transformação de Dados
Padrão: Shared utility

Lógica complexa de normalização de dados encapsulada. Todos os workflows que precisam converter formatos chamam o mesmo sub-workflow.

🔐 Autenticação/Tokens
Padrão: Token manager

Sub-workflow que verifica se o token OAuth ainda é válido, renova se necessário e retorna o token atualizado para o workflow pai.

📊 Relatórios Automáticos
Padrão: Report builder

Sub-workflow que recebe dados brutos, monta um relatório formatado (HTML, PDF ou planilha) e retorna a URL do arquivo gerado.

04 · Self-hosting

Rodando n8n no Seu Próprio Servidor

Self-hosting significa instalar e manter o n8n na sua própria infraestrutura — um VPS, servidor dedicado ou mesmo um Raspberry Pi. Você tem controle total sobre dados, sem limites de execuções e sem custo mensal crescente.

Implementação Completa

Vamos criar um custom node que consulta a ViaCEP API (busca endereço por CEP brasileiro) — um exemplo real e útil:

nodes/ViaCep/ViaCep.node.ts
import {
  IExecuteFunctions,
  INodeExecutionData,
  INodeType,
  INodeTypeDescription,
  NodeOperationError,
} from 'n8n-workflow';

export class ViaCep implements INodeType {
  // ── Metadados do node (aparece na UI do n8n) ──────────────────
  description: INodeTypeDescription = {
    displayName: 'ViaCEP',
    name: 'viaCep',
    icon: 'fa:map-marker-alt',
    group: ['transform'],
    version: 1,
    description: 'Busca endereço a partir de um CEP brasileiro via ViaCEP API',
    defaults: { name: 'ViaCEP' },
    inputs: ['main'],
    outputs: ['main'],

    // ── Campos que aparecem na UI do node ─────────────────────
    properties: [
      {
        displayName: 'CEP',
        name: 'cep',
        type: 'string',
        default: '',
        placeholder: '01310-100',
        description: 'CEP a ser consultado (com ou sem hífen)',
        required: true,
      },
      {
        displayName: 'Usar CEP do item de entrada',
        name: 'useCepFromInput',
        type: 'boolean',
        default: false,
        description: 'Se ativado, usa o campo "cep" do item de entrada ao invés do valor fixo acima',
      },
      {
        displayName: 'Campo CEP no item',
        name: 'cepField',
        type: 'string',
        default: 'cep',
        description: 'Nome do campo no item de entrada que contém o CEP',
        displayOptions: {
          show: { useCepFromInput: [true] },
        },
      },
    ],
  };

  // ── Lógica de execução ────────────────────────────────────────
  async execute(this: IExecuteFunctions): Promise {
    const items = this.getInputData();
    const returnData: INodeExecutionData[] = [];

    for (let i = 0; i < items.length; i++) {
      try {
        // Pegar o CEP — do parâmetro fixo ou do item de entrada
        let cep: string;
        const useCepFromInput = this.getNodeParameter('useCepFromInput', i) as boolean;

        if (useCepFromInput) {
          const cepField = this.getNodeParameter('cepField', i) as string;
          cep = items[i].json[cepField] as string;
        } else {
          cep = this.getNodeParameter('cep', i) as string;
        }

        // Limpar CEP (remover hífen e espaços)
        const cepClean = cep.replace(/\D/g, '');

        if (cepClean.length !== 8) {
          throw new NodeOperationError(this.getNode(), `CEP inválido: ${cep}`, { itemIndex: i });
        }

        // Fazer chamada à API ViaCEP
        const response = await this.helpers.request({
          method: 'GET',
          url: `https://viacep.com.br/ws/${cepClean}/json/`,
          json: true,
        });

        // ViaCEP retorna { erro: true } quando CEP não existe
        if (response.erro) {
          throw new NodeOperationError(this.getNode(), `CEP não encontrado: ${cep}`, { itemIndex: i });
        }

        // Retornar dados do endereço junto com o item original
        returnData.push({
          json: {
            ...items[i].json,   // mantém dados do item original
            endereco: {
              cep: response.cep,
              logradouro: response.logradouro,
              complemento: response.complemento,
              bairro: response.bairro,
              cidade: response.localidade,
              estado: response.uf,
              ibge: response.ibge,
              ddd: response.ddd,
            },
          },
          pairedItem: { item: i },
        });

      } catch (error) {
        // Continue on error = não parar o workflow se um item falhar
        if (this.continueOnFail()) {
          returnData.push({
            json: { ...items[i].json, error: (error as Error).message },
            pairedItem: { item: i },
          });
          continue;
        }
        throw error;
      }
    }

    return [returnData];
  }
}

06 · Hands-on

Projeto Prático: Sistema de Notificações Modulares

Construa um sistema onde qualquer workflow pode enviar notificações via um único sub-workflow centralizado.

1
Crie o sub-workflow "Enviar Notificação"
Novo workflow com trigger "When called by another workflow" e campos: canal (string: email/telegram/slack), mensagem (string), destinatario (string)
// Code node no sub-workflow — roteia para o canal correto
const { canal, mensagem, destinatario } = $('When called').item.json

// Preparar payload para o próximo node (IF/Switch)
return [{
  json: {
    canal,
    mensagem,
    destinatario,
    timestamp: new Date().toISOString()
  }
}]

Adicione um Switch node para rotear por canal → cada saída conecta ao node correto (Send Email, Telegram, etc.)

2
Crie o workflow principal
Webhook trigger → processa dados → chama o sub-workflow de notificação
// No Execute Workflow node:
// Source: Database
// Workflow: "Enviar Notificação" (selecionar)
// Input Fields:
{
  "canal": "telegram",
  "mensagem": "Novo pedido recebido: #{{ $json.orderId }}",
  "destinatario": "{{ $json.userId }}"
}
3
Teste com múltiplos canais
Chame o mesmo sub-workflow duas vezes com canais diferentes para testar o fan-out
✅ Critério de sucesso
O mesmo sub-workflow deve funcionar tanto para email quanto para telegram, sem duplicar lógica. Modificar o sub-workflow afeta todos os workflows que o chamam automaticamente.
07 · Armadilhas Comuns

⚠️ Erros que Todo Mundo Comete

⚠️ Sub-workflow sem "When called by another workflow"

Se o trigger do sub-workflow for um Webhook ou Schedule, o Execute Workflow node não consegue chamá-lo. Use sempre o trigger correto: "When called by another workflow". Caso contrário, o Execute Workflow retornará erro "Workflow trigger not found".

⚠️ n8n self-hosted sem HTTPS

Webhooks externos (Telegram, GitHub, etc.) exigem HTTPS. Sem SSL, os webhooks não funcionam em produção. Use sempre Traefik ou Nginx com Let's Encrypt. Nunca exponha o n8n na porta 5678 diretamente para a internet sem HTTPS e autenticação.

⚠️ Custom node sem pairedItem

Ao criar custom nodes, sempre inclua pairedItem: {'{'} item: i {'}'} no objeto retornado. Sem isso, o n8n não consegue rastrear qual item de entrada gerou qual item de saída, quebrando operações de merge e causando erros confusos.

⚠️ Senhas no docker-compose.yml

Nunca coloque senhas diretamente no docker-compose.yml — use sempre variáveis de ambiente via arquivo .env e adicione .env ao .gitignore. Repositórios públicos com senhas são um vetor de ataque clássico.
