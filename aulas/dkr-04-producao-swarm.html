Por que isso importa em produção?

Nas aulas anteriores, você aprendeu a criar imagens Docker, escrever Dockerfiles e orquestrar múltiplos containers com Compose. Mas quando chega a hora de levar para produção, surgem perguntas reais:

📦

Onde armazenar imagens?

Docker Hub, GitHub Container Registry, Amazon ECR — como escolher e usar cada um.

🔄

Como automatizar o deploy?

Cada push no código deve automaticamente construir, testar e publicar a imagem.

🌐

Como escalar sem Kubernetes?

Docker Swarm oferece orquestração nativa, simples e poderosa para equipes menores.

💡 Por que um engenheiro sênior usa isso hoje?
Um pipeline CI/CD com Docker elimina o clássico "funciona na minha máquina". Docker Swarm é a escolha de times que precisam de orquestração sem a complexidade do Kubernetes — startups, projetos internos e VPS de baixo custo.

A Fábrica, o Armazém e a Frota de Caminhões

🏭 Analogia
Imagine uma fábrica de produtos (seu código) que a cada nova versão sai da linha de montagem como uma caixa lacrada (imagem Docker). Essa caixa vai para um armazém central (Registry) onde fica catalogada com versão e etiqueta. Quando é hora de distribuir, uma frota de caminhões (Docker Swarm) pega as caixas do armazém e as entrega nos pontos de venda (servidores). O gerente de logística (manager node) coordena a frota, e os motoristas (worker nodes) fazem as entregas. Se um motorista quebra, o gerente automaticamente redistribui a carga.
🏭 Fábrica → código-fonte / Dockerfile
→ docker build
📦 Caixa lacrada → imagem versionada
→ Docker Image + tag
🏬 Armazém central → Registry
→ Docker Hub / GHCR / ECR
🚚 Frota de caminhões → cluster Swarm
→ Manager + Worker nodes
👔 Gerente de logística → manager node
→ docker service
🔁 Nova coleção → versão nova do app
→ Rolling update

Docker Registries — Onde Vivem Suas Imagens

Um Registry é um servidor que armazena e distribui imagens Docker. Toda imagem que você baixa com docker pull vem de um registry. Por padrão, o Docker usa o Docker Hub — mas você pode usar registries privados ou gerenciados na nuvem.

🐳 Docker Hub — hub.docker.com
Registry padrão. Gratuito com limitações (1 push/hora em conta free).

Ideal para imagens públicas e projetos open source. Todas as imagens oficiais (nginx, postgres, redis) ficam aqui. Endereço implícito: docker.io/usuario/imagem:tag

🐙 GitHub Container Registry (GHCR) — ghcr.io
Integrado ao GitHub. Free para repos públicos, incluído no GitHub Free para privados (até limite de Actions).

Melhor escolha para projetos no GitHub. Imagens ficam na mesma conta do repositório, com controle de acesso por token. Endereço: ghcr.io/usuario/imagem:tag

☁️ Amazon ECR — AWS Elastic Container Registry
Registry privado na AWS. Pago por GB armazenado + transferência.

Padrão em projetos que usam ECS/EKS. Autenticação via AWS IAM. Endereço: 123456789.dkr.ecr.us-east-1.amazonaws.com/minha-app:tag

Pipeline Automatizado: Build → Push → Deploy

Um pipeline CI/CD com Docker segue sempre o mesmo padrão: a cada novo commit, a imagem é construída, testada e publicada no registry — podendo ainda disparar o deploy automático no servidor.

Pipeline CI/CD Docker com GitHub Actions
sequenceDiagram participant Dev as 💻 Developer participant GH as 🐙 GitHub participant CI as ⚙️ GitHub Actions participant Reg as 📦 GHCR participant Srv as 🖥️ Servidor Dev->>GH: git push main GH->>CI: trigger workflow CI->>CI: checkout code CI->>CI: docker build CI->>CI: run tests CI->>Reg: docker push :sha + :latest CI->>Srv: SSH → docker pull + restart Srv-->>CI: ✅ deploy ok

GitHub Actions — Workflow Completo

.github/workflows/docker-cicd.yml
name: Docker CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}   # ex: org/minha-app

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write   # necessário para push no GHCR

    steps:
      # 1. Checkout do código
      - name: Checkout
        uses: actions/checkout@v4

      # 2. Login no GHCR usando token automático do Actions
      - name: Login no Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      # 3. Extrai metadados: tags automáticas (branch, sha, semver)
      - name: Extrai metadados (tags e labels)
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=sha,prefix=sha-
            type=semver,pattern={{version}}
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}

      # 4. Build multi-plataforma + push em um único passo
      - name: Build e Push da imagem
        uses: docker/build-push-action@v5
        with:
          context: .
          push: ${{ github.event_name != 'pull_request' }}  # não faz push em PRs
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha       # cache do GitHub Actions (mais rápido)
          cache-to: type=gha,mode=max

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'   # deploy só no main

    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            # Pull a nova imagem e reinicia o container
            docker pull ghcr.io/${{ github.repository }}:latest
            docker stop minha-app || true
            docker rm minha-app || true
            docker run -d \
              --name minha-app \
              --restart unless-stopped \
              -p 3000:3000 \
              -e DATABASE_URL=$DATABASE_URL \
              ghcr.io/${{ github.repository }}:latest
🔥 Pro Tip: Cache do build
cache-from: type=gha usa o cache do GitHub Actions para evitar rebuilds desnecessários. Em projetos grandes, pode reduzir o tempo de build de 5 minutos para 30 segundos.

Configurando os Secrets no GitHub

Vá em Settings → Secrets and variables → Actions → New repository secret e adicione:

SecretValorObservação
SERVER_HOSTIP ou domínio do servidorEx: 192.168.1.100
SERVER_USERUsuário SSHEx: ubuntu, root
SSH_PRIVATE_KEYChave privada SSHConteúdo do arquivo ~/.ssh/id_rsa
GITHUB_TOKENAutomáticoGerado pelo Actions — não precisa criar
⚠️ Não commite secrets no código!
Nunca coloque credenciais, senhas ou chaves diretamente no código ou no Dockerfile. Use sempre ARG + --build-arg para build-time e variáveis de ambiente em runtime. Secrets no GitHub Actions são injetados de forma segura e nunca aparecem nos logs.

Orquestração Nativa: Docker Swarm

Docker Swarm é o modo de orquestração nativo do Docker — baked in, sem instalação adicional. Com um único comando, você transforma múltiplos hosts Docker em um cluster gerenciado, capaz de distribuir containers, fazer rolling updates e se recuperar de falhas automaticamente.

💡 Swarm vs Kubernetes
Kubernetes é mais poderoso, mas também mais complexo (curva de aprendizado enorme). Docker Swarm é ideal para equipes menores, projetos sem budget para cloud gerenciada, ou quando você precisa de orquestração sem dedicar semanas de estudo.

No Swarm, você não cria containers diretamente — você cria services. O Swarm garante que o número de réplicas definido esteja sempre rodando.

Rolling Updates — Zero Downtime

Rolling update é a estratégia de atualizar containers um a um (ou em grupos), mantendo a aplicação disponível durante o processo.

rolling update e rollback
# Atualizar image (rolling update — 1 réplica por vez, espera 10s):
$ docker service update \
--image ghcr.io/meuorg/minha-app:v2.0.0 \
--update-parallelism 1 \
--update-delay 10s \
--update-failure-action rollback \

# Se algo der errado — rollback manual:
$ docker service rollback minha-app
minha-app rolled back to previous spec

Stacks — Compose para o Swarm

Uma stack é um conjunto de services definidos em um arquivo docker-compose.yml e implantado no Swarm com um único comando. É o equivalente ao Compose para produção.

docker-stack.yml — App Node.js + Nginx + Redis
version: '3.9'

services:
  app:
    image: ghcr.io/meuorg/minha-app:latest
    deploy:
      replicas: 3
      update_config:
        parallelism: 1       # atualiza 1 réplica por vez
        delay: 10s
        failure_action: rollback
        order: start-first   # sobe nova réplica antes de derrubar antiga
      restart_policy:
        condition: on-failure
        max_attempts: 3
      resources:
        limits:
          cpus: '0.5'
          memory: 256M
    networks:
      - app-network
    secrets:
      - db_password
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgres://app:FILE[/run/secrets/db_password]@db:5432/app

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    deploy:
      replicas: 2
      placement:
        constraints:
          - node.role == manager   # nginx só nos managers
    configs:
      - source: nginx_conf
        target: /etc/nginx/conf.d/default.conf
    networks:
      - app-network

  redis:
    image: redis:7-alpine
    deploy:
      replicas: 1
      placement:
        constraints:
          - node.labels.redis == true   # só no nó com label redis
    volumes:
      - redis-data:/data
    networks:
      - app-network

networks:
  app-network:
    driver: overlay   # overlay conecta containers em hosts diferentes

volumes:
  redis-data:

secrets:
  db_password:
    external: true   # secret criado previamente com docker secret create

configs:
  nginx_conf:
    external: true
deploy de stack
# Criar secret antes do deploy:
$ echo "minhasenhasupersecreta" | docker secret create db_password -
x1y2z3w4v5u6t7s8r9q0

# Deploy da stack (ou atualização se já existir):
$ docker stack deploy -c docker-stack.yml minha-stack
Creating network minha-stack_app-network
Creating service minha-stack_app
Creating service minha-stack_nginx
Creating service minha-stack_redis

# Listar services da stack:
$ docker stack services minha-stack
NAME REPLICAS IMAGE
minha-stack_app 3/3 ghcr.io/meuorg/minha-app:latest
minha-stack_nginx 2/2 nginx:alpine
minha-stack_redis 1/1 redis:7-alpine

# Remover a stack inteira:
$ docker stack rm minha-stack

Secrets — Credenciais Seguras no Swarm

Secrets no Swarm são armazenados criptografados no Raft log do manager e injetados nos containers como arquivos em /run/secrets/. Nunca ficam em variáveis de ambiente (que podem vazar em logs).

docker secret
# Criar secrets:
$ echo "minha_senha" | docker secret create db_password -
$ docker secret create ssl_cert ./certificado.pem

# cat /run/secrets/db_password → "minha_senha"

Projeto Prático: API + CI/CD + Swarm Local

Vamos montar do zero um pipeline completo: API Node.js → Registry → Swarm local (simulado com um único nó).

1
Crie a aplicação de exemplo
Uma API Node.js simples com health check para validar o deploy
// app.js
const http = require('http')
const PORT = process.env.PORT || 3000
const VERSION = process.env.APP_VERSION || '1.0.0'

const server = http.createServer((req, res) => {
  if (req.url === '/health') {
    res.writeHead(200, {'Content-Type': 'application/json'})
    res.end(JSON.stringify({ status: 'ok', version: VERSION }))
    return
  }
  res.writeHead(200, {'Content-Type': 'text/plain'})
  res.end(`🐳 Olá do Docker Swarm! Versão ${VERSION}\n`)
})

server.listen(PORT, () => console.log(`Rodando na porta ${PORT} — v${VERSION}`))
# Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY app.js .
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s \
  CMD wget -qO- http://localhost:3000/health || exit 1
CMD ["node", "app.js"]
2
Inicialize o Swarm local
Com um único nó, o manager também executa tarefas (modo desenvolvimento)
$ docker swarm init
Swarm initialized: current node is now a manager.
$ docker node ls
ID HOSTNAME STATUS MANAGER STATUS
abc123 * mypc Ready Leader
3
Build, tag e push para Docker Hub
$ docker login
$ docker build -t seuusuario/swarm-demo:v1.0.0 .
$ docker push seuusuario/swarm-demo:v1.0.0
4
Deploy como service no Swarm
$ docker service create \
--name swarm-demo \
--replicas 3 \
--publish 8080:3000 \
-e APP_VERSION=v1.0.0 \
seuusuario/swarm-demo:v1.0.0

$ curl localhost:8080
🐳 Olá do Docker Swarm! Versão v1.0.0
$ curl localhost:8080/health
{"status":"ok","version":"v1.0.0"}
5
Simule um rolling update
Mude a versão no Dockerfile (APP_VERSION=v2.0.0), faça build e push, depois atualize o service
$ docker build -t seuusuario/swarm-demo:v2.0.0 .
$ docker push seuusuario/swarm-demo:v2.0.0
$ docker service update \
--image seuusuario/swarm-demo:v2.0.0 \
--update-delay 5s \
swarm-demo

# Em outro terminal, fique testando durante o update:
$ watch -n1 "curl -s localhost:8080 | head -1"
🐳 Olá do Docker Swarm! Versão v2.0.0

⚠️ Erros que Todo Mundo Comete

⚠️ Tag "latest" em produção

Usar :latest no Swarm é perigoso: cada worker pode baixar versões diferentes dependendo do cache local. Use sempre tags semânticas (:v1.2.3) ou SHA do commit (:sha-abc1234) para garantir reprodutibilidade.

⚠️ Swarm sem múltiplos managers

Com um único manager, se ele cair, o cluster inteiro para. Para alta disponibilidade, use 3 ou 5 managers (número ímpar para consenso Raft). Regra: quorum = (n/2) + 1

⚠️ Volumes em clusters multi-host

Volumes Docker padrão são locais — se o container migrar para outro nó, os dados somem. Para dados persistentes em Swarm, use NFS, GlusterFS, ou volumes de cloud (AWS EFS, etc.) que sejam acessíveis de todos os nós.

⚠️ Ignorar healthchecks

Sem HEALTHCHECK no Dockerfile, o Swarm considera o container "healthy" assim que inicia, mesmo que a aplicação ainda esteja inicializando. Isso causa falhas em rolling updates. Sempre implemente healthcheck.

Desafio: Pipeline Completo do Zero

Construa um pipeline CI/CD com Docker Swarm para uma aplicação simples (pode ser a API de exemplo desta aula ou qualquer projeto seu):

  • Escreva um Dockerfile com HEALTHCHECK
  • Configure o push automático para GHCR via GitHub Actions
  • Inicialize um Swarm local com docker swarm init
  • Faça deploy como service com pelo menos 2 réplicas
  • Execute um rolling update (v1 → v2) sem derrubar o serviço
