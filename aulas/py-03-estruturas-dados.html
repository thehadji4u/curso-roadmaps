// motivação

Por que isso importa?

Se funções são os verbos do Python — os fazeres — as estruturas de dados são os substantivos: os recipientes que guardam, organizam e relacionam informações. Escolher a estrutura errada é um dos erros mais comuns e mais caros em termos de performance em código Python real.

Um loop que verifica if item in lista em uma lista de 1 milhão de elementos é O(n). O mesmo teste em um set é O(1). Em análise de dados com pandas, DataFrames são construídos sobre dicionários. Em APIs REST, quase toda resposta JSON se converte diretamente em dict e list. Dominar essas quatro estruturas é dominar 80% do código Python do dia a dia.

💼 Por que um engenheiro sênior domina isso profundamente
Sêniors escolhem estruturas com intenção: usam tuple para deixar explícito que dados são imutáveis, set para deduplicação e testes de pertencimento eficientes, collections.Counter para frequências, e collections.defaultdict para evitar o clássico KeyError ao agrupar dados. A escolha certa comunica intenção — e é o que separa código bom de código que "apenas funciona".
// analogia

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense em uma cozinha profissional bem organizada. Cada tipo de recipiente existe por uma razão: a prateleira numerada guarda itens em ordem de chegada e você pode inserir ou remover qualquer um (lista). O armário rotulado com etiquetas e conteúdos associados é acessado pelo rótulo, não pela posição (dicionário). A cesta de ingredientes únicos elimina duplicatas automaticamente — se você jogar dois alhos, tem um (conjunto). E a receita impressa em papel é imutável: você pode lê-la, mas não pode alterar o original (tupla).

Usar a lista quando precisa de acesso por rótulo é como tentar encontrar "manjericão" contando posições na prateleira. Usar um dict torna isso instantâneo.
📦 Prateleira numerada e ordenada → list — mutável, ordenada, indexada 🏷️ Armário com etiquetas → dict — chave → valor 🧺 Cesta de itens únicos → set — sem duplicatas, não ordenado 📋 Receita impressa e selada → tuple — imutável, ordenada

// conceito 2

list — A Estrutura mais Versátil

list é um array dinâmico: pode crescer e encolher, aceita qualquer tipo de dado em qualquer posição, e suporta slicing poderoso. É a estrutura que você vai usar mais frequentemente.

Criação e Slicing

// criação, indexação e slicing

# Criação
frutas = ["maçã", "banana", "cereja", "damasco", "figo"]
numeros = list(range(1, 11))  # [1, 2, 3, ..., 10]
misto   = [1, "dois", 3.0, True, None]  # tipos heterogêneos ✓

# Indexação — começa em 0, negativo conta do fim
frutas[0]    # "maçã"
frutas[-1]   # "figo"  ← último elemento
frutas[-2]   # "damasco"

# Slicing: lista[inicio:fim:passo]  — fim é EXCLUSIVO
frutas[1:3]   # ["banana", "cereja"]
frutas[:3]    # ["maçã", "banana", "cereja"]   ← do início até índice 2
frutas[2:]    # ["cereja", "damasco", "figo"]  ← do índice 2 até o fim
frutas[::2]   # ["maçã", "cereja", "figo"]     ← passo 2 (elementos pares)
frutas[::-1]  # ["figo", "damasco", "cereja", "banana", "maçã"]  ← invertida!

# Modificação
frutas[0] = "abacaxi"      # substitui primeiro elemento
frutas[1:3] = ["blueberry"] # substitui fatia por uma lista diferente

⚠️ sort() vs sorted() — diferença crítica
lista.sort() modifica a lista original e retorna None. sorted(lista) retorna uma nova lista ordenada sem modificar a original. Um erro comum é fazer lista = lista.sort(), o que define lista como None.
// conceito 3

dict — O Coração do Python

Dicionários são hash maps — acesso por chave em O(1) independente do tamanho. Desde Python 3.7, a ordem de inserção é garantida. São a base de kwargs, objetos JSON, configs, caches e muito mais.

// criação, acesso e métodos essenciais

# Formas de criar
usuario = {"nome": "Ana", "idade": 29, "ativo": True}
vazio   = {}
do_zip  = dict(zip(["a", "b", "c"], [1, 2, 3]))  # {"a":1, "b":2, "c":3}
do_comp = {k: k**2 for k in range(1, 6)}          # {1:1, 2:4, 3:9, 4:16, 5:25}

# Acesso — dois jeitos
usuario["nome"]            # "Ana" — levanta KeyError se a chave não existir
usuario.get("email")       # None  — seguro, sem KeyError
usuario.get("email", "")  # ""    — com valor default

# Modificação
usuario["email"] = "ana@email.com"  # adiciona ou atualiza
usuario.update({"cidade": "SP", "idade": 30})  # merge de múltiplos campos
del usuario["ativo"]                # remove chave

# Iteração — três visões
for chave in usuario:             # itera sobre chaves
for chave in usuario.keys():      # equivalente, mais explícito
for valor in usuario.values():    # itera sobre valores
for chave, valor in usuario.items():  # tuplas (chave, valor) — mais comum

# Verificação de existência
"nome" in usuario    # True  — O(1)
"cpf"  in usuario   # False

# Remoção segura
removido = usuario.pop("cidade", None)  # remove e retorna, sem KeyError se ausente
usuario.setdefault("score", 0)          # define valor só se chave não existe

Merge de Dicionários (Python 3.9+)

❌ Antes do Python 3.9

merged = {**dict1, **dict2}
# ou
merged = dict1.copy()
merged.update(dict2)
✅ Python 3.9+ (operadores | e |=)

merged = dict1 | dict2   # novo dict
dict1 |= dict2            # atualiza dict1 in-place

defaultdict e Counter — quando o dict nativo não basta

// collections — ferramentas de nível sênior

from collections import defaultdict, Counter

# defaultdict: nunca levanta KeyError — define valor padrão para chaves novas
grupos = defaultdict(list)
dados  = [("A", 1), ("B", 2), ("A", 3), ("B", 4)]
for chave, valor in dados:
    grupos[chave].append(valor)  # sem precisar verificar "chave in grupos"
# grupos == {"A": [1, 3], "B": [2, 4]}

# Counter: conta frequências automaticamente
texto  = "abracadabra"
contagem = Counter(texto)   # Counter({"a": 5, "b": 2, "r": 2, "c": 1, "d": 1})
contagem.most_common(2)     # [("a", 5), ("b", 2)]

palavras = "o gato comeu o rato".split()
freq = Counter(palavras)    # Counter({"o": 2, "gato": 1, "comeu": 1, "rato": 1})
// conceito 4

set — Unicidade e Operações Matemáticas

set é uma hash table sem valores — apenas chaves. Isso garante unicidade automática e torna testes de pertencimento O(1). Ele também suporta operações de teoria dos conjuntos: união, interseção, diferença.

// operações fundamentais com set

# Criação — atenção: {} vazio cria dict, não set!
frutas_a = {"maçã", "banana", "cereja"}
frutas_b = {"banana", "damasco", "cereja"}
vazio    = set()  # ← correto para set vazio

# Pertencimento — O(1), mais rápido que lista
"maçã" in frutas_a   # True
"kiwi" in frutas_a   # False

# Deduplicação instantânea
duplicados = [1, 2, 2, 3, 3, 3, 4]
unicos     = list(set(duplicados))  # [1, 2, 3, 4] — ordem não garantida

# Operações de conjunto
frutas_a | frutas_b   # União:      {"maçã", "banana", "cereja", "damasco"}
frutas_a & frutas_b   # Interseção: {"banana", "cereja"}
frutas_a - frutas_b   # Diferença:  {"maçã"}  (em A mas não em B)
frutas_a ^ frutas_b   # Diferença simétrica: {"maçã", "damasco"} (só em um)

# Modificação
frutas_a.add("kiwi")         # adiciona elemento
frutas_a.discard("melão")    # remove se existir, sem erro se não existir
frutas_a.remove("banana")    # remove — KeyError se não existir

# Frozenset — set imutável (pode ser chave de dict ou elemento de outro set)
imutavel = frozenset({"a", "b", "c"})
🔥 Pro tip — use set para encontrar diferenças entre listas
set(lista_nova) - set(lista_antiga) retorna os elementos que estão na nova mas não na antiga. Muito útil para detectar o que mudou entre duas versões de dados — operação que seria O(n²) com loops ingênuos e fica O(n) com sets.
// conceito 5

tuple — Imutabilidade como Comunicação

tuple é como uma lista, mas imutável. Isso não é apenas uma restrição — é uma intenção comunicada ao leitor do código: "estes dados não devem mudar". Tuplas são mais eficientes em memória que listas e podem ser usadas como chaves de dicionários.

// quando usar tuple

# Uso 1 — retorno múltiplo de funções (o mais comum)
def min_max(numeros):
    return min(numeros), max(numeros)  # retorna uma tuple implicitamente

menor, maior = min_max([4, 1, 7, 2])  # unpacking
print(menor, maior)  # 1 7

# Uso 2 — coordenadas e dados estruturados imutáveis
ponto = (3.5, -2.1)
rgb   = (255, 128, 0)

# Uso 3 — chave composta de dicionário
cache = {}
cache[("São Paulo", "Rio de Janeiro")] = 429  # km
cache[("SP", "BSB")] = 1015

# Uso 4 — namedtuple — tuple com acesso por nome
from collections import namedtuple

Ponto = namedtuple("Ponto", ["x", "y"])
p = Ponto(x=3.5, y=-2.1)
print(p.x, p.y)   # 3.5 -2.1
print(p[0])        # 3.5  ← acesso por índice ainda funciona
print(p._asdict()) # {"x": 3.5, "y": -2.1}

# namedtuple é excelente para dados sem comportamento (substituindo dicts simples)
Produto = namedtuple("Produto", ["nome", "preco", "estoque"])
item = Produto("Notebook", 7000.0, 5)
print(f"{item.nome}: R$ {item.preco:.2f} ({item.estoque} em estoque)")
💡 dataclass vs namedtuple
Para dados simples sem comportamento, namedtuple é mais eficiente. Para dados que precisam de valores default, métodos ou mutabilidade controlada, prefira @dataclass (Python 3.7+). Veremos dataclasses na aula de Orientação a Objetos.

collections.deque — Quando list Não Basta

list.insert(0, x) e list.pop(0) são O(n) — todos os elementos são reindexados. Para filas e pilhas que adicionam/removem nas duas extremidades, collections.deque faz essas operações em O(1).

// deque — double-ended queue

from collections import deque

fila = deque()

# Adicionar
fila.append("João")      # lado direito — O(1)
fila.appendleft("Maria") # lado esquerdo — O(1)  ← list faria isso em O(n)!

# Remover
fila.pop()               # remove da direita — O(1)
fila.popleft()           # remove da esquerda — O(1)

# Deque com tamanho máximo — buffer circular
historico = deque(maxlen=3)
for i in range(6):
    historico.append(i)
    print(list(historico))
# [0]
# [0, 1]
# [0, 1, 2]
# [1, 2, 3]  ← 0 foi descartado
# [2, 3, 4]
# [3, 4, 5]  ← mantém sempre os últimos 3
🔥 Pro tip — deque para histórico de undo/redo
deque(maxlen=50) é a estrutura perfeita para histórico de ações de usuário — mantém apenas os últimos N estados sem gerenciar limpeza manual.

Exemplo Completo — Analisador de Logs

Um script que lê uma lista de eventos de log e usa todas as estruturas para gerar um relatório:

// analisador_logs.py — usa list, dict, set, Counter e namedtuple

from collections import Counter, namedtuple, defaultdict

# ── Definição de estrutura de dado ──────────────────────────────────────
Evento = namedtuple("Evento", ["timestamp", "nivel", "usuario", "mensagem"])

# ── Dados de entrada (simulando parse de arquivo de log) ─────────────────
logs_raw = [
    ("2024-01-15 08:00", "INFO",  "ana",    "login"),
    ("2024-01-15 08:01", "INFO",  "bruno",  "login"),
    ("2024-01-15 08:05", "ERROR", "ana",    "falha ao acessar /admin"),
    ("2024-01-15 08:10", "INFO",  "carla",  "login"),
    ("2024-01-15 08:15", "WARN",  "bruno",  "tentativa suspeita"),
    ("2024-01-15 08:20", "ERROR", "carla",  "senha incorreta"),
    ("2024-01-15 08:21", "ERROR", "carla",  "senha incorreta"),
    ("2024-01-15 08:25", "INFO",  "ana",    "logout"),
    ("2024-01-15 08:30", "WARN",  "carla",  "conta bloqueada"),
]

# ── Processamento: list comprehension para converter raw → namedtuple ────
eventos = [Evento(*linha) for linha in logs_raw]

# ── Filtragem com list comprehension ────────────────────────────────────
erros  = [e for e in eventos if e.nivel == "ERROR"]
avisos = [e for e in eventos if e.nivel == "WARN"]

print(f"Total eventos: {len(eventos)}")
print(f"Erros:  {len(erros)} | Avisos: {len(avisos)}")

# ── Usuários únicos com set ───────────────────────────────────────────────
usuarios_ativos = {e.usuario for e in eventos}  # set comprehension
print(f"Usuários únicos: {sorted(usuarios_ativos)}")

# ── Contagem por nível com Counter ───────────────────────────────────────
por_nivel = Counter(e.nivel for e in eventos)
print("\nDistribuição por nível:")
for nivel, contagem in por_nivel.most_common():
    print(f"  {nivel:5s}: {'█' * contagem} ({contagem})")

# ── Agrupamento por usuário com defaultdict ──────────────────────────────
por_usuario = defaultdict(list)
for evento in eventos:
    por_usuario[evento.usuario].append(evento)

print("\nRelatório por usuário:")
for usuario in sorted(por_usuario):
    eventos_u = por_usuario[usuario]
    niveis_u  = Counter(e.nivel for e in eventos_u)
    erros_u   = niveis_u.get("ERROR", 0)
    status    = "🔴 CRÍTICO" if erros_u >= 2 else ("🟡 ATENÇÃO" if erros_u == 1 else "🟢 OK")
    print(f"  {usuario:8s} — {len(eventos_u)} eventos | {status}")

# ── Usuários sem erros (diferença de sets) ───────────────────────────────
com_erros  = {e.usuario for e in erros}
sem_erros  = usuarios_ativos - com_erros
print(f"\nUsuários sem erros: {sem_erros}")

⚠️ Pitfalls — Erros Frequentes

❌ Pitfall 1 — Cópia rasa vs cópia profunda

Em Python, b = a para listas cria uma referência, não uma cópia. b = a.copy() cria uma cópia rasa — listas aninhadas ainda são compartilhadas. Use copy.deepcopy(a) quando tiver estruturas aninhadas.


import copy

# Problema
a = [[1, 2], [3, 4]]
b = a.copy()          # cópia rasa
b[0].append(99)       # modifica a lista interna
print(a)              # [[1, 2, 99], [3, 4]] ← a também foi modificada!

# Solução
c = copy.deepcopy(a)
c[0].append(88)
print(a)              # não modificada ✓
❌ Pitfall 2 — Modificar lista durante iteração

# ERRADO — comportamento imprevisível
numeros = [1, 2, 3, 4, 5]
for n in numeros:
    if n % 2 == 0:
        numeros.remove(n)   # modifica a lista enquanto itera

# CORRETO — list comprehension ou itere sobre cópia
numeros = [n for n in numeros if n % 2 != 0]  # filtra em nova lista
# ou: for n in numeros[:]:  ← itera sobre slice (cópia)
❌ Pitfall 3 — Usar list como argumento default de função

# ERRADO — a lista default é criada UMA VEZ e compartilhada entre chamadas!
def adicionar(item, lista=[]):
    lista.append(item)
    return lista

print(adicionar("a"))   # ["a"]
print(adicionar("b"))   # ["a", "b"]  ← não era o esperado!

# CORRETO — usar None como default e criar dentro da função
def adicionar(item, lista=None):
    if lista is None:
        lista = []
    lista.append(item)
    return lista
❌ Pitfall 4 — Criar set vazio com {}

# ERRADO — isso cria um dict vazio, não um set!
meu_conjunto = {}
type(meu_conjunto)  # <class 'dict'>

# CORRETO
meu_conjunto = set()
type(meu_conjunto)  # <class 'set'>

Glossário

Hash Table
Estrutura de dados interna usada por dict e set. Transforma a chave em um índice numérico via função hash, permitindo acesso O(1) médio.
Slicing
Sintaxe seq[inicio:fim:passo] para extrair sub-sequências de listas, tuplas e strings. Retorna sempre uma nova cópia da fatia.
Shallow Copy (Cópia Rasa)
Cópia que duplica o container externo, mas as referências internas ainda apontam para os mesmos objetos. lista.copy() e lista[:] fazem cópia rasa.
Deep Copy (Cópia Profunda)
Cópia recursiva que duplica o container e todos os objetos internos. Feita com copy.deepcopy(). Necessária quando se trabalha com estruturas aninhadas mutáveis.
Hashável (Hashable)
Objeto que possui um valor de hash fixo durante sua vida útil — condição para ser chave de dict ou elemento de set. Tipos imutáveis (int, str, tuple) são hashável; mutáveis (list, dict, set) não são.
defaultdict
Subclasse de dict do módulo collections que aceita uma função "factory" para criar valores default automaticamente quando uma chave inexistente é acessada.
Counter
Subclasse de dict especializada em contar frequências de elementos. Possui métodos como most_common(n) e suporta operações aritméticas entre contadores.
namedtuple
Fábrica de subclasses de tuple com campos nomeados. Combina eficiência de tupla com acesso por nome (ponto.x em vez de ponto[0]).
deque
Double-ended queue do módulo collections. Operações de inserção/remoção nas duas extremidades são O(1), diferente de list que é O(n) na extremidade esquerda.
