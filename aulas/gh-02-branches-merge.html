Git — Branches, Merge & Conflitos

⏱ ~30 min de leitura 🎯 Requer Aula 01 🔧 Hands-on incluído 💥 Inclui resolução de conflitos
git branch git merge git switch git stash git restore git reset git revert conflitos
// 01 — motivação

Por que isso importa?

A aula anterior te deu o básico de commits. Mas trabalhar em projeto real tem um problema sério: você precisa desenvolver uma nova feature sem quebrar o que já funciona, enquanto corrige um bug urgente, enquanto um colega trabalha no mesmo arquivo. Sem branches, isso é impossível.

Branches são o que transformam o Git de uma ferramenta de backup em uma plataforma de colaboração. Toda empresa de tecnologia opera com branches — entender merge e resolução de conflitos é o que separa quem usa Git de quem domina Git.

💼 Por que um engenheiro sênior usa isso hoje
Em qualquer time profissional, main é protegida — ninguém commita direto nela. Todo trabalho acontece em branches: feat/user-auth, fix/login-crash, chore/update-deps. Quando a feature está pronta, abre-se um Pull Request, outro engenheiro revisa, e o merge acontece de forma controlada. Saber resolver conflitos rápido e sem estresse é uma das habilidades mais valorizadas em code reviews.
// 02 — analogia

Antes do código: a linha do tempo paralela

🎬 Mundo Real → Git
Imagine que você está escrevendo um roteiro de filme. Você tem a versão principal já aprovada — o rascunho final. Agora o diretor pede para você testar um final alternativo, mas sem apagar o original. Você tira uma fotocópia e trabalha nela.

Se o final alternativo for aprovado, você mescla as mudanças de volta ao original. Se não funcionar, simplesmente descarta a cópia. O original nunca foi tocado.

Quando dois roteiristas editam a mesma cena ao mesmo tempo e as versões são incompatíveis, alguém precisa sentar e decidir qual versão prevalece — isso é resolver um conflito de merge.
Rascunho principal aprovado → branch main Tirar uma fotocópia para editar → git checkout -b minha-feature Mesclar de volta ao original → git merge Dois roteiristas editando a mesma cena → conflito de merge Guardar rascunho inacabado na gaveta → git stash
// 03 — conceito

O que é uma branch — de verdade

Uma branch no Git não é uma cópia de arquivos. É simplesmente um ponteiro leve para um commit. Quando você cria uma branch, o Git cria apenas um arquivo de ~41 bytes com o hash do commit atual. É por isso que criar branches no Git é instantâneo, ao contrário de outros sistemas de versão.

O HEAD é um ponteiro especial que diz ao Git em qual branch (e commit) você está trabalhando agora.

// branches são ponteiros para commits

Os tipos de merge

Tipo Quando ocorre Histórico Quando usar
Fast-forward main não avançou desde o branch Linear, sem commit extra Features simples, commits recentes
Merge commit main e branch divergiram Preserva contexto de quando as coisas foram mergeadas Padrão do GitHub ao fechar PRs
Squash merge Branch com muitos commits bagunçados Comprime tudo num único commit limpo Features com histórico de WIP descartável
Rebase + merge Quer histórico totalmente linear Sem merge commits, árvore limpa Times que valorizam histórico linear
// 04 — mãos na massa: branches

Criando e navegando entre branches

1
Crie uma branch para sua feature
Sempre crie branches a partir da main atualizada. O padrão de nomenclatura tipo/descricao-curta é usado em 90% dos times profissionais.
terminalbash 
# Ver branches existentes (o * indica onde você está)
git branch

# Criar branch e já entrar nela — forma moderna
git switch -c feat/adicionar-contato

# Forma antiga (ainda muito usada, funciona igual)
git checkout -b feat/adicionar-contato

# Confirmar que você está na nova branch
git branch
# Saída:
#   main
# * feat/adicionar-contato
2
Trabalhe na branch e faça commits
Tudo que você commitar aqui fica isolado — a main não é afetada.
terminalbash 
# Crie ou edite arquivos normalmente
echo "Nome: João · Email: joao@exemplo.com" > contato.txt

git add contato.txt
git commit -m "feat: add contact page with basic info"

# Faça mais um commit
echo "Tel: (11) 99999-9999" >> contato.txt
git add contato.txt
git commit -m "feat: add phone number to contact"

# Ver o log da branch atual
git log --oneline
3
Volte para main e faça o merge
Primeiro você vai para a branch de destino, depois faz o merge da branch de origem. A direção é importante: você "puxa" a branch para dentro da atual.
terminalbash 
# Voltar para a main
git switch main

# Verificar que contato.txt NÃO existe aqui ainda
ls

# Fazer o merge
git merge feat/adicionar-contato

# Se for fast-forward, saída será:
# Updating a3f9c21..d8e4f12
# Fast-forward
#  contato.txt | 2 +++

# Depois do merge, contato.txt agora existe na main
ls

# Deletar a branch (trabalho concluído)
git branch -d feat/adicionar-contato
// 05 — conflitos

Resolvendo conflitos — sem pânico

Um conflito acontece quando duas branches modificaram a mesma linha do mesmo arquivo de formas diferentes. O Git não sabe qual versão você quer — então ele para e pede que você decida. Não é erro, é uma feature.

⚠️ Conflito não é catástrofe
Engenheiros iniciantes ficam com medo de conflitos. Seniores encaram como parte normal do trabalho. Um conflito significa que duas pessoas estavam trabalhando produtivamente no mesmo código — é um sinal de colaboração, não de falha.

Quando um conflito ocorre, o Git insere marcadores no arquivo para mostrar as duas versões:

⚠️ contato.txt — CONFLITO DE MERGE
<<<<<<< HEAD (sua versão — branch atual)
Nome: João · Email: joao@exemplo.com
Tel: (11) 99999-9999
======= (separador)
Nome: João Silva · Email: joao.silva@empresa.com
Tel: (11) 88888-8888
LinkedIn: linkedin.com/in/joaosilva

>>>>>>> feat/atualizar-contato (versão deles — branch que está sendo mergeada)
1
Identifique todos os arquivos em conflito
terminalbash 
# Git mostra o status com os conflitos
git status

# Saída:
# both modified:   contato.txt
# Unmerged paths:
#   (use "git add ..." to mark resolution)
#     both modified:   contato.txt
2
Abra o arquivo e escolha o que manter
Edite o arquivo manualmente: remova os marcadores <<<<<<<, ======= e >>>>>>> e deixe apenas o conteúdo final desejado. Você pode manter uma versão, a outra, ou combinar as duas.
contato.txt — após resoluçãotexto
# Versão final que você decidiu manter (sem marcadores):
Nome: João Silva · Email: joao.silva@empresa.com
Tel: (11) 99999-9999
LinkedIn: linkedin.com/in/joaosilva
🔥 Pro tip — VS Code resolve conflitos com cliques
O VS Code destaca os conflitos com botões visuais: Accept Current Change (HEAD), Accept Incoming Change (branch mergeada), Accept Both Changes, ou Compare Changes. Você raramente precisa editar os marcadores manualmente.
3
Marque como resolvido e finalize o merge
terminalbash 
# Após editar o arquivo, marque como resolvido
git add contato.txt

# Confirme que não há mais conflitos
git status

# Finalize o merge com um commit
git commit -m "merge: resolve contact info conflict, keep linkedin field"

# Ver o resultado no histórico
git log --oneline --graph --all
// 06 — desfazendo mudanças

Desfazendo mudanças — a ferramenta certa para cada situação

Git tem três comandos diferentes para desfazer coisas. Usar o errado pode causar dor de cabeça. A regra de ouro: nunca reescreva história que já foi pushed para o remote.

Comando O que faz Seguro em remote? Quando usar
git restore Descarta mudanças no working directory (não commitadas) ✓ Não mexe em commits Jogou fora edições indesejadas antes de commitar
git reset --soft Move HEAD para commit anterior, mantém mudanças na staging ⚠️ Só em branches locais Quer refazer o commit com mensagem melhor
git reset --hard Move HEAD E descarta todas as mudanças ✗ Nunca em commits pushados Quer apagar completamente os últimos commits locais
git revert Cria um novo commit que desfaz um commit anterior ✓ Sempre seguro Desfazer mudança que já foi para o remote/main
desfazendo-mudancas.shbash
### ─── ANTES DE COMMITAR ──────────────────────────────
# Descartar mudanças em um arquivo específico
git restore contato.txt

# Descartar TUDO no working directory (cuidado!)
git restore .

# Remover um arquivo da staging (sem perder as edições)
git restore --staged contato.txt


### ─── APÓS COMMITAR (local, não pushed) ──────────────
# Desfazer o último commit MAS manter as mudanças na staging
git reset --soft HEAD~1

# Desfazer o último commit MAS manter as mudanças no working dir
git reset HEAD~1

# Desfazer os últimos 2 commits E apagar as mudanças (PERIGOSO)
git reset --hard HEAD~2


### ─── APÓS PUSHED (seguro, não reescreve história) ───
# Criar um commit que desfaz o commit abc1234
git revert abc1234

# Revert sem abrir o editor de mensagem
git revert abc1234 --no-edit
// 07 — stash

git stash — a gaveta de rascunhos

Você está no meio de uma feature, working directory cheio de mudanças, quando chega um bug urgente para corrigir em outra branch. Você não pode commitar um trabalho incompleto, e não quer perder tudo. É exatamente para isso que existe o git stash.

// fluxo típico de uso do stash
          sequenceDiagram
          participant WD as Working Dir
          participant ST as Stash
          participant BR as Branch fix/bug

          Note over WD: Trabalhando em feat/nova-feature
          WD->>ST: git stash (guarda rascunho)
          Note over WD: Working dir limpo
          WD->>BR: git switch fix/bug
          Note over BR: Corrige bug urgente, commita
          BR->>WD: git switch feat/nova-feature
          ST->>WD: git stash pop (recupera rascunho)
          Note over WD: Continua de onde parou
git-stash-guia.shbash 
# Guardar mudanças no stash (com descrição — sempre use!)
git stash push -m "feat: metade da página de contato implementada"

# Ver o que está na gaveta
git stash list
# stash@{0}: On feat/contato: feat: metade da página de contato implementada
# stash@{1}: On main: wip: testando nova config

# Recuperar o stash mais recente (e remove da lista)
git stash pop

# Recuperar um stash específico (sem remover da lista)
git stash apply stash@{1}

# Ver o que está num stash sem aplicar
git stash show stash@{0} -p

# Deletar um stash específico
git stash drop stash@{1}

# Limpar todos os stashes
git stash clear
🔥 Pro tip — stash também salva arquivos novos
Por padrão, git stash não guarda arquivos untracked (novos). Use git stash push -u para incluir arquivos novos também. Use git stash push -a para incluir até arquivos no .gitignore.
// 08 — estratégias

Estratégias de branch — como os times se organizam

🌊
GitHub Flow
Só main + branches de feature. Simples e direto. Feature pronta → PR → merge → deploy imediato.
Ideal para: SaaS, deploy contínuo, times ágeis
🌲
Git Flow
main, develop, feature/*, release/*, hotfix/*. Mais controle, mais cerimônia.
Ideal para: produtos com releases versionadas
🛤️
Trunk-Based
Todo mundo commita direto na main (ou branches curtíssimas de 1-2 dias). Feature flags controlam o que está ativo.
Ideal para: grandes times com CI/CD maduro
✅ Qual usar agora?
Para 90% dos projetos pessoais e profissionais: GitHub Flow. É o que o próprio GitHub usa. Simples de entender, fácil de executar, funciona muito bem.
// 09 — referência

Referência rápida — o que você vai usar todo dia

git-branches-cheatsheet.shbash
### ─── BRANCHES ──────────────────────────────────────
git branch                       # listar branches locais
git branch -a                    # listar locais + remotas
git switch main                  # trocar de branch
git switch -c feat/minha-feature # criar e entrar
git branch -d feat/concluida     # deletar (seguro — falha se não mergeado)
git branch -D feat/abandonada    # deletar à força

### ─── MERGE ──────────────────────────────────────────
git merge feat/minha-feature     # merge normal
git merge --no-ff feat/feature   # força merge commit (sem fast-forward)
git merge --squash feat/feature  # comprime todos os commits em um
git merge --abort                # cancela merge com conflito

### ─── DESFAZENDO ────────────────────────────────────
git restore arquivo.txt          # descarta edição (working dir)
git restore --staged arquivo.txt # remove da staging
git reset --soft HEAD~1          # desfaz commit, mantém na staging
git reset HEAD~1                 # desfaz commit, mantém no working dir
git reset --hard HEAD~1          # desfaz commit E apaga mudanças
git revert abc1234               # cria commit que desfaz (seguro no remote)

### ─── STASH ──────────────────────────────────────────
git stash push -m "descrição"    # guardar rascunho
git stash list                   # ver gaveta
git stash pop                    # recuperar + remover
git stash apply stash@{0}        # recuperar sem remover
git stash drop stash@{0}         # deletar item da gaveta

### ─── DIAGNÓSTICO ───────────────────────────────────
git log --oneline --graph --all  # histórico visual completo
git diff main..feat/feature      # diferença entre duas branches
git branch --merged              # branches já mergeadas (podem ser deletadas)
git branch --no-merged           # branches ainda não mergeadas
// 10 — armadilhas

Erros clássicos com branches

⚠️ git reset --hard em commits já pushed

Se você fez reset --hard em commits que já foram para o GitHub, a sua branch local e a remota divergiram. Forçar um git push --force pode apagar trabalho de colegas. Use sempre git revert para desfazer coisas públicas.

⚠️ Branches de longa duração acumulam conflitos

Quanto mais tempo uma branch fica sem sincronizar com a main, mais conflitos você vai ter no merge. A solução: faça git merge main (ou git rebase main) periodicamente na sua feature branch para trazer as mudanças mais recentes.

⚠️ Criar branches a partir de outra branch (que não é main)

Se você criar feat/b a partir de feat/a, você vai incluir todos os commits de feat/a quando fizer o merge de feat/b. Sempre crie branches a partir da main atualizada, a não ser que haja dependência explícita.

⚠️ Esquecer que stash é global, não por branch

O git stash não pertence a uma branch — ele fica numa pilha global. Se você fizer stash na feat/a e aplicar na main, vai funcionar (e pode causar confusão). Sempre adicione uma mensagem descritiva ao stash.

// 12 — desafio

🏁 Checkpoint — simule um fluxo de trabalho real

Use o repositório criado na Aula 01 (checkpoint-git) ou crie um novo. Execute este fluxo completo:

  • Crie uma branch feat/nova-secao a partir da main e adicione um arquivo novo com 2 commits
  • Volte para a main, edite o mesmo arquivo de uma forma diferente e faça 1 commit
  • Faça o merge da feat/nova-secao na main — vai gerar conflito
  • Resolva o conflito manualmente (edite o arquivo, remova os marcadores, decida o conteúdo final)
  • Finalize o merge e confirme o histórico com git log --oneline --graph
  • Crie outra branch feat/experimento, faça mudanças, use git stash para guardar rascunho, mude para a main, volte e recupere o stash
  • Use git reset --soft HEAD~1 para desfazer o último commit (sem perder as mudanças) e recomitar com mensagem melhor
💬 Dica: Para provocar o conflito intencionalmente, edite a linha 1 do mesmo arquivo nas duas branches antes de mergear. O git log --oneline --graph --all vai mostrar visualmente a bifurcação e o merge — é muito satisfatório ver isso funcionando.
