Git — Branches, Merge & Conflitos

⏱ ~40 min de leitura 🎯 Fundamentos → Intermediário 🔧 Hands-on incluído
branch merge conflitos checkout rebase stash reset desfazer mudanças
// MOTIVAÇÃO

Por que isso importa?

Branches são o coração do desenvolvimento colaborativo. Elas permitem que você (e seu time) trabalhem em múltiplas funcionalidades ao mesmo tempo — sem nunca tocar no código estável em produção. É o que possibilita que 50 engenheiros trabalhem no mesmo repositório simultaneamente, cada um em sua própria linha do tempo de código.

Saber resolver conflitos de merge é uma habilidade que distingue devs júnior de pleno. Conflitos são inevitáveis em qualquer time real — e abordá-los com calma e método, em vez de entrar em pânico, é sinal de maturidade técnica.

💼 Por que um engenheiro sênior domina branches profundamente
Times sêniors têm estratégias de branching bem definidas (GitHub Flow, GitFlow, trunk-based development). Cada pull request é uma branch. Saber criar, navegar, mergear e resolver conflitos com fluidez é pré-requisito para contribuir em qualquer codebase profissional. E git rebase, git stash e git reset são ferramentas que sêniors usam diariamente para manter o histórico limpo.
// ANALOGIA

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Imagine um estúdio de música gravando um álbum. A faixa principal (main) já está gravada e aprovada. Mas o produtor quer testar uma versão com guitarra elétrica, o vocalista quer experimentar uma linha de voz diferente, e o baterista quer adicionar um solo novo — tudo ao mesmo tempo, sem destruir a versão oficial.

Cada músico trabalha em sua própria trilha isolada (branch). Se o experimento funcionar, a trilha é mixada de volta na faixa principal (merge). Se não funcionar, a trilha é simplesmente descartada, sem nenhum dano ao álbum original. E quando duas trilhas têm notas que batem exatamente no mesmo lugar, o produtor precisa decidir qual versão usar — isso é um conflito de merge.
🎵 Faixa principal aprovada → branch main 🎸 Trilha experimental isolada → feature branch 🎚️ Mixar trilha na faixa principal → git merge 🎯 Duas notas batem no mesmo lugar → Conflito de merge
// CONCEITO 1

O que é uma branch — internamente

Uma branch no Git é simplesmente um ponteiro leve para um commit. Nada mais. Não é uma cópia dos arquivos — é apenas um arquivo de texto com o hash de um commit. Isso torna branches em Git incrivelmente baratas e rápidas de criar.

HEAD é o ponteiro especial que indica em qual branch (ou commit) você está trabalhando agora. Quando você faz um commit em uma branch, o ponteiro dessa branch avança automaticamente para o novo commit.

// branches como ponteiros para commits
gitGraph commit id: "chore: init" commit id: "feat: auth base" branch feature/login checkout feature/login commit id: "feat: login form" commit id: "feat: validação" checkout main commit id: "fix: typo README" branch hotfix/crash checkout hotfix/crash commit id: "fix: null pointer" checkout main merge hotfix/crash id: "merge hotfix"
🔥 Branches são baratas — crie sem medo
Em Git, criar uma branch é criar um arquivo de 41 bytes. Custa praticamente zero. A filosofia do Git é: crie uma branch para cada tarefa, por menor que seja. Isso mantém o main sempre estável e o histórico organizado.
// CONCEITO 2

Criando e navegando em branches

// tudo sobre branches — do básico ao avançado
# ── Listar branches ────────────────────────────────────────────
git branch              # branches locais (* = branch atual)
git branch -r           # branches remotas
git branch -a           # todas (locais + remotas)
git branch -v           # mostra último commit de cada branch

# ── Criar branch ───────────────────────────────────────────────
git branch feature/login           # cria mas NÃO muda para ela
git checkout -b feature/login      # cria E muda — forma clássica
git switch -c feature/login        # cria E muda — forma moderna (Git 2.23+)

# ── Navegar entre branches ────────────────────────────────────
git checkout main                  # muda para a branch main (clássico)
git switch main                    # muda para main (moderno)
git switch -                       # volta para a branch anterior (útil!)

# ── Renomear branch ───────────────────────────────────────────
git branch -m nome-antigo nome-novo   # renomeia branch local
git branch -m novo-nome               # renomeia a branch atual

# ── Deletar branch ────────────────────────────────────────────
git branch -d feature/login        # deleta (só se já foi merged — seguro)
git branch -D feature/login        # força deleção (mesmo sem merge — cuidado!)

# ── Ver relação entre branches ────────────────────────────────
git log --oneline --graph --all    # visualiza todas as branches no terminal
git branch --merged                # branches já mergeadas em main (podem ser deletadas)
git branch --no-merged             # branches ainda não mergeadas

Merge — unindo históricos

Merge integra o histórico de uma branch em outra. Existem dois tipos principais, e o Git escolhe automaticamente o mais adequado — mas é importante entender a diferença.

Fast-Forward Merge

Quando a branch de destino não teve commits novos desde a criação da feature branch. Git simplesmente move o ponteiro para frente — sem criar commit de merge. Histórico linear e limpo.

main ──→ A → B → C
feature ────→ A → B → C → D → E
após merge: main ────→ ... → D → E

3-Way Merge

Quando ambas as branches tiveram commits. Git encontra o ancestral comum e cria um merge commit que une os dois históricos. Preserva o contexto de quando a feature foi desenvolvida.

main ──→ A → B → C → F (merge)
feature ──→ A → D → E ─────→ F
// fast-forward vs 3-way merge
gitGraph commit id: "A: init" commit id: "B: auth base" branch feature/perfil checkout feature/perfil commit id: "C: página perfil" commit id: "D: upload avatar" checkout main commit id: "E: fix hotfix" merge feature/perfil id: "F: merge perfil" type: HIGHLIGHT commit id: "G: continua..."
// fluxo completo de merge
# ── Fluxo padrão de merge ────────────────────────────────────
# 1. Garanta que main está atualizado
git switch main
git pull origin main          # puxa mudanças remotas (veremos na aula 03)

# 2. Crie e trabalhe na feature branch
git switch -c feature/pagamento
# ... faz commits ...

# 3. Volte para main e faça o merge
git switch main
git merge feature/pagamento

# ── Opções importantes de merge ──────────────────────────────
git merge feature/x --no-ff        # FORÇA criação de merge commit
                                   # (mesmo se fosse fast-forward)
                                   # Preserva a história da feature

git merge feature/x --squash       # Combina todos os commits da feature
                                   # em um único commit no main
                                   # (não faz o commit automaticamente)

git merge feature/x --abort        # Aborta um merge em andamento
                                   # (útil quando conflitos são complexos)

# ── Ver diferenças antes de mergear ─────────────────────────
git diff main..feature/x          # o que a feature tem que main não tem
git log main..feature/x --oneline # commits que serão mergeados
🔥 Pro tip: --no-ff para preservar contexto
Em times profissionais, é comum usar --no-ff (no fast-forward) ao mergear feature branches para sempre criar um merge commit. Isso deixa claro no histórico quais commits fazem parte de uma feature, permitindo reverter uma feature inteira com um único git revert.
// CONCEITO 4

Conflitos de Merge — entendendo e resolvendo

Um conflito acontece quando duas branches modificam exatamente as mesmas linhas de um arquivo, e o Git não consegue decidir automaticamente qual versão é a correta. Ele para e pede que você decida.

Conflitos não são erros — são situações normais que o Git sinaliza para intervenção humana. Com calma e método, qualquer conflito é resolvível.

Como um conflito aparece no arquivo

<<<<<<< HEAD
def calcular_desconto(preco, percentual):
return preco - (preco * percentual / 100)
======= (separador entre as duas versões)
def calcular_desconto(preco, percentual, maximo=None):
desconto = preco * percentual / 100
if maximo: desconto = min(desconto, maximo)
return preco - desconto
>>>>>>> feature/desconto-maximo

Os marcadores significam:

MarcadorSignificado
<<<<<<< HEADInício da versão da branch atual (onde você está)
=======Separador entre as duas versões
>>>>>>> feature/xInício da versão que veio via merge
// passo a passo para resolver conflitos
# ── 1. O merge gera conflito ──────────────────────────────────
git switch main
git merge feature/desconto-maximo
# Auto-merging calculadora.py
# CONFLICT (content): Merge conflict in calculadora.py
# Automatic merge failed; fix conflicts and then commit the result.

# ── 2. Ver quais arquivos têm conflito ────────────────────────
git status
# Unmerged paths:
#   both modified:   calculadora.py

# ── 3. Abrir o arquivo e decidir o que manter ─────────────────
# Opções:
#   a) Manter só a versão do HEAD (sua branch)
#   b) Manter só a versão que veio (theirs)
#   c) Combinar as duas versões manualmente — mais comum!

# Após editar, o arquivo deve ficar SEM os marcadores:
# def calcular_desconto(preco, percentual, maximo=None):
#     desconto = preco * percentual / 100
#     if maximo: desconto = min(desconto, maximo)
#     return preco - desconto

# ── 4. Marcar como resolvido e commitar ───────────────────────
git add calculadora.py         # marca como resolvido
git status                     # confirma que não há mais conflitos
git commit                     # cria o merge commit (mensagem pré-preenchida)

# ── 5. Se decidir abortar o merge ────────────────────────────
git merge --abort              # volta ao estado antes do merge

Resolvendo conflitos no VS Code

VS Code tem uma interface visual excelente para resolver conflitos. Ao abrir um arquivo com conflito, ele mostra botões contextuais sobre cada bloco:

Ctrl/Cmd + Shift + P → "Git: Open Merge Editor" abre uma view de 3 painéis: Current (HEAD), Incoming (theirs) e Result (o que você decide). Muito mais confortável para conflitos complexos do que editar o arquivo diretamente.
// CONCEITO 5

Rebase — histórico linear e limpo

git rebase é uma alternativa ao merge que reescreve o histórico para manter uma linha do tempo linear. Em vez de criar um merge commit, ele "replanta" os commits da feature branch no topo da branch de destino — como se você tivesse começado a trabalhar na feature a partir do estado mais recente do main.

// merge vs rebase — diferença visual
gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C: minha feat" commit id: "D: ajustes" checkout main commit id: "E: outro fix" merge feature id: "F: MERGE commit" type: HIGHLIGHT tag: "com merge"
// rebase — quando e como usar
# ── Rebase básico ────────────────────────────────────────────
# Cenário: você está em feature/login e main teve novos commits
git switch feature/login
git rebase main
# Git "transplanta" seus commits no topo do main atualizado
# Resultado: histórico linear, sem merge commit

# ── Rebase interativo — o mais poderoso ──────────────────────
git rebase -i HEAD~3    # edita os últimos 3 commits interativamente
git rebase -i main      # edita commits desde que divergiu do main

# No editor, cada commit pode receber uma ação:
# pick   = manter como está
# reword = manter, mas editar a mensagem
# edit   = pausar para editar o commit
# squash = fundir com o commit anterior (combina)
# fixup  = como squash, mas descarta a mensagem
# drop   = REMOVE o commit do histórico

# Exemplo prático: squash de 3 commits "WIP" em um commit limpo
# pick abc123 feat: estrutura inicial
# squash def456 WIP: mais stuff
# squash ghi789 WIP: continuando
# → resulta em um único commit com mensagem editada

# ── Resolução de conflitos durante rebase ────────────────────
# Se houver conflito durante rebase:
git status                # ver conflitos
# ... resolve conflitos manualmente ...
git add arquivo-resolvido
git rebase --continue     # continua o rebase
# ou
git rebase --abort        # cancela e volta ao estado original
🚨 A regra de ouro do rebase
NUNCA faça rebase de commits que já foram pushed para uma branch compartilhada. Rebase reescreve o histórico (gera novos hashes). Se outros devs têm esses commits, seus históricos ficarão incompatíveis e você vai criar um caos no repositório remoto. Rebase é seguro apenas em branches locais que ainda não foram compartilhadas.
// CONCEITO 6

Stash & Reset — ferramentas de salvamento

git stash — guardar trabalho temporariamente

git stash salva suas mudanças não commitadas em uma "pilha" temporária e restaura o working directory para o estado do último commit. Perfeito quando você precisa trocar de branch urgentemente sem commitar trabalho incompleto.

// git stash completo
# ── Guardar mudanças ─────────────────────────────────────────
git stash                          # guarda tracked files modificados
git stash push -m "WIP: login form" # com mensagem descritiva
git stash push --include-untracked  # inclui arquivos não rastreados (novos)
git stash push -u                   # atalho para --include-untracked

# ── Ver o que está guardado ───────────────────────────────────
git stash list
# stash@{0}: WIP on feature/login: abc1234 feat: form base
# stash@{1}: On main: fix temporário

# ── Recuperar mudanças ────────────────────────────────────────
git stash pop                      # aplica o mais recente E remove da pilha
git stash apply                    # aplica mas MANTÉM na pilha (útil para aplicar em várias branches)
git stash apply stash@{1}          # aplica um stash específico

# ── Gerenciar stashes ─────────────────────────────────────────
git stash drop stash@{0}           # deleta um stash específico
git stash clear                    # limpa TODOS os stashes
git stash show -p stash@{0}        # mostra o diff de um stash

git reset — desfazendo commits

git reset move o ponteiro HEAD (e da branch) para um commit anterior. Tem três modos com comportamentos muito diferentes:

ModoHistóricoStagingWorking DirQuando usar
--soft Move HEAD Mantém Mantém Desfazer commits mantendo mudanças no staging (para reformular)
--mixed (padrão) Move HEAD Limpa Mantém Desfazer commits + unstage, mas mantém as edições nos arquivos
--hard Move HEAD Limpa Limpa ⚠️ Descarta completamente os commits e todas as mudanças
// git reset na prática
# ── --soft: desfaz commit, mantém tudo staged ────────────────
git reset --soft HEAD~1    # desfaz o último commit
git reset --soft HEAD~3    # desfaz os últimos 3 commits
# Os arquivos ainda estão no staging → você pode reformular e recomitar

# ── --mixed (padrão): desfaz commit e o staging ──────────────
git reset HEAD~1           # equivale a --mixed
git reset --mixed HEAD~2   # desfaz 2 commits, arquivos voltam a "modified"

# ── --hard: desfaz tudo — CUIDADO ────────────────────────────
git reset --hard HEAD~1    # ⚠️ PERDE as mudanças dos commits desfeitos
git reset --hard abc1234   # vai para um commit específico, descarta tudo depois
git reset --hard origin/main  # reseta para igual ao remoto (usado para sincronizar)

# ── git revert: alternativa segura para histórico público ─────
# reset reescreve histórico — problemático em branches compartilhadas
# revert cria um NOVO commit que desfaz as mudanças — seguro em produção
git revert HEAD            # cria commit que desfaz o último
git revert abc1234         # cria commit que desfaz um commit específico
git revert HEAD~3..HEAD    # reverte os últimos 3 commits (um por um)
git revert HEAD~3..HEAD --no-commit  # reverte tudo de uma vez, sem commitar ainda
⚠️ reset vs revert — qual usar?
Use reset apenas em commits locais (que não foram pushed). Use revert para desfazer commits em branches compartilhadas ou em main — ele adiciona ao histórico em vez de apagar, mantendo o histórico íntegro para todos.

⚠️ Armadilhas Comuns

1. Fazer rebase em branch compartilhada (pushed)

Rebase reescreve hashes de commits. Se alguém já fez pull da sua branch, os históricos ficam incompatíveis e você vai forçar um git push --force que pode apagar o trabalho de outras pessoas. Use rebase apenas em branches locais.

2. git reset --hard sem backup

--hard descarta mudanças permanentemente do working directory. Se você não tinha commitado nem feito stash, as mudanças se perdem. Sempre verifique git status antes e, se precisar de segurança extra, crie uma branch temporária antes do reset.

3. Deixar marcadores de conflito no código

Após resolver conflitos, é fácil esquecer algum bloco <<<<<<< ou ======= no arquivo. Esses marcadores quebram o código. Após resolver, sempre rode o projeto e faça uma busca por <<<<< no codebase antes de commitar.

4. Trabalhar direto no main por tempo demais

O main deve sempre refletir código estável e funcional. Nunca desenvolva features diretamente nele. Crie uma branch para cada tarefa, por menor que seja. Isso também facilita code review via pull requests.

5. Acumular branches velhas no repositório

Branches mergeadas devem ser deletadas. Um repo com dezenas de branches abandonadas é difícil de navegar. Use git branch --merged | grep -v main | xargs git branch -d para limpar branches locais já mergeadas de uma vez.

// GLOSSÁRIO

Termos desta Aula

Branch
Ponteiro leve para um commit. Permite linhas de desenvolvimento paralelas e independentes no mesmo repositório.
HEAD
Ponteiro especial que indica o commit atual. Normalmente aponta para o topo de uma branch; em "detached HEAD" aponta diretamente para um commit.
Fast-Forward Merge
Merge sem commit extra, quando a branch de destino não divergiu. Git apenas move o ponteiro para frente.
3-Way Merge
Merge que cria um commit extra unindo dois históricos divergentes, usando o ancestral comum como base de comparação.
Merge Conflict
Situação onde duas branches modificaram as mesmas linhas de um arquivo e o Git precisa de intervenção humana para decidir a versão final.
Rebase
Reescreve o histórico "transplantando" commits de uma branch no topo de outra. Cria histórico linear mas altera hashes dos commits.
Stash
Pilha temporária para guardar mudanças não commitadas sem criar um commit. Permite trocar de branch com trabalho em andamento.
git revert
Cria um novo commit que desfaz as mudanças de um commit anterior. Seguro para branches compartilhadas — não reescreve histórico.

🏁 Checkpoint — Desafio Final

Simule um fluxo de desenvolvimento real com múltiplas branches, merge com conflito e limpeza de histórico:

  • Crie um repo com pelo menos 3 commits na main (arquivo Python com funções)
  • Crie feature/funcionalidade-a e feature/funcionalidade-b — ambas modificando a mesma função no arquivo principal de formas diferentes
  • Mergee a feature A em main sem conflito (ela foi criada antes de qualquer divergência)
  • Tente mergear a feature B — resolva o conflito deliberadamente combinando as duas versões de forma inteligente
  • Use git rebase -i para squash dos commits da feature B em 1 único commit limpo antes do merge
  • Ao final: rode git log --oneline --graph --all, delete branches mergeadas e capture o histórico organizado
💬 Dica: antes de fazer qualquer reset ou rebase, crie uma branch de backup com git branch backup/antes-do-rebase. Se algo der errado, você pode recuperar com git reset --hard backup/antes-do-rebase. Esta é uma prática de sênior.
