01 · Motivação

Por que integrar APIs de LLM diretamente?

ChatGPT e Claude.ai são interfaces para usuários finais. Um engenheiro sênior vai além: ele integra capacidades de IA diretamente nas suas aplicações via API — dando controle total sobre modelo, contexto, temperatura, custo e comportamento.

Seja um pipeline de processamento de documentos, um chatbot corporativo, um sistema de análise de código ou um agente autônomo, tudo começa com uma chamada HTTP para um dos três grandes provedores: OpenAI, Anthropic e Google Gemini.

💼 Por que um engenheiro sênior usa APIs de LLM em produção hoje

Três razões principais movem equipes técnicas a integrar LLMs via API:

  • Automação de tarefas cognitivas — classificação, extração, sumarização, geração de código: tarefas que antes exigiam humanos agora rodam em lote com custo de centavos.
  • Produto diferenciado — features como "explique esse erro", "gere um relatório personalizado" ou "resuma este PDF" elevam o valor percebido do produto sem meses de ML custom.
  • Velocidade de experimentação — trocar o modelo subjacente (GPT-4o → Claude Opus → Gemini) é questão de mudar uma string, permitindo A/B tests de qualidade e custo em horas.
🎯

Controle total

Você define system prompt, temperatura, max tokens, stop sequences, seed para reprodutibilidade — nenhuma UI faz isso.

💰

Custo previsível

Pague por token consumido. Com caching e escolha de modelo certo, o custo real de produção é 10–100× menor que parece.

🔒

Privacidade e dados

APIs enterprise têm acordos de não-treinamento com seus dados. Vital para projetos corporativos com dados sensíveis.

🔌

Integração nativa

Encadeie LLMs com databases, filas, webhooks, outros microserviços — impossível via interface web.

02 · Analogia

Especialistas por hora vs. funcionário fixo

🌍 Mundo Real → Tecnologia
Imagine que você precisa de análises jurídicas, traduções e código revisado para o seu projeto. Você tem duas opções: contratar um funcionário fixo para cada área (caro, lento, trabalhista) ou contratar especialistas por hora via plataforma — você envia a tarefa, eles entregam em segundos, você paga pelo esforço exato.

APIs de LLM são exatamente essa plataforma. OpenAI, Anthropic e Gemini são agências com especialistas diferentes — você não precisa "possuir" a inteligência, apenas consumi-la sob demanda. A diferença entre os provedores é como escolher entre advogados: cada um tem seu estilo, especialidade e precificação. Seu trabalho como engenheiro é saber quando chamar qual especialista, como formular o briefing (prompt), e como gerenciar o custo.

03 · Arquitetura

Como uma chamada de API funciona de ponta a ponta

Antes de ver código, entenda o fluxo completo. Toda chamada a uma API de LLM segue o mesmo padrão: sua aplicação usa um SDK para serializar a requisição, enviar via HTTPS com autenticação, e receber a resposta — seja completa (JSON) ou em chunks (streaming SSE).

Arquitetura de chamada — request/response e streaming

(JSON + Bearer API Key) API->>Model: Tokenize + Forward pass Model-->>API: Generated tokens (stream) alt Modo Streaming API-->>SDK: SSE chunk 1: {"delta": "Ol"} API-->>SDK: SSE chunk 2: {"delta": "á, "} API-->>SDK: SSE chunk N: {"delta": "mundo"} API-->>SDK: SSE: [DONE] SDK-->>App: Iterador de chunks else Modo Normal API-->>SDK: JSON completo: {"choices": [...]} SDK-->>App: ChatCompletion object end App->>App: response.choices[0].message.content
🔍 Conceitos-chave de tokenização

Um token é aproximadamente 4 caracteres em inglês (3–4 em português). "Hello world" = 2 tokens. "Olá, como vai você?" ≈ 7 tokens. A contagem de tokens determina o custo e os limites de contexto. Use o Tokenizer da OpenAI para estimar. Regra prática: 1.000 tokens ≈ 750 palavras ≈ 1,5 páginas A4.

04 · OpenAI

OpenAI API — Chat Completions

O
OpenAI
GPT-4o · gpt-4o-mini · o1 · text-embedding-3 · dall-e-3

A OpenAI popularizou o formato de Chat Completions — uma lista de mensagens com roles (system, user, assistant) que representa o histórico da conversa. O modelo gera a próxima mensagem do assistente com base nesse contexto.

Instalação e configuração

1
Instalar o SDK
A biblioteca oficial openai é mantida pela própria OpenAI com suporte a Python 3.8+.
pip install openai python-dotenv
2
Criar arquivo .env com a API key
Nunca coloque chaves no código. Use variáveis de ambiente — mais sobre isso na seção de segurança.
# .env (adicione ao .gitignore!)
OPENAI_API_KEY=sk-proj-...

Chat Completions — exemplo básico completo

O endpoint /v1/chat/completions é o coração da API. O parâmetro messages é uma lista de turnos de conversa. O system define a persona e as regras do assistente.

import os
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "Você é um assistente técnico especializado em Python."},
        {"role": "user", "content": "Explique o que é um decorator em Python."}
    ],
    temperature=0.7,   # 0 = determinístico, 2 = caótico; 0.7 é o equilíbrio padrão
    max_tokens=500     # limite de tokens no output; sem isso o modelo pode gerar muito
)

# Acessar o conteúdo gerado
print(response.choices[0].message.content)

# Monitorar uso de tokens (vital para controle de custo)
print(f"Tokens usados: {response.usage.total_tokens}")
print(f"  Input:  {response.usage.prompt_tokens}")
print(f"  Output: {response.usage.completion_tokens}")
🔥 Pro tip: multi-turn conversation
Para manter contexto entre mensagens, accumule o histórico manualmente — a API é stateless. Cada chamada recebe todo o histórico do zero. 
messages = [{"role": "system", "content": "Você é um assistente Python."}]

def chat(user_input: str) -> str:
    messages.append({"role": "user", "content": user_input})
    response = client.chat.completions.create(model="gpt-4o", messages=messages)
    reply = response.choices[0].message.content
    messages.append({"role": "assistant", "content": reply})  # guardar para próxima chamada
    return reply

print(chat("O que é uma list comprehension?"))
print(chat("Me dê um exemplo com filtro."))

Streaming — output em tempo real

Sem streaming, o usuário espera o modelo gerar tudo antes de ver qualquer coisa. Com stream=True, cada token chega assim que é gerado — essencial para UX de chatbots.

stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Escreva um poema sobre código limpo."}],
    stream=True  # ativa SSE (Server-Sent Events)
)

# Cada chunk chega assim que o modelo gera o próximo pedaço de texto
for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)
        # flush=True força a impressão imediata — importante em terminais e UIs
print()  # nova linha ao final

Function Calling — integre o LLM com seu código

Function Calling permite que o modelo decida chamar uma função do seu sistema ao invés de responder diretamente. O modelo não executa a função — ele retorna um JSON estruturado com o nome e os argumentos, e você executa.

tools = [
    {
        "type": "function",
        "function": {
            "name": "buscar_clima",
            "description": "Obtém o clima atual de uma cidade",
            "parameters": {
                "type": "object",
                "properties": {
                    "cidade": {"type": "string", "description": "Nome da cidade"},
                    "unidade": {"type": "string", "enum": ["celsius", "fahrenheit"]}
                },
                "required": ["cidade"]
            }
        }
    }
]

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Qual o clima em São Paulo?"}],
    tools=tools,
    tool_choice="auto"  # "auto" = modelo decide; "required" = sempre usa ferramenta
)

# Checar se o modelo quer chamar uma ferramenta
message = response.choices[0].message
if message.tool_calls:
    tool_call = message.tool_calls[0]
    function_name = tool_call.function.name    # "buscar_clima"
    arguments = json.loads(tool_call.function.arguments)  # {"cidade": "São Paulo"}

    # Executar a função real do seu sistema
    resultado = buscar_clima(**arguments)

    # Enviar o resultado de volta ao modelo para que ele formule a resposta final
    messages = [
        {"role": "user", "content": "Qual o clima em São Paulo?"},
        message,  # mensagem do assistente com o tool_call
        {
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": str(resultado)
        }
    ]
    final_response = client.chat.completions.create(model="gpt-4o", messages=messages)
    print(final_response.choices[0].message.content)
⚠️ Function Calling não executa código
O modelo apenas decide que quer chamar a função e retorna os argumentos em JSON. Você é responsável por executar, tratar erros e retornar o resultado. Isso é intencional — mantém você no controle total das operações.
05 · Anthropic

Anthropic API — Messages com Claude

A
Anthropic — Claude
claude-opus-4-5 · claude-sonnet-4-5 · claude-haiku-3-5

A Anthropic usa a Messages API — ligeiramente diferente da OpenAI. O system é um parâmetro de topo (não uma mensagem), e o conteúdo pode ser uma lista de blocos tipados. Claude se destaca em contextos longos (200K tokens), raciocínio e seguimento preciso de instruções.

Instalação

pip install anthropic python-dotenv

Messages API — exemplo completo

Note as diferenças em relação à OpenAI: system é um parâmetro separado, não uma mensagem; max_tokens é obrigatório; e o conteúdo retorna como lista de blocos de conteúdo.

import anthropic
import os

client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

message = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,           # obrigatório na API Anthropic — sem valor padrão
    system="Você é um especialista em arquitetura de software.",  # system separado
    messages=[
        {"role": "user", "content": "Quais são os princípios SOLID?"}
    ]
)

# Conteúdo é uma lista de blocos — para texto, acessar o primeiro
print(message.content[0].text)

# Monitorar uso de tokens
print(f"Input tokens:  {message.usage.input_tokens}")
print(f"Output tokens: {message.usage.output_tokens}")
print(f"Stop reason:   {message.stop_reason}")  # "end_turn" | "max_tokens" | "stop_sequence"

Turnos multi-round — user e assistant alternados

# A Anthropic exige que mensagens alternem entre "user" e "assistant"
# O primeiro e último devem ser "user"
message = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    system="Você é um professor de Python paciente e detalhado.",
    messages=[
        {"role": "user",      "content": "O que é uma classe em Python?"},
        {"role": "assistant", "content": "Uma classe é um template para criar objetos..."},
        {"role": "user",      "content": "Pode mostrar um exemplo com herança?"}
    ]
)

Streaming com context manager

A Anthropic oferece streaming via context manager with client.messages.stream() — mais ergonômico que o loop manual da OpenAI.

with client.messages.stream(
    model="claude-opus-4-5",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Liste 5 padrões de design essenciais."}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
    print()

    # Após o stream, metadados ficam disponíveis
    final_message = stream.get_final_message()
    print(f"\nTokens: {final_message.usage.input_tokens} in / {final_message.usage.output_tokens} out")

Tool Use (equivalente ao Function Calling)

tools = [
    {
        "name": "buscar_documentacao",
        "description": "Busca na documentação oficial do Python por um termo",
        "input_schema": {
            "type": "object",
            "properties": {
                "termo": {
                    "type": "string",
                    "description": "Termo a buscar na documentação"
                }
            },
            "required": ["termo"]
        }
    }
]

response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "Como funciona o itertools.groupby?"}]
)

# Verificar se o modelo quer usar uma ferramenta
if response.stop_reason == "tool_use":
    for block in response.content:
        if block.type == "tool_use":
            print(f"Ferramenta: {block.name}")
            print(f"Input: {block.input}")  # {"termo": "itertools.groupby"}
🔥 Claude para contextos longos
Claude Opus 4.5 suporta até 200K tokens de contexto. Isso equivale a ~150.000 palavras — praticamente um romance inteiro. Ideal para análise de documentos jurídicos, bases de código completas, ou históricos longos de conversa.
06 · Google Gemini

Google Gemini API — Texto e Multimodal

G
Google Gemini
gemini-1.5-pro · gemini-1.5-flash · gemini-2.0-flash · gemini-2.5-pro

O Gemini da Google é o único dos três com suporte nativo a contexto de 1 milhão de tokens e input multimodal que inclui vídeo e áudio além de imagens. O SDK google-generativeai tem uma interface mais simples que os concorrentes para casos de uso básicos.

Instalação

pip install google-generativeai python-dotenv

Geração de texto — exemplo básico

import google.generativeai as genai
import os

genai.configure(api_key=os.environ["GEMINI_API_KEY"])

# Instanciar o modelo (pode ser reutilizado)
model = genai.GenerativeModel("gemini-1.5-pro")

response = model.generate_content("Explique machine learning em 3 parágrafos.")
print(response.text)

# Informações sobre o uso
print(f"Candidatos: {len(response.candidates)}")
print(f"Tokens prompt: {response.usage_metadata.prompt_token_count}")
print(f"Tokens output: {response.usage_metadata.candidates_token_count}")

Multi-turn com histórico — ChatSession

# Gemini tem uma abstração de chat com histórico gerenciado automaticamente
chat = model.start_chat(history=[])

response1 = chat.send_message("Quais são os tipos de ML?")
print(response1.text)

response2 = chat.send_message("Explique mais sobre aprendizado por reforço.")
print(response2.text)

# O histórico é mantido automaticamente na sessão
print(f"\nHistórico com {len(chat.history)} mensagens")

Multimodal — texto + imagem

O Gemini pode analisar imagens diretamente junto com texto. Útil para análise de screenshots, gráficos, documentos escaneados e qualquer caso que combine visão e linguagem.

import PIL.Image
import google.generativeai as genai

genai.configure(api_key=os.environ["GEMINI_API_KEY"])
model = genai.GenerativeModel("gemini-1.5-pro")

# Carregar uma imagem local
imagem = PIL.Image.open("grafico_vendas.png")

# O prompt pode combinar texto e imagem na mesma chamada
response = model.generate_content([
    "Analise este gráfico de vendas e identifique:",
    "1. A tendência principal",
    "2. Picos ou quedas significativos",
    "3. Uma recomendação de ação",
    imagem  # a imagem vai junto no mesmo array
])

print(response.text)

# Com URL remota (sem baixar o arquivo)
import httpx
image_url = "https://example.com/chart.png"
image_data = httpx.get(image_url).content

response = model.generate_content([
    "Descreva esta imagem em detalhes:",
    {"mime_type": "image/png", "data": image_data}
])

08 · Segurança

Autenticação segura — nunca hardcode uma API key

Esse ponto parece óbvio, mas chaves de API da OpenAI com billing ativo vazadas em repositórios públicos geram contas de milhares de dólares em horas. Isso acontece com frequência — GitHub scans detectam e bots exploram automaticamente.

🚫 NUNCA faça isso
# ❌ ERRADO — hardcode direto no código
client = OpenAI(api_key="sk-proj-abc123xyz...")

# ❌ ERRADO — variável no topo do arquivo commitada
API_KEY = "sk-proj-abc123xyz..."
client = OpenAI(api_key=API_KEY)

# ❌ ERRADO — em arquivo de config commitado
# config.json: {"openai_key": "sk-proj-..."}

A abordagem correta: variáveis de ambiente + .env

# Passo 1: Crie o arquivo .env na raiz do projeto
# .env (NUNCA commitar — adicione ao .gitignore!)
# OPENAI_API_KEY=sk-proj-...
# ANTHROPIC_API_KEY=sk-ant-api03-...
# GEMINI_API_KEY=AIzaSy...

# Passo 2: Adicione .env ao .gitignore
# echo ".env" >> .gitignore

# Passo 3: No código Python, use python-dotenv
from dotenv import load_dotenv
import os

load_dotenv()  # carrega automaticamente o .env do diretório atual

# Usar os.environ com colchetes — falha explicitamente se a variável não existir
# Isso é fail-fast intencional: melhor descobrir na inicialização do que em runtime
openai_key = os.environ["OPENAI_API_KEY"]
anthropic_key = os.environ["ANTHROPIC_API_KEY"]
gemini_key = os.environ["GEMINI_API_KEY"]

# Alternativa: os.getenv com valor padrão — mas cuidado, pode esconder erros
openai_key = os.getenv("OPENAI_API_KEY", "")
if not openai_key:
    raise ValueError("OPENAI_API_KEY não definida! Verifique o .env")

Em produção: serviços de secret management

1
AWS Secrets Manager / SSM Parameter Store
Para aplicações em AWS — chaves injetadas na runtime do container, nunca em variáveis de ambiente hardcoded.
2
GitHub Actions Secrets

09 · Resiliência

Rate limiting e retry com exponential backoff

APIs de LLM têm limites de requisições por minuto (RPM) e tokens por minuto (TPM). Em produção, você vai atingir esses limites. A solução padrão é exponential backoff com jitter — esperar cada vez mais tempo entre tentativas, com um delay aleatório para evitar thundering herd.

Exponential backoff com jitter
graph LR REQ[📤 Requisição] --> TRY1[Tentativa 1] TRY1 -- 429 Rate Limit --> W1[⏳ Esperar 4s + jitter] W1 --> TRY2[Tentativa 2] TRY2 -- 500 Server Error --> W2[⏳ Esperar 8s + jitter] W2 --> TRY3[Tentativa 3] TRY3 -- ✅ 200 OK --> RESP[📥 Resposta] TRY3 -- Falha --> W3[⏳ Esperar 16s + jitter] W3 --> TRY4[Tentativa 4] TRY4 -- ✅ 200 OK --> RESP style RESP fill:#10b981,color:#000 style W1 fill:#f59e0b,color:#000 style W2 fill:#f59e0b,color:#000 style W3 fill:#ef4444,color:#fff

Implementação com tenacity

from tenacity import (
    retry,
    stop_after_attempt,
    wait_exponential,
    retry_if_exception_type
)
from openai import RateLimitError, APIError
import logging

logger = logging.getLogger(__name__)

@retry(
    # Quais exceções devem disparar o retry (não tente em erros de autenticação!)
    retry=retry_if_exception_type((RateLimitError, APIError)),
    # Esperar: min 4s, max 60s, multiplicando por 2 a cada tentativa
    wait=wait_exponential(multiplier=1, min=4, max=60),
    # Desistir após 5 tentativas
    stop=stop_after_attempt(5),
    # Log de cada tentativa (útil para debugging em produção)
    before_sleep=lambda retry_state: logger.warning(
        f"Tentativa {retry_state.attempt_number} falhou, aguardando {retry_state.next_action.sleep:.1f}s..."
    )
)
def chamar_llm(prompt: str, model: str = "gpt-4o-mini") -> str:
    """
    Chama a API da OpenAI com retry automático em caso de rate limit ou erro.
    Não faz retry em: AuthenticationError, BadRequestError (erros do cliente).
    """
    response = client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content


# Versão assíncrona para aplicações com asyncio
from tenacity import AsyncRetrying

async def chamar_llm_async(prompt: str) -> str:
    async for attempt in AsyncRetrying(
        wait=wait_exponential(min=2, max=30),
        stop=stop_after_attempt(4)
    ):
        with attempt:
            response = await async_client.chat.completions.create(
                model="gpt-4o-mini",
                messages=[{"role": "user", "content": prompt}]
            )
            return response.choices[0].message.content
⚠️ Não faça retry em todos os erros
  • RateLimitError (429) — retry faz sentido, espere e tente de novo
  • APIError (5xx) — erro do servidor, retry pode resolver
  • AuthenticationError (401) — chave inválida, retry não adianta
  • BadRequestError (400) — seu prompt tem problema, retry sem mudança é inútil
  • PermissionDeniedError (403) — sem acesso ao modelo, retry não resolve
10 · Avaliação

Avaliando a qualidade das respostas — LLM-as-Judge

Como você sabe se suas respostas são boas? Métricas clássicas de NLP (BLEU, ROUGE) são ruins para LLMs — textos com mesmas informações têm pontuações baixas por diferença de vocabulário. A abordagem moderna é LLM-as-Judge: usar outro LLM (ou o mesmo) para avaliar as respostas.

G-Eval — avaliação estruturada por critérios

def avaliar_resposta_geval(
    pergunta: str,
    resposta: str,
    criterios: list[str] = None
) -> dict:
    """
    G-Eval: usa um LLM para avaliar a qualidade de uma resposta em múltiplos critérios.
    Retorna scores 1-5 para cada critério.
    """
    if criterios is None:
        criterios = ["Relevância", "Precisão factual", "Clareza", "Completude"]

    criterios_texto = "\n".join(f"- {c}" for c in criterios)

    prompt = f"""Você é um avaliador especializado. Avalie a resposta abaixo segundo cada critério.

Pergunta: {pergunta}

Resposta avaliada:
{resposta}

Para cada critério, atribua uma nota de 1 a 5 onde:
1 = Muito ruim | 3 = Aceitável | 5 = Excelente

Critérios:
{criterios_texto}

Responda APENAS em JSON:
{{"scores": {{"Relevância": 4, "Precisão factual": 5, ...}}, "feedback": "..."}}"""

    resultado = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"}
    )

    import json
    return json.loads(resultado.choices[0].message.content)


# Uso:
avaliacao = avaliar_resposta_geval(
    pergunta="O que é um decorator em Python?",
    resposta="Um decorator é uma função que modifica outra função..."
)
print(avaliacao)
# {"scores": {"Relevância": 5, "Precisão factual": 4, "Clareza": 4, "Completude": 3},
#  "feedback": "Resposta relevante e precisa, mas poderia incluir exemplos."}

Estratégia 1: Prompt Compression

def comprimir_prompt(texto: str, max_chars: int = 2000) -> str:
    """
    Técnicas de compressão de prompt sem perder informação crítica.
    """
    # 1. Remover espaços duplos e quebras de linha desnecessárias
    import re
    texto = re.sub(r'\s+', ' ', texto).strip()

    # 2. Truncar documentos longos preservando início e fim (mais relevantes)
    if len(texto) > max_chars:
        meio = max_chars // 2
        texto = texto[:meio] + "\n...[conteúdo intermediário omitido]...\n" + texto[-meio:]

    return texto

# OpenAI Prompt Caching — automático para prompts > 1024 tokens
# Tokens em cache custam 50% menos

# Estruturar o prompt para maximizar cache hits:
# Coloque a parte estática (system prompt longo, documentos de referência)
# PRIMEIRO e a parte dinâmica (pergunta do usuário) POR ÚLTIMO

SYSTEM_PROMPT_LONGO = """
Você é um especialista em análise de contratos jurídicos brasileiros.
[... 2000 tokens de instruções detalhadas ...]
"""  # Esta parte vai para o cache automaticamente

def analisar_clausula(clausula: str) -> str:
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT_LONGO},  # cacheado após 1a chamada
            {"role": "user", "content": clausula}  # único que muda
        ]
    )
    # Verificar cache hits
    usage = response.usage
    if hasattr(usage, 'prompt_tokens_details'):
        cached = usage.prompt_tokens_details.cached_tokens
        print(f"Cache hit: {cached} tokens em cache (50% de desconto)")
    return response.choices[0].message.content


# Anthropic — cache explícito com cache_control
message = anthropic_client.messages.create(

            "text": SYSTEM_PROMPT_LONGO,
            "cache_control": {"type": "ephemeral"}  # cache por 5 minutos
        }
    ],
    messages=[{"role": "user", "content": clausula}]
)

Estratégia 3: Escolha dinâmica de modelo por complexidade

def escolher_modelo(tarefa: str, complexidade: str = "auto") -> str:
    """
    Roteamento inteligente de modelo baseado na complexidade da tarefa.
    """
    tarefas_simples = ["classificar", "extrair", "categorizar", "resumir brevemente"]
    tarefas_complexas = ["raciocinar", "planejar", "debugar", "arquitetar", "analisar profundamente"]

    if complexidade == "auto":
        tarefa_lower = tarefa.lower()
        if any(t in tarefa_lower for t in tarefas_simples):
            complexidade = "baixa"
        elif any(t in tarefa_lower for t in tarefas_complexas):
            complexidade = "alta"
        else:
            complexidade = "media"

    modelos = {
        "baixa":  "gpt-4o-mini",     # ~50x mais barato que GPT-4o
        "media":  "gpt-4o",
        "alta":   "gpt-4o"           # ou claude-opus para raciocínio avançado
    }

    return modelos[complexidade]

# Uso:
model = escolher_modelo("Classifique este email como spam ou não-spam")
# → "gpt-4o-mini" (tarefa simples)

13 · Armadilhas

Erros comuns ao usar APIs de LLM em produção

🚫 Não tratar o custo como métrica de produto

Equipes que não monitoram custo por requisição descobrem contas inesperadas. Instrumente cada chamada com o custo em tokens. Alerte quando superar um threshold (ex: > 2.000 tokens por operação quando o esperado é 500).

🚫 Assumir que o modelo sempre retorna JSON válido

Mesmo com response_format={"type": "json_object"}, o conteúdo pode ser JSON válido mas semanticamente errado. Sempre valide com Pydantic ou marshmallow.

from pydantic import BaseModel, ValidationError

class Resultado(BaseModel):
    score: int
    categoria: str

try:
    parsed = Resultado.model_validate_json(response_text)
except ValidationError as e:
    # Log o erro e a resposta original para debugging
    logger.error(f"JSON inválido do LLM: {e}\nResposta: {response_text}")
    # Fallback ou re-tentativa com prompt diferente
🚫 Context window overflow silencioso

Quando a conversa acumula mais tokens que o contexto suportado, a API silenciosamente trunca as mensagens mais antigas. Implemente gerenciamento explícito de contexto em chats longos — resuma ou comprima o histórico periodicamente.

🚫 Temperature 0 para criatividade, Temperature 1+ para extração

Use temperature=0 para extração de dados, classificação e geração de código — você quer determinismo. Use temperature=0.7–1.0 para escrita criativa, brainstorming e diversidade de respostas. A maioria dos bugs de "respostas inconsistentes" vem de temperature alta em tarefas estruturadas.

🏁 Checkpoint Final

Desafio: Abstração multi-provider com fallback automático

Construa uma classe LLMClient que abstraia OpenAI, Anthropic e Gemini com as seguintes funcionalidades:

  • Método generate(prompt, provider="openai") que funciona nos 3 provedores
  • Retry automático com tenacity em caso de rate limit
  • Fallback automático: se OpenAI falhar, tentar Anthropic; se falhar, tentar Gemini
  • Log de custo estimado em tokens para cada chamada
  • Autenticação via .env com falha explícita se a chave não existir
  • Bônus: Implement cache local com diskcache para evitar chamadas repetidas ao mesmo prompt
💬 Dica: Comece criando uma interface abstrata (classe base ou Protocol) com o método generate. Depois implemente uma subclasse por provedor. O fallback pode ser um decorator ou um wrapper que tenta cada provedor em ordem. Meça a diferença de latência e custo entre os provedores para o mesmo prompt.
