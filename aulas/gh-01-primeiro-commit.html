Git — Do zero ao primeiro commit

⏱ ~35 min de leitura 🎯 Fundamentos 🔧 Hands-on incluído
controle de versão git init git commit staging area git status git log git diff .gitignore
// motivação

Por que isso importa?

Git é a habilidade mais transversal da engenharia de software moderna. Não importa se você trabalha com frontend, backend, dados ou DevOps — todo time usa Git. É o sistema que permite que dezenas de pessoas trabalhem no mesmo projeto simultaneamente, sem destruir o trabalho uma da outra.

Mais do que uma ferramenta de backup, Git é uma máquina do tempo para código. Você pode voltar a qualquer ponto da história do projeto, comparar o que mudou, entender por que uma mudança foi feita semanas atrás, e reverter erros com segurança. Nenhum desenvolvedor profissional trabalha sem ele.

💼 Por que um engenheiro sênior domina Git profundamente
Sêniors usam Git como extensão do pensamento. Commits atômicos bem escritos formam uma documentação viva do projeto. Um git log bem estruturado conta a história de cada decisão técnica. Em code reviews, o histórico de commits revela o raciocínio por trás das mudanças. Git não é opcional — é o idioma universal do desenvolvimento colaborativo.
// analogia

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Imagine escrever um livro com co-autores. Sem Git, você teria um único documento Word compartilhado onde todo mundo edita ao mesmo tempo — um caos. Com Git, cada autor tem sua própria cópia completa do livro (repositório local), faz suas edições no rascunho (working directory), revisa o que vai incluir no próximo capítulo (staging area), e então "salva oficialmente" um capítulo finalizado (commit) com uma mensagem descritiva.

O Git guarda um snapshot completo de cada versão salva, com data, autor e descrição. A qualquer momento, qualquer co-autor pode ver todos os rascunhos históricos, comparar versões e entender exatamente o que mudou entre dois capítulos.
📖 Livro completo com histórico → Repositório Git ✏️ Rascunho em edição → Working Directory 📋 Capítulos prontos para publicar → Staging Area (Index) 📸 Capítulo finalizado e numerado → Commit (snapshot permanente)
// conceito 1

Git vs GitHub — não confunda

Essa confusão é universal entre iniciantes. São coisas completamente diferentes:

Git

Sistema de controle de versão

Software local, gratuito e open-source criado por Linus Torvalds em 2005. Roda na sua máquina. Funciona completamente offline. Gerencia o histórico de qualquer projeto.

GitHub

Plataforma de hospedagem

Serviço web (da Microsoft) que armazena repositórios Git na nuvem. Adiciona features de colaboração: pull requests, issues, Actions, etc. Existem alternativas: GitLab, Bitbucket.

💡 A relação entre os dois
Git é o motor. GitHub é a garagem na nuvem. Você usa Git localmente para gerenciar versões, e usa GitHub (ou outra plataforma) para compartilhar, colaborar e fazer backup na nuvem. Nesta aula foco em Git puro — GitHub vem na aula 03.

As três áreas do Git

Entender as três áreas é o insight mais importante desta aula. Todo arquivo em um repositório Git está em um desses três estados — e compreender as transições entre eles elimina a maioria das confusões de iniciantes.

Working Directory
— editar arquivo —
Onde você edita. Modificações ainda não rastreadas.
Staging Area
git add
Arquivos "preparados" para o próximo commit.
Repositório Local
git commit
Snapshots permanentes com hash único.
Repositório Remoto
git push
GitHub/GitLab — compartilhado com o time.
// ciclo de vida de um arquivo no git
stateDiagram-v2 [*] --> Untracked: novo arquivo criado Untracked --> Staged: git add Staged --> Committed: git commit Committed --> Modified: editar arquivo Modified --> Staged: git add Staged --> Modified: git restore --staged Modified --> Unmodified: git restore Committed --> Unmodified: sem mudanças Unmodified --> Modified: editar arquivo
🔥 Por que existe a Staging Area?
À primeira vista parece redundante — por que não ir direto de "editado" para "commitado"? A staging area permite selecionar exatamente o que vai no próximo commit. Você pode ter mudanças em 5 arquivos mas fazer um commit só com 2 deles, mantendo os outros para o próximo. Isso resulta em commits atômicos e histórico limpo.

Comandos core — do init ao commit

// fluxo completo comentado
# ── 1. Inicializar um repositório ───────────────────────────────
git init                    # cria a pasta .git/ — repositório vazio
git init meu-projeto        # cria a pasta e inicializa dentro

# ── 2. Ver o estado atual do repositório ─────────────────────────
git status                  # arquivos modificados, staged, untracked
git status -s               # versão compacta (short)
#  M = modified, A = added/staged, ?? = untracked, D = deleted

# ── 3. Rastrear e preparar arquivos (staging) ─────────────────────
git add arquivo.txt          # adiciona arquivo específico
git add src/                 # adiciona pasta inteira
git add *.py                 # adiciona todos os .py
git add .                    # adiciona TUDO na pasta atual
git add -p                   # modo interativo — escolhe pedaços (hunks)

# ── 4. Remover da staging area sem desfazer mudanças ─────────────
git restore --staged arquivo.txt   # retira do staging, mantém edições
git reset HEAD arquivo.txt         # alternativa mais antiga

# ── 5. Ver o que mudou ────────────────────────────────────────────
git diff                    # working directory vs staging (o que ainda não fiz add)
git diff --staged           # staging vs último commit (o que vai no próximo commit)
git diff HEAD               # working directory vs último commit (tudo)
git diff abc123 def456      # entre dois commits específicos

# ── 6. Criar um commit ──────────────────────────────────────────
git commit -m "feat: adiciona página de login"     # mensagem inline
git commit                                          # abre editor configurado
git commit -am "fix: corrige validação do email"   # add + commit (só modified, não untracked)

# ── 7. Verificar o histórico ──────────────────────────────────────
git log                     # log completo
git log --oneline           # uma linha por commit (muito útil)
git log --oneline --graph   # grafo de branches
git log --oneline -10       # últimos 10 commits
git log --author="Ana"      # commits de um autor específico
git log -- arquivo.txt      # histórico de um arquivo específico

# ── 8. Inspecionar um commit específico ──────────────────────────
git show abc1234            # detalhes + diff de um commit
git show HEAD               # detalhe do commit mais recente
git show HEAD~2             # dois commits antes do HEAD

# ── 9. Desfazer mudanças no working directory ─────────────────────
git restore arquivo.txt     # descarta edições não staged (CUIDADO: irreversível)
git restore .               # descarta TODAS as edições não staged

Anatomia de uma boa mensagem de commit

Mensagens de commit são documentação. Um histórico com mensagens descritivas vale mais do que comentários no código — explica o porquê, não apenas o quê.

// conventional commits — padrão da indústria
# Formato: tipo(escopo): descrição
# Linha em branco
# Corpo opcional (mais detalhes)
# Linha em branco
# Rodapé opcional (BREAKING CHANGE, refs, etc.)

# ── Tipos principais ──────────────────────────────────────────────
feat:     # nova funcionalidade
fix:      # correção de bug
docs:     # mudança na documentação
style:    # formatação, sem mudança de lógica
refactor: # refatoração sem nova feature nem fix
test:     # adiciona ou corrige testes
chore:    # tarefas de manutenção, deps, build

# ── Exemplos RUINS (evite) ────────────────────────────────────────
git commit -m "fix"
git commit -m "alterações"
git commit -m "WIP"
git commit -m "corrigindo bug da autenticação finalmente"

# ── Exemplos BOM ──────────────────────────────────────────────────
git commit -m "feat(auth): adiciona login com Google OAuth"
git commit -m "fix(cart): corrige total quando desconto > 100%"
git commit -m "docs(readme): atualiza instruções de instalação"
git commit -m "refactor(api): extrai lógica de paginação para helper"

# ── Commit com corpo detalhado ────────────────────────────────────
git commit -m "feat(payment): implementa pagamento via PIX

Integra com a API do Banco Central para geração de QR codes.
O pagamento expira em 30 minutos conforme regulamentação do BCB.

Closes #142"

⚠️ Segredos no Git são para sempre
Se você commitar um arquivo .env com senhas, tokens de API ou chaves privadas, esses segredos ficam no histórico do Git — mesmo depois de deletar o arquivo. Repositórios públicos são escaneados por bots 24/7 procurando exatamente isso. Sempre verifique com git status antes de commitar. Use gitignore.io para gerar .gitignore completo para seu stack.
🔥 Pro tip: arquivo já rastreado mas precisa ignorar
Se um arquivo já foi commitado e você quer que o Git passe a ignorá-lo: git rm --cached arquivo.txt (remove do tracking sem deletar o arquivo) → adicione ao .gitignore → commite as mudanças.

// pitfalls

⚠️ Armadilhas Comuns

1. git add . sem checar o git status antes

Rodar git add . cegamente pode incluir arquivos temporários, senhas em .env ou builds gerados. Sempre confira git status antes e tenha um .gitignore atualizado.

2. Commits gigantes com mensagem "várias mudanças"

Commits devem ser atômicos — uma mudança lógica por commit. Commits grandes são difíceis de revisar, reverter e entender. Se você está com dificuldade de descrever o commit em uma frase, ele provavelmente deve ser dividido.

3. Commitar segredos acidentalmente

Se acontecer: não basta deletar o arquivo e fazer outro commit — o segredo ainda está no histórico. Você precisa usar git filter-repo para reescrever o histórico E revogar/rotacionar a credencial vazada imediatamente.

4. Confundir git restore com git reset

git restore arquivo descarta mudanças no working directory (perigoso, irreversível para mudanças não commitadas). git restore --staged arquivo apenas remove do staging. git reset tem comportamentos diferentes com --soft, --mixed e --hard — veremos na próxima aula.

5. Não configurar o user.name e user.email

Sem configuração, Git pode usar defaults incorretos ou recusar commits. Toda linha de git log mostra o autor — configure corretamente antes do primeiro commit de qualquer projeto.

// glossário

Termos desta Aula

Repositório (repo)
Pasta gerenciada pelo Git, contendo a pasta oculta .git/ com todo o histórico do projeto.
Working Directory
A pasta do projeto onde você edita arquivos. Estado "sujo" antes de fazer add.
Staging Area (Index)
Área intermediária onde arquivos são preparados antes de virar commit. Permite commits precisos e atômicos.
Commit
Snapshot imutável do estado do projeto em um momento. Tem hash SHA-1 único, autor, data e mensagem.
HEAD
Ponteiro para o commit atual (geralmente o mais recente da branch). HEAD~1 = commit anterior, HEAD~2 = dois antes.
.gitignore
Arquivo de texto com padrões de arquivos e pastas que o Git deve ignorar completamente.
SHA-1 / Hash
Identificador único de 40 caracteres gerado para cada commit (ex: a1b2c3d). Garante integridade do histórico.

🏁 Checkpoint — Desafio Final

Crie um repositório Git para um projeto fictício de sua escolha e construa um histórico de pelo menos 6 commits atômicos:

  • Commit 1 — chore: .gitignore + README com descrição do projeto
  • Commit 2 — feat: estrutura inicial de pastas e arquivo principal
  • Commit 3 — feat: primeira funcionalidade (qualquer função Python)
  • Commit 4 — feat: segunda funcionalidade independente
  • Commit 5 — docs: atualiza README com instruções de uso
  • Commit 6 — fix: corrige um bug real ou simulado na funcionalidade 1

Após os commits, rode git log --oneline e verifique se cada mensagem descreve claramente o que mudou. Use git show <hash> para inspecionar pelo menos 2 commits e confirmar que o diff corresponde à mensagem.

💬 Dica: use git add -p em pelo menos um commit para praticar o modo interativo de staging. Ele permite selecionar exatamente quais linhas vão para o próximo commit — um hábito de engenheiro sênior.
