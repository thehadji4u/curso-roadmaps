// motivação

Por que RAG é a técnica dominante de grounding para LLMs?

LLMs são impressionantes mas têm um problema fundamental: eles só sabem o que estava nos dados de treino. Sua empresa tem documentos internos, manuais atualizados semanalmente, contratos, wikis, bases de suporte — nada disso está no GPT-4o. RAG (Retrieval-Augmented Generation) é a ponte entre o conhecimento privado da sua organização e a capacidade de raciocínio do LLM.

Mais do que isso, RAG resolve o problema das alucinações de forma sistemática. Em vez de perguntar "o que você sabe sobre X?", você passa os documentos relevantes diretamente no prompt: "dado este contexto específico, responda X". O modelo não precisa inventar — ele sintetiza a partir de fontes reais que você fornece.

💼 O que um engenheiro sênior constrói com RAG

RAG é o arquétipo de aplicação LLM em produção. Casos de uso reais:

  • Chatbot corporativo — responde perguntas sobre RH, políticas internas, benefícios. Base: PDFs e wikis da empresa.
  • Q&A sobre documentação técnica — desenvolvedor pergunta "como configuro CORS no FastAPI?" e recebe a resposta exata com trecho do manual.
  • Assistente jurídico — analisa contratos, encontra cláusulas relevantes, compara com legislação vigente sem vazar dados para o LLM cloud.
  • Suporte ao cliente nível 1 — RAG sobre base de conhecimento do produto responde 80% dos tickets automaticamente, escalando apenas os complexos.
  • Assistente de código — indexa toda a codebase da empresa, responde "onde está a lógica de precificação?" com arquivo e linha.
🧠

Sem alucinações

O LLM responde com base em documentos reais que você fornece. Se a informação não está no contexto, ele diz isso — em vez de inventar.

🔒

Dados privados seguros

O conhecimento fica no seu vector store local. O LLM só vê os trechos relevantes em cada query — não toda a base de uma vez.

🔄

Atualização sem re-treino

Para atualizar o conhecimento, basta reindexar os documentos novos. Nenhum fine-tuning necessário — o LLM permanece o mesmo.

📎

Citação de fontes

RAG permite retornar não só a resposta, mas de qual documento e em qual trecho ela foi baseada. Rastreabilidade total.

"RAG é como um advogado que, antes de responder uma pergunta do cliente, vai à biblioteca buscar os casos relevantes, anota os trechos específicos, e então formula a resposta baseada nos documentos. A diferença crítica: um advogado respondendo de memória é um LLM sem RAG. Impressionante, mas falível. Um advogado que pesquisa antes de responder é um LLM com RAG. Mais lento, porém fundamentado e confiável.

// arquitetura

Visão completa dos dois pipelines

Um sistema RAG tem dois pipelines distintos com momentos de execução diferentes. O pipeline de ingestão roda offline (uma vez, ou quando documentos mudam). O pipeline de consulta roda online, em tempo real a cada pergunta do usuário.

🔵 Ingestão (offline) — roda 1x por batch de documentos 🟢 Consulta (online) — roda a cada pergunta do usuário
Diagrama — Pipelines de ingestão e consulta RAG
flowchart TD subgraph INGESTAO ["🔵 Pipeline de Ingestão (offline)"] A["📄 Documentos\nPDF/TXT/HTML/CSV"] --> B[Document Loader] B --> C["Text Splitter\nchunk_size=500\noverlap=50"] C --> D["Embedding Model\ntext-embedding-3-small"] D --> E[("Vector Store\nChroma/Pinecone")] end subgraph CONSULTA ["🟢 Pipeline de Consulta (online)"] F["💬 Pergunta do usuário"] --> G[Embedding Model] G --> H["Vector Search\ntop-k=5"] H --> E E --> I[Chunks relevantes] I --> J["Reranker\nCross-encoder"] J --> K["Prompt Template\nContext + Question"] K --> L["LLM\nGPT-4o / Claude"] L --> M["✅ Resposta com fontes"] end style INGESTAO fill:#0d1117,stroke:#1f6feb,color:#c9d1d9 style CONSULTA fill:#0d1117,stroke:#238636,color:#c9d1d9
Insight arquitetural

O Vector Store é o elo entre os dois pipelines. A ingestão escreve nele; a consulta lê dele. Esta separação é o que torna RAG escalável: você pode ter milhões de documentos indexados e cada query ainda resolve em milissegundos.

// implementação

Construindo o pipeline passo a passo

1

Document Loading — Ingerir qualquer fonte

LangChain oferece loaders para dezenas de formatos. O objetivo é normalizar tudo para uma lista de Document com page_content e metadata.

python · langchain document loaders
from langchain_community.document_loaders import (
    PyPDFLoader,
    TextLoader,
    WebBaseLoader,
    CSVLoader,
    DirectoryLoader
)

# PDF — carrega página a página, metadata inclui número da página
loader = PyPDFLoader("documentos/manual.pdf")
docs = loader.load()
print(f"Páginas carregadas: {len(docs)}")
print(f"Metadados: {docs[0].metadata}")
# {'source': 'documentos/manual.pdf', 'page': 0}

# Diretório completo — indexa recursivamente
loader = DirectoryLoader(
    "documentos/",
    glob="**/*.pdf",
    loader_cls=PyPDFLoader
)
docs = loader.load()
print(f"Total de páginas: {len(docs)}")

# Web scraping — útil para documentação online
loader = WebBaseLoader([
    "https://docs.python.org/3/",
    "https://fastapi.tiangolo.com/"
])
docs = loader.load()
Atenção aos metadados

Os metadados que você adiciona durante o loading aparecem na resposta final como citação de fonte. Inclua sempre source (nome do arquivo), page (número da página), created_at, author — tudo que ajuda o usuário a verificar a informação.

2

Chunking — Dividir documentos estrategicamente

Documentos grandes precisam ser divididos em pedaços (chunks) que caibam no context window do LLM e sejam semanticamente coerentes.

python · recursive character text splitter
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # tamanho máximo de cada chunk (em chars)
    chunk_overlap=50,     # sobreposição entre chunks adjacentes
    separators=["\n\n", "\n", ".", " ", ""],  # ordem de prioridade
    length_function=len,
)

chunks = splitter.split_documents(docs)
print(f"Documentos originais: {len(docs)}")
print(f"Chunks gerados: {len(chunks)}")

# Adicionar metadados ricos para rastreabilidade
for i, chunk in enumerate(chunks):
    chunk.metadata["chunk_id"] = i
    chunk.metadata["chunk_size"] = len(chunk.page_content)

Estratégias de chunking

A escolha errada de chunk strategy é uma das causas mais comuns de RAG com performance ruim. A tabela abaixo guia a decisão:

Estratégia chunk_size Quando usar Trade-off
Fixed-size 200–500 chars FAQ, respostas curtas, artigos de suporte Rápido, pode cortar frases no meio
Sentence ~3–5 frases Documentação técnica, tutoriais Preserva frases completas, tamanho variável
Recursive Recomendado 500 tokens Propósito geral — primeira escolha Respeita estrutura do texto, configurável
Semantic Variável Textos narrativos longos, livros Melhor qualidade, mais lento e caro
Regra prática para chunk_size

Chunks menores = melhor precisão no retrieval (menos ruído). Chunks maiores = mais contexto por chunk (menos fragmentação). Para começar: 500 chars com 50 de overlap funciona para 80% dos casos. Ajuste baseado em avaliação com RAGAS (veja Fase 7).

3

Ingestão no Vector Store — Indexar os chunks

Gerar embeddings para cada chunk e persistir no vector store. Esta fase pode demorar minutos a horas dependendo do volume.

python · chroma vector store com persistência
import os
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",  # 1536 dims, barato e eficiente
    api_key=os.environ["OPENAI_API_KEY"]
)

# Criar vector store com persistência em disco
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",
    collection_name="base_conhecimento"
)

print(f"Chunks indexados: {vectorstore._collection.count()}")

# Carregar vector store existente (sem re-indexar)
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings,
    collection_name="base_conhecimento"
)
4

Retrieval — Buscar os chunks mais relevantes

Transformar a query do usuário em embedding e buscar os k chunks mais similares no vector store.

python · retriever — similarity, mmr e busca com score
# Retriever básico — similaridade cosseno, top-k
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 5}
)

# MMR — Maximal Marginal Relevance
# Equilibra relevância COM diversidade (evita chunks repetitivos)
retriever_mmr = vectorstore.as_retriever(
    search_type="mmr",
    search_kwargs={
        "k": 5,           # chunks retornados
        "fetch_k": 20,    # candidatos avaliados antes do MMR
        "lambda_mult": 0.5  # 0=máx diversidade, 1=máx similaridade
    }
)

# Busca com score de similaridade (útil para debug e thresholding)
docs_com_score = vectorstore.similarity_search_with_score(
    "Como configurar autenticação JWT?",
    k=3
)

for doc, score in docs_com_score:
    print(f"Score: {score:.3f} | Fonte: {doc.metadata.get('source', 'N/A')}")
    print(f"Trecho: {doc.page_content[:100]}...")
    print()
Similarity vs MMR — quando usar cada um?

Similarity puro: rápido, direto, bom quando os chunks são bem distintos. MMR: essencial quando a base tem documentos redundantes (ex.: vários tutoriais sobre o mesmo tópico). MMR garante que os 5 chunks retornados cubram perspectivas diferentes, não a mesma informação repetida 5 vezes.

5

Reranking — Refinar a ordem dos resultados

O retrieval por embedding é rápido mas aproximado. O reranker usa um modelo mais poderoso (cross-encoder) para reordenar os candidatos com maior precisão.

python · reranking com cohere (cloud) e cross-encoder local
# Opção 1: Reranking com Cohere — excelente qualidade, pago
from langchain.retrievers import ContextualCompressionRetriever
from langchain_cohere import CohereRerank

compressor = CohereRerank(
    model="rerank-multilingual-v3.0",
    top_n=3  # retorna apenas os 3 melhores após reranking
)

compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=retriever  # usa o retriever base para candidatos iniciais
)

# Opção 2: Cross-encoder local — gratuito, rodando na sua máquina
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank_local(query: str, docs: list, top_n: int = 3) -> list:
    """Reordena docs por relevância usando cross-encoder."""
    pairs = [(query, doc.page_content) for doc in docs]
    scores = reranker.predict(pairs)
    ranked = sorted(zip(docs, scores), key=lambda x: x[1], reverse=True)
    return [doc for doc, _ in ranked[:top_n]]

# Uso: primeiro retrieval amplo, depois rerank
candidatos = retriever.invoke("Como implementar JWT em FastAPI?")  # top-10
top_docs = rerank_local("Como implementar JWT em FastAPI?", candidatos, top_n=3)
Por que reranking importa?

Embedding models comparam query e chunk independentemente (bi-encoder). Cross-encoders analisam query + chunk juntos — mais lento, mas dramaticamente mais preciso para capturar nuances de relevância. O padrão recomendado é: retrieval amplo (top-20 com embedding) → reranking (top-3 com cross-encoder) → geração.

6

Geração — RAG Chain completo com LCEL

Montar o pipeline final: retriever → formatar contexto → prompt → LLM → resposta. Com LangChain Expression Language (LCEL).

python · rag chain completo com lcel
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

llm = ChatOpenAI(model="gpt-4o", temperature=0)

PROMPT_TEMPLATE = """Você é um assistente especializado. Responda a pergunta com base EXCLUSIVAMENTE no contexto fornecido.
Se a informação não estiver no contexto, diga explicitamente: "Não encontrei essa informação na base de conhecimento."

Contexto:
{context}

Pergunta: {question}

Resposta (cite a fonte quando possível):"""

prompt = ChatPromptTemplate.from_template(PROMPT_TEMPLATE)

def formatar_docs(docs):
    """Formata a lista de chunks com referência à fonte."""
    return "\n\n".join([
        f"[Fonte: {doc.metadata.get('source', 'desconhecida')}]\n{doc.page_content}"
        for doc in docs
    ])

# RAG Chain usando LCEL — composição declarativa de pipelines
rag_chain = (
    {
        "context": retriever | formatar_docs,
        "question": RunnablePassthrough()
    }
    | prompt
    | llm
    | StrOutputParser()
)

# Invocar
resposta = rag_chain.invoke("Como implementar autenticação JWT em FastAPI?")
print(resposta)
O que é LCEL e por que usar?

LCEL (LangChain Expression Language) é uma DSL de composição de pipelines usando o operador | (pipe). Vantagens: streaming nativo, suporte a async, paralelismo automático, e facilidade para trocar componentes (ex.: trocar o LLM sem reescrever o pipeline). Todo RAG moderno em LangChain usa LCEL.

Retornando documentos fonte

Em produção, você quase sempre quer retornar não só a resposta, mas também as fontes que embasaram ela — para o usuário poder verificar.

python · retrieval qa com sources
from langchain.chains import RetrievalQAWithSourcesChain

chain = RetrievalQAWithSourcesChain.from_chain_type(
    llm=llm,
    chain_type="stuff",  # coloca todos os chunks no prompt (simples e eficaz)
    retriever=retriever,
    return_source_documents=True
)

resultado = chain.invoke({"question": "Como criar um endpoint POST em FastAPI?"})

print("Resposta:", resultado["answer"])
print("\nFontes:")
for doc in resultado["source_documents"]:
    print(f"  - {doc.metadata['source']} (página {doc.metadata.get('page', 'N/A')})")

Medir a qualidade do RAG com RAGAS

"Parece estar funcionando bem" não é avaliação. RAGAS (RAG Assessment) é o framework padrão para medir objetivamente a qualidade de sistemas RAG com quatro métricas complementares. Sem isso, você não sabe se suas mudanças melhoraram ou pioraram o sistema.

> 0.80

> 0.70
Context Precision alvo
> 0.75
Context Recall alvo
python · avaliação com ragas
from ragas import evaluate
from ragas.metrics import (
    faithfulness,        # resposta fiel ao contexto?
    answer_relevancy,   # resposta relevante à pergunta?
    context_precision,  # contexto recuperado é preciso?
    context_recall      # contexto recuperado é completo?
)
from datasets import Dataset

# Montar dataset de avaliação (mínimo 5-10 pares)
test_data = {
    "question": [
        "Como instalar FastAPI?",
        "O que é async/await em Python?"
    ],
    "answer": [
        rag_chain.invoke("Como instalar FastAPI?"),
        rag_chain.invoke("O que é async/await em Python?")
    ],
    "contexts": [
        [doc.page_content for doc in retriever.invoke("Como instalar FastAPI?")],
        [doc.page_content for doc in retriever.invoke("O que é async/await em Python?")]
    ],
    "ground_truth": [
        "pip install fastapi uvicorn",
        "async/await permite programação assíncrona sem bloquear o event loop"
    ]
}

dataset = Dataset.from_dict(test_data)

resultado = evaluate(
    dataset=dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall]
)

print(resultado.to_pandas())
# Saída esperada:
# faithfulness=0.92 | answer_relevancy=0.88 | context_precision=0.75 | context_recall=0.83

As quatro métricas RAGAS explicadas

Métrica O que mede Faixa ideal Quando está baixo
Faithfulness Resposta é fiel ao contexto recuperado? (sem alucinação) > 0.85 LLM está "completando" além do contexto. Ajuste o prompt para ser mais restritivo.
Answer Relevancy Resposta é relevante à pergunta feita? > 0.80 LLM está divagando. Reduza o contexto ou melhore o prompt template.
Context Precision Os chunks recuperados são de fato úteis para a resposta? > 0.70 Retrieval está trazendo "lixo". Ajuste chunk size ou adicione reranking.
Context Recall Todos os chunks necessários foram recuperados? > 0.75 Aumente k no retriever ou melhore a estratégia de chunking.
// técnicas avançadas

Além do RAG básico

RAG básico funciona. RAG avançado funciona bem. Estas técnicas resolvem problemas específicos que aparecem em produção:

HyDE — Hypothetical Document Embeddings

Em vez de embeddar a query diretamente, pede ao LLM para gerar um "documento hipotético" que responderia a pergunta — então embeda esse documento para buscar. Funciona melhor porque queries são curtas e documentos são longos: o HyDE resolve essa assimetria de distribuição.

Self-Query Retriever

O LLM analisa a query e gera automaticamente filtros de metadados. Exemplo: "documentos de segurança criados depois de 2023" → LLM gera filter={type: 'segurança', date: {$gt: '2023'}}. Combina busca semântica com filtros estruturados.

Parent Document Retriever

Indexa chunks pequenos para retrieval preciso, mas retorna o documento pai maior para ter mais contexto na geração. Melhor dos dois mundos: alta precisão no match + contexto rico para o LLM.

Multi-Query Retriever

Gera múltiplas variações da pergunta original e busca cada uma separadamente. Une os resultados e remove duplicatas. Resolve o problema de queries ambíguas ou mal formuladas pelo usuário.

python · multi-query retriever
from langchain.retrievers.multi_query import MultiQueryRetriever

retriever_mq = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
    llm=llm
)

# Para a query "Como configurar JWT no FastAPI?", o LLM gera automaticamente:
# Variação 1: "Qual é o processo de configurar JWT no FastAPI?"
# Variação 2: "Como adicionar autenticação Bearer token ao FastAPI?"
# Variação 3: "Exemplo de middleware de autenticação FastAPI"
# → Busca as 3, une os resultados únicos → maior cobertura

docs = retriever_mq.invoke("Como configurar JWT no FastAPI?")

Erros comuns que destroem RAG pipelines

Chunk size inadequado para o tipo de documento

Chunk de 200 chars para contratos de 50 páginas fragmente informações cruciais. Chunk de 2000 chars para FAQ de uma linha coloca ruído demais no contexto. Teste com RAGAS para encontrar o tamanho certo para o seu corpus.

Metadados insuficientes nos chunks

Se o chunk não tem source, page e section, você não consegue citar a fonte nem filtrar por tipo de documento. Adicione metadados ricos no momento do loading — depois fica difícil.

Prompt template que não instrui a usar só o contexto

Se o prompt diz "responda a pergunta", o LLM mistura contexto recuperado com conhecimento próprio — voltando a ter alucinações. Use a instrução explícita: "responda exclusivamente com base no contexto fornecido".

Vector store desatualizado após mudanças nos documentos

Documento atualizado no disco não atualiza o vector store automaticamente. Implemente um pipeline de re-ingestão incremental (detectar documentos modificados e re-indexar só esses).

Avaliar RAG "na intuição" sem métricas quantitativas

Sem RAGAS ou framework equivalente, você não sabe se sua mudança de chunk_size de 500 para 750 melhorou ou piorou. Defina um test set com ground_truth antes de iterar.

Context stuffing — ultrapassar a context window

Recuperar k=20 chunks de 500 chars cada = 10.000 tokens só de contexto. Com prompt template + pergunta, você pode ultrapassar os 128k do GPT-4o em casos com muita formatação. Monitore o tamanho do prompt.

Checkpoint · Aula 43

Construir um chatbot RAG para a documentação do curso

Aplique tudo desta aula construindo um assistente que responde perguntas sobre as 50 aulas do curso. O sistema deve recuperar a aula certa e citar o link.

  • Criar um arquivo JSON ou texto com título, descrição e link das 50 aulas como documentos
  • Fazer chunking (cada aula como um chunk) e indexar no Chroma localmente
  • Criar RAG chain com gpt-4o-mini (mais barato para desenvolvimento)
  • Testar: "Qual aula me ensina Docker Compose?" → retorna título + link correto
  • Montar dataset RAGAS com 5+ pares de Q&A com ground_truth
  • Avaliar com as 4 métricas e atingir faithfulness > 0.85
  • Iterar: ajustar chunk_size ou k até Context Recall > 0.75
Dica: use os metadados para incluir o número da aula, trilha e URL. Assim o RAG chain pode retornar um link clicável junto com a resposta — não apenas texto.
