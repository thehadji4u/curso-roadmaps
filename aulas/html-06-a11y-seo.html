01 · Motivação

Por que acessibilidade e SEO andam juntos?

Sites acessíveis tendem a ter melhor SEO — e vice-versa. Isso não é coincidência: tanto leitores de tela quanto crawlers do Google entendem melhor um HTML semântico, com estrutura clara e conteúdo significativo.

♿

1 em 7 pessoas

Aproximadamente 15% da população mundial tem algum tipo de deficiência. Sites inacessíveis excluem essa audiência e podem gerar processos legais (ADA nos EUA, Lei Brasileira de Inclusão).

🔍

SEO = acessibilidade

Googlebot é essencialmente um usuário cego: lê texto, segue links, entende semântica HTML. Tudo que melhora acessibilidade melhora indexação.

💡 Por que um engenheiro sênior cuida disso?
Empresas sérias exigem conformidade WCAG AA em contratos. Core Web Vitals impactam diretamente o ranking no Google. Acessibilidade e SEO são diferencial competitivo — não "nice to have".
02 · Analogia

O Prédio com Rampa e Placa Visível

🏢 Analogia
Um site é como um prédio público. Para ser acessível, precisa de rampa (navegação por teclado), elevador (skip links), braille nos botões (aria-labels), sinalização clara (headings em ordem), e contraste adequado nas placas. O SEO é a fachada e placa de identificação — se não estiverem visíveis e legíveis, o Google (e os usuários) não encontram o prédio.
♿ Rampa de acesso
→ Navegação por teclado / Tab
🔤 Braille nos botões
→ aria-label / aria-labelledby
🪧 Placa do prédio
→ <title> + meta description
📢 Aviso sonoro do elevador
→ aria-live regions
🗺️ Mapa do prédio
→ schema.org / sitemap.xml
03 · WCAG 2.1

WCAG — As Diretrizes de Acessibilidade

WCAG (Web Content Accessibility Guidelines) é o padrão internacional de acessibilidade web, mantido pelo W3C. Organizado em 4 princípios: Perceptível, Operável, Compreensível, Robusto (POUR).

NívelO que significaObrigatório?
AMínimo absoluto — sem isso, partes do site são inutilizáveis para pessoas com deficiênciaSempre
AAPadrão de mercado — exigido por leis e contratos corporativosRecomendado
AAAIdeal — difícil de atingir 100% do site; buscar em componentes críticosAspiracional

Critérios mais importantes (nível A/AA)

NÍVEL A
1.1.1 — Conteúdo não-textual

Toda imagem deve ter texto alternativo significativo. Imagens decorativas: alt="". Imagens informativas: descreva o conteúdo.

<!-- ❌ Errado -->
<img src="grafico-vendas.png">
<img src="foto.jpg" alt="imagem">

<!-- ✅ Correto -->
<img src="grafico-vendas.png" alt="Gráfico de barras mostrando crescimento de 40% nas vendas em Q3 2024">
<img src="decoracao.png" alt=""> <!-- decorativa: alt vazio -->
NÍVEL AA
1.4.3 — Contraste mínimo

Texto normal: contraste mínimo 4.5:1. Texto grande (18pt+ ou 14pt bold): 3:1.

Texto cinza claro
❌ 2.1:1 — FALHA
Texto branco
✅ 16:1 — PASSA
Azul links
✅ 5.9:1 — PASSA
NÍVEL A
2.4.3 — Ordem de foco lógica

A ordem de Tab deve seguir a lógica visual da página. Nunca use tabindex > 0 para forçar ordem — isso causa confusão.

<!-- ❌ Errado: tabindex positivo bagunça a ordem natural -->
<button tabindex="3">Cancelar</button>
<button tabindex="1">Confirmar</button>

<!-- ✅ Correto: ordem segue o DOM -->
<button>Confirmar</button>
<button>Cancelar</button>
<!-- Exceção: tabindex="0" para tornar elemento focável; tabindex="-1" para foco via JS -->
04 · ARIA

ARIA — Accesssible Rich Internet Applications

ARIA adiciona semântica a elementos HTML que não têm significado acessível nativo. A regra de ouro: use HTML semântico primeiro — ARIA é para quando o HTML nativo não é suficiente.

⚠️ Regra número 1 do ARIA
"Não use ARIA se puder usar HTML semântico." Um <button> é sempre melhor que <div role="button">. ARIA incorreto é pior que sem ARIA.

Roles, States e Properties

<!-- ── ARIA Roles ────────────────────────────────────────── -->
<nav aria-label="Menu principal">...</nav>
<aside aria-label="Navegação lateral">...</aside>
<div role="dialog" aria-modal="true" aria-labelledby="dialog-title">
  <h2 id="dialog-title">Confirmar exclusão</h2>
</div>

<!-- ── ARIA States ──────────────────────────────────────── -->
<button aria-expanded="false" aria-controls="menu-dropdown">Menu ▼</button>
<ul id="menu-dropdown" hidden>...</ul>

<input type="text" aria-invalid="true" aria-describedby="email-error">
<span id="email-error" role="alert">Email inválido</span>

<!-- ── ARIA Live Regions (atualizações dinâmicas) ─────── -->
<div aria-live="polite" aria-atomic="true">
  <!-- Conteúdo aqui é anunciado pelo leitor de tela quando muda -->
  Status: Carregando...
</div>

<div role="status">3 resultados encontrados</div>  <!-- live="polite" implícito -->
<div role="alert">Erro ao salvar!</div>           <!-- live="assertive" implícito -->

<!-- ── Landmarks importantes ─────────────────────────── -->
<header role="banner">...</header>
<main role="main">...</main>
<footer role="contentinfo">...</footer>
<form aria-label="Formulário de contato">
  <label for="nome">Nome completo</label>
  <input id="nome" type="text" required aria-required="true">
</form>

Skip Links — Navegação rápida por teclado

<!-- Primeiro elemento do <body> — visível apenas no foco -->
<a href="#conteudo-principal" class="skip-link">Pular para o conteúdo</a>

<main id="conteudo-principal">...</main>
/* Skip link visível apenas no foco — não polui o design */
.skip-link {
  position: absolute;
  top: -100%;
  left: 16px;
  background: #000;
  color: #fff;
  padding: 8px 16px;
  border-radius: 0 0 8px 8px;
  font-size: 14px;
  z-index: 9999;
  transition: top 0.2s;
}
.skip-link:focus { top: 0; }
05 · SEO Técnico

SEO — Sendo Encontrado pelos Motores de Busca

Meta tags essenciais

<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

  <!-- SEO básico -->
  <title>Título da Página — Nome do Site | máx 60 caracteres</title>
  <meta name="description" content="Descrição clara do conteúdo. Entre 150-160 caracteres. Aparece nos resultados do Google.">
  <link rel="canonical" href="https://seusite.com/pagina">

  <!-- Open Graph (Facebook, LinkedIn, WhatsApp) -->
  <meta property="og:title" content="Título para redes sociais">
  <meta property="og:description" content="Descrição para redes sociais">
  <meta property="og:image" content="https://seusite.com/og-image.jpg"> <!-- 1200x630px -->
  <meta property="og:url" content="https://seusite.com/pagina">
  <meta property="og:type" content="website">
  <meta property="og:site_name" content="Nome do Site">

  <!-- Twitter Card -->
  <meta name="twitter:card" content="summary_large_image">
  <meta name="twitter:title" content="Título para Twitter">
  <meta name="twitter:description" content="Descrição para Twitter">
  <meta name="twitter:image" content="https://seusite.com/og-image.jpg">
</head>

Schema.org — Dados Estruturados

Schema.org permite que o Google entenda o tipo de conteúdo — artigo, produto, FAQ, receita, evento — e exiba rich snippets nos resultados.

<!-- JSON-LD (recomendado pelo Google) -->
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Como usar Docker em Produção",
  "description": "Guia completo de Docker Swarm, Registry e CI/CD",
  "author": {
    "@type": "Person",
    "name": "Vinicius Silva"
  },
  "datePublished": "2024-03-14",
  "image": "https://seusite.com/post-docker.jpg",
  "publisher": {
    "@type": "Organization",
    "name": "Curso Roadmaps",
    "logo": { "@type": "ImageObject", "url": "https://seusite.com/logo.png" }
  }
}
</script>

<!-- FAQ Schema — aparece expandido no Google -->
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [{
    "@type": "Question",
    "name": "O que é Docker Swarm?",
    "acceptedAnswer": {
      "@type": "Answer",
      "text": "Docker Swarm é o modo de orquestração nativo do Docker..."
    }
  }]
}
</script>

Boas práticas de performance/SEO no HTML
<!-- Preload do recurso mais importante (LCP) -->
<link rel="preload" as="image" href="hero-image.webp" fetchpriority="high">

<!-- Fontes sem bloco de renderização -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="stylesheet" href="fonts.css" media="print" onload="this.media='all'">

<!-- Imagens: tamanho explícito evita CLS -->
<img src="hero.webp" width="1200" height="600"
     alt="Dashboard da aplicação mostrando métricas em tempo real"
     loading="lazy"   <!-- lazy para imagens abaixo do fold -->
     decoding="async">

<!-- JS não-crítico: defer não bloqueia parsing -->
<script src="analytics.js" defer></script>

<!-- Hierarquia de headings correta (afeta SEO e a11y) -->
<h1>Título principal — apenas UM por página</h1>
  <h2>Seção principal</h2>
    <h3>Sub-seção</h3>
  <h2>Outra seção</h2>
06 · Armadilhas

⚠️ Erros Comuns

⚠️ alt="imagem" ou alt="foto"

Texto alternativo genérico é inútil para leitores de tela. Descreva o conteúdo informativo da imagem. Se for decorativa, use alt="" (vazio) — o leitor de tela irá ignorá-la completamente.

⚠️ Múltiplos <h1> por página

Cada página deve ter exatamente um <h1> descrevendo o conteúdo principal. Múltiplos confundem tanto leitores de tela quanto o Google (dilui a relevância do título).

⚠️ Foco invisível (outline: none)

Remover o outline de foco dos botões/links (muito comum em resets CSS) torna o site inacessível para usuários de teclado. Estilize o foco — não o remova: :focus-visible { outline: 2px solid #58a6ff; }

⚠️ Meta description duplicada

Cada página deve ter uma meta description única e descritiva. Duplicadas fazem o Google ignorá-las e gerar um snippet automático (geralmente pior).

🏁 Checkpoint Final

Desafio: Auditoria Completa de Acessibilidade & SEO

Escolha qualquer página HTML que você já criou neste curso e realize uma auditoria completa:

  • Rode o Lighthouse no Chrome DevTools (F12 → Lighthouse → Accessibility + SEO)
  • Corrija todos os problemas de contraste encontrados (use o Chrome DevTools Color Picker)
  • Adicione alt descritivo em todas as imagens informativas
  • Adicione meta tags completas: title, description, Open Graph e Twitter Card
  • Adicione pelo menos um schema.org JSON-LD adequado ao conteúdo
  • Bônus: Navegue pela página usando apenas o teclado (Tab, Enter, Esc) e corrija qualquer armadilha de foco
💬 Dica: A extensão axe DevTools (gratuita) encontra problemas de acessibilidade que o Lighthouse não detecta. Instale no Chrome para uma auditoria mais completa. Ferramenta de contraste: webaim.org/resources/contrastchecker
