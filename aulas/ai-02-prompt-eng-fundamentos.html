01 · Motivação

Por que Prompt Engineering importa?

O mesmo modelo de IA pode dar respostas completamente diferentes dependendo de como você formula a instrução. Prompt Engineering é a habilidade de comunicar intenção com precisão para um LLM — a diferença entre um assistente medíocre e um especialista virtual.

💰

ROI imediato

Um prompt bem feito economiza iterações, reduz tokens e entrega resultado direto na primeira tentativa.

🔬

Técnica, não magia

Prompt Engineering tem princípios concretos e técnicas testadas cientificamente em papers.

🏗️

Base para agentes

Tudo que vem depois — RAG, agentes, function calling — depende de saber prompting bem.

💡 Por que um engenheiro sênior domina isso?
Prompt Engineering determina a qualidade de produtos de IA. Um engenheiro que sabe prompting bem pode fazer um modelo de $0.01/1k tokens superar um de $0.30/1k tokens — 30x mais barato, mesma qualidade.
02 · Analogia

O Assistente Brilhante mas Literal

🧑‍💼 Analogia
Imagine contratar um assistente brilhante mas extremamente literal. Se você diz "me ajuda com esse documento", ele não sabe se quer resumo, revisão, tradução ou formatação. Mas se você diz "você é revisor sênior de textos técnicos, revise o parágrafo a seguir focando em clareza e precisão técnica, entregue como lista de sugestões", o resultado é profissional. O modelo tem o conhecimento — você tem que dar o contexto, o papel, o formato e o objetivo.

📋 Mostrar exemplos do que quer

03 · System vs User Prompt

Anatomia de uma Conversa com LLM

A maioria das APIs de LLM tem dois (ou três) tipos de mensagem: system, user e assistant. Entender a diferença é fundamental.

Estrutura de mensagens na API

from openai import OpenAI
client = OpenAI()

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {
            "role": "system",
            "content": """Você é um especialista em Python com 15 anos de experiência.
Suas respostas são:
- Concisas e diretas (máximo 3 parágrafos)
- Sempre incluem exemplo de código executável
- Usam boas práticas (type hints, docstrings)
- Em português do Brasil"""
        },
        {
            "role": "user",
            "content": "Como funciona o decorator @property?"
        }
    ]
)
🔥 System prompt é o seu contrato com o modelo
O system prompt define a "persona" e as regras permanentes. Invista tempo nele — um bom system prompt elimina a necessidade de repetir instruções em cada mensagem do usuário.

Anatomia de um System Prompt eficaz

❌ System prompt fraco
Você é um assistente útil.
✅ System prompt forte
## Papel
Você é um revisor de código Python sênior na empresa Acme Corp.

## Responsabilidades
- Revisar código quanto a: bugs, performance, segurança e legibilidade
- Sugerir refatorações baseadas em PEP 8 e boas práticas modernas (Python 3.11+)

## Formato de resposta
Sempre responda com:
1. **Resumo** (1-2 frases)
2. **Problemas encontrados** (lista numerada, cada item com severidade: 🔴 crítico / 🟡 aviso / 🟢 sugestão)
3. **Código corrigido** (bloco Python completo)

## Restrições
- Não modifique a lógica de negócio, apenas a qualidade do código
- Se o código estiver bom, diga explicitamente "Nenhum problema encontrado"
- Responda sempre em português
04 · Técnicas Fundamentais

Zero-shot, Few-shot e Chain-of-Thought

🎯 Zero-shot
Quando usar: tarefas simples e bem definidas, modelos grandes (GPT-4, Claude 3.5)

Instrução direta sem exemplos. Funciona bem quando o modelo já foi treinado extensivamente naquele tipo de tarefa.

Classifique o sentimento do texto como Positivo, Negativo ou Neutro.

Texto: "A entrega chegou antes do prazo e o produto superou minhas expectativas!"
Sentimento:
📚 Few-shot
Quando usar: formato específico, tarefas incomuns, quando zero-shot falha consistentemente

Fornece 2-5 exemplos de entrada → saída antes da pergunta real. O modelo aprende o padrão pelos exemplos.

Extraia o produto e preço de cada texto. Formato: PRODUTO | PREÇO

Texto: "Comprei um notebook Dell por R$ 3.200,00 no shopping"
Extração: Notebook Dell | R$ 3.200,00

Texto: "Meu celular Samsung custou 1850 reais"
Extração: Celular Samsung | R$ 1.850,00

Texto: "Adquiri uma televisão LG 55 polegadas por dois mil e quinhentos"
Extração:
🧠 Chain-of-Thought (CoT)
Quando usar: problemas de raciocínio, matemática, lógica multi-passo, debugging

Peça ao modelo para "pensar passo a passo" antes de dar a resposta final. Reduz erros em até 40% em tarefas de raciocínio.

Resolva o problema abaixo. Pense passo a passo antes de dar a resposta final.

Problema: Uma API recebe 1.200 requisições por minuto.
Cada requisição custa 0.002 tokens em média e o limite de rate é
800k tokens por hora. Estamos dentro do limite?

Raciocínio passo a passo:
## Resposta do modelo com CoT:

Vou calcular passo a passo:

**Passo 1:** Requisições por hora
1.200 req/min × 60 min = 72.000 req/hora

**Passo 2:** Tokens por hora
72.000 req × 0.002 tokens/req = 144 tokens/hora

**Passo 3:** Comparar com o limite
144 tokens/hora vs 800.000 tokens/hora de limite

**Conclusão:** Sim, estamos muito dentro do limite (apenas 0,018% do limite).
🔥 "Let's think step by step" — o prompt mais poderoso
Adicionar apenas "Pense passo a passo" ou "Let's think step by step" ao final do prompt melhora significativamente a acurácia em tarefas de raciocínio. É grátis e funciona em todos os modelos.
05 · Role Prompting

Role Prompting — Dando Persona ao Modelo

Atribuir um papel específico ao modelo ativa padrões de resposta associados a esse papel no treinamento. "Você é um médico" faz o modelo ser mais cauteloso; "Você é um hacker ético" faz o modelo ser mais técnico e direto.

PersonaEfeitoExemplo de uso
Especialista sênior em XRespostas técnicas, sem explicações básicasCode review, arquitetura
Professor universitário de XExplicações didáticas, progressivas, com exemplosAprender conceitos novos
Advogado do diaboQuestiona suposições, aponta falhas no argumentoRevisar planos, ideias
Editor técnico do NYTPrioriza clareza, elimina jargão desnecessárioRevisar documentação
Usuário leigo de 60 anosAponta onde a explicação é confusa ou técnica demaisTestar UX copy
 
# Combinando role + task + format + constraints
system_prompt = """
## Papel
Você é um arquiteto de software sênior com experiência em sistemas distribuídos de alta escala.

## Tarefa ao receber código
Analise o código fornecido e identifique:
1. Problemas de performance em escala
2. Single points of failure
3. Problemas de segurança

## Formato obrigatório
```
### Análise de Performance
[lista de problemas]

### Resiliência
[lista de problemas]

### Segurança
[lista de problemas]

### Recomendação principal
[1 frase com a mudança mais impactante]
```

## Estilo
- Direto e técnico (audiência: engenheiros sêniors)
- Referencie padrões do setor (CAP theorem, circuit breaker, etc.)
- Em português
"""
06 · Controle de Output

Controlando o Formato da Resposta

Especificar o formato de saída elimina pós-processamento e torna a integração em código muito mais simples.

## Exemplo: forçar JSON estruturado

prompt = """
Analise o seguinte texto e extraia as entidades mencionadas.

Texto: "João Silva, CEO da TechBR em São Paulo, anunciou parceria com a Microsoft."

Retorne APENAS um JSON válido neste formato exato:
{
  "pessoas": ["nome1", "nome2"],
  "organizacoes": ["org1", "org2"],
  "locais": ["local1"],
  "cargos": [{"pessoa": "nome", "cargo": "titulo"}]
}

Não inclua explicações, apenas o JSON."""

# Com GPT-4o, você também pode usar response_format:
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": prompt}],
    response_format={"type": "json_object"}  # garante JSON válido
)

Templates de prompt reutilizáveis

from string import Template

# Template genérico para análise de código
CODE_REVIEW_TEMPLATE = Template("""
Você é um $role com $years anos de experiência em $language.

Revise o código abaixo focando em: $focus_areas

Código:
```$language
$code
```

Formato de resposta:
$output_format
""")

# Usar o template
prompt = CODE_REVIEW_TEMPLATE.substitute(
    role="desenvolvedor Python sênior",
    years="10",
    language="python",
    focus_areas="performance, legibilidade e type hints",
    code=my_code,
    output_format="- Lista de pontos de melhoria\n- Código refatorado"
)
07 · Armadilhas

⚠️ Erros Comuns em Prompting

⚠️ Prompt ambíguo — instrução que pode ser interpretada de várias formas

"Melhore meu texto" → O modelo não sabe se quer gramática, estilo, concisão ou estrutura. Seja específico: "Corrija erros gramaticais e melhore a clareza sem mudar o tom informal. Máximo 150 palavras."

⚠️ Instruções negativas ("não faça X")

LLMs têm dificuldade com negações. "Não use linguagem técnica" é menos eficaz que "use linguagem simples, nível ensino médio". Reformule negações como afirmações do que você QUER.

⚠️ Prompt enorme sem estrutura

Um prompt de 2 páginas sem seções claras confunde o modelo. Use cabeçalhos (##), listas (-) e separadores. O modelo processa melhor quando a instrução está bem estruturada.

⚠️ Não testar variações

Pequenas mudanças de wording podem mudar drasticamente o resultado. Teste pelo menos 3-5 variações do mesmo prompt antes de usar em produção. Use temperature=0.0 para resultados reproduzíveis nos testes.

Desafio: Construir um Assistente Especializado
