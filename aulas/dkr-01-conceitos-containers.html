Docker — Conceitos, Imagens & Containers

// MOTIVAÇÃO

Por que isso importa?

Você já ouviu a frase "Na minha máquina funciona"? Esse é um dos problemas mais frustrantes e custosos do desenvolvimento de software. Um desenvolvedor configura o ambiente local, tudo funciona perfeitamente — e quando o código chega no servidor de produção ou na máquina de outro dev, quebra completamente.

O motivo? Diferenças de sistema operacional, versão do Python, dependências do sistema, variáveis de ambiente, portas em conflito. Times desperdiçam horas — às vezes dias — nisso. Docker elimina esse problema de forma definitiva: seu app roda exatamente igual em qualquer máquina com Docker instalado.

💡 Por que um engenheiro sênior usa Docker hoje
Docker garante paridade absoluta entre desenvolvimento, CI/CD e produção. É o pré-requisito direto para Kubernetes, microserviços e toda stack moderna de deploy. Saber Docker deixou de ser diferencial — virou requisito mínimo.

O contêiner marítimo que mudou o mundo

// analogia do mundo real
Antes dos contêineres marítimos (anos 1950), carregar um navio era um caos. Cada tipo de mercadoria precisava de método diferente de acondicionamento. Estivadores precisavam saber tudo sobre cada produto — e qualquer erro era catastrófico.

O contêiner padronizado revolucionou a logística: não importa o que está dentro. Se couber no contêiner, pode ser carregado em qualquer navio, empilhado em qualquer porto, colocado em qualquer caminhão. O mundo todo fala a mesma língua.

Docker é exatamente isso para software. Seu app, junto com todas as dependências, fica dentro de um container padronizado. Não importa se é Python, Node ou Java — se tem Docker, roda.

VM vs Container — a diferença fundamental

Muita gente confunde containers com máquinas virtuais. São coisas fundamentalmente diferentes. VMs virtualizam hardware completo — cada uma tem seu próprio kernel de sistema operacional, tornando-as pesadas e lentas. Containers compartilham o kernel do host e isolam apenas o processo.

CaracterísticaMáquina Virtual (VM)Container Docker
O que virtualiza Hardware completo + OS inteiro Apenas o processo / userspace
Tamanho Gigabytes (OS completo) Megabytes (só o app)
Tempo de startup Minutos Segundos (ou menos)
Isolamento Total (kernel próprio) Compartilha kernel do host
Overhead Alto (hypervisor) Mínimo (namespaces + cgroups)
Portabilidade Boa, porém pesada Excelente — leve e rápida
💡 Como o isolamento funciona
Containers usam Linux namespaces para isolamento (processo, rede, filesystem, usuários) e cgroups para limitar CPU e memória. Não há hypervisor — o container roda diretamente no kernel do host, por isso é tão rápido e leve.
// CONCEITO 2

Arquitetura Docker

Docker segue uma arquitetura cliente-servidor. O Docker CLI (o comando que você digita no terminal) se comunica via API REST com o Docker Daemon (dockerd), que é quem realmente cria e gerencia containers, imagens, volumes e redes.

Camadas de uma Imagem (Layers)

Imagens Docker são construídas em camadas imutáveis. Cada instrução no Dockerfile cria uma camada. Camadas são compartilhadas entre imagens — economizando espaço e acelerando downloads. Apenas a camada superior (do container em execução) é gravável e temporária.

// camadas de uma imagem docker (layers)
graph BT L1["Camada 1 — ubuntu:22.04 (base, ~72MB)"] L2["Camada 2 — apt install python3 (+45MB)"] L3["Camada 3 — pip install requirements (+120MB)"] L4["Camada 4 — COPY . /app (+2MB — seu código)"] L5["Camada R/W — Container em execução (gravável, temporária)"] L1 --> L2 --> L3 --> L4 --> L5 style L5 fill:#0e2e0e,stroke:#10b981,color:#6ee7b7
🔥 Pro tip: cache de layers
Camadas são cacheadas. Se você mudar apenas o código da aplicação (camada 4), o Docker reutiliza as camadas 1–3 do cache — o rebuild é quase instantâneo. Por isso, a ordem das instruções no Dockerfile importa muito. Veremos isso a fundo na próxima aula.

Volumes — dados que sobrevivem ao container

Por padrão, tudo dentro de um container é efêmero. Quando o container é removido, os dados somem. Volumes são áreas de armazenamento gerenciadas pelo Docker que existem fora do ciclo de vida do container — os dados persistem mesmo que você destrua e recrie o container.

Named Volume
-v nome:/caminho
Docker gerencia o storage. Recomendado para produção.
Bind Mount
-v $(pwd)/src:/app
Pasta do host mapeada. Ideal para desenvolvimento.
tmpfs Mount
--tmpfs /tmp
Apenas em memória. Dados não persistem nunca.
// volumes na prática
# ── Named Volume (recomendado para dados de app) ─────────────────
docker volume create pgdata

docker run -d \
  --name postgres-dev \
  -e POSTGRES_PASSWORD=senha123 \
  -v pgdata:/var/lib/postgresql/data \
  postgres:15-alpine

# Remove o container — os dados ficam no volume!
docker rm -f postgres-dev

# Cria novo container com o mesmo volume → dados intactos ✓
docker run -d \
  --name postgres-dev2 \
  -v pgdata:/var/lib/postgresql/data \
  postgres:15-alpine

# ── Bind Mount (desenvolvimento / hot-reload) ────────────────────
docker run -d \
  --name meu-app \
  -p 3000:3000 \
  -v $(pwd)/src:/app/src \
  node:18-alpine

# ── Gerenciando volumes ───────────────────────────────────────────
docker volume ls                   # lista todos os volumes
docker volume inspect pgdata       # detalhes: mountpoint, driver
docker volume rm pgdata            # remove (falha se em uso)
docker volume prune                # remove volumes sem containers associados
⚠️ Quando usar cada tipo
Use bind mounts em desenvolvimento para hot-reload imediato. Use named volumes em produção — Docker gerencia o armazenamento de forma otimizada e o caminho físico no host não importa.
// CONCEITO 5

Redes — containers que se conversam

Containers na mesma rede Docker custom se referenciam pelo nome do container como hostname — sem precisar hardcodar IPs que mudam a cada restart. É o DNS interno do Docker em ação.

// containers na mesma rede se comunicam por nome
graph LR HOST["🖥️ Host\nlocalhost:8000"] subgraph NET ["Rede: dev-network"] API["📦 api\n(container)"] DB["📦 db\n(postgres)"] CACHE["📦 cache\n(redis)"] end HOST -- "porta 8000" --> API API -- "hostname: db\nporta 5432" --> DB API -- "hostname: cache\nporta 6379" --> CACHE style NET fill:#0a1a1a,stroke:#06b6d4
// criando e usando redes docker
# Cria rede customizada (melhor que a rede bridge padrão)
docker network create dev-network

# Sobe containers na mesma rede
docker run -d \
  --name db \
  --network dev-network \
  -e POSTGRES_PASSWORD=senha123 \
  postgres:15-alpine

docker run -d \
  --name api \
  --network dev-network \
  -p 8000:8000 \
  -e DATABASE_URL=postgresql://postgres:senha123@db:5432/postgres \
  minha-api-image

# "api" resolve "db" como hostname automaticamente!
# DNS interno do Docker — zero configuração de IP necessária.

docker network ls
docker network inspect dev-network

⚠️ Armadilhas Comuns

1. Esquecer o -d (detached mode)

docker run nginx sem -d bloqueia o terminal. Use sempre -d para serviços que ficam em background. Omita apenas quando quiser ver o output diretamente — scripts, jobs únicos.

2. Perder dados por não usar volumes

Salvar dados dentro do container sem volume é receita para desastre. Tudo na camada gravável some quando o container é removido. Use -v sempre que os dados precisem persistir — banco de dados, uploads, logs.

3. Containers zumbis acumulando espaço

docker run sem --rm cria containers parados que acumulam espaço. Para scripts e tarefas únicas, sempre use --rm. Rode docker ps -a para ver os zumbis e docker container prune para limpar.

4. Usar a tag latest em produção

nginx:latest pode mudar de versão sem aviso entre deploys. Sempre use tags específicas: nginx:1.25.3-alpine. Isso garante reprodutibilidade total e evita surpresas no próximo deploy.

5. Ignorar o .dockerignore

Sem .dockerignore, o COPY . . no Dockerfile copia node_modules, .git, arquivos de teste e secrets. Imagem gigante e potencialmente insegura. Crie o .dockerignore antes do primeiro build.

6. Rodar como root dentro do container

Por padrão, processos rodam como root no container. Em produção, isso é risco de segurança. Use a instrução USER no Dockerfile — veremos em detalhes na próxima aula.

Termos desta Aula

Container
Instância isolada e em execução de uma imagem Docker. Efêmero por natureza — não guarda estado entre execuções sem volumes.
Imagem (Image)
Template somente-leitura formado por camadas (layers) que define o que vai dentro de um container. É o "molde"; o container é a "peça".
Dockerfile
Arquivo de texto com instruções para construir uma imagem Docker. Cada instrução gera uma camada imutável e cacheável.
Registry
Servidor que armazena e distribui imagens Docker. Docker Hub é o registry público padrão. GHCR (GitHub Container Registry) é muito usado em projetos open-source.
Volume
Mecanismo de persistência de dados gerenciado pelo Docker, que existe fora do ciclo de vida do container. Dados em volumes sobrevivem a remoções.
Docker Daemon (dockerd)
Processo servidor que roda em background e gerencia containers, imagens, volumes e redes. O CLI se comunica com ele via API REST.
Layer (camada)
Cada instrução do Dockerfile gera uma camada imutável e cacheável. Camadas são compartilhadas entre imagens para economizar espaço.
Namespace + cgroup
Primitivas do kernel Linux usadas pelo Docker. Namespaces fornecem isolamento (processos, rede, filesystem). Cgroups limitam recursos como CPU e memória.

🏁 Checkpoint — Missão Docker

Suba um ambiente de desenvolvimento com banco de dados e UI de administração usando apenas comandos Docker — sem Docker Compose ainda. Tudo no terminal.

  • Crie uma rede Docker chamada dev-network
  • Suba um container PostgreSQL 15 Alpine nessa rede, com senha e banco configurados via variáveis de ambiente, e dados persistidos em um named volume pgdata
  • Suba um container adminer na mesma rede, exposto na porta 8080 do host
  • Acesse http://localhost:8080, conecte ao PostgreSQL usando o hostname db e confirme que funciona
  • Pare e remova o container do PostgreSQL, crie um novo com o mesmo volume — os dados devem estar intactos
  • Rode docker system df e interprete o que cada linha significa
💬 Dica: a imagem do Adminer é adminer (oficial). No campo "servidor", coloque o nome do container do Postgres (ex: db). Isso só funciona se ambos estiverem na mesma rede Docker — é o DNS interno em ação!
