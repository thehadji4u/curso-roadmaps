Por que isso importa?

React é a biblioteca de UI mais usada no mundo — presente em Facebook, Instagram, Airbnb, Netflix, Shopify e milhares de startups. Não é uma trend passageira: é a habilidade mais procurada em vagas de front-end desde 2015. Em 2024, a versão 18 trouxe mudanças fundamentais com o Concurrent Mode e os Server Components.

Mas React é difícil de aprender mal. A maioria dos devs júniors aprende os hooks de forma mecânica sem entender o modelo mental por trás. Nesta aula vamos construir a fundação correta: como o React realmente funciona, o que são componentes de verdade, como o estado muda a UI e como usar cada hook no momento certo.

💼 Por que um engenheiro sênior domina React profundamente
Sêniors não apenas "fazem funcionar" — eles entendem quando o React re-renderiza, como evitar renders desnecessários com useMemo e useCallback, como estruturar o estado para evitar prop drilling, e como usar Suspense para UIs que carregam dados de forma elegante. A diferença entre um app lento e um app rápido está em entender esses detalhes.
// analogia

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense numa interface React como um conjunto de peças de LEGO. Cada peça (componente) tem uma forma específica, encaixes padronizados e pode ser combinada com outras peças de infinitas maneiras. Uma peça de LEGO não sabe onde vai ser usada — ela só sabe o que ela é e como ela se conecta.

O estado é como a cor da peça — pode mudar, e quando muda, qualquer pessoa que olha para ela enxerga a mudança imediatamente. As props são os encaixes — o que você passa de uma peça para outra para que elas se conectem corretamente. E o React é o arquiteto que decide quando reconstruir partes do conjunto baseado nas mudanças de estado.
🧱 Peça de LEGO reutilizável → Componente React 🔌 Encaixes da peça → Props (entrada de dados) 🎨 Cor que pode mudar → State (useState) 🏗️ Arquiteto que reconstrói → React Reconciler
// conceito central

Como o React realmente funciona

O React mantém uma Virtual DOM (árvore de objetos JavaScript representando a UI). Quando o estado muda, o React cria uma nova árvore virtual, compara com a anterior (processo chamado reconciliation), e atualiza no DOM real apenas o que mudou.

💡 A regra de ouro do React
O React re-renderiza um componente quando: (1) seu estado muda via setState, (2) suas props mudam, ou (3) o componente pai re-renderiza. Entender isso é a chave para otimizar performance.

JSX — HTML no JavaScript

JSX não é HTML. É açúcar sintático que o compilador (Babel/SWC) transforma em chamadas React.createElement(). Entender isso explica por que você precisa usar className em vez de class, por que JSX precisa de um elemento raiz e por que você pode usar JavaScript dentro de {}.

jsx × o que o compilador gera
// O que você escreve (JSX):
function Saudacao({ nome, ativo }) {
  return (
    <div className={ativo ? 'ativo' : 'inativo'}>
      <h1>Olá, {nome}!</h1>
      <p>{ativo ? 'Online agora' : 'Offline'}</p>
    </div>
  );
}

// O que o compilador gera:
function Saudacao({ nome, ativo }) {
  return React.createElement(
    'div',
    { className: ativo ? 'ativo' : 'inativo' },
    React.createElement('h1', null, 'Olá, ', nome, '!'),
    React.createElement('p', null, ativo ? 'Online agora' : 'Offline')
  );
}

Componentes & Props

Todo componente React é uma função que retorna JSX. Props são os argumentos dessa função — dados que o componente pai passa para o filho. A regra: props são somente leitura (imutáveis dentro do componente que as recebe).

padrões de componentes — do simples ao composto
// 1. Componente puro — sem estado, só UI
function Avatar({ src, nome, tamanho = 40 }) {
  return (
    <div style={{ display: 'flex', alignItems: 'center', gap: 8 }}>
      <img
        src={src}
        alt={nome}
        style={{ width: tamanho, height: tamanho, borderRadius: '50%' }}
      />
      <span>{nome}</span>
    </div>
  );
}

// 2. Componente com children (composição)
function Card({ titulo, children, variante = 'default' }) {
  const estilos = {
    default: { border: '1px solid #1e1e2e', background: '#111118' },
    destaque: { border: '1px solid #f59e0b', background: 'rgba(245,158,11,0.05)' },
  };
  return (
    <div style={{ ...estilos[variante], borderRadius: 12, padding: 24 }}>
      <h3 style={{ marginBottom: 12 }}>{titulo}</h3>
      {children}
    </div>
  );
}

function PerfilUsuario({ usuario }) {

// hooks

Guia completo de Hooks

Hooks são funções especiais que permitem usar estado, ciclo de vida e contexto em componentes funcionais. A regra de ouro: só use hooks no nível raiz do componente (nunca dentro de if, loop ou função aninhada).

HookPara que serveQuando usar
useState Estado local do componente Todo dado que, quando muda, deve re-renderizar o componente
useEffect Side effects (API calls, subscriptions, DOM) Buscar dados, sincronizar com sistemas externos, eventos do DOM
useContext Consumir contexto global Dados compartilhados sem prop drilling (tema, usuário logado, idioma)
useReducer Estado complexo com múltiplas ações Quando o próximo estado depende do anterior ou há muitas transições
useMemo Memoizar valor computado Cálculos caros que não devem ser refeitos a cada render
useCallback Memoizar referência de função Funções passadas como props para componentes filhos memoizados
useRef Referência mutável que não causa re-render Acessar elementos DOM, guardar valores entre renders sem re-render
useId 18+ Gerar IDs únicos estáveis IDs para formulários e acessibilidade (sem usar Math.random)
useTransition 18+ Marcar updates de baixa prioridade Filtros, buscas — manter UI responsiva durante renders pesados

useState — o mais fundamental

useState — padrões e armadilhas
import { useState } from 'react';

// PADRÃO 1: estado simples
function Contador() {
  const [count, setCount] = useState(0);

  // ⚠️ ERRADO: ler state diretamente em updates sequenciais
  const incrementarDuas = () => {
    setCount(count + 1); // usa o valor atual
    setCount(count + 1); // ainda usa o mesmo valor — só incrementa 1!
  };

  // ✅ CERTO: usar função de atualização (recebe prev state)
  const incrementarDuasCerto = () => {
    setCount(prev => prev + 1); // prev = valor mais atual
    setCount(prev => prev + 1); // agora incrementa 2!
  };

  return <button onClick={incrementarDuasCerto}>{count}</button>;
}

// PADRÃO 2: estado de objeto — sempre spread!
function Formulario() {
  const [form, setForm] = useState({ nome: '', email: '', aceito: false });

  const handleChange = (e) => {
    const { name, value, type, checked } = e.target;
    setForm(prev => ({
      ...prev, // ← preservar outros campos!
      [name]: type === 'checkbox' ? checked : value
    }));
  };

  return (
    <form>
      <input name="nome" value={form.nome} onChange={handleChange} />
      <input name="email" value={form.email} onChange={handleChange} />
      <input name="aceito" type="checkbox" checked={form.aceito} onChange={handleChange} />
    </form>
  );
}

useEffect — side effects com controle

useEffect — os 3 padrões essenciais
import { useState, useEffect } from 'react';

// PADRÃO 1: buscar dados na montagem
function ListaUsuarios() {
  const [usuarios, setUsuarios] = useState([]);
  const [carregando, setCarregando] = useState(true);
  const [erro, setErro] = useState(null);

  useEffect(() => {
    let cancelado = false; // evita setState em componente desmontado

    async function buscar() {
      try {
        const res = await fetch('/api/usuarios');
        const dados = await res.json();
        if (!cancelado) setUsuarios(dados);
      } catch (e) {
        if (!cancelado) setErro(e.message);
      } finally {
        if (!cancelado) setCarregando(false);
      }
    }

    buscar();

    // Cleanup: cancela se o componente desmontar antes de receber resposta
    return () => { cancelado = true; };
  }, []); // [] = roda só na montagem

  if (carregando) return <p>Carregando...</p>;
  if (erro) return <p>Erro: {erro}</p>;
  return <ul>{usuarios.map(u => <li key={u.id}>{u.nome}</li>)}</ul>;
}

// PADRÃO 2: reagir a mudança de prop/estado (dependências)
function BuscaUsuario({ id }) {
  const [usuario, setUsuario] = useState(null);

  useEffect(() => {
    if (!id) return;
    fetch(`/api/usuarios/${id}`)
      .then(r => r.json())
      .then(setUsuario);
  }, [id]); // re-executa toda vez que `id` mudar

  return usuario ? <p>{usuario.nome}</p> : null;
}

// PADRÃO 3: subscription com cleanup (ex: event listener)
function TamanhoJanela() {
  const [tamanho, setTamanho] = useState({
    w: window.innerWidth, h: window.innerHeight
  });

  useEffect(() => {
    const handleResize = () => setTamanho({ w: window.innerWidth, h: window.innerHeight });
    window.addEventListener('resize', handleResize);
    // Cleanup OBRIGATÓRIO — remove o listener quando o componente desmontar
    return () => window.removeEventListener('resize', handleResize);
  }, []); // sem deps = registra uma vez, limpa ao desmontar

  return <p>{tamanho.w} x {tamanho.h}</p>;
}

useContext — compartilhar estado globalmente

contexto completo — criação, provider e consumo
import { createContext, useContext, useState } from 'react';

// 1. Criar o contexto
const TemaContext = createContext(null);

// 2. Criar o Provider (envolve a árvore que precisa do tema)
export function TemaProvider({ children }) {
  const [tema, setTema] = useState('escuro');

  const alternarTema = () => setTema(t => t === 'escuro' ? 'claro' : 'escuro');

  return (
    <TemaContext.Provider value={{ tema, alternarTema }}>
      {children}
    </TemaContext.Provider>
  );
}

// 3. Hook customizado para consumir (boa prática!)
export function useTema() {
  const ctx = useContext(TemaContext);
  if (!ctx) throw new Error('useTema deve ser usado dentro de TemaProvider');
  return ctx;
}

// 4. Usar em qualquer componente da árvore — sem prop drilling!
function BotaoTema() {
  const { tema, alternarTema } = useTema();
  return (
    <button onClick={alternarTema}>
      Tema atual: {tema}
    </button>
  );
}

// 5. Na raiz do app:
function App() {
  return (
    <TemaProvider>
      <Header />     {/* pode usar useTema() */}
      <Main />       {/* pode usar useTema() */}
      <Footer />     {/* pode usar useTema() */}
    </TemaProvider>
  );
}

useMemo & useCallback — otimização de performance

quando e como usar memoização
import { useState, useMemo, useCallback, memo } from 'react';

// useMemo — evitar recalcular valor caro a cada render
function TabelaProdutos({ produtos, filtro }) {
  // ✅ só recalcula quando produtos ou filtro mudar
  const produtosFiltrados = useMemo(() => {
    console.log('Filtrando...'); // deve aparecer poucas vezes
    return produtos.filter(p =>
      p.nome.toLowerCase().includes(filtro.toLowerCase())
    ).sort((a, b) => b.preco - a.preco);
  }, [produtos, filtro]);

  return <ul>{produtosFiltrados.map(p => <li key={p.id}>{p.nome}</li>)}</ul>;
}

// useCallback + memo — evitar re-render de filhos
const ItemLista = memo(function ItemLista({ item, onDeletar }) {
  console.log('Renderizando item:', item.id); // deve aparecer só quando necessário
  return (
    <li>
      {item.nome}
      <button onClick={() => onDeletar(item.id)}>Deletar</button>
    </li>
  );
});

function ListaItens() {
  const [itens, setItens] = useState([...]);
  const [outraCoisa, setOutraCoisa] = useState(false);

  // ✅ mesma referência de função entre renders — ItemLista não re-renderiza à toa
  const handleDeletar = useCallback((id) => {
    setItens(prev => prev.filter(item => item.id !== id));
  }, []); // sem deps = referência estável para sempre

  return (
    <>
      <button onClick={() => setOutraCoisa(v => !v)}>Toggle</button>
      {itens.map(item => (
        <ItemLista key={item.id} item={item} onDeletar={handleDeletar} />
      ))}
    </>
  );
}
// react 18 — novidades

React 18 — Concurrent Mode & Server Components

Concurrent Mode

Renderização interrompível

React 18 pode pausar, retomar e abandonar renders. Isso significa que a UI pode continuar responsiva mesmo durante renders pesados.

Server Components

Componentes no servidor

RSC (React Server Components) rodam apenas no servidor — zero bundle size, acesso direto ao banco, mas sem hooks de estado ou eventos.

Suspense Data

Suspense para dados

Suspense agora funciona com data fetching (antes era só lazy loading). Componentes declaram que "suspendem" enquanto aguardam dados.

Automatic Batching

Batch automático

No React 18, múltiplos setState em qualquer contexto (async, setTimeout) são batched automaticamente em um único re-render.

useTransition — manter UI responsiva

busca com useTransition — UI nunca trava
import { useState, useTransition } from 'react';

function BuscaProdutos({ todosOsProdutos }) {
  const [query, setQuery] = useState('');
  const [resultados, setResultados] = useState(todosOsProdutos);
  const [isPending, startTransition] = useTransition();

  const handleBusca = (e) => {
    const valor = e.target.value;
    setQuery(valor); // atualização urgente — input responde imediatamente

    startTransition(() => {
      // atualização não-urgente — pode ser interrompida se novo input chegar
      const filtrado = todosOsProdutos.filter(p =>
        p.nome.toLowerCase().includes(valor.toLowerCase())
      );
      setResultados(filtrado);
    });
  };

  return (
    <>
      <input value={query} onChange={handleBusca} placeholder="Buscar..." />
      {isPending && <span>Filtrando...</span>}
      <ul style={{ opacity: isPending ? 0.5 : 1 }}>
        {resultados.map(p => <li key={p.id}>{p.nome}</li>)}
      </ul>
    </>
  );
}

src/todoReducer.js
// Estado inicial
export const initialState = {
  tarefas: [],
  filtro: 'todas', // 'todas' | 'ativas' | 'concluidas'
};

// Reducer — todas as transições de estado em um lugar
export function todoReducer(state, action) {
  switch (action.type) {
    case 'ADICIONAR':
      return {
        ...state,
        tarefas: [...state.tarefas, {
          id: Date.now(),
          texto: action.payload,
          concluida: false,
          criadaEm: new Date().toISOString(),
        }]
      };
    case 'TOGGLE':
      return {
        ...state,
        tarefas: state.tarefas.map(t =>
          t.id === action.payload ? { ...t, concluida: !t.concluida } : t
        )
      };
    case 'DELETAR':
      return { ...state, tarefas: state.tarefas.filter(t => t.id !== action.payload) };
    case 'LIMPAR_CONCLUIDAS':
      return { ...state, tarefas: state.tarefas.filter(t => !t.concluida) };
    case 'SET_FILTRO':
      return { ...state, filtro: action.payload };
    default:
      return state;
  }
}

Contexto + Provider
Crie src/TodoContext.jsx — contexto que expõe estado e dispatch + persistência no localStorage:
src/TodoContext.jsx
import { createContext, useContext, useReducer, useEffect } from 'react';
import { todoReducer, initialState } from './todoReducer';

const TodoContext = createContext(null);

export function TodoProvider({ children }) {
  const [state, dispatch] = useReducer(
    todoReducer,
    initialState,
    // Inicializar com dados do localStorage (lazy initializer)
    (init) => {
      const salvo = localStorage.getItem('todos');
      return salvo ? JSON.parse(salvo) : init;
    }
  );

  // Persistir no localStorage a cada mudança
  useEffect(() => {
    localStorage.setItem('todos', JSON.stringify(state));
  }, [state]);

  return (
    <TodoContext.Provider value={{ state, dispatch }}>
      {children}
    </TodoContext.Provider>
  );
}

export const useTodo = () => {
  const ctx = useContext(TodoContext);
  if (!ctx) throw new Error('useTodo fora do TodoProvider');
  return ctx;
};

Componente principal App
Crie src/App.jsx usando todos os hooks:
src/App.jsx
import { useState, useMemo, useCallback } from 'react';
import { useTodo } from './TodoContext';

function App() {
  const { state, dispatch } = useTodo();
  const [inputValue, setInputValue] = useState('');

  // Filtrar tarefas — recalcula só quando tarefas ou filtro mudam
  const tarefasFiltradas = useMemo(() => {
    switch (state.filtro) {
      case 'ativas':    return state.tarefas.filter(t => !t.concluida);
      case 'concluidas': return state.tarefas.filter(t => t.concluida);
      default:          return state.tarefas;
    }
  }, [state.tarefas, state.filtro]);

  // Callbacks memoizados
  const handleAdicionar = useCallback((e) => {
    e.preventDefault();
    if (!inputValue.trim()) return;
    dispatch({ type: 'ADICIONAR', payload: inputValue.trim() });
    setInputValue('');
  }, [inputValue, dispatch]);

  const handleToggle = useCallback((id) => {
    dispatch({ type: 'TOGGLE', payload: id });
  }, [dispatch]);

  const handleDeletar = useCallback((id) => {
    dispatch({ type: 'DELETAR', payload: id });
  }, [dispatch]);

  const pendentes = state.tarefas.filter(t => !t.concluida).length;

  return (
    <div className="app">
      <h1>Minhas Tarefas <span>({pendentes} pendentes)</span></h1>

      <form onSubmit={handleAdicionar}>
        <input
          value={inputValue}
          onChange={e => setInputValue(e.target.value)}
          placeholder="Nova tarefa..."
          autoFocus
        />
        <button type="submit">Adicionar</button>
      </form>

      <div className="filtros">
        {['todas', 'ativas', 'concluidas'].map(f => (
          <button
            key={f}
            className={state.filtro === f ? 'ativo' : ''}
            onClick={() => dispatch({ type: 'SET_FILTRO', payload: f })}
          >
            {f.charAt(0).toUpperCase() + f.slice(1)}
          </button>
        ))}
      </div>

      <ul>
        {tarefasFiltradas.map(tarefa => (
          <li key={tarefa.id}>
            <input
              type="checkbox"
              checked={tarefa.concluida}
              onChange={() => handleToggle(tarefa.id)}
            />
            <span style={{ textDecoration: tarefa.concluida ? 'line-through' : 'none' }}>
              {tarefa.texto}
            </span>
            <button onClick={() => handleDeletar(tarefa.id)}>✕</button>
          </li>
        ))}
      </ul>

      {state.tarefas.some(t => t.concluida) && (
        <button onClick={() => dispatch({ type: 'LIMPAR_CONCLUIDAS' })}>
          Limpar concluídas
        </button>
      )}
    </div>
  );
}
// armadilhas

Erros clássicos com React

⚠️ Loop infinito no useEffect — a armadilha mais comum

useEffect com um objeto ou função como dependência causa loop infinito porque cada render cria uma nova referência para esses valores.

// ❌ Loop infinito — 'config' é novo objeto a cada render
useEffect(() => { fetchData(config); }, [config]);

// ✅ Extrair os valores primitivos das dependências
useEffect(() => { fetchData(config); }, [config.url, config.method]);
⚠️ Stale closure — função "esquecida" no tempo

Uma função criada dentro de useEffect com deps vazia captura os valores de quando foi criada — não os valores atuais.

// ❌ count sempre será 0 dentro do setInterval
useEffect(() => {
  const id = setInterval(() => {
    setCount(count + 1); // count está "stale" — sempre 0
  }, 1000);
  return () => clearInterval(id);
}, []);

// ✅ Usar função de atualização — não captura o valor, recebe o atual
useEffect(() => {
  const id = setInterval(() => {
    setCount(prev => prev + 1); // sempre incrementa corretamente
  }, 1000);
  return () => clearInterval(id);
}, []);
⚠️ useMemo/useCallback em tudo — premature optimization

Adicionar useMemo e useCallback em todo componente é um erro comum de devs que leram sobre performance mas não entendem o custo de memoização (que também tem custo de memória e tempo).

✅ Regra: só use useMemo/useCallback quando (1) o cálculo é comprovadamente caro, (2) a função é passada para componente filho memoizado com memo(), ou (3) é dependência de outro hook. Profile primeiro, otimize depois.

⚠️ Atualizar state de componente desmontado

Se um fetch termina depois que o componente desmontou (usuário navegou para outra página), setar state nele gera um warning e pode causar memory leaks.

✅ Solução: use a flag de cancelamento no useEffect (como mostrado nos exemplos acima) ou use AbortController para cancelar o fetch diretamente.
