// MOTIVAÇÃO

Por que isso importa?

Qualquer pessoa aprende git add, git commit e git push em um dia. O que separa um desenvolvedor júnior de um sênior no uso de Git não é conhecer esses comandos básicos — é saber reescrever o histórico com precisão cirúrgica, marcar versões de forma profissional e escolher a estratégia de branching certa para cada tipo de time e projeto.

Rebase interativo é a ferramenta que transforma um histórico caótico de WIP commits em uma narrativa limpa e revisável. Tags são o contrato formal entre o seu repositório e o mundo — elas marcam exatamente o commit que foi para produção. E estratégias de branch determinam se seu time consegue fazer deploys com confiança ou vive apagando incêndios.

💼 Por que um engenheiro sênior domina esses recursos
Sêniors sabem que o histórico do Git é documentação. Um git log --oneline bem estruturado conta a história de uma feature em 5 linhas. Rebase interativo permite que o desenvolvedor entregue commits atômicos e revisáveis mesmo tendo trabalhado de forma iterativa e messy durante o desenvolvimento. Tags e releases criam pontos de referência que o time de infra, QA e suporte usam para rastrear exatamente o que está em produção.

A Analogia do Mundo Real

🎬 Analogia: Edição de Filme
Imagine que você está filmando um documentário. Durante as gravações, você captura muita coisa: takes ruins, ensaios, imagens de câmera na mão tremendo, cenas fora de ordem. O produto final — o filme editado — é uma narrativa coesa e linear que o público vai assistir.

Git rebase é a ilha de edição. Você pega todos aqueles takes brutos (commits de "fix typo", "wip", "tentei de novo") e reorganiza, comprime e aperfeiçoa até criar uma sequência limpa. Git tags são o lançamento do filme: a versão v1.0 é o que chegou nos cinemas, registrado para sempre. Estratégias de branch são os acordos de produção: quem filma o que, quando cada equipe entrega suas cenas e como tudo se junta na montagem final.

git commit durante dev (WIP commits)
🎬 Ilha de edição
git rebase -i (rebase interativo)
🎥 Lançamento nos cinemas
git tag v1.0.0 + Release
📋 Acordo de produção entre equipes
Estratégia de branching (GitFlow, trunk-based)

Git Rebase — Reescrevendo o Histórico

O git rebase move ou reescreve commits de uma branch como se eles tivessem sido criados em outro ponto da história. Existem duas formas principais de uso: o rebase simples (para integrar mudanças de outra branch mantendo histórico linear) e o rebase interativo (para reorganizar, squash e editar commits).

Rebase Simples vs. Merge

// merge vs rebase — diferença no histórico resultante
gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C" commit id: "D" checkout main commit id: "E"
✅ git merge

Histórico preservado

Cria um "merge commit" que une os dois lados. O histórico mostra exatamente o que aconteceu — incluindo o ponto de divergência. Ideal quando o histórico da integração importa.

⟳ git rebase

Histórico linear

Reaplica os commits da feature como se tivessem começado a partir do último commit da main. O resultado é um histórico reto, sem bifurcações — mais fácil de ler com git log.

🚫 A Regra de Ouro do Rebase
Nunca faça rebase de commits que já foram enviados para um repositório compartilhado (push). Rebase reescreve o histórico criando novos commits com hashes diferentes. Se outra pessoa já baixou os commits originais, isso causa conflitos sérios e confusão para todo o time.

Rebase Interativo — A ferramenta mais poderosa

O git rebase -i (interactive) abre um editor onde você decide o que fazer com cada commit de uma branch. É aqui que a magia acontece.

bash
$ git rebase -i HEAD~5
# Abre o editor com os últimos 5 commits
# Cada linha tem um comando + hash + mensagem
// o que aparece no editor (lista de commits a editar)
pick 3a1b2c4 feat: adiciona autenticação OAuth
pick 7d8e9f0 fix typo in variable name
pick 1a2b3c4 wip: trying to fix the bug
pick 5e6f7a8 WIP continue
pick 9b0c1d2 feat: autenticação funcionando agora

# Comandos disponíveis:
# p, pick   = usa o commit como está
# r, reword = usa o commit mas edita a mensagem
# e, edit   = usa o commit mas para para edição
# s, squash = une ao commit anterior
# f, fixup  = como squash mas descarta a mensagem
# d, drop   = remove o commit completamente
# reorder   = basta reordenar as linhas

O mais comum em times profissionais é transformar isso em:

// após editar: commits organizados e limpos
pick 3a1b2c4 feat: adiciona autenticação OAuth
fixup 7d8e9f0 fix typo in variable name
squash 1a2b3c4 wip: trying to fix the bug
squash 5e6f7a8 WIP continue
squash 9b0c1d2 feat: autenticação funcionando agora

# Resultado: 1 commit limpo com mensagem editável
# "feat: adiciona autenticação OAuth"
🔥 Pro tip: squash antes de abrir PR
Antes de abrir um Pull Request, rode git rebase -i origin/main para fazer squash dos commits de WIP em commits atômicos e bem descritos. Seu reviewer vai agradecer — e o histórico do projeto vai ficar limpo para sempre.
// TAGS & RELEASES

Tags — Marcando Versões com Semver

Tags são ponteiros permanentes para commits específicos. Diferente de branches (que avançam com novos commits), uma tag é imutável — ela sempre aponta para o mesmo commit. Tags são a forma padrão da indústria de marcar versões de software prontas para release.

Versionamento Semântico (SemVer)

O padrão MAJOR.MINOR.PATCH (ex: v2.1.3) é o contrato que informa à comunidade o impacto de cada atualização:

Componente Quando incrementar Exemplo Impacto
MAJOR Breaking changes — API incompatível com versão anterior v1.x → v2.0.0 ⚠️ Quebra
MINOR Nova funcionalidade compatível com versão anterior v2.0 → v2.1.0 ✓ Seguro
PATCH Bug fixes compatíveis — sem novas funcionalidades v2.1 → v2.1.3 ✓ Seguro
PRE-RELEASE Versões instáveis para testes v3.0.0-beta.1 ⚡ Instável

Tipos de Tag: Lightweight vs Annotated

🏷️

Lightweight

Apenas um ponteiro para um commit. Sem metadados. Útil para tags temporárias ou internas. git tag v1.0.0

📦

Annotated (recomendada)

Objeto Git completo com autor, data, mensagem e pode ser assinada. Use sempre em releases públicas. git tag -a v1.0.0 -m "Release"

// histórico com tags em pontos de release
gitGraph commit id: "init" commit id: "feat: login" tag: "v0.1.0" commit id: "feat: dashboard" commit id: "fix: crash" tag: "v0.1.1" commit id: "feat: payments" commit id: "feat: reports" tag: "v1.0.0" commit id: "feat: export"

Criando e Publicando Tags

bash
# Criar tag annotated
$ git tag -a v1.2.0 -m "feat: adiciona exportação CSV e PDF"

# Publicar tag no remote (push não envia tags por padrão!)
$ git push origin v1.2.0

# Ou publicar todas as tags de uma vez
$ git push origin --tags

# Listar todas as tags
$ git tag -l
v0.1.0
v0.1.1
v1.0.0
v1.2.0

# Ver detalhes de uma tag annotated
$ git show v1.2.0

# Deletar tag local e remote (se errou)
$ git tag -d v1.2.0
$ git push origin --delete v1.2.0
💡 GitHub Releases
No GitHub, ao criar uma tag você pode associá-la a uma Release: uma página com changelog, assets para download (binários, ZIPs) e notas de versão. Isso é automático se você criar a tag via interface do GitHub ou usando gh release create com o GitHub CLI.
// ESTRATÉGIAS

Estratégias de Branching

Escolher a estratégia errada de branching pode destruir a produtividade de um time. As três abordagens mais usadas na indústria são GitFlow, GitHub Flow e Trunk-Based Development — cada uma com trade-offs claros dependendo do ritmo de deploy e tamanho do time.

🌊
GitFlow
Branches main, develop, feature/*, release/*, hotfix/*. Ótimo para software com releases marcadas e suporte a múltiplas versões. Mais overhead, mas muito controle.
🚀
GitHub Flow
Só main + feature branches de curta duração. PR → review → merge → deploy contínuo. Simples, rápido, ideal para times com CI/CD maduro e deploy frequente.
🌱
Trunk-Based Dev
Todos commitam direto na main (ou branches de max. 2 dias). Feature flags escondem código incompleto. Máxima velocidade de integração, mínimo de conflitos. Padrão em Google e Netflix.

Comparativo para ajudar na escolha

Critério GitFlow GitHub Flow Trunk-Based
Frequência de deploy Semanal/mensal Diária Várias vezes/dia
Complexidade Alta Baixa Média
Conflitos de merge Frequentes Ocasionais Raros
Suporte múltiplas versões ✓ Sim Não ideal Não ideal
Ideal para Produtos com versões LTS, apps mobile SaaS, APIs, times pequenos Times maduros com CI/CD + feature flags
// gitflow — fluxo típico com hotfix e release branch
gitGraph commit id: "init" tag: "v1.0.0" branch develop checkout develop commit id: "dev start" branch feature/login checkout feature/login commit id: "login: form" commit id: "login: auth" checkout develop merge feature/login id: "merge login" branch release/1.1 checkout release/1.1 commit id: "release prep" checkout main merge release/1.1 id: "release" tag: "v1.1.0" checkout develop merge release/1.1 id: "sync" branch hotfix/crash checkout hotfix/crash commit id: "fix: crash" checkout main merge hotfix/crash id: "hotfix" tag: "v1.1.1"

Outros Comandos Avançados Essenciais

git stash — Guardando trabalho em progresso

Quando você precisa trocar de branch mas tem mudanças não commitadas que não estão prontas para commit ainda, o git stash salva essas mudanças em uma pilha temporária.

bash
# Guardar mudanças com descrição
$ git stash push -m "feature: login em progresso"

# Listar stashes
$ git stash list
stash@{0}: On main: feature: login em progresso
stash@{1}: On develop: fix parcial do bug

# Aplicar o stash mais recente e remover da pilha
$ git stash pop

# Aplicar stash específico sem remover
$ git stash apply stash@{1}

git cherry-pick — Transplantando commits individuais

Permite aplicar um commit específico de qualquer branch para a branch atual. Muito usado para aplicar hotfixes em múltiplas branches de release ou trazer uma feature específica sem fazer merge completo.

bash
# Pegar o hash do commit que queremos
$ git log --oneline feature/payments
a1b2c3d fix: corrige cálculo de taxas
d4e5f6g feat: integra gateway de pagamento

# Aplicar apenas o fix na branch atual (ex: hotfix/v1.1)
$ git cherry-pick a1b2c3d

# Cherry-pick de um range de commits
$ git cherry-pick a1b2c3d^..d4e5f6g

git bisect — Caçando o commit que introduziu o bug

O git bisect usa busca binária para encontrar qual commit introduziu um bug. Você informa um commit "bom" (sem bug) e um "ruim" (com bug), e o Git navega automaticamente pelos commits do meio para você testar.

bash
$ git bisect start
$ git bisect bad # commit atual tem o bug
$ git bisect good v1.0.0 # v1.0.0 estava ok
Bisecting: 23 revisions left to test...
# Git vai para o commit do meio — você testa
$ git bisect good # ou: git bisect bad
# Repete até encontrar o commit exato
a1b2c3d is the first bad commit
$ git bisect reset # volta ao HEAD original

Passo a Passo: Rebase Interativo na Prática

Vamos simular um fluxo real de desenvolvimento com commits bagunçados e organizá-los com rebase interativo.

1
Criar repositório de prática
Monte um repo com commits propositalmente bagunçados para praticar rebase
$ mkdir git-avancado-lab && cd git-avancado-lab
$ git init && git commit --allow-empty -m "chore: init"
$ git checkout -b feature/user-profile
$ echo "# User Profile" > profile.py && git add . && git commit -m "wip"
$ echo "class User: pass" >> profile.py && git add . && git commit -m "fix"
$ echo " def __init__(self): pass" >> profile.py && git add . && git commit -m "more stuff"
$ echo " def get_name(self): return self.name" >> profile.py && git add . && git commit -m "feat: user profile class"
$ git log --oneline
2
Iniciar rebase interativo dos últimos 4 commits
Vamos compactar os 3 primeiros commits WIP em um único commit descritivo
$ git rebase -i HEAD~4
# O editor abre. Edite para ficar assim:
pick abc1234 wip
squash def5678 fix
squash ghi9012 more stuff
squash jkl3456 feat: user profile class
# Salve e feche o editor
# Um segundo editor abre para a mensagem final
# Escreva: "feat: adiciona classe User com getter de nome"
# Salve novamente
3
Verificar o resultado
Confirmar que os 4 commits viraram 1 commit limpo
$ git log --oneline
a9b8c7d feat: adiciona classe User com getter de nome
1234567 chore: init
# Perfeito! 2 commits limpos ao invés de 5 bagunçados
4
Criar uma tag de release
Marcar o primeiro release do módulo de User Profile
# Mergear na main primeiro
$ git checkout main && git merge feature/user-profile --ff-only

# Criar tag annotated
$ git tag -a v0.1.0 -m "feat: módulo inicial de User Profile"

# Verificar
$ git log --oneline --decorate
a9b8c7d (HEAD -> main, tag: v0.1.0, feature/user-profile) feat: adiciona classe User
1234567 chore: init
5
Praticar stash e cherry-pick
Simular o fluxo de guardar trabalho e trazer commits específicos
# Criar mudanças não commitadas
$ echo " def get_email(self): pass" >> profile.py
$ git stash push -m "wip: adiciona getter de email"
$ git status
nothing to commit, working tree clean

# Recuperar o stash
$ git stash pop
$ git status
modified: profile.py

Script de Automação de Tags e Releases

Um script Bash prático para automatizar o processo de criação de tags e releases seguindo SemVer.

// release.sh — script de release com semver automático
#!/usr/bin/env bash
# release.sh — automatiza o fluxo de criação de tags e releases
# Uso: ./release.sh [major|minor|patch] "Descrição do release"
# Exemplo: ./release.sh minor "adiciona exportação CSV"

set -euo pipefail

BUMP=${1:-patch}       # tipo de bump: major, minor ou patch
MESSAGE=${2:-""}       # mensagem do release

# ─── Validações ───────────────────────────────────────────────
if [[ -n "$(git status --porcelain)" ]]; then
  echo "❌ Erro: working tree não está limpa. Commit ou stash suas mudanças."
  exit 1
fi

BRANCH=$(git rev-parse --abbrev-ref HEAD)
if [[ "$BRANCH" != "main" && "$BRANCH" != "master" ]]; then
  echo "⚠️  Aviso: você está na branch '$BRANCH', não em main."
  read -p "Continuar mesmo assim? (s/N): " confirm
  [[ "$confirm" != "s" ]] && exit 0
fi

# ─── Buscar última tag e calcular próxima versão ──────────────
LAST_TAG=$(git describe --tags --abbrev=0 2>/dev/null || echo "v0.0.0")
echo "📌 Última tag: $LAST_TAG"

# Remove o 'v' e separa os números
VERSION="${LAST_TAG#v}"
MAJOR=$(echo "$VERSION" | cut -d. -f1)
MINOR=$(echo "$VERSION" | cut -d. -f2)
PATCH=$(echo "$VERSION" | cut -d. -f3)

# Incrementa o componente correto
case "$BUMP" in
  major) MAJOR=$((MAJOR + 1)); MINOR=0; PATCH=0 ;;
  minor) MINOR=$((MINOR + 1)); PATCH=0 ;;
  patch) PATCH=$((PATCH + 1)) ;;
  *)
    echo "❌ Tipo de bump inválido. Use: major, minor ou patch"
    exit 1
    ;;
esac

NEW_TAG="v${MAJOR}.${MINOR}.${PATCH}"
echo "🚀 Nova versão: $NEW_TAG"

# ─── Gerar changelog automático desde a última tag ────────────
CHANGELOG=$(git log "${LAST_TAG}..HEAD" --oneline --no-merges)
echo ""
echo "📝 Commits desde $LAST_TAG:"
echo "$CHANGELOG"
echo ""

# ─── Confirmar antes de criar ─────────────────────────────────
read -p "Criar tag $NEW_TAG? (s/N): " confirm
[[ "$confirm" != "s" ]] && exit 0

# ─── Criar tag annotated ──────────────────────────────────────
FULL_MESSAGE="${MESSAGE:-"Release $NEW_TAG"}

$CHANGELOG"

git tag -a "$NEW_TAG" -m "$FULL_MESSAGE"
echo "✅ Tag $NEW_TAG criada localmente"

# ─── Push da tag ──────────────────────────────────────────────
read -p "Fazer push de $NEW_TAG para origin? (s/N): " push_confirm
if [[ "$push_confirm" == "s" ]]; then
  git push origin "$NEW_TAG"
  echo "✅ Tag publicada em origin/$NEW_TAG"
  echo ""
  echo "💡 Para criar um GitHub Release:"
  echo "   gh release create $NEW_TAG --notes \"$MESSAGE\""
fi

echo ""
echo "🎉 Release $NEW_TAG concluído!"
🔥 Pro tip: GitHub CLI para releases
Com o GitHub CLI instalado, você pode criar releases completas direto do terminal: gh release create v1.0.0 --generate-notes — o --generate-notes gera automaticamente as release notes a partir dos PRs mergeados.
// ⚠️ PITFALLS

Armadilhas Comuns

⚠️ Rebase em branch compartilhada
Fazer git rebase em uma branch que outros desenvolvedores já baixaram causa conflitos sérios. Os hashes dos commits mudam, e os outros desenvolvedores ficam com um histórico incompatível. Regra absoluta: nunca rebaze o que já foi pushed para um remote compartilhado — a não ser que todos os colaboradores concordem e saibam o que fazer.
⚠️ Tags sem push ficam apenas no local
git push não envia tags automaticamente. Se você criar uma tag e esquecer de fazer git push origin --tags, a tag existe apenas na sua máquina. O CI/CD, os outros devs e o GitHub não vão vê-la. Sempre use git push origin <tagname> após criar uma tag de release.
⚠️ Squash antes de confirmar com o time
Squash interativo que engloba commits já mergeados em branches compartilhadas pode criar duplicatas e confusão. Faça squash apenas de commits que ainda não foram integrados em branches de longa duração (main, develop).
⚠️ Branch strategy sem CI/CD automatizado
Trunk-Based Development sem CI/CD sólido é uma receita para quebrar produção constantemente. Só adote estratégias de integração contínua agressiva quando o pipeline de testes estiver robusto e cobrindo a maior parte do código.

// GLOSSÁRIO

Termos desta Aula

Rebase
Operação que reaplica commits de uma branch sobre outra, reescrevendo o histórico para criar uma linha de desenvolvimento linear.
Squash
Técnica de combinar múltiplos commits em um único commit, usada para limpar o histórico antes de um PR ou merge.
Tag Annotated
Objeto Git completo que armazena nome do tagger, data, mensagem e pode ser assinado com GPG. Recomendado para releases oficiais.
SemVer (Versionamento Semântico)
Convenção MAJOR.MINOR.PATCH que comunica o impacto de cada versão: MAJOR quebra compatibilidade, MINOR adiciona features, PATCH corrige bugs.
GitFlow
Modelo de branching com branches dedicadas para main, develop, features, releases e hotfixes. Mais estruturado, ideal para releases com ciclos definidos.
Trunk-Based Development
Estratégia onde todos commitam na branch principal frequentemente, usando feature flags para esconder código incompleto. Favorece CI/CD de alta frequência.
Cherry-pick
Comando que copia um commit específico de qualquer branch para a branch atual, sem fazer merge completo de toda a branch.
git bisect
Ferramenta de busca binária no histórico do Git para encontrar o commit exato que introduziu um bug.

🏁 Checkpoint — Desafio Final

Você gerencia um projeto Python chamado task-manager. Faça o seguinte fluxo completo numa sessão de terminal:

  • Iniciar um repositório com 1 commit inicial vazio (chore: init)
  • Criar uma branch feature/task-crud e fazer 5 commits com mensagens de WIP
  • Usar git rebase -i HEAD~5 para compactar em 1 commit com mensagem no estilo Conventional Commits: feat: implementa CRUD de tarefas
  • Mergear na main com --ff-only para garantir histórico linear
  • Criar uma tag annotated v0.1.0 com mensagem descritiva
  • Criar uma segunda feature branch, fazer algumas mudanças, usar git stash para salvar o trabalho em progresso e trocar de branch
  • Recuperar o stash com git stash pop
💬 Dica: use git log --oneline --decorate --graph --all para visualizar todo o histórico com branches e tags ao mesmo tempo. É o comando preferido dos sêniors para entender o estado do repositório.
