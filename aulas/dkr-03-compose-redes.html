// MOTIVAÇÃO

Por que isso importa?

Nenhuma aplicação real roda em um único container. Uma API precisa de um banco de dados, um sistema de cache, talvez um worker de background e um proxy reverso. Sem Docker Compose, subir esse ambiente significa executar cinco comandos docker run com dezenas de flags, na ordem certa, esperando cada serviço iniciar antes do próximo — e torcer para não errar nenhuma variável de ambiente.

O Docker Compose resolve isso com um único arquivo declarativo docker-compose.yml que descreve toda a infraestrutura da aplicação: quais containers rodam, como se conectam, quais variáveis recebem, quais volumes compartilham. Um único comando — docker compose up — sobe tudo. E docker compose down derruba tudo limpo.

💼 Por que um engenheiro sênior domina Docker Compose
Em times de produto, o docker-compose.yml é o "contrato de ambiente" — qualquer dev clona o repo, roda docker compose up e tem o mesmo ambiente que produção em minutos. Sem "funciona na minha máquina". Sem horas configurando PostgreSQL localmente. Compose é também a base para entender Kubernetes: os conceitos de serviços, redes e volumes são diretamente análogos.

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense numa produção teatral na Broadway. Um ator solo consegue fazer um monólogo sozinho — isso é um único container. Mas uma peça completa precisa de elenco, cenário, iluminação, sonoplastia, figurino e direção — todos trabalhando juntos, em sincronia, num ambiente controlado.

O roteiro de produção descreve cada papel, onde cada pessoa fica no palco, como se comunicam e em que ordem cada cena acontece. Você não telefona para cada ator individualmente toda noite — o diretor lê o roteiro e coordena tudo. O docker-compose.yml é esse roteiro de produção: ele descreve cada serviço (ator), a rede interna do palco (como se ouvem), as variáveis de ambiente (o script de cada um) e os volumes (os figurinos e adereços compartilhados).
📋 Roteiro de produção completo → docker-compose.yml 🎭 Cada ator no palco → Cada service no Compose 🎙️ Palco onde todos se ouvem → Rede interna (bridge network) 📜 Script personalizado de cada ator → environment variables por serviço 🎁 Adereços compartilhados entre cenas → volumes compartilhados 🎬 "Cena 3 só após cena 2" → depends_on com healthcheck

Docker Compose — o que é e como funciona

O Docker Compose é uma ferramenta que lê um arquivo docker-compose.yml e gerencia o ciclo de vida de múltiplos containers como uma unidade. Ele cria automaticamente uma rede privada compartilhada para todos os serviços definidos, permitindo que se comuniquem pelo nome do serviço (DNS interno).

Ciclo de vida: docker compose up → running → docker compose down

O ponto-chave da rede interna do Compose: quando a api quer se conectar ao banco de dados, ela não usa localhost nem um IP — ela usa o nome do serviço como hostname. Se o serviço se chama postgres no Compose, a connection string é postgresql://user:pass@postgres:5432/db. O Docker resolve o DNS internamente.

🔥 Compose V2 — use sempre docker compose (sem hífen)
O Docker Compose V1 era um binário separado (docker-compose com hífen). Desde o Docker Desktop 3.4 e Docker Engine 20.10, o Compose V2 é integrado ao CLI do Docker como plugin (docker compose sem hífen). A V2 é mais rápida, suporta BuildKit nativo e é a versão mantida. Se você ainda usa docker-compose, atualize.
// ANATOMIA

Anatomia do docker-compose.yml

O arquivo Compose usa YAML e tem quatro seções principais: services, networks, volumes e configs. A seção services é obrigatória — as demais são criadas automaticamente se não declaradas.

docker-compose.yml — estrutura anotada
# Versão do schema do Compose (omitir aceita a versão mais recente)
# A partir do Compose V2, a chave "version" é obsoleta

services:          # ← Seção principal: define cada container

  api:             # ← Nome do serviço (vira hostname na rede interna)
    build: .       # ← Builda a imagem do Dockerfile do diretório atual
    # OU: image: python:3.12-slim  ← Usa imagem pronta do registry
    
    ports:
      - "8000:8000"  # ← "HOST:CONTAINER" — publica para o mundo externo
    
    environment:     # ← Variáveis de ambiente inline (evite segredos aqui)
      - DEBUG=false
      - DATABASE_URL=postgresql://user:senha@postgres:5432/mydb
    
    env_file:        # ← Carrega variáveis de um arquivo .env
      - .env
    
    depends_on:      # ← Garante ordem de inicialização
      postgres:
        condition: service_healthy   # ← Aguarda o healthcheck passar (não só "started")
      redis:
        condition: service_started   # ← Apenas aguarda o container iniciar
    
    volumes:
      - ./app:/app   # ← Bind mount: sincroniza pasta local com container (dev)
    
    networks:
      - backend      # ← Coloca o serviço nesta rede
    
    restart: unless-stopped   # ← Reinicia automaticamente em falhas

  postgres:
    image: postgres:16-alpine       # ← Pin de versão: sempre!
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: senha      # ← Em produção: use secrets ou env_file
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data   # ← Volume nomeado: dados persistem
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d mydb"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    networks:
      - backend

  redis:
    image: redis:7-alpine
    volumes:
      - redisdata:/data
    networks:
      - backend

networks:
  backend:           # ← Rede privada — apenas containers nesta rede se comunicam
    driver: bridge   # ← Padrão; isola da rede do host

volumes:
  pgdata:            # ← Volume nomeado: gerenciado pelo Docker, persiste entre restarts
  redisdata:
// NETWORKING

Redes no Docker — como os containers se comunicam

Por padrão, o Docker Compose cria uma rede bridge compartilhada para todos os serviços definidos no arquivo. Containers na mesma rede podem se resolver pelo nome do serviço como DNS — mas containers em redes diferentes não se enxergam.

O nginx acessa a api (mesma rede frontend). A api acessa postgres e redis (mesma rede backend). O postgres não é acessível pelo nginx — redes diferentes. A porta :5432 não está publicada no host — zero acesso externo.

Driver de RedeComportamentoQuando usar
bridge Rede privada isolada no host. Containers se comunicam pelo nome. Padrão do Compose. Uso geral — praticamente sempre
host Container compartilha a rede do host diretamente. Sem isolamento de porta. Performance máxima em cenários específicos (monitoramento, proxies)
none Container sem acesso à rede. Completamente isolado. Jobs batch que não precisam de rede. Segurança extrema.
overlay Rede que span múltiplos hosts Docker. Requer Docker Swarm ou Kubernetes. Produção distribuída — não no Compose local

Ports vs Expose — a diferença fundamental

ports:

Publica para o mundo externo

"8000:8000" mapeia a porta 8000 do container para a porta 8000 do seu laptop. Qualquer processo no host (ou na rede local) pode acessar. Use para serviços que precisam de acesso externo (nginx, api principal).

expose:

Apenas documentação interna

expose: ["5432"] documenta que o container usa essa porta, mas não a abre para o host. Containers na mesma rede ainda se comunicam — é apenas uma sinalização. Para banco de dados e caches: não use ports.

Variáveis de Ambiente — o jeito certo

Gerenciar configuração e segredos via variáveis de ambiente é o padrão da indústria (The Twelve-Factor App). O Compose oferece três formas de injetar variáveis, cada uma com seu caso de uso específico.

⚠️ .env no .gitignore — sempre!
O arquivo .env contém segredos reais e nunca deve ser commitado. Adicione ao .gitignore imediatamente. Crie um .env.example com os mesmos nomes de variáveis mas sem valores (ou com valores fictícios) para documentar o que o projeto precisa. Esse arquivo sim vai para o Git.
Referenciando variáveis do .env no docker-compose.yml
services:
  api:
    build: .
    environment:
      # Interpolação: o Compose substitui ${VAR} pelo valor do .env ou do shell
      - DATABASE_URL=${DATABASE_URL}
      - SECRET_KEY=${SECRET_KEY}
      - DEBUG=${DEBUG:-false}   # Valor padrão "false" se DEBUG não estiver definido
    env_file:
      - .env    # Carrega TODAS as variáveis do arquivo diretamente

  postgres:
    image: postgres:16-alpine
    environment:
      # Passagem direta: o Compose expande ${} na hora do up
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
// VOLUMES

Volumes — persistência e compartilhamento de dados

Por padrão, dados criados dentro de um container existem apenas enquanto o container existe. Volumes quebram esse ciclo — eles persistem no host independentemente do container, permitindo que dados sobrevivam a reinicializações, atualizações e recriações de containers.

TipoSintaxeOnde ficaQuando usar
Named volume pgdata:/var/lib/postgresql/data Gerenciado pelo Docker (/var/lib/docker/volumes/) Bancos de dados, dados de produção
Bind mount ./src:/app/src Pasta específica do seu sistema de arquivos Desenvolvimento — hot reload de código
tmpfs mount type: tmpfs, target: /tmp Memória RAM — nunca no disco Dados temporários sensíveis (tokens, caches voláteis)
Anonymous - /var/lib/mysql Gerenciado pelo Docker, sem nome Evitar — difícil de gerenciar
❌ Desenvolvimento sem bind mount
services:
  api:
    build: .
    # Sem volume: a cada mudança no código,
    # precisa fazer novo docker build!
    ports:
      - "8000:8000"
✅ Desenvolvimento com bind mount
services:
  api:
    build: .
    volumes:
      # Sincroniza código local com o container
      # Hot reload funciona sem rebuild!
      - ./app:/app/app
    ports:
      - "8000:8000"
// ORDEM E RESILIÊNCIA

depends_on & Healthchecks — ordem de inicialização

O depends_on simples garante apenas que o container do serviço dependente foi iniciado — não que está pronto para receber conexões. Um PostgreSQL pode levar 15 segundos para aceitar conexões após o processo iniciar. Sem a condição service_healthy, a API tenta conectar ao banco antes que ele esteja pronto e falha.

❌ depends_on simples — race condition
services:
  api:
    build: .
    depends_on:
      - postgres   # Apenas espera o container INICIAR
      # postgres pode não estar pronto ainda!
      # A API vai tentar conectar e falhar
✅ depends_on com healthcheck — seguro
services:
  api:
    build: .
    depends_on:
      postgres:
        condition: service_healthy  # Espera o HEALTHCHECK passar
      redis:
        condition: service_started  # Redis inicia rápido, ok aqui
🔥 Sempre defina healthcheck no serviço do banco de dados
Para a condição service_healthy funcionar, o serviço dependido (postgres, redis, etc.) precisa ter um healthcheck definido. Imagens oficiais como postgres e redis não têm healthcheck por padrão — você precisa adicionar no seu Compose.

Stack completa — API + Postgres + Redis + Worker

Um arquivo Compose real de produção para uma aplicação FastAPI com banco de dados, cache e worker de background.

docker-compose.yml — stack completa com boas práticas
# docker-compose.yml — Stack completa: API + PostgreSQL + Redis + Worker

services:

  # ─── API Principal ──────────────────────────────────────────────────────────
  api:
    build:
      context: .
      dockerfile: Dockerfile
      target: production          # Multi-stage: usa o estágio "production"
    image: myapp-api:latest
    container_name: myapp_api
    restart: unless-stopped
    ports:
      - "${API_PORT:-8000}:8000"  # Porta configurável via .env, padrão 8000
    environment:
      - DATABASE_URL=${DATABASE_URL}
      - REDIS_URL=${REDIS_URL}
      - SECRET_KEY=${SECRET_KEY}
      - DEBUG=${DEBUG:-false}
    env_file:
      - .env
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      # Bind mount APENAS em dev — remova em prod!
      # - ./app:/app/app
      - static_files:/app/staticfiles
    networks:
      - frontend
      - backend
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
    logging:
      driver: "json-file"
      options:
        max-size: "10m"     # Limita logs a 10 MB por arquivo
        max-file: "3"       # Mantém 3 arquivos rotacionados

  # ─── Banco de Dados ──────────────────────────────────────────────────────────
  postgres:
    image: postgres:16-alpine
    container_name: myapp_postgres
    restart: unless-stopped
    # NÃO expõe porta 5432 para o host em produção!
    # Em dev, descomente a linha abaixo para acessar com pgAdmin/DBeaver:
    # ports:
    #   - "5432:5432"
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    volumes:
      - pgdata:/var/lib/postgresql/data        # Dados persistem aqui
      - ./scripts/init.sql:/docker-entrypoint-initdb.d/init.sql:ro  # Init script
    networks:
      - backend
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

  # ─── Cache ───────────────────────────────────────────────────────────────────
  redis:
    image: redis:7-alpine
    container_name: myapp_redis
    restart: unless-stopped
    command: redis-server --appendonly yes --requirepass ${REDIS_PASSWORD}
    volumes:
      - redisdata:/data
    networks:
      - backend
    healthcheck:
      test: ["CMD", "redis-cli", "-a", "${REDIS_PASSWORD}", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5

  # ─── Worker de Background ────────────────────────────────────────────────────
  worker:
    build:
      context: .
      dockerfile: Dockerfile
      target: production
    container_name: myapp_worker
    restart: unless-stopped
    command: ["celery", "-A", "app.celery", "worker", "--loglevel=info"]
    environment:
      - DATABASE_URL=${DATABASE_URL}
      - REDIS_URL=${REDIS_URL}
    env_file:
      - .env
    depends_on:
      redis:
        condition: service_healthy
      postgres:
        condition: service_healthy
    networks:
      - backend
    profiles:
      - full    # Só sobe com: docker compose --profile full up

  # ─── Proxy Reverso (produção) ─────────────────────────────────────────────
  nginx:
    image: nginx:alpine
    container_name: myapp_nginx
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - static_files:/var/www/static:ro
      - ./ssl:/etc/nginx/ssl:ro
    depends_on:
      api:
        condition: service_healthy
    networks:
      - frontend
    profiles:
      - full

# ─── Redes ────────────────────────────────────────────────────────────────────
networks:
  frontend:
    driver: bridge
    # API e nginx se comunicam aqui
  backend:
    driver: bridge
    # API, postgres, redis e worker se comunicam aqui
    # O banco NUNCA é exposto para a rede frontend

# ─── Volumes ──────────────────────────────────────────────────────────────────
volumes:
  pgdata:         # Dados do PostgreSQL
    driver: local
  redisdata:      # Dados do Redis (AOF persistence)
    driver: local
  static_files:   # Arquivos estáticos compartilhados entre api e nginx
    driver: local
// TÉCNICA AVANÇADA

Profiles e Override Files — ambientes diferentes

Profiles — suba apenas o que precisa

Profiles permitem marcar serviços opcionais que só sobem quando explicitamente solicitados. Perfeito para serviços de produção que não precisam rodar no ambiente de desenvolvimento local.

Terminal
# Sobe apenas os serviços sem profile (api, postgres, redis)
$ docker compose up -d

# Sobe TUDO incluindo nginx e worker (profile "full")
$ docker compose --profile full up -d

# Verifica status de todos os serviços
$ docker compose ps
NAME STATUS PORTS
myapp_api Up (healthy) 0.0.0.0:8000->8000/tcp
myapp_postgres Up (healthy) 5432/tcp
myapp_redis Up (healthy) 6379/tcp

Override files — dev vs produção

O Compose merge automaticamente docker-compose.yml com docker-compose.override.yml quando presente. Use esse padrão para ter configurações diferentes por ambiente:

docker-compose.yml

Base — configuração comum

Definição dos serviços, imagens, volumes, redes. Valores que são iguais em todos os ambientes. Vai para o Git.

docker-compose.override.yml

Dev — carregado automaticamente

Bind mounts para hot reload, portas extras expostas para debug, variáveis de desenvolvimento. Vai para o Git (sem segredos).

docker-compose.override.yml — específico para desenvolvimento
# docker-compose.override.yml — carregado automaticamente em desenvolvimento
# NÃO use em produção: docker compose -f docker-compose.yml up

services:
  api:
    build:
      target: builder    # Estágio de build (tem compiladores, útil para debug)
    volumes:
      - ./app:/app/app   # Hot reload: código do host sincronizado
    environment:
      - DEBUG=true
      - LOG_LEVEL=debug
    command: ["uvicorn", "app.main:app", "--reload", "--host", "0.0.0.0", "--port", "8000"]

  postgres:
    ports:
      - "5432:5432"      # Expõe o banco para o host (acesso via DBeaver/pgAdmin)

  redis:
    ports:
      - "6379:6379"      # Expõe Redis para debug local
Terminal — gerenciando com override explícito
# Desenvolvimento (merge automático com override.yml)
$ docker compose up -d

# Produção (ignora override.yml)
$ docker compose -f docker-compose.yml up -d

# Staging (arquivo específico)
$ docker compose -f docker-compose.yml -f docker-compose.staging.yml up -d

from fastapi import FastAPI
import asyncpg
import redis.asyncio as aioredis
import os

app = FastAPI(title="Compose Demo")

DB_URL = os.getenv("DATABASE_URL", "postgresql://user:senha@postgres:5432/appdb")
REDIS_URL = os.getenv("REDIS_URL", "redis://redis:6379/0")

@app.get("/health")
async def health():
    return {"status": "healthy"}

@app.get("/")
async def root():
    return {"message": "Stack rodando!", "services": ["api", "postgres", "redis"]}

@app.get("/db-check")
async def db_check():
    """Testa conexão com o banco"""
    conn = await asyncpg.connect(DB_URL)
    version = await conn.fetchval("SELECT version()")
    await conn.close()
    return {"postgres": version[:50]}

@app.get("/cache-check")
async def cache_check():
    """Testa conexão com o Redis"""
    r = aioredis.from_url(REDIS_URL)
    await r.set("ping", "pong", ex=10)
    value = await r.get("ping")
    await r.aclose()
    return {"redis": value.decode()}

⚠️ Armadilhas Comuns

1. depends_on sem condition: service_healthy

O depends_on sem condição garante apenas que o container do serviço dependido foi criado e iniciado — não que o processo dentro está pronto. Postgres leva alguns segundos para aceitar conexões. Sempre use condition: service_healthy junto com um healthcheck definido no serviço do banco.

2. Usar localhost para se conectar a outros serviços

Dentro de um container, localhost refere-se ao próprio container — não ao host nem aos outros serviços. Para a API se conectar ao banco, use o nome do serviço: postgresql://user:pass@postgres:5432/db. O Docker resolve o DNS internamente na rede bridge.

3. Commitar o arquivo .env com segredos

O .env contém senhas e tokens e nunca deve ir para o Git. Adicione ao .gitignore imediatamente. Crie um .env.example com os nomes das variáveis mas sem valores — esse arquivo vai para o Git e serve como documentação.

4. Expor porta do banco de dados em produção

Em produção, bancos de dados nunca devem ter ports: publicadas. ports: ["5432:5432"] expõe o banco para qualquer processo no host e, dependendo das configurações de rede, para a internet. Deixe o banco acessível apenas via rede interna do Compose.

5. docker compose down -v acidentalmente

O flag -v no docker compose down -v remove todos os volumes nomeados — incluindo os dados do banco de dados. Em produção, isso significa perda de dados irreversível. Sempre faça backup antes de usar down -v e prefira docker compose down (sem -v) para operações rotineiras.

6. Usar bind mounts em produção

Bind mounts (- ./app:/app) são ótimos para desenvolvimento com hot reload, mas não devem ser usados em produção — o container passa a depender de um path específico do host, quebrando a portabilidade. Em produção, o código deve estar dentro da imagem, não montado do host.

Termos desta Aula

docker-compose.yml
Arquivo YAML declarativo que define todos os serviços, redes e volumes de uma aplicação multi-container. É o "contrato de ambiente" do projeto.
Service (Serviço)
No contexto do Compose, um serviço é a definição de um container: qual imagem usa, quais variáveis recebe, a quais redes pertence. Um serviço pode ter múltiplas réplicas.
Bridge Network
Rede privada virtual criada automaticamente pelo Compose. Containers na mesma rede se comunicam pelo nome do serviço como hostname (DNS interno). Isolada da rede do host.
Named Volume
Volume gerenciado pelo Docker com um nome explícito. Persiste entre docker compose down e up. Os dados ficam em /var/lib/docker/volumes/ no host.
Bind Mount
Mapeamento direto entre uma pasta do host e um caminho dentro do container. Mudanças nos arquivos do host refletem imediatamente no container (hot reload). Ideal para desenvolvimento.
depends_on
Diretiva que controla a ordem de inicialização dos serviços. Com condition: service_healthy, aguarda o healthcheck do serviço dependido passar antes de iniciar.
profiles
Mecanismo para marcar serviços opcionais que só são iniciados quando o profile é explicitamente ativado com --profile nome. Útil para serviços de produção ou debugging.
Override file
Arquivo docker-compose.override.yml que é automaticamente merged com o Compose principal. Permite configurações diferentes por ambiente sem duplicar o arquivo base.

🏁 Checkpoint — Desafio Final

Crie uma stack Docker Compose completa para uma aplicação de sua escolha com pelo menos três serviços que se comunicam entre si:

  • Um serviço de aplicação (Python/Node/qualquer linguagem) com Dockerfile próprio
  • Um banco de dados (PostgreSQL ou MySQL) com volume nomeado para persistência
  • Um serviço de cache (Redis) ou outro serviço de suporte
  • depends_on com condition: service_healthy e healthchecks nos serviços dependidos
  • Variáveis de ambiente carregadas via .env (com .env.example no Git)
  • Duas redes separadas: uma pública (api + proxy) e uma privada (api + db + cache)
  • Comprove que o banco não está acessível de fora da rede backend

Bônus: crie um docker-compose.override.yml com bind mounts para hot reload em desenvolvimento e portas expostas do banco para uso com DBeaver ou pgAdmin.

💬 Dica: use docker compose config para ver o arquivo Compose final com todas as variáveis interpoladas — é o melhor jeito de debugar problemas com .env. E docker network inspect compose-demo_backend para ver todos os containers conectados a cada rede.
