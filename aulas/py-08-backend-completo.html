// motivação

A diferença entre um CRUD e uma API de produção

Na aula anterior, construímos uma API FastAPI com dados em memória. Funcionou — mas sem persistência (reiniciou o servidor, perdeu tudo), sem autenticação (qualquer um acessa tudo) e sem testes (como garantir que nada quebrou?). Esta aula fecha essas três lacunas.

SQLAlchemy é o ORM padrão do Python — traduz objetos Python para SQL sem escrever queries na mão. JWT é o protocolo de autenticação stateless mais usado em APIs REST. E pytest com fixtures transforma testes em algo prazeroso de escrever.

💼 Por que um engenheiro sênior usa isso hoje

APIs de produção precisam de: banco relacional com migrações versionadas (Alembic), autenticação stateless que escala horizontalmente (JWT), e uma suíte de testes que roda em segundos no CI (pytest). Esses três pilares juntos transformam um projeto de hobby em software maintível por um time.

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Imagine um cofre de banco. O cofre é o banco de dados — os dados persistem lá mesmo sem ninguém usando. O gerente de banco é o SQLAlchemy: você diz "quero abrir essa conta" e ele converte para as ações físicas corretas. O crachá de acesso com chip é o JWT: você autentica uma vez, recebe um crachá, e ele te identifica em todas as entradas seguintes sem precisar digitar senha de novo. E o auditório de testes da agência é o pytest: antes de abrir qualquer cofre real, todo procedimento é testado num ambiente simulado idêntico.
🏦 Cofre do banco → PostgreSQL/SQLite (banco real) 👔 Gerente de operações → SQLAlchemy ORM 📇 Crachá com chip → JWT Token 🏋️ Sala de treino simulada → pytest + fixtures

SQLAlchemy — ORM que pensa como Python

Um ORM (Object-Relational Mapper) permite trabalhar com tabelas de banco de dados como se fossem classes Python. Você define um modelo, e o ORM cuida do SQL. Com Alembic, as mudanças no schema são versionadas — como git para o banco.

// models/database.py — configuração
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import DeclarativeBase, sessionmaker

# Para SQLite (dev): "sqlite+aiosqlite:///./app.db"
# Para PostgreSQL (prod): "postgresql+asyncpg://user:pass@host/db"
DATABASE_URL = "sqlite+aiosqlite:///./todo.db"

engine = create_async_engine(DATABASE_URL, echo=True)

AsyncSessionLocal = sessionmaker(
    engine, class_=AsyncSession, expire_on_commit=False
)

class Base(DeclarativeBase):
    pass

# Dependency — injetada via Depends() nas rotas
async def get_db():
    async with AsyncSessionLocal() as session:
        yield session
        # session.close() é chamado automaticamente
// models/todo.py — modelo SQLAlchemy
from sqlalchemy import String, Integer, Boolean, DateTime, func
from sqlalchemy.orm import Mapped, mapped_column
from .database import Base

class Todo(Base):
    __tablename__ = "todos"

    # mapped_column é a sintaxe moderna (SQLAlchemy 2.0+)
    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    title: Mapped[str] = mapped_column(String(200), nullable=False)
    description: Mapped[str | None] = mapped_column(String(1000))
    priority: Mapped[int] = mapped_column(Integer, default=1)
    completed: Mapped[bool] = mapped_column(Boolean, default=False)
    owner_id: Mapped[int] = mapped_column(Integer, index=True)

    # auto-preenchido pelo banco
    created_at: Mapped[DateTime] = mapped_column(
        DateTime, server_default=func.now()
    )

Alembic — Migrações versionadas

bash — workflow de migrações
$ pip install alembic aiosqlite
$ alembic init alembic # cria pasta alembic/
# edite alembic/env.py para apontar para seus models
$ alembic revision --autogenerate -m "create todos table"
Generating alembic/versions/001_create_todos_table.py
$ alembic upgrade head # aplica ao banco
Running upgrade -> 001_create_todos_table
🔥 Pro tip: nunca edite o banco manualmente em produção

Toda mudança de schema deve passar por uma migration Alembic. Isso garante que dev, staging e prod ficam sempre sincronizados — e permite fazer rollback com alembic downgrade -1.

Autenticação JWT — stateless e escalável

JWT (JSON Web Token) é um token que carrega informações do usuário de forma auto-suficiente e assinada. O servidor não precisa guardar sessões — basta verificar a assinatura do token. Isso é fundamental para APIs que escalam horizontalmente.

// core/security.py — JWT + bcrypt
from datetime import datetime, timedelta
from typing import Optional
import bcrypt
from jose import JWTError, jwt

SECRET_KEY = "sua-chave-super-secreta"  # use env var em produção!
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 30

def hash_password(password: str) -> str:
    """Gera hash bcrypt. Nunca armazene senha em texto puro."""
    salt = bcrypt.gensalt()
    return bcrypt.hashpw(password.encode(), salt).decode()

def verify_password(plain: str, hashed: str) -> bool:
    """Verifica senha contra o hash bcrypt armazenado."""
    return bcrypt.checkpw(plain.encode(), hashed.encode())

def create_access_token(data: dict, expires_delta: Optional[timedelta] = None):
    """Cria um JWT assinado com expiração."""
    to_encode = data.copy()
    expire = datetime.utcnow() + (expires_delta or timedelta(minutes=15))
    to_encode.update({"exp": expire})
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

def decode_token(token: str) -> dict:
    """Decodifica e valida o JWT. Lança JWTError se inválido."""
    return jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
// core/deps.py — Dependency para rotas protegidas
from fastapi import Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer
from sqlalchemy.ext.asyncio import AsyncSession
from .security import decode_token
from .database import get_db
from jose import JWTError

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="auth/login")

async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db)
):
    """
    Dependency que extrai e valida o usuário do JWT.
    Use: user = Depends(get_current_user) em qualquer rota.
    """
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Token inválido ou expirado",
        headers={"WWW-Authenticate": "Bearer"},
    )
    try:
        payload = decode_token(token)
        user_id: str = payload.get("sub")
        if user_id is None:
            raise credentials_exception
    except JWTError:
        raise credentials_exception

    # Em produção: buscar usuário do banco aqui
    return {"id": int(user_id)}

pytest — Testes que dão confiança para refatorar

pytest é o framework de testes padrão do Python. Com fixtures você cria estados reutilizáveis (banco de teste, cliente HTTP), e com httpx.AsyncClient testa endpoints FastAPI sem precisar de um servidor rodando.

Tipo de TesteO que testaVelocidadeQuando usar
Unitário Função isolada (ex: hash_password) Muito rápido Sempre — lógica de negócio pura
Integração Rota + banco + auth juntos Médio Fluxos críticos (login, CRUD)
E2E Sistema completo com HTTP real Lento Smoke tests em staging/prod
// tests/conftest.py — fixtures compartilhadas
import pytest
import pytest_asyncio
from httpx import AsyncClient, ASGITransport
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import sessionmaker

from main import app
from models.database import Base, get_db

# Banco de dados em memória para testes — nunca toca o banco real
TEST_DB = "sqlite+aiosqlite:///:memory:"

@pytest_asyncio.fixture(scope="session")
async def test_db():
    engine = create_async_engine(TEST_DB, echo=False)
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    yield engine
    await engine.dispose()

@pytest_asyncio.fixture
async def db_session(test_db):
    TestSession = sessionmaker(test_db, class_=AsyncSession,
                               expire_on_commit=False)
    async with TestSession() as session:
        yield session

@pytest_asyncio.fixture
async def client(db_session):
    # Sobrescreve a dependency de banco para usar o de teste
    app.dependency_overrides[get_db] = lambda: db_session
    async with AsyncClient(
        transport=ASGITransport(app=app),
        base_url="http://test"
    ) as ac:
        yield ac
    app.dependency_overrides.clear()
// tests/test_todos.py — testes de integração
import pytest

@pytest.mark.asyncio
async def test_create_todo(client):
    """Criar uma tarefa retorna 201 com os dados corretos."""
    response = await client.post("/todos", json={
        "title": "Estudar FastAPI",
        "priority": 3
    })
    assert response.status_code == 201
    data = response.json()
    assert data["title"] == "Estudar FastAPI"
    assert data["priority"] == 3
    assert data["completed"] is False
    assert "id" in data

@pytest.mark.asyncio
async def test_get_todo_not_found(client):
    """Buscar tarefa inexistente retorna 404."""
    response = await client.get("/todos/99999")
    assert response.status_code == 404
    assert "não encontrada" in response.json()["detail"]

@pytest.mark.asyncio
async def test_todo_validation_error(client):
    """Dados inválidos retornam 422."""
    response = await client.post("/todos", json={
        "title": "",        # min_length=1
        "priority": 10      # max=5
    })
    assert response.status_code == 422

@pytest.mark.asyncio
async def test_security_hash_verify():
    """Testa bcrypt sem banco — teste unitário puro."""
    from core.security import hash_password, verify_password
    hashed = hash_password("senha123")
    assert verify_password("senha123", hashed) is True
    assert verify_password("senha_errada", hashed) is False

Docker + Uvicorn — da máquina ao cloud

Com os arquivos anteriores organizados, o deploy é direto: um Dockerfile multi-stage cria uma imagem enxuta, e qualquer plataforma que aceite Docker (Railway, Fly.io, AWS, Render) recebe sua API.

// Dockerfile — multi-stage para produção
# Stage 1 — builder (instala dependências)
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

# Stage 2 — runner (imagem final enxuta)
FROM python:3.12-slim AS runner
WORKDIR /app
# Copia só as dependências compiladas do stage anterior
COPY --from=builder /root/.local /root/.local
COPY . .

ENV PATH=/root/.local/bin:$PATH
ENV PYTHONUNBUFFERED=1

# Uvicorn com múltiplos workers para produção
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]
bash — build e run local
$ docker build -t todo-api .
Successfully built 89ab12cd
$ docker run -p 8000:8000 todo-api
INFO: Uvicorn running on http://0.0.0.0:8000

Erros comuns

⚠️ Guardar SECRET_KEY no código-fonte

Nunca commite chaves secretas. Use variáveis de ambiente com python-decouple ou pydantic-settings. Em produção, use secrets manager (AWS Secrets Manager, Vault, etc.).

# ✅ Use pydantic-settings
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    secret_key: str
    database_url: str
    class Config:
        env_file = ".env"  # lê do .env em dev

settings = Settings()
⚠️ Token JWT sem expiração curta

Access tokens devem expirar em 15–30 minutos. Para sessões longas, use o padrão refresh token: um token de vida curta (access) + um de vida longa (refresh) armazenado em cookie HttpOnly. Nunca armazene tokens JWT em localStorage.

⚠️ Usar o banco de produção nos testes

Sempre use SQLite em memória (sqlite+aiosqlite:///:memory:) ou um banco de teste separado. O fixture dependency_overrides do FastAPI foi criado exatamente para isso — sem ele, testes que inserem dados poluem o banco real.
