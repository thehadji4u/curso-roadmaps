GitHub Avançado — Segurança & Packages

⏱ ~50 min de leitura 🎯 Avançado 🔧 Hands-on incluído

// motivação

Por que isso importa?

Um repositório sem proteções é uma porta aberta. Qualquer desenvolvedor pode dar push direto na main, uma dependência desatualizada pode ter uma vulnerabilidade crítica, e um token exposto no código pode comprometer toda a infraestrutura. Em ambientes de produção, segurança não é opcional — é requisito de entrada.

Nesta aula você vai aprender as 5 camadas de segurança e governança do GitHub: proteger branches, automatizar atualizações de dependências, escanear código em busca de vulnerabilidades, publicar pacotes de forma controlada e automatizar o próprio GitHub com a CLI e API. Tudo isso são habilidades que qualquer tech lead ou engenheiro sênior precisa dominar.

💼 Por que um engenheiro sênior usa isso hoje
Em times grandes, você não pode confiar que todos vão seguir as boas práticas manualmente. Engenheiros sêniors automatizam a governança: branch rules garantem que código não revisado nunca chega à produção, Dependabot fecha CVEs antes que virem incidentes, CodeQL detecta SQL injection e XSS antes do deploy. Isso se chama DevSecOps — segurança integrada no pipeline, não checada depois.
// analogia

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense no seu repositório como uma agência bancária. O banco tem múltiplas camadas de segurança que operam em paralelo, não em sequência: o cofre tem travas que requerem duas chaves ao mesmo tempo (branch protection + required reviews), um sistema detecta automaticamente transações suspeitas (Dependabot + security alerts), guardas inspecionam cada pacote que entra no cofre (CodeQL), e há uma central de monitoramento que registra tudo e pode ser acionada remotamente (GitHub API & CLI).

Nenhuma dessas camadas protege sozinha — é a combinação delas que torna o sistema robusto.
🔒 Travas do cofre com 2 chaves → Branch Protection + Required Reviews 🚨 Detector de transações suspeitas → Dependabot Security Alerts 🔍 Inspeção de cada pacote → CodeQL / SAST Scanning 📡 Central de monitoramento remota → GitHub API & gh CLI

As 5 camadas de segurança do GitHub

🛡️
1. Branch Protection Rules
Impede push direto na main, exige reviews obrigatórios, bloqueia merge sem CI passando. A primeira linha de defesa contra código não revisado.
🤖
2. Dependabot
Monitora dependências do projeto e abre PRs automáticos quando há atualizações de segurança ou versões novas disponíveis.
🔬
3. Code Scanning (CodeQL)
Análise estática automática que detecta vulnerabilidades como SQL injection, XSS, path traversal e credenciais expostas no código.
📦
4. GitHub Packages
Registry privado ou público para publicar e consumir pacotes npm, Docker, PyPI e Maven. Integrado ao repositório e à autenticação do GitHub.
⚡
5. GitHub API & gh CLI
Automação do próprio GitHub: criar issues, gerenciar PRs, configurar repositórios e orquestrar workflows — tudo programaticamente.

Branch Protection Rules

Branch protection rules são configurações aplicadas a branches específicos (como main ou release/*) que definem quem pode fazer o quê — e o que deve acontecer antes de um merge ser permitido.

RegraO que fazQuando usar
Require PR reviews Bloqueia merge sem N aprovações de outros devs Sempre em projetos com time ≥ 2 pessoas
Require status checks Bloqueia merge se CI falhar (testes, lint, build) Todo projeto com Actions configurados
Require conversation resolution Exige que todos os comentários de review sejam resolvidos Times que usam review como checklist
Restrict who can push Apenas pessoas/teams específicos podem fazer push Branches de release e hotfix
Require signed commits Apenas commits com GPG assinados são aceitos Projetos com requisitos de compliance
Require linear history Proíbe merge commits — só rebase ou squash Times que preferem histórico limpo

Configurando via gh CLI

gh CLI
# Criar branch protection rule via API
$ gh api repos/OWNER/REPO/branches/main/protection \
  -X PUT \
  --field required_status_checks='{"strict":true,"contexts":["ci/tests"]}' \
  --field enforce_admins=true \
  --field required_pull_request_reviews='{"required_approving_review_count":2}' \
  --field restrictions=null

# Ver status das regras de um branch
$ gh api repos/OWNER/REPO/branches/main/protection

Dependabot — atualizações automáticas

O Dependabot monitora o arquivo de dependências do seu projeto (package.json, requirements.txt, Dockerfile, etc.) e abre PRs automaticamente quando:

  • Uma vulnerabilidade de segurança (CVE) é encontrada numa dependência
  • Uma versão nova da dependência é publicada (se configurado)
Security Alerts

Alertas automáticos

Notifica quando uma dependência do seu projeto tem CVE conhecido. Aparece em Security → Dependabot alerts no repositório.

Version Updates

PRs de atualização

Configura via dependabot.yml para abrir PRs semanalmente/mensalmente com atualizações de patch, minor ou major.

.github/dependabot.yml — configuração completa
version: 2
updates:
  # Dependências npm/Node.js
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
      time: "09:00"
      timezone: "America/Sao_Paulo"
    open-pull-requests-limit: 5
    reviewers:
      - "seu-usuario"
    labels:
      - "dependencies"
      - "javascript"
    ignore:
      # Evitar major bumps automáticos (pode quebrar)
      - dependency-name: "react"
        update-types: ["version-update:semver-major"]

  # Dependências Python
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
    labels:
      - "dependencies"
      - "python"

  # Actions do GitHub — crítico para segurança!
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
    labels:
      - "dependencies"
      - "github-actions"

  # Imagens Docker base
  - package-ecosystem: "docker"
    directory: "/"
    schedule:
      interval: "monthly"
    labels:
      - "dependencies"
      - "docker"
⚠️ Ative também o Dependabot para GitHub Actions
Actions que usam actions/checkout@v2 (versão fixada) podem ter vulnerabilidades. Configure o Dependabot para github-actions também — é um vetor de ataque frequentemente ignorado.

Code Scanning & CodeQL — SAST no CI

CodeQL é a engine de análise estática do GitHub. Ela trata o código como um banco de dados e executa queries de segurança para encontrar padrões de vulnerabilidade — SQL injection, XSS, path traversal, credenciais hardcoded, deserialização insegura.

💡 SAST vs DAST
SAST (Static Analysis): analisa o código fonte sem executar. Detecta vulnerabilidades antes do deploy. Rápido, sem falsos negativos de runtime.
DAST (Dynamic Analysis): executa a aplicação e testa em tempo real. Complementar ao SAST. Mais lento, detecta problemas de configuração e runtime.
.github/workflows/codeql.yml — análise automática em cada PR
name: "CodeQL Security Analysis"

on:
  push:
    branches: [ "main", "develop" ]
  pull_request:
    branches: [ "main" ]
  schedule:
    # Rodar toda segunda às 8h — detecta CVEs novas em código existente
    - cron: '0 8 * * 1'

jobs:
  analyze:
    name: Analyze
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write  # necessário para publicar alertas

    strategy:
      fail-fast: false
      matrix:
        language: [ 'javascript', 'python' ]
        # Suporte: javascript, python, java, go, csharp, cpp, ruby, swift

    steps:
    - name: Checkout repository
      uses: actions/checkout@v4

    - name: Initialize CodeQL
      uses: github/codeql-action/init@v3
      with:
        languages: ${{ matrix.language }}
        # Queries extras de segurança (mais abrangentes)
        queries: security-extended

    - name: Autobuild
      uses: github/codeql-action/autobuild@v3

    - name: Perform CodeQL Analysis
      uses: github/codeql-action/analyze@v3
      with:
        category: "/language:${{matrix.language}}"

Após o workflow rodar, os resultados aparecem em Security → Code scanning alerts. Cada alerta indica o arquivo, linha, severidade (critical/high/medium/low) e uma explicação da vulnerabilidade com sugestão de fix.

GitHub Packages — seu registry privado

GitHub Packages é um registry integrado ao GitHub que suporta npm, Docker, Maven, NuGet, RubyGems e Gradle. A vantagem é que a autenticação usa o próprio token do GitHub — sem precisar gerenciar credenciais separadas.

FormatoURL do RegistryCaso de uso
npm npm.pkg.github.com Bibliotecas JS/TS compartilhadas entre projetos
Docker ghcr.io Imagens Docker privadas para CI/CD e deploy
PyPI (pip) pypi.pkg.github.com Pacotes Python internos (utils, SDKs)

Publicar imagem Docker no GHCR

.github/workflows/docker-publish.yml
name: Build & Publish Docker Image

on:
  push:
    branches: [ "main" ]
  release:
    types: [published]

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write  # necessário para push no GHCR

    steps:
    - name: Checkout
      uses: actions/checkout@v4

    - name: Log in to GHCR
      uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}  # token automático do Actions

    - name: Extract metadata (tags, labels)
      id: meta
      uses: docker/metadata-action@v5
      with:
        images: ghcr.io/${{ github.repository }}
        tags: |
          type=ref,event=branch
          type=semver,pattern={{version}}
          type=sha,prefix=sha-

    - name: Build and push
      uses: docker/build-push-action@v5
      with:
        context: .
        push: true
        tags: ${{ steps.meta.outputs.tags }}
        labels: ${{ steps.meta.outputs.labels }}
        cache-from: type=gha
        cache-to: type=gha,mode=max
Pull da imagem do GHCR
# Autenticar com token pessoal
$ echo $GITHUB_TOKEN | docker login ghcr.io -u seu-usuario --password-stdin

# Fazer pull da imagem
$ docker pull ghcr.io/OWNER/REPO:main
main: Pulling from OWNER/REPO
Digest: sha256:abc123...
Status: Downloaded newer image for ghcr.io/OWNER/REPO:main

GitHub CLI (gh) — automatizar o GitHub

O gh é a CLI oficial do GitHub que permite fazer praticamente tudo que você faz na interface web — criar PRs, gerenciar issues, rodar workflows, ver logs — tudo no terminal e scriptável.

gh CLI — comandos essenciais
# Autenticar
$ gh auth login

# Criar PR com template
$ gh pr create --title "feat: add payment gateway" --body "## Summary\n- Integra Stripe\n- Adiciona testes" --reviewer colega1,colega2

# Ver status de todos os PRs
$ gh pr list --state open
ID TITLE BRANCH STATUS
#42 feat: add payment gateway feature/stripe Open

# Rodar workflow manualmente
$ gh workflow run deploy.yml --field environment=staging

# Ver logs de uma run específica
$ gh run view 12345 --log

# Criar issue a partir de template
$ gh issue create --title "Bug: login falha em Safari" --label "bug,priority:high" --assignee @me

# Clonar e abrir PR em um comando
$ gh repo clone OWNER/REPO && cd REPO && gh pr checkout 42

GitHub API — scripting avançado

Para casos mais complexos, a API REST e GraphQL do GitHub permitem qualquer operação programaticamente. O gh api é um atalho conveniente que já inclui autenticação:

exemplos de uso da GitHub API via gh
# Listar todos os repositórios de uma organização
gh api orgs/MINHA-ORG/repos --paginate --jq '.[].name'

# Criar repository secret
gh secret set DATABASE_URL --body "postgresql://user:pass@host/db" --repo OWNER/REPO

# Obter métricas de CI (tempo médio de build dos últimos 10 runs)
gh api repos/OWNER/REPO/actions/workflows/ci.yml/runs \
  --jq '[.workflow_runs[:10] | .[].run_duration_ms] | add / length / 1000 | "\(.) segundos"'

# Fechar todas as issues com label "wontfix"
gh issue list --label "wontfix" --json number --jq '.[].number' | \
  xargs -I{} gh issue close {}

# Criar release com changelog automático
gh release create v1.2.0 \
  --title "Release 1.2.0" \
  --generate-notes \
  --target main

Mãos na massa: blindar um repositório do zero

Vamos configurar todas as camadas de segurança em um repositório real:

1
Ativar Branch Protection na main
Settings → Branches → Add rule → main → marque: Require PR reviews (2), Require status checks, Require conversation resolution, Include administrators
2
Criar .github/dependabot.yml
Use o template acima. Commit → aguarde o primeiro PR do Dependabot (normalmente aparece em horas)
3
Criar .github/workflows/codeql.yml
Use o template acima com as linguagens do seu projeto. Após o push, veja os resultados em Security → Code scanning alerts
4
Ativar Secret Scanning
Settings → Security → Secret scanning → Enable. O GitHub vai varrer o histórico inteiro em busca de tokens, API keys e passwords expostos
5
Configurar CODEOWNERS
Crie .github/CODEOWNERS para definir revisores obrigatórios por área do código
# .github/CODEOWNERS
# Revisores globais
*                   @OWNER/team-leads

# Backend requer review do time de backend
/src/api/           @OWNER/backend-team
/src/database/      @OWNER/backend-team

# Frontend requer review do time de design
/src/components/    @OWNER/frontend-team

# Infra requer review do time de DevOps
/infrastructure/    @OWNER/devops-team
/.github/           @OWNER/devops-team

GitHub Codespaces — dev environment na nuvem

Codespaces cria um ambiente de desenvolvimento completo na nuvem, configurado via devcontainer.json. Qualquer colaborador pode contribuir com um clique — sem instalar nada localmente.

.devcontainer/devcontainer.json — configuração de ambiente
{
  "name": "Meu Projeto Dev",
  "image": "mcr.microsoft.com/devcontainers/python:3.11",
  "features": {
    "ghcr.io/devcontainers/features/node:1": { "version": "20" },
    "ghcr.io/devcontainers/features/docker-in-docker:2": {},
    "ghcr.io/devcontainers/features/github-cli:1": {}
  },
  "postCreateCommand": "pip install -r requirements.txt && npm install",
  "customizations": {
    "vscode": {
      "extensions": [
        "ms-python.python",
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode"
      ],
      "settings": {
        "editor.formatOnSave": true,
        "python.defaultInterpreterPath": "/usr/local/bin/python"
      }
    }
  },
  "forwardPorts": [3000, 8000, 5432],
  "remoteEnv": {
    "DATABASE_URL": "${localEnv:DATABASE_URL}"
  }
}

Erros comuns em segurança no GitHub

⚠️ Branch protection sem "Include administrators" — admin faz push direto

Por padrão, administradores conseguem contornar as regras de branch protection. Em incidentes de pressão, isso vira atalho perigoso.

✅ Solução: sempre marque "Include administrators" nas branch protection rules. Ninguém está acima das regras de segurança.

⚠️ Dependabot abrindo 50 PRs de uma vez — todo o time ignora

Se configurado sem limites, o Dependabot pode abrir dezenas de PRs simultaneamente, causando fadiga de alertas e sendo ignorado.

✅ Solução: defina open-pull-requests-limit: 5 no dependabot.yml. Priorize atualizações de segurança — configure grouping para agrupar várias atualizações de um mesmo ecossistema em um único PR.

⚠️ GITHUB_TOKEN com permissões desnecessárias no Actions

Por padrão em alguns repos, o GITHUB_TOKEN tem permissões amplas de read/write. Se uma Action maliciosa for injetada via supply chain, ela tem acesso completo.

✅ Solução: defina permissões mínimas no workflow. Use permissions: read-all no topo do arquivo e adicione explicitamente apenas o que precisar (ex: packages: write).

⚠️ Usar Actions de terceiros por branch em vez de hash

uses: actions/checkout@main — se o repositório da action for comprometido, o código malicioso roda nos seus pipelines imediatamente.

✅ Solução: sempre pin com hash: uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2. O Dependabot mantém os hashes atualizados automaticamente.

🏁 Checkpoint — valide seu aprendizado

Configure um repositório de produção seguro do zero:

  • Criar branch protection na main: 2 reviews obrigatórios, CI deve passar, include admins
  • Adicionar .github/dependabot.yml cobrindo pelo menos 2 ecossistemas + github-actions
  • Adicionar .github/workflows/codeql.yml para a linguagem principal do projeto
  • Criar .github/CODEOWNERS com pelo menos 2 regras por área do código
  • Fazer um PR de teste e verificar que todas as proteções estão funcionando
  • Usar gh pr list e gh run list no terminal
