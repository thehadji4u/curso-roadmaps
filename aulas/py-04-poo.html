// motivação

Por que isso importa?

Você já escreveu código que funciona — mas que, ao crescer, vira uma bagunça difícil de manter. A Programação Orientada a Objetos (POO) resolve exatamente esse problema: ela organiza código em blocos autossuficientes que modelam entidades do mundo real, facilitando manutenção, reutilização e trabalho em equipe.

Em Python, tudo já é um objeto — listas, strings, funções. Aprender a criar seus próprios tipos de objeto é o salto que separa scripts rápidos de aplicações reais.

💼 Como engenheiros sênior usam isso hoje

No desenvolvimento de APIs com FastAPI ou Django, cada endpoint manipula modelos de dados representados como classes. Uma class Usuario encapsula validação, lógica de negócio e acesso ao banco — tudo junto. Sem POO, cada novo endpoint repete a mesma lógica de forma dispersa e inconsistente. Com POO, você altera o comportamento em um único lugar e todos os endpoints se atualizam.

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Imagine uma fábrica de carros. A fábrica não produz carros um a um de forma improvisada — ela tem um molde (uma planta técnica) que define o que todo carro terá: motor, rodas, cor, velocímetro. Usando esse molde, ela pode fabricar milhares de carros distintos, cada um com suas características específicas (cor vermelha, motor 1.0), mas todos seguindo a mesma estrutura.

Em Python, a classe é a planta/molde e cada objeto (instância) é um carro fabricado a partir dela.
🏭 Planta da Fábrica (molde) → class Carro 🚗 Carro produzido (unidade real) → meu_carro = Carro(...) 📋 Características do carro → Atributos (self.cor) ⚙️ O que o carro faz (acelerar) → Métodos (def acelerar())

Os 4 pilares da POO — na prática

POO não é só "classes e objetos". São 4 pilares que, juntos, permitem construir sistemas complexos de forma organizada:

Pilar 1

🔒 Encapsulamento

Esconder detalhes internos. O usuário da classe não precisa saber como ela funciona, só o que ela faz. Em Python: atributos com _ (protegido) ou __ (privado) + @property.

Pilar 2

🧬 Herança

Uma classe filha herda atributos e métodos da classe pai, reutilizando código sem repetição. class Cachorro(Animal) herda tudo de Animal e adiciona o que é específico.

Pilar 3

🎭 Polimorfismo

Objetos diferentes respondem ao mesmo método de formas distintas. cachorro.falar() retorna "Au!" e gato.falar() retorna "Miau!" — mesma interface, comportamentos diferentes.

Pilar 4

💡 Abstração

Modelar apenas o que é relevante para o problema. Uma class ContaBancaria abstrai o sistema do banco — você só vê depositar() e sacar(), não os 200 sistemas internos do banco.

1
Defina a classe base com encapsulamento
Crie a classe com atributo privado __saldo e use @property para acesso seguro.
conta_bancaria.py Python 
class ContaBancaria:
    banco = "Banco PyDev"  # atributo de CLASSE (compartilhado)

    def __init__(self, titular: str, saldo_inicial: float = 0):
        self.titular = titular        # atributo público
        self.__saldo = saldo_inicial  # atributo PRIVADO (encapsulado)

    @property
    def saldo(self) -> float:
        """Getter: acessa __saldo com segurança."""
        return self.__saldo

    def depositar(self, valor: float) -> None:
        if valor <= 0:
            raise ValueError("Valor do depósito deve ser positivo.")
        self.__saldo += valor
        print(f"✅ Depósito de R${valor:.2f} realizado. Saldo: R${self.__saldo:.2f}")

    def sacar(self, valor: float) -> bool:
        if valor <= 0:
            raise ValueError("Valor do saque deve ser positivo.")
        if valor > self.__saldo:
            print(f"❌ Saldo insuficiente. Saldo atual: R${self.__saldo:.2f}")
            return False
        self.__saldo -= valor
        print(f"💸 Saque de R${valor:.2f} realizado. Saldo: R${self.__saldo:.2f}")
        return True

    def __str__(self) -> str:
        """Dunder method: representação em string."""
        return f"[{self.banco}] {self.titular} | Saldo: R${self.__saldo:.2f}"

conta_bancaria.py (continuação) Python 
class ContaCorrente(ContaBancaria):
    def __init__(self, titular: str, saldo_inicial: float = 0, limite: float = 500):
        super().__init__(titular, saldo_inicial)  # chama __init__ do pai
        self.limite = limite

    def sacar(self, valor: float) -> bool:
        """Polimorfismo: sobrescreve sacar() para incluir limite."""
        if valor > self.saldo + self.limite:
            print(f"❌ Limite excedido. Saldo + limite: R${self.saldo + self.limite:.2f}")
            return False
        # Acessa __saldo do pai via método (encapsulamento mantido)
        self._ContaBancaria__saldo -= valor  # name mangling — evite isso normalmente!
        print(f"💸 Saque com limite: R${valor:.2f} | Saldo: R${self.saldo:.2f}")
        return True

    @classmethod
    def criar_conta_premium(cls, titular: str) -> "ContaCorrente":
        """Factory method: cria conta com configurações premium."""
        return cls(titular, saldo_inicial=1000, limite=5000)


class ContaPoupanca(ContaBancaria):
    TAXA_JUROS_MENSAL = 0.005  # 0.5% ao mês

    def render_juros(self) -> None:
        """Aplica rendimento mensal ao saldo."""
        rendimento = self.saldo * self.TAXA_JUROS_MENSAL
        self.depositar(rendimento)
        print(f"📈 Juros aplicados: R${rendimento:.2f}")

    @staticmethod
    def calcular_rendimento_projetado(valor: float, meses: int) -> float:
        """Utilitário estático: sem precisar de instância."""
        return valor * ((1 + ContaPoupanca.TAXA_JUROS_MENSAL) ** meses)

Exemplo avançado: Decorators & Dunder Methods

Python POO vai além do básico: dunder methods (métodos mágicos como __len__, __eq__, __repr__) permitem que suas classes se comportem como tipos nativos do Python.

biblioteca.py Python 
"""
Exemplo: Sistema de Biblioteca com Dunder Methods, Properties e Decorators
Execute: python biblioteca.py
"""

from dataclasses import dataclass, field
from datetime import date


@dataclass
class Livro:
    """
    @dataclass gera __init__, __repr__ e __eq__ automaticamente.
    Ideal para classes que são principalmente contêineres de dados.
    """
    titulo: str
    autor: str
    ano: int
    isbn: str
    disponivel: bool = True

    def __str__(self) -> str:
        status = "✅ Disponível" if self.disponivel else "❌ Emprestado"
        return f'"{self.titulo}" — {self.autor} ({self.ano}) [{status}]'


class Biblioteca:
    def __init__(self, nome: str):
        self.nome = nome
        self.__acervo: list[Livro] = []  # encapsulado

    # ── Dunder Methods ────────────────────────────────
    def __len__(self) -> int:
        """len(biblioteca) funciona graças a este método."""
        return len(self.__acervo)

    def __contains__(self, isbn: str) -> bool:
        """'isbn in biblioteca' funciona graças a este método."""
        return any(livro.isbn == isbn for livro in self.__acervo)

    def __repr__(self) -> str:
        return f"Biblioteca('{self.nome}', {len(self)} livros)"

    # ── Properties ────────────────────────────────────
    @property
    def livros_disponiveis(self) -> list[Livro]:
        return [l for l in self.__acervo if l.disponivel]

    @property
    def total_disponiveis(self) -> int:
        return len(self.livros_disponiveis)

    # ── Métodos de instância ──────────────────────────
    def adicionar(self, livro: Livro) -> None:
        if livro.isbn in self:  # usa __contains__
            print(f"⚠️ ISBN {livro.isbn} já cadastrado.")
            return
        self.__acervo.append(livro)
        print(f"📚 '{livro.titulo}' adicionado ao acervo.")

    def emprestar(self, isbn: str, usuario: str) -> bool:
        livro = self.__buscar_por_isbn(isbn)
        if not livro:
            print(f"❌ Livro com ISBN {isbn} não encontrado.")
            return False
        if not livro.disponivel:
            print(f"❌ '{livro.titulo}' já está emprestado.")
            return False
        livro.disponivel = False
        print(f"✅ '{livro.titulo}' emprestado para {usuario}.")
        return True

    def devolver(self, isbn: str) -> bool:
        livro = self.__buscar_por_isbn(isbn)
        if livro and not livro.disponivel:
            livro.disponivel = True
            print(f"📖 '{livro.titulo}' devolvido com sucesso.")
            return True
        print(f"⚠️ Livro não encontrado ou já disponível.")
        return False

    # ── Método privado (encapsulamento) ───────────────
    def __buscar_por_isbn(self, isbn: str) -> Livro | None:
        """Método interno: não exposto para uso externo."""
        return next((l for l in self.__acervo if l.isbn == isbn), None)

    # ── Classmethod: factory ──────────────────────────
    @classmethod
    def de_lista(cls, nome: str, livros: list[Livro]) -> "Biblioteca":
        """Cria biblioteca com acervo inicial."""
        bib = cls(nome)
        for livro in livros:
            bib.adicionar(livro)
        return bib


# ── Execução ──────────────────────────────────────────
if __name__ == "__main__":
    livro1 = Livro("Clean Code", "Robert Martin", 2008, "978-01")
    livro2 = Livro("Python Fluente", "Ramalho", 2022, "978-02")
    livro3 = Livro("The Pragmatic Programmer", "Hunt & Thomas", 1999, "978-03")

    bib = Biblioteca.de_lista("Biblioteca PyDev", [livro1, livro2, livro3])

    print(f"\n{repr(bib)}")                    # usa __repr__
    print(f"Total: {len(bib)} livros")         # usa __len__
    print(f"'978-01' no acervo: {'978-01' in bib}")  # usa __contains__

    print(f"\nDisponíveis: {bib.total_disponiveis}")

    bib.emprestar("978-01", "Alice")
    bib.emprestar("978-01", "Bob")  # já emprestado!

    print(f"\nDisponíveis após empréstimo: {bib.total_disponiveis}")
    for livro in bib.livros_disponiveis:
        print(f"  • {livro}")

    bib.devolver("978-01")

Erros comuns em POO Python

⚠️ Esquecer o self nos métodos

Em Python, self não é automático — você precisa declarar explicitamente. def depositar(valor) vai falhar; o correto é def depositar(self, valor). É verboso, mas intencional — reforça que o método pertence à instância.

⚠️ Atributo mutável como default no __init__

Nunca faça def __init__(self, lista=[]). Listas são mutáveis e esse objeto vazio é compartilhado entre todas as instâncias. O correto: def __init__(self, lista=None) e depois self.lista = lista or [].

⚠️ Abusar de herança quando composição seria melhor

Herança ("é um") vs composição ("tem um"). Um Carro não deve herdar de Motor — ele tem um motor. Use self.motor = Motor(). Prefira composição sobre herança sempre que possível.
