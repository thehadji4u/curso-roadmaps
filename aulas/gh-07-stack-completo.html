GitHub como centro nervoso do stack

Nas aulas anteriores estudamos cada ferramenta isoladamente. Nesta aula, vamos ver como o GitHub conecta tudo — cada push, PR ou tag pode disparar pipelines que envolvem Docker, Python, n8n e GitHub Pages em uma única automação coesa.

🐳

GitHub + Docker

Build, push para GHCR e deploy automático a cada commit no main.

🔀

GitHub + n8n

Webhook do GitHub dispara workflows n8n — notificações, deploys, integrações.

🐍

GitHub + Python

CI com pytest, ruff, mypy e cobertura de testes a cada PR.

💡 Por que um engenheiro sênior usa isso hoje?
Empresas de alto nível operam com pipelines que cruzam todas as ferramentas. Saber conectar GitHub ao seu stack completo é o que separa um dev que "sabe as ferramentas" de um que "sabe construir sistemas".

O GitHub como Torre de Controle

✈️ Analogia
Pense no GitHub como a torre de controle de um aeroporto. Cada avião (código) que decola (push) precisa de autorização e coordenação. A torre coordena: notifica o hangar (Docker build), avisa o pessoal de terra (n8n webhook), verifica a segurança (testes Python) e atualiza o painel de voos ao vivo (GitHub Pages). Nenhum avião pousa sem passar pela torre.
✈️ Avião decolando
→ git push / PR merge
🏗️ Hangar de manutenção
→ Docker build + GHCR push
📡 Comunicação com solo
→ n8n webhook trigger
🔍 Inspeção de segurança
→ pytest + ruff CI
🖥️ Painel de voos
→ GitHub Pages atualizado

GitHub + Docker — CI/CD Completo

O workflow mais comum em produção: a cada push no main, o Actions constrói a imagem Docker, publica no GHCR e faz deploy no servidor via SSH.

GitHub → Docker → Servidor
graph LR A[git push main] --> B[⚙️ Actions trigger] B --> C[docker build] C --> D{testes OK?} D -->|sim| E[docker push GHCR] D -->|não| F[❌ falha — notifica] E --> G[SSH deploy] G --> H[🟢 produção] style H fill:#10b981,color:#000 style F fill:#ef4444,color:#fff
.github/workflows/docker-deploy.yml
name: Build, Test & Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-test-deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Login GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build image
        run: docker build -t ghcr.io/${{ github.repository }}:${{ github.sha }} .

      - name: Run tests inside container
        run: |
          docker run --rm \
            ghcr.io/${{ github.repository }}:${{ github.sha }} \
            npm test   # ou pytest, etc.

      - name: Push to GHCR
        if: github.ref == 'refs/heads/main'
        run: |
          docker tag ghcr.io/${{ github.repository }}:${{ github.sha }} \
                     ghcr.io/${{ github.repository }}:latest
          docker push ghcr.io/${{ github.repository }}:${{ github.sha }}
          docker push ghcr.io/${{ github.repository }}:latest

      - name: Deploy via SSH
        if: github.ref == 'refs/heads/main'
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            docker pull ghcr.io/${{ github.repository }}:latest
            docker compose -f /opt/app/docker-compose.yml up -d --no-deps app

GitHub + n8n — Webhooks e Automações

O GitHub pode disparar eventos para o n8n em tempo real via webhooks: novo PR, novo issue, release publicada, commit no main — tudo pode acionar um workflow n8n.

GitHub Webhook → n8n Workflow
sequenceDiagram participant GH as 🐙 GitHub participant N8N as 🔀 n8n participant TG as 💬 Telegram participant SH as 🖥️ Servidor GH->>N8N: POST /webhook/deploy (push event) N8N->>N8N: Processa payload N8N->>TG: "🚀 Deploy iniciado por @usuario" N8N->>SH: SSH → docker pull && restart SH-->>N8N: exit 0 N8N->>TG: "✅ Deploy concluído em 23s"

Configurando o Webhook do GitHub

1
No n8n: crie um workflow com Webhook trigger
Adicione o node Webhook, método POST, copie a URL gerada: https://n8n.seudominio.com/webhook/github-deploy
2
No GitHub: Settings → Webhooks → Add webhook
Payload URL: a URL do n8n. Content type: application/json. Secret: uma chave aleatória para validar a origem.
3
No n8n: valide a assinatura e processe o evento
// Code node — validar assinatura HMAC do GitHub
const crypto = require('crypto')
const secret = 'sua-chave-secreta'
const signature = $request.headers['x-hub-signature-256']
const body = JSON.stringify($input.item.json)

const expected = 'sha256=' + crypto
  .createHmac('sha256', secret)
  .update(body)
  .digest('hex')

if (signature !== expected) {
  throw new Error('Assinatura inválida — possível ataque')
}

// Processar o evento
const event = $request.headers['x-github-event']
const payload = $input.item.json

return [{ json: { event, repo: payload.repository.name, pusher: payload.pusher?.name, commits: payload.commits?.length || 0 } }]
🔥 Eventos úteis do GitHub Webhook
push (commits), pull_request (aberto/merged/fechado), release (nova versão publicada), issues (criado/fechado), workflow_run (Actions concluídas). Configure quais eventos disparar em "Which events would you like to trigger this webhook?".

GitHub + Python — CI com Qualidade de Código

Para projetos Python, o Actions garante que todo PR passa por testes, análise estática e verificação de cobertura antes de ser mergeado.

.github/workflows/python-ci.yml
name: Python CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  quality:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.11", "3.12"]   # testa em múltiplas versões

    steps:
      - uses: actions/checkout@v4

      - name: Setup Python ${{ matrix.python-version }}
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
          cache: pip   # cache das dependências

      - name: Instalar dependências
        run: |
          pip install -e ".[dev]"   # instala o projeto + deps de dev

      - name: Lint com ruff (ultrarrápido, substitui flake8+isort)
        run: ruff check . --output-format=github

      - name: Type check com mypy
        run: mypy src/ --strict

      - name: Testes com pytest + cobertura
        run: |
          pytest tests/ \
            --cov=src \
            --cov-report=xml \
            --cov-report=term-missing \
            --cov-fail-under=80   # falha se cobertura < 80%

      - name: Upload cobertura para Codecov
        uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          file: coverage.xml
🔥 ruff: o linter Python do futuro
ruff é 10–100x mais rápido que flake8 e substitui: flake8, isort, pyupgrade, bandit e mais. Escrito em Rust. Configure em pyproject.toml: [tool.ruff] line-length = 88

GitHub Pages — Deploy Estático Automático

GitHub Pages publica qualquer conteúdo estático (HTML/CSS/JS) diretamente do repositório, com HTTPS gratuito. Ideal para documentação, portfolios, sites de curso e landing pages.

FonteUsoBranch/Pasta
Raiz do repositórioSites simples — HTML diretomain /
Pasta /docsDocumentação junto ao códigomain /docs
Branch gh-pagesBuild gerado (React, Next, VitePress)gh-pages /
GitHub ActionsControle total do processo de buildActions → Pages
.github/workflows/pages-deploy.yml — Deploy de site estático gerado
name: Deploy GitHub Pages

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write   # necessário para deploy no Pages

concurrency:
  group: "pages"
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npm run build   # gera pasta dist/
      - uses: actions/upload-pages-artifact@v3
        with: { path: dist }

  deploy:
    needs: build
    environment:
      name: github-pages
      url: ${{ steps.deploy.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to GitHub Pages
        id: deploy
        uses: actions/deploy-pages@v4

Monorepo — Tudo em Um Só Repositório

Um monorepo é um único repositório Git que contém múltiplos projetos ou pacotes relacionados (frontend, backend, libs compartilhadas). Usado por Google, Meta, Vercel e muitas empresas de referência.

Estrutura de Monorepo com Turborepo
graph TB Root[📁 meu-monorepo/] --> Apps[📁 apps/] Root --> Packages[📁 packages/] Root --> T[⚙️ turbo.json] Root --> PJ[📄 package.json workspaces] Apps --> Web[📁 web/ Next.js] Apps --> API[📁 api/ FastAPI] Apps --> Docs[📁 docs/ VitePress] Packages --> UI[📁 ui/ components compartilhados] Packages --> Config[📁 config/ tsconfig/eslint] Packages --> Utils[📁 utils/ funções compartilhadas] Web -.->|usa| UI Web -.->|usa| Utils API -.->|usa| Utils style Root fill:#58a6ff,color:#000
package.json — workspaces (npm/pnpm/yarn)
{
  "name": "meu-monorepo",
  "private": true,
  "workspaces": ["apps/*", "packages/*"],
  "scripts": {
    "build": "turbo run build",
    "test":  "turbo run test",
    "lint":  "turbo run lint",
    "dev":   "turbo run dev --parallel"
  },
  "devDependencies": {
    "turbo": "^2.0.0"
  }
}
turbo.json — pipeline de tarefas com cache inteligente
{
  "$schema": "https://turbo.build/schema.json",
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "dist/**", "build/**"]
    },
    "test": {
      "dependsOn": ["build"],
      "outputs": ["coverage/**"]
    },
    "lint": {
      "outputs": []
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
🔥 Por que Turborepo?
Turborepo faz cache inteligente das tarefas — se o código de um pacote não mudou, ele não reconstrói nem roda os testes de novo. Em monorepos grandes, isso reduz CI de 20 minutos para 2 minutos.

CI para Monorepo — só o que mudou

.github/workflows/ci-monorepo.yml
name: Monorepo CI

on: [push, pull_request]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 2 }   # necessário para turbo diff

      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: pnpm }

      - run: pnpm install --frozen-lockfile

      - name: Build & test afetados (turbo + cache remoto)
        run: pnpm turbo run build test lint --filter=...[HEAD^1]
        env:
          TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}   # cache remoto Vercel
          TURBO_TEAM: ${{ vars.TURBO_TEAM }}

⚠️ Erros Comuns

⚠️ Secrets expostos em logs do Actions

Nunca use echo $SECRET em workflows — o Actions oculta valores de secrets, mas comandos como env ou erros de runtime podem vazar. Use --password-stdin em vez de passar senhas como argumento.

⚠️ Monorepo sem filtro no CI

Sem --filter do Turborepo, todo push reconstrói e retesta todos os apps, mesmo os que não mudaram. Use --filter=...[HEAD^1] para rodar apenas o que foi afetado pelas mudanças.

⚠️ GitHub Pages sem index.html

GitHub Pages precisa de um arquivo index.html na raiz da branch/pasta configurada. Se o arquivo se chama roadmaps_new.html, acesse diretamente pela URL completa — não apenas pelo domínio raiz.

Desafio: Stack Integrado do Zero

Monte um monorepo com dois apps e um CI/CD completo:

  • Crie um monorepo com apps/web (HTML estático) e apps/api (FastAPI)
  • Configure CI Python com pytest + ruff para a API
  • Configure deploy do apps/web para GitHub Pages via Actions
  • Adicione um webhook do GitHub para notificar via n8n quando há deploy bem-sucedido
  • Bônus: Adicione Turborepo e verifique que apenas o app modificado é reconstruído no CI
💬 Dica: Comece pelo mais simples — GitHub Pages primeiro, depois adicione o CI Python, depois o webhook n8n. Construa incrementalmente.
