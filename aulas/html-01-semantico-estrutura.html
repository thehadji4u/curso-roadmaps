HTML Semântico & Estrutura

⏱ ~30 min de leitura 🎯 Fundamentos 🔧 Hands-on incluído
semântica estrutura tags de seção formulários media acessibilidade SEO
// motivação

Por que isso importa?

HTML é a espinha dorsal de qualquer página web. Mas há uma diferença enorme entre escrever HTML que funciona e HTML que é semântico. HTML semântico comunica o significado do conteúdo, não apenas sua aparência — e isso tem impacto direto em SEO, acessibilidade e manutenção.

Um screen reader (leitor de tela, usado por pessoas com deficiência visual) depende completamente das tags certas para narrar a página corretamente. Mecanismos de busca como o Google usam a estrutura semântica para entender e indexar o conteúdo. E qualquer dev que abrir seu código depois de você vai entender muito mais rápido uma página com <article>, <nav> e <main> do que uma cheia de <div> sem sentido.

💼 Por que um engenheiro sênior se preocupa com isso
Acessibilidade e SEO não são "extras" — são requisitos em qualquer produto sério. Além disso, HTML semântico reduz drasticamente a necessidade de classes CSS e JavaScript extras para descrever estrutura. Code review em empresas de nível médio-alto rejeitam ativamente "div soup" (páginas com <div> para tudo) e exigem uso correto de elementos semânticos.
// analogia

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense em um jornal impresso. Ele não é só um monte de texto e imagens jogados aleatoriamente. Ele tem uma estrutura clara: o cabeçalho com o nome do jornal e data, a navegação com as editorias (Política, Esportes, Cultura), os artigos com título, subtítulo, corpo e autor, a barra lateral com destaques, e o rodapé com informações de contato.

HTML semântico faz exatamente isso: dá a cada parte da página um nome que descreve seu papel, não só sua posição visual. Assim como qualquer pessoa entende a estrutura de um jornal sem precisar de uma legenda, qualquer ferramenta (browser, screen reader, Google) entende a estrutura de uma página semântica.
📰 Nome e data do jornal → <header> 📑 Editorias (Política, Esportes…) → <nav> 📄 Cada notícia independente → <article> 📌 Barra de destaques → <aside> 📋 Rodapé com contato → <footer>
// conceito 1

Anatomia de um documento HTML

Todo documento HTML começa com o mesmo esqueleto. Cada parte tem um propósito específico — entendê-las evita erros comuns de iniciantes como colocar scripts no lugar errado ou esquecer o charset.

// esqueleto completo e comentado
<!DOCTYPE html>
<!-- Instrui o browser a usar HTML5 (não é uma tag HTML, é uma declaração) -->

<html lang="pt-BR">
<!-- lang: essencial para screen readers e SEO.
     Use "pt-BR" para português do Brasil, "en" para inglês -->

<head>
  <!-- Metadados: não aparecem na página, mas são vitais -->

  <meta charset="UTF-8">
  <!-- Codificação de caracteres: suporta acentos, emojis, etc.
       DEVE ser o primeiro elemento do head -->

  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <!-- Controla o layout em dispositivos móveis.
       Sem isso, o browser renderiza a página como se fosse desktop -->

  <meta name="description" content="Descrição da página para SEO (até 160 caracteres)">

  <title>Título da Página | Nome do Site</title>
  <!-- Aparece na aba do browser e nos resultados de busca -->

  <link rel="stylesheet" href="styles.css">
  <!-- CSS no head: carrega antes da renderização (evita flash sem estilo) -->

  <link rel="icon" type="image/svg+xml" href="/favicon.svg">
  <!-- Ícone na aba do browser -->
</head>

<body>
  <!-- Todo conteúdo visível fica aqui -->

  <script src="app.js" defer></script>
  <!-- Scripts no final do body (ou com defer/async no head).
       defer: executa após o HTML ser parseado, mantendo a ordem.
       async: executa assim que carrega, sem garantia de ordem -->
</body>
</html>
🔥 Pro tip: defer vs async
Use defer para scripts que dependem do DOM ou de outros scripts (é o padrão seguro). Use async para scripts independentes como analytics. Nunca coloque scripts pesados no <head> sem um desses atributos — vai bloquear a renderização da página.

O HTML5 introduziu elementos que descrevem papéis e regiões da página. Eles substituem o uso indiscriminado de <div> e comunicam intenção a browsers, leitores de tela e crawlers de busca.

Elemento Papel semântico Pode ter múltiplos? Dica de uso
<header> Cabeçalho de página ou seção Sim Logo, título, nav, hero. Pode existir dentro de <article> também
<nav> Bloco de navegação Sim Use para navegação principal e secundária; adicione aria-label se houver dois
<main> Conteúdo principal único Não (1 por página) Nunca dentro de <header>, <footer> ou <aside>
<article> Conteúdo autônomo e redistribuível Sim Post de blog, notícia, produto, comentário — faz sentido sozinho fora de contexto
<section> Agrupamento temático com título Sim Sempre deve ter um <h2>-<h6> filho. Não é um container genérico!
<aside> Conteúdo tangencialmente relacionado Sim Sidebar, callouts, anúncios, links relacionados
<footer> Rodapé de página ou seção Sim Autor, copyright, links de contato
<div> Container genérico sem semântica Sim (use com moderação) Apenas para layout ou quando nenhum elemento semântico se aplica

Hierarquia de títulos e texto

Os títulos <h1>–<h6> formam o outline (esquema) do documento — é como o índice de um livro. Screen readers permitem navegar diretamente entre títulos, e o Google os usa fortemente para entender o tema da página.

⚠️ Regras críticas para títulos
Um único <h1> por página — é o título principal, deve descrever o assunto da página inteira. Nunca pule níveis: não vá de <h2> para <h4> sem um <h3> no meio. Escolha o nível pelo significado, não pelo tamanho visual — use CSS para estilizar.
// hierarquia correta de títulos + elementos de texto
<!-- ── Hierarquia de títulos ───────────────────────── -->
<h1>Guia Completo de JavaScript</h1>       <!-- 1 por página -->
  <h2>Fundamentos da Linguagem</h2>
    <h3>Variáveis e Tipos</h3>
    <h3>Funções</h3>
      <h4>Arrow Functions</h4>
  <h2>JavaScript no Browser</h2>
    <h3>DOM Manipulation</h3>

<!-- ── Elementos de texto semânticos ─────────────── -->
<p>Parágrafo normal de texto corrido.</p>

<strong>Texto importante (bold semântico)</strong>   <!-- importância -->
<em>Ênfase (itálico semântico)</em>                  <!-- ênfase -->
<b>Bold visual apenas (sem importância)</b>           <!-- estético -->
<i>Itálico visual (termos técnicos, etc.)</i>         <!-- estético -->

<mark>Texto destacado (highlight)</mark>
<small>Texto secundário, notas de rodapé</small>

<abbr title="HyperText Markup Language">HTML</abbr>

<!-- ── Código e preformatado ─────────────────────── -->

  // bloco de código preserva espaços e quebras

  }
</code></pre>

<!-- ── Citações ──────────────────────────────────── -->

  <p>Semântica é o estudo do significado.</p>
  <footer>— <cite>MDN Web Docs</cite></footer>
</blockquote>

<!-- ── Listas ────────────────────────────────────── -->

Formulários — o coração da interação

Formulários são o principal mecanismo de entrada de dados do usuário. HTML nativo já oferece validação, tipos de input específicos para mobile, acessibilidade embutida e estados semânticos — tudo sem JavaScript. Aproveitá-los bem reduz código e melhora a experiência.

// formulário semântico e acessível completo
<form action="/cadastro" method="POST" novalidate>
  <!-- novalidate: desabilita validação nativa do browser (quando usamos JS customizado) -->

  <!-- ── Fieldset agrupa campos relacionados ─────────── -->
  <fieldset>
    <legend>Dados Pessoais</legend>

    <div class="campo">
      <!-- label SEMPRE associado ao input (via for + id ou aninhamento) -->
      <label for="nome">Nome completo *</label>
      <input
        type="text"
        id="nome"
        name="nome"
        placeholder="Ana Silva"
        required
        minlength="3"
        maxlength="100"
        autocomplete="name"
      >
    </div>

    <div class="campo">
      <label for="email">E-mail *</label>
      <input
        type="email"           <!-- valida formato email, abre teclado correto no mobile -->
        id="email"
        name="email"
        required
        autocomplete="email"
        aria-describedby="email-hint"
      >
      <span id="email-hint" class="hint">Usado para login e comunicações</span>
    </div>

    <div class="campo">
      <label for="telefone">Telefone</label>
      <input type="tel" id="telefone" name="telefone"
             pattern="[\d\s\-\(\)]{10,15}"
             placeholder="(11) 99999-9999"
             autocomplete="tel">
    </div>

    <div class="campo">
      <label for="nascimento">Data de nascimento</label>
      <input type="date" id="nascimento" name="nascimento"
             min="1920-01-01" max="2010-12-31">
    </div>
  </fieldset>

  <fieldset>
    <legend>Preferências</legend>

    <!-- Select / dropdown ──────────────────────────── -->
    <div class="campo">
      <label for="plano">Plano</label>
      <select id="plano" name="plano">
        <option value="">Selecione...</option>
        <optgroup label="Pessoal">
          <option value="free">Gratuito</option>
          <option value="pro" selected>Pro — R$ 29/mês</option>
        </optgroup>
        <optgroup label="Empresarial">
          <option value="team">Team — R$ 99/mês</option>
        </optgroup>
      </select>
    </div>

    <!-- Textarea ──────────────────────────────────── -->
    <div class="campo">
      <label for="bio">Bio</label>
      <textarea id="bio" name="bio" rows="4"
                maxlength="500" placeholder="Conte um pouco sobre você..."></textarea>
    </div>

    <!-- Radio buttons (grupo exclusivo) ───────────── -->
    <fieldset>
      <legend>Gênero</legend>
      <label><input type="radio" name="genero" value="f"> Feminino</label>
      <label><input type="radio" name="genero" value="m"> Masculino</label>
      <label><input type="radio" name="genero" value="nb"> Não-binário</label>
      <label><input type="radio" name="genero" value="na"> Prefiro não informar</label>
    </fieldset>

    <!-- Checkboxes (múltipla escolha) ─────────────── -->
    <fieldset>
      <legend>Interesses</legend>
      <label><input type="checkbox" name="interesses" value="frontend"> Frontend</label>
      <label><input type="checkbox" name="interesses" value="backend"> Backend</label>
      <label><input type="checkbox" name="interesses" value="devops"> DevOps</label>
    </fieldset>

    <!-- Range e number ─────────────────────────────── -->
    <label for="experiencia">Anos de experiência: <output id="exp-val">3</output></label>
    <input type="range" id="experiencia" name="experiencia"
           min="0" max="20" value="3" step="1"
           oninput="document.getElementById('exp-val').value = this.value">
  </fieldset>

  <!-- Botões ─────────────────────────────────────── -->
  <div class="acoes">
    <button type="submit">Criar conta</button>
    <button type="reset">Limpar</button>
    <button type="button" onclick="preview()">Pré-visualizar</button>
  </div>
</form>

Mídia — imagens, vídeo e áudio

O HTML moderno tem elementos ricos para mídia, com suporte a múltiplos formatos, lazy loading nativo, imagens responsivas e acessibilidade embutida.

// imagens responsivas e acessíveis
<!-- ── Imagem simples — atributos essenciais ─────────────── -->
<img
  src="foto.jpg"
  alt="Foto de Ana Silva sorrindo em frente a um computador"
  width="800"
  height="600"
  loading="lazy"
>
<!--
  alt: OBRIGATÓRIO para acessibilidade e SEO.
       Se a imagem for decorativa (sem conteúdo), use alt=""
  width + height: previne layout shift (CLS) durante o carregamento
  loading="lazy": adia o carregamento até a imagem entrar na viewport
-->

<!-- ── Imagens responsivas com srcset ────────────────────── -->
<img
  src="foto-800.jpg"
  srcset="
    foto-400.jpg  400w,
    foto-800.jpg  800w,
    foto-1600.jpg 1600w
  "
  sizes="(max-width: 600px) 100vw, (max-width: 1200px) 50vw, 800px"
  alt="Paisagem da Serra da Mantiqueira ao amanhecer"
  loading="lazy"
  decoding="async"
>

<!-- ── <picture> para art direction e formatos modernos ───── -->
<picture>
  <source type="image/avif" srcset="hero.avif">  <!-- melhor compressão -->
  <source type="image/webp" srcset="hero.webp">  <!-- amplo suporte -->
  <img src="hero.jpg" alt="Hero da landing page" width="1200" height="600">
  <!-- fallback para browsers antigos -->
</picture>

<!-- ── Figure com legenda ────────────────────────────────── -->
<figure>
  <img src="grafico-vendas.png" alt="Gráfico de barras: vendas crescem 40% no Q3">
  <figcaption>Fig. 1 — Crescimento de vendas Q1–Q3 2024</figcaption>
</figure>

<!-- ── Vídeo ─────────────────────────────────────────────── -->
<video
  controls
  width="800"
  poster="thumbnail.jpg"
  preload="metadata"
>
  <source src="aula.webm" type="video/webm">
  <source src="aula.mp4"  type="video/mp4">
  <track kind="subtitles" src="aula-pt.vtt" srclang="pt" label="Português" default>
  <p>Seu browser não suporta vídeo HTML5. <a href="aula.mp4">Baixar</a></p>
</video>

<!-- ── Áudio ─────────────────────────────────────────────── -->
<audio controls preload="none">
  <source src="podcast.ogg" type="audio/ogg">
  <source src="podcast.mp3" type="audio/mpeg">
</audio>

<!-- ── Embed e iframe ─────────────────────────────────────── -->
<iframe
  src="https://www.youtube.com/embed/VIDEO_ID"
  title="Título descritivo do vídeo incorporado"
  width="560"
  height="315"
  allow="accelerometer; autoplay; clipboard-write; encrypted-media"
  allowfullscreen
  loading="lazy"
></iframe>
⚠️ Alt text — a regra de ouro
O atributo alt é a descrição textual da imagem para quem não pode vê-la. Se a imagem carrega informação (gráfico, foto de produto, captura de tela), descreva o conteúdo. Se é puramente decorativa (ícone de fundo, separador visual), use alt="" — screen readers vão ignorá-la. Nunca omita o atributo.

⚠️ Armadilhas Comuns

1. Usar <div> e <span> para tudo ("div soup")

Antes de adicionar um <div>, pergunte: "existe um elemento semântico para isso?" Na maioria das vezes a resposta é sim. <div> deve ser o último recurso, não o primeiro.

2. Múltiplos <h1> na mesma página

Existe exatamente um <h1> por página — o título principal. Usar múltiplos confunde screen readers e prejudica SEO. O framework que você usa (React, Next.js) não muda essa regra.

3. <input> sem <label> associado

Todo campo de formulário precisa de um <label> com for igual ao id do input. placeholder não substitui label — desaparece ao digitar e tem contraste ruim. Sem label, screen readers não conseguem descrever o campo.

4. target="_blank" sem rel="noopener noreferrer"

Sem esses atributos, a página aberta em nova aba pode acessar window.opener e redirecionar a página original — uma vulnerabilidade de segurança chamada "reverse tabnapping".

5. Omitir width e height em imagens

Sem dimensões, o browser não reserva espaço para a imagem antes de carregá-la, causando layout shift (CLS) — a página "pula" enquanto carrega. Isso prejudica a experiência do usuário e o Core Web Vitals.

6. Usar <b> e <i> para ênfase semântica

Use <strong> para importância real e <em> para ênfase semântica. <b> e <i> são apenas visuais, sem semântica — screen readers podem não os diferenciar.

Termos desta Aula

HTML Semântico
Uso de elementos HTML que descrevem o significado e papel do conteúdo, não apenas sua aparência visual.
DOCTYPE
Declaração no topo do HTML que instrui o browser a usar o modo de renderização HTML5, evitando o "quirks mode" legado.
Accessibility Tree
Representação paralela ao DOM que o browser constrói para tecnologias assistivas, baseada na semântica do HTML.
ARIA (Accessible Rich Internet Applications)
Conjunto de atributos HTML (aria-label, aria-describedby, role…) que complementam a semântica nativa para casos complexos.
CLS (Cumulative Layout Shift)
Métrica de Core Web Vitals que mede a instabilidade visual da página. Imagens sem width/height são uma causa comum de CLS alto.
Lazy Loading
Técnica de adiar o carregamento de recursos (imagens, iframes) até que entrem na viewport, melhorando o tempo de carregamento inicial.
srcset
Atributo de <img> que oferece múltiplas versões de uma imagem em diferentes tamanhos para o browser escolher a mais adequada.

🏁 Checkpoint — Desafio Final

Construa uma página de artigo de blog completa e semanticamente correta:

  • Estrutura com <header> (nav com logo + links), <main>, <aside> e <footer>
  • Dentro do <main>, um <article> com: <header> interno (título, autor, data com <time datetime="...">), pelo menos 3 seções com subtítulos, uma imagem com figure/figcaption e lazy loading, e uma <blockquote> com <cite>
  • <aside> com uma lista de artigos relacionados (links) e um mini formulário de newsletter (email + botão)
  • Todos os links externos com rel="noopener noreferrer", todos os inputs com labels associadas
  • Validar no validator.w3.org e corrigir todos os erros
💬 Dica: use <time datetime="2024-11-15">15 de novembro de 2024</time> para datas legíveis por máquinas. O atributo datetime usa o formato ISO 8601 e é importante para SEO e agregadores de conteúdo.
