HTML Semântico & Estrutura

⏱ ~30 min de leitura 🎯 Iniciante 🔧 Hands-on incluído ♿ Acessibilidade & SEO incluídos
<header> <main> <article> <section> <form> <input> <picture> srcset ARIA
// 01 — motivação

Por que isso importa?

Todo mundo consegue fazer uma página web funcionar com <div> para tudo. O problema é que um navegador, um leitor de tela e um crawler do Google não têm a menor ideia do que cada parte da sua página significa — e isso custa caro: em acessibilidade, em SEO, e em manutenibilidade do código.

HTML semântico é usar o elemento certo para o significado certo. É a diferença entre um arquiteto que usa plantas detalhadas e um pedreiro que improvisa. O resultado pode parecer igual visualmente, mas internamente é completamente diferente.

💼 Por que um engenheiro sênior se importa com isso
HTML semântico impacta diretamente três áreas que empresas medem: SEO (o Google entende melhor a hierarquia do conteúdo), acessibilidade (leitores de tela navegam por landmarks semânticos — header, main, nav), e Core Web Vitals (estrutura limpa ajuda parsers de renderização). Em auditorias de código, um HTML cheio de divs sem semântica é sinal imediato de código legado com dívida técnica.
// 02 — analogia

Antes do código: a planta de um edifício

🏛️ Mundo Real → HTML
Imagine um edifício. Você consegue entrar numa sala qualquer e perceber: "isso é a recepção", "aqui é a área de espera", "lá no fundo é a sala de reuniões". Como? Pelo contexto e pelos sinais visuais — recepcionista, cadeiras enfileiradas, mesa grande.

Agora imagine o mesmo edifício sem nenhuma indicação. Todas as salas com as mesmas paredes brancas, sem sinalização, sem móveis identificadores. Para um humano com tempo, dá para descobrir. Para um robô (Google, leitor de tela) — impossível.

HTML semântico é a sinalização do edifício. Cada elemento diz ao navegador, ao crawler e ao leitor de tela: "esta área é a entrada principal", "aqui estão os artigos de conteúdo".
Entrada principal do edifício → <header> Placa com menu de navegação → <nav> Área principal de trabalho → <main> Sala individual de projeto → <article> Áreas temáticas dentro da sala → <section> Sala de utilidades lateral → <aside> Placa de saída / rodapé → <footer>
// 03 — conceito

A anatomia de uma página semântica

Veja como uma página semântica bem estruturada se organiza visualmente — e o que cada elemento comunica:

<header> logo, título do site, identidade visual
<nav> links de navegação principal — Home · Sobre · Contato
<main> — conteúdo único desta página
<article> — conteúdo independente (post, produto)
<section> — grupo temático dentro do article
<article> — segundo item de conteúdo
<aside> sidebar: relacionados, anúncios, bio
<footer> copyright, links secundários, redes sociais

Quando usar cada elemento — a regra definitiva

Elemento Significado semântico Pode ser reutilizado? Substitui o quê
<header> Cabeçalho — da página ou de uma seção ✓ Vários por página <div id="header">
<nav> Bloco de links de navegação principal ✓ Menu principal + breadcrumb <div class="menu">
<main> Conteúdo único e principal da página ✗ Apenas 1 por página <div id="content">
<article> Conteúdo independente e redistribuível (post, produto, card) ✓ Quantos quiser <div class="card">
<section> Agrupamento temático com título próprio ✓ Quantos quiser <div class="section">
<aside> Conteúdo relacionado mas periférico (sidebar, callout) ✓ Quantos quiser <div class="sidebar">
<footer> Rodapé — da página ou de uma seção ✓ Vários por página <div id="footer">
<div> Nenhum significado — container genérico ✓ Use como último recurso Nada específico
🔥 Regra de ouro para article vs section
Pergunte: "esse conteúdo faz sentido sozinho, fora desta página?" Se sim → <article> (um post de blog, uma review de produto faz sentido isolado). Se só faz sentido dentro do contexto da página → <section>. Ainda em dúvida? Use <div> sem culpa — semântica ruim não quebra nada, mas semântica certa agrega valor.
// 04 — código

Uma página semântica do zero

index.html html 
<!DOCTYPE html>
<html lang="pt-BR">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

  <!-- SEO básico -->
  <title>Blog de Tecnologia — Artigos sobre Dev</title>
  <meta name="description" content="Artigos práticos sobre desenvolvimento web, Python e boas práticas de engenharia.">

  <!-- Open Graph (WhatsApp, LinkedIn, Twitter) -->
  <meta property="og:title"       content="Blog de Tecnologia">
  <meta property="og:description" content="Artigos práticos sobre desenvolvimento web">
  <meta property="og:image"       content="https://meusite.com/og-image.jpg">
  <meta property="og:type"        content="website">

  <link rel="stylesheet" href="style.css">
</head>

<body>

  <!-- Cabeçalho global do site -->
  <header>
    <a href="/" aria-label="Página inicial">
      <img src="logo.svg" alt="Logo do Blog" width="120" height="40">
    </a>

    <!-- Navegação principal -->
    <nav aria-label="Navegação principal">
      <ul>
        <li><a href="/">Home</a></li>
        <li><a href="/artigos">Artigos</a></li>
        <li><a href="/sobre">Sobre</a></li>
        <li><a href="/contato">Contato</a></li>
      </ul>
    </nav>
  </header>

  <!-- Conteúdo único desta página -->
  <main>

    <!-- Seção de destaque -->
    <section aria-labelledby="hero-title">
      <h1 id="hero-title">Os melhores artigos de dev</h1>
      <p>Conteúdo prático para quem está aprendendo a programar.</p>
    </section>

    <!-- Lista de artigos -->
    <section aria-labelledby="artigos-titulo">
      <h2 id="artigos-titulo">Artigos recentes</h2>

      <!-- Cada post é um article — conteúdo independente -->
      <article>
        <header>
          <h3><a href="/artigos/git-branches">Git Branches na Prática</a></h3>
          <time datetime="2025-02-15">15 de fevereiro de 2025</time>
        </header>
        <p>Aprenda a trabalhar com branches, merge e resolução de conflitos...</p>
        <footer>
          <a href="/artigos/git-branches">Ler artigo →</a>
        </footer>
      </article>

      <article>
        <header>
          <h3><a href="/artigos/html-semantico">HTML Semântico — Por que importa</a></h3>
          <time datetime="2025-02-10">10 de fevereiro de 2025</time>
        </header>
        <p>Entenda a diferença entre div e elementos semânticos...</p>
        <footer>
          <a href="/artigos/html-semantico">Ler artigo →</a>
        </footer>
      </article>

    </section>
  </main>

  <!-- Sidebar -->
  <aside aria-label="Conteúdo relacionado">
    <section>
      <h2>Artigos populares</h2>
      <ul>
        <li><a href="/artigos/python-funcoes">Python — Funções avançadas</a></li>
        <li><a href="/artigos/css-grid">CSS Grid do zero</a></li>
      </ul>
    </section>
  </aside>

  <!-- Rodapé global -->
  <footer>
    <nav aria-label="Links do rodapé">
      <a href="/privacidade">Privacidade</a>
      <a href="/termos">Termos</a>
    </nav>
    <p><small>&copy; 2025 Blog de Tecnologia. Todos os direitos reservados.</small></p>
  </footer>

</body>
</html>
// 05 — formulários

Formulários — todos os inputs que você precisa saber

Formulários são uma das partes mais negligenciadas do HTML. O navegador oferece validação, tipos especializados e acessibilidade gratuitamente — mas só se você usar os elementos e atributos corretos.

Preview ao vivo — formulário real com HTML puro
formulario-completo.html html 
<!-- SEMPRE: label conectado ao input via for/id -->
<label for="nome">Nome completo *</label>
<input type="text"
  id="nome"
  name="nome"
  required                    <!-- validação nativa -->
  autocomplete="name"         <!-- hint para autopreenchimento -->
  placeholder="João Silva"
  aria-describedby="nome-hint" >
<span id="nome-hint">Use seu nome como aparece no documento.</span>

<!-- TIPOS ESPECIAIS -->
<input type="email"    >  <!-- valida formato email -->
<input type="password" >  <!-- oculta texto -->
<input type="number" min="0" max="100" step="5">
<input type="date"     >  <!-- seletor de data nativo -->
<input type="tel"      >  <!-- teclado numérico no mobile -->
<input type="url"      >  <!-- valida formato URL -->
<input type="search"   >  <!-- com botão de limpar -->
<input type="range" min="0" max="10">  <!-- slider -->
<input type="color"    >  <!-- color picker nativo -->
<input type="file" accept=".pdf,.jpg">
<input type="checkbox" >  <!-- caixa de seleção -->
<input type="radio"    >  <!-- seleção exclusiva -->
<input type="hidden"   >  <!-- dados invisíveis ao usuário -->

<!-- TEXTAREA -->
<label for="msg">Mensagem</label>
<textarea id="msg" name="msg" rows="4" maxlength="500"
  placeholder="Escreva sua mensagem..."></textarea>

<!-- SELECT com grupos -->
<select id="linguagem" name="linguagem">
  <option value="">Escolha uma linguagem</option>
  <optgroup label="Web">
    <option value="html">HTML/CSS</option>
    <option value="js">JavaScript</option>
  </optgroup>
  <optgroup label="Backend">
    <option value="python">Python</option>
  </optgroup>
</select>

<!-- FIELDSET para agrupar inputs relacionados -->
<fieldset>
  <legend>Preferências de contato</legend>
  <label><input type="radio" name="contato" value="email"> E-mail</label>
  <label><input type="radio" name="contato" value="whatsapp"> WhatsApp</label>
</fieldset>

<!-- BUTTON com tipos -->
<button type="submit">Enviar formulário</button>
<button type="reset">Limpar</button>
<button type="button">Ação customizada (JS)</button>
⚠️ Nunca use <div> ou <span> como botão clicável
<div onclick="..."> quebra acessibilidade (não é focável por teclado), não dispara em Enter/Space, e não tem estado disabled. Use sempre <button> ou <a href>. Se precisar de estilo customizado, estilize o button — não invente substitutos.
// 06 — media

Media & Embeds — imagens, vídeo e audio modernos

Imagens responsivas e vídeo bem implementados são um dos maiores impactos em performance web. O HTML moderno tem elementos e atributos específicos para cada caso — e usar img src simples para tudo é deixar performance na mesa.

🖼️
<img>
Imagem simples. Sempre com alt descritivo (ou alt="" se decorativa).
loading="lazy" · width · height · decoding="async"
📐
<picture>
Imagem com múltiplos formatos e breakpoints. Para arte direcionada (mudar o enquadramento em mobile).
source · media · srcset · type
🎬
<video>
Vídeo nativo sem player externo. Múltiplos formatos via <source>.
controls · autoplay · muted · loop · poster · preload
🎵
<audio>
Player de áudio nativo. Para podcasts, efeitos sonoros, músicas.
controls · loop · preload · src
media-modernos.html html 
<!-- ─── IMG BÁSICO — SEMPRE com alt ──────────────── -->
<img
  src="hero.jpg"
  alt="Desenvolvedor digitando em laptop num café moderno"
  width="800"
  height="600"
  loading="lazy"
  decoding="async">

<!-- alt="" em imagens decorativas (leitores de tela pulam) -->
<img src="divider-line.svg" alt="" role="presentation">


<!-- ─── SRCSET — imagens responsivas ──────────────── -->
<img
  src="hero-800.jpg"
  srcset="
    hero-400.jpg  400w,
    hero-800.jpg  800w,
    hero-1200.jpg 1200w"
  sizes="
    (max-width: 600px) 100vw,
    (max-width: 900px) 50vw,
    800px"
  alt="Hero banner do produto"
  loading="eager">  <!-- acima da dobra: eager, não lazy -->


<!-- ─── PICTURE — arte direcionada + formatos modernos ── -->
<picture>
  <!-- Formato AVIF (menor, mais moderno) -->
  <source type="image/avif" srcset="foto.avif">

  <!-- Formato WebP (fallback) -->
  <source type="image/webp" srcset="foto.webp">

  <!-- Mobile: versão vertical -->
  <source media="(max-width: 600px)" srcset="foto-mobile.jpg">

  <!-- Fallback universal -->
  <img src="foto.jpg" alt="Foto do produto em destaque" width="800" height="600">
</picture>


<!-- ─── VIDEO ──────────────────────────────────────── -->
<video
  width="800"
  height="450"
  poster="thumbnail.jpg"   <!-- imagem antes do play -->
  controls
  preload="metadata"        <!-- carrega só metadados -->
  muted                     <!-- necessário para autoplay -->>
  <source src="video.mp4"  type="video/mp4">
  <source src="video.webm" type="video/webm">
  <p>Seu navegador não suporta vídeo HTML5.
     <a href="video.mp4">Baixe aqui</a>.</p>
</video>


<!-- ─── AUDIO ──────────────────────────────────────── -->
<audio controls preload="none">
  <source src="episodio.mp3" type="audio/mpeg">
  <source src="episodio.ogg" type="audio/ogg">
</audio>


<!-- ─── IFRAME (embeds externos) ──────────────────── -->
<iframe
  src="https://www.google.com/maps/embed?..."
  width="600"
  height="400"
  loading="lazy"
  title="Mapa com localização do escritório"  <!-- obrigatório para acessibilidade -->
  allowfullscreen>
</iframe>
🔥 Pro tip — loading="lazy" em tudo abaixo da dobra
Adicionar loading="lazy" em todas as imagens que não aparecem na tela inicial é uma das melhorias de performance mais fáceis que existem — e é 1 atributo a mais no HTML. O navegador só vai baixar a imagem quando o usuário estiver chegando perto dela no scroll. Não use em imagens acima da dobra (hero, logo) — isso piora o LCP.
// 07 — hierarquia

A hierarquia de headings — H1 a H6

Os headings (<h1> a <h6>) não são só "texto grande" — eles formam o outline do documento, usado por leitores de tela para navegação e pelo Google para entender a estrutura do conteúdo.

// outline correto de uma página de blog
graph TD H1["H1 — Título da página (único)"] H2a["H2 — Seção: Artigos Recentes"] H2b["H2 — Seção: Sobre o Autor"] H3a["H3 — Título do Artigo 1"] H3b["H3 — Título do Artigo 2"] H4a["H4 — Subtópico dentro do artigo"] H1 --> H2a H1 --> H2b H2a --> H3a H2a --> H3b H3a --> H4a style H1 fill:#ef4444,color:#fff,stroke:none style H2a fill:#16161f,stroke:#ef4444,color:#e8e8f0 style H2b fill:#16161f,stroke:#ef4444,color:#e8e8f0 style H3a fill:#16161f,stroke:#8585a8,color:#e8e8f0 style H3b fill:#16161f,stroke:#8585a8,color:#e8e8f0 style H4a fill:#16161f,stroke:#1e1e2e,color:#8585a8
⚠️ Não pule níveis de heading
Ir de <h2> direto para <h4> é como pular capítulos num livro. Quebre a hierarquia e leitores de tela perdem o contexto. Se você quer um texto menor visualmente, use CSS — não mude o nível do heading.

// 09 — armadilhas

Erros clássicos de HTML

⚠️ Div soup — o pesadelo semântico

Páginas com dezenas de <div class="wrapper"><div class="container"><div class="inner"> são a norma em código legado. Cada div extra é um elemento sem significado que polui o DOM. Revise: pode ser substituído por um elemento semântico?

⚠️ Imagens sem alt ou com alt genérico

alt="imagem" ou alt="foto" é pior do que um alt vazio. Descreva o que a imagem comunica: alt="Gráfico mostrando crescimento de 40% em receita no Q3 2024". Imagens puramente decorativas? Use alt="" — leitores de tela vão pular.

⚠️ <label> sem conexão com o <input>

Um <label> solto sem o atributo for apontando para o id do input não tem efeito nenhum — nem visual (clicar no label não foca o input) nem de acessibilidade. É um dos erros mais comuns em formulários e um dos mais impactantes.

⚠️ Múltiplos <h1> numa página

Antes do HTML5, múltiplos <h1> eram tecnicamente válidos em contextos de sectioning elements. Na prática hoje: use um único <h1> por página. Todos os motores de busca e leitores de tela modernos esperam isso. Segundo título? Use <h2>.

⚠️ <b> e <i> ao invés de <strong> e <em>

<b> e <i> são elementos de apresentação — apenas mudam a aparência. <strong> e <em> têm significado semântico (importância e ênfase) e são lidos com entonação diferente por leitores de tela. Use sempre os semânticos.

// 11 — desafio

🏁 Checkpoint — construa uma página de portfólio

Crie um arquivo portfolio.html do zero com a seguinte estrutura — sem usar CSS ainda, só HTML semântico:

  • Um <header> com seu nome em <h1> e um <nav> com 3 links internos (âncoras com #)
  • Um <main> contendo: uma <section> "Sobre mim", uma <section> "Projetos" com pelo menos 2 <article> dentro
  • Cada article com: <h3>, <p>, e uma imagem com alt descritivo e loading="lazy"
  • Um formulário de contato com: campo nome (type="text"), email (type="email"), mensagem (<textarea>), e botão submit — todos com <label> conectado
  • Um <aside> com lista de habilidades
  • Um <footer> com copyright e links de redes sociais
  • Valide em validator.w3.org e corrija todos os erros
💬 Dica: Comece pelo esqueleto (header, main, aside, footer) antes de preencher o conteúdo. Use a extensão Live Server no VS Code para ver as mudanças em tempo real. O HTML sem CSS vai parecer feio — isso é totalmente normal e esperado. Na próxima aula você vai estilizar tudo isso com CSS.
