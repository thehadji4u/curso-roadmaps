⏱ ~40 min de leitura 🎯 Intermediário 🔧 Hands-on incluído

// MOTIVAÇÃO

Por que isso importa?

Qualquer pessoa com Google consegue escrever um Dockerfile. Mas um Dockerfile ruim gera imagens de 1 GB que demoram 10 minutos para buildar, que contêm vulnerabilidades críticas e que quebram silenciosamente em produção sem nenhum sinal. A diferença entre um container que "funciona no meu laptop" e um que roda de forma confiável em produção por meses está inteiramente no Dockerfile.

Nesta aula vamos aprender a escrever Dockerfiles que são rápidos de buildar (layer caching inteligente), pequenos (imagens mínimas), seguros (sem root, sem segredos expostos) e resilientes (health checks). Essas práticas são o que separa um DevOps júnior de um sênior.

💼 Por que um engenheiro sênior se preocupa com cada byte do Dockerfile
Em um pipeline de CI/CD com 50 builds por dia, uma imagem de 200 MB vs 1 GB representa horas de diferença em tempo de deploy. Um container rodando como root é uma vulnerabilidade de segurança crítica. Um health check mal configurado significa que o orchestrator (Kubernetes, ECS) não consegue detectar quando a aplicação travou. Cada instrução do Dockerfile tem consequências reais de custo, velocidade e segurança.

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense num Dockerfile como uma receita de bolo profissional de uma confeitaria. Uma receita amadora lista os ingredientes e diz "misture tudo e asse". Já a receita de um chef Michelin detalha: a temperatura exata, a ordem de adição dos ingredientes, os ingredientes mínimos necessários (sem desperdício), os checkpoints de verificação ("espete um palito no minuto 25") e as instruções de segurança alimentar ("higienize as mãos antes").

A receita amadora produz um bolo. A receita profissional produz um bolo consistente, mais rápido, mais barato e que pode ser reproduzido por qualquer confeiteiro do mundo com o mesmo resultado. O Dockerfile é exatamente essa receita — e escrever bem essa receita é o que diferencia uma imagem de produção de uma imagem de "funciona aqui".
📋 Receita com lista de passos → Dockerfile (instrução por instrução) 🧁 Cada camada de montagem do bolo → Cada instrução = uma layer de imagem 🥚 Ingredientes pré-pesados e organizados → Base image já com dependências do SO 🍰 Bolo finalizado pronto para entrega → Docker image pronta para deploy ✅ Palito para testar se está cozido → HEALTHCHECK no container 🙅 Chef não lambe os dedos na cozinha → Container roda como non-root user

Anatomia de um Dockerfile

Um Dockerfile é uma sequência de instruções que o Docker executa de cima para baixo para construir uma imagem. Cada instrução cria uma nova layer (camada) imutável, empilhada sobre a anterior. O Docker reutiliza layers do cache sempre que possível — o que torna a ordem das instruções fundamental para builds rápidos.

Instruções Fundamentais

1
FROM
FROM python:3.12-slim
Define a imagem base. Obrigatória — sempre a primeira instrução. Escolha a menor imagem que atenda suas necessidades.
2
WORKDIR
WORKDIR /app
Define o diretório de trabalho dentro do container. Cria o diretório se não existir. Use sempre em vez de RUN mkdir && cd.
3
COPY / ADD
COPY requirements.txt .
Copia arquivos do host para a imagem. COPY é preferível — apenas copia. ADD tem comportamentos extras (extrai tarballs, baixa URLs) — use somente quando necessário.
4
RUN
RUN pip install -r requirements.txt
Executa comandos durante o build. Cada RUN separado cria uma nova layer. Encadeie comandos com && para reduzir layers e tamanho.
5
ENV
ENV PORT=8000
Define variáveis de ambiente disponíveis no build e em runtime. Use para configurações não-sensíveis. Nunca para senhas ou tokens.
6
EXPOSE
EXPOSE 8000
Documenta qual porta o container usa. Não publica a porta — é apenas metadado para quem lê o Dockerfile e para ferramentas como Docker Compose.
7
USER
USER appuser
Define o usuário para execução. Essencial para segurança — o processo não deve rodar como root. Crie um usuário sem privilégios antes desta instrução.
8
HEALTHCHECK
HEALTHCHECK CMD curl -f http://localhost:8000/health
Define como o Docker/orchestrator verifica se o container está saudável. Containers com falha no health check recebem status unhealthy e são reiniciados.
9
CMD / ENTRYPOINT
CMD ["python", "app.py"]
CMD define o comando padrão (pode ser sobrescrito em docker run). ENTRYPOINT define o executável fixo. Prefira a forma JSON array ["cmd", "arg"] — evita interpretação por shell.
// OTIMIZAÇÃO

Layer Caching — a regra de ouro

O Docker reutiliza layers do cache quando uma instrução e todos os seus inputs são idênticos ao build anterior. Quando uma layer muda, todas as layers subsequentes são invalidadas e reexecutadas. Isso torna a ordem das instruções crítica para performance.

⚠️ A Regra de Ouro do Cache
Coloque instruções que mudam raramente no topo e instruções que mudam frequentemente no fundo. O código da aplicação muda a cada commit — ele deve ser o último a ser copiado. As dependências mudam menos — instale-as antes.
❌ Ordem ruim — rebuilda tudo sempre
FROM python:3.12-slim
WORKDIR /app

# Copia TUDO de uma vez
COPY . .

# Instala dependências DEPOIS de copiar o código
# Toda vez que qualquer arquivo mudar,
# o pip install é reexecutado!
RUN pip install -r requirements.txt

CMD ["python", "app.py"]
✅ Ordem correta — aproveita cache
FROM python:3.12-slim
WORKDIR /app

# 1. Copia APENAS o arquivo de dependências
COPY requirements.txt .

# 2. Instala dependências (cacheia se o arquivo não mudou)
RUN pip install --no-cache-dir -r requirements.txt

# 3. Copia o código por último (muda com frequência)
COPY . .

CMD ["python", "app.py"]

Com a ordem correta, a cada novo commit o Docker pula todas as layers de instalação de dependências e rebuild apenas a camada do código — um build que antes demorava 4 minutos passa a demorar 8 segundos.

Dockerfile Completo — Python API com Boas Práticas

Veja um Dockerfile de produção completo com todas as práticas aplicadas: cache otimizado, usuário não-root, health check, .dockerignore e instruções comentadas.

Dockerfile — API Python/FastAPI
# ─── Estágio de Build ────────────────────────────────────────────────────────
# Usamos slim para ter compiladores disponíveis durante o build
FROM python:3.12.3-slim-bookworm AS builder

# Variáveis para desabilitar bytecode e buffering (boa prática em containers)
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1

WORKDIR /app

# ── Instalar dependências do sistema (somente o necessário) ──────────────────
# Encadeamos em um único RUN para minimizar layers
# --no-install-recommends evita instalar pacotes desnecessários
# Limpamos o cache do apt na mesma layer para não inflar a imagem
RUN apt-get update && apt-get install -y \
    --no-install-recommends \
    gcc \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# ── Dependências Python ───────────────────────────────────────────────────────
# Copiamos APENAS o arquivo de requirements ANTES do código da aplicação
# Isso garante que o pip install seja cacheado quando o código mudar
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# ─── Estágio Final (imagem de produção) ──────────────────────────────────────
# Partimos de uma imagem limpa, sem os compiladores do estágio anterior
FROM python:3.12.3-slim-bookworm AS production

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PORT=8000

WORKDIR /app

# ── Copiar apenas os pacotes instalados do estágio de build ──────────────────
# Não copiamos os compiladores (gcc etc.) — apenas as libs Python instaladas
COPY --from=builder /usr/local/lib/python3.12/site-packages /usr/local/lib/python3.12/site-packages
COPY --from=builder /usr/local/bin /usr/local/bin

# ── Criar usuário sem privilégios ────────────────────────────────────────────
# NUNCA rode a aplicação como root em produção
# --no-create-home e --disabled-password minimizam a superfície de ataque
RUN groupadd --gid 1001 appgroup && \
    useradd --uid 1001 --gid 1001 --no-create-home --shell /bin/false appuser

# ── Copiar o código da aplicação por último ──────────────────────────────────
# Esta é a camada que muda com mais frequência — fica por último para
# maximizar o aproveitamento do cache das layers anteriores
COPY --chown=appuser:appgroup . .

# ── Mudar para usuário sem privilégios ───────────────────────────────────────
USER appuser

# ── Documentar porta (não publica — apenas metadado) ─────────────────────────
EXPOSE 8000

# ── Health check ─────────────────────────────────────────────────────────────
# --interval: verifica a cada 30s
# --timeout: desiste após 10s sem resposta
# --start-period: aguarda 40s antes de começar a verificar (tempo de startup)
# --retries: declara unhealthy após 3 falhas consecutivas
HEALTHCHECK --interval=30s --timeout=10s --start-period=40s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" \
    || exit 1

# ── Comando de inicialização ─────────────────────────────────────────────────
# Forma JSON array (sem shell intermediário) — melhor para tratamento de sinais
# O processo recebe SIGTERM diretamente, permitindo shutdown gracioso
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

Multi-Stage Builds — o segredo das imagens pequenas

Multi-stage builds permitem usar múltiplos FROM num único Dockerfile. O estágio final copia apenas o que precisa dos estágios anteriores — sem trazer compiladores, dependências de desenvolvimento ou arquivos temporários de build.

Multi-Stage Build: Build Stage vs Production Stage
flowchart TD subgraph "Estágio 1: builder (descartado)" B1["FROM python:3.12-slim AS builder"] --> B2["apt-get install gcc libpq-dev"] B2 --> B3["COPY requirements.txt"] B3 --> B4["RUN pip install (com compiladores)"] B4 --> B5["Resultado: 450 MB\n(com gcc, headers, cache)"] end subgraph "Estágio 2: production (imagem final)" P1["FROM python:3.12-slim AS production"] --> P2["COPY --from=builder\nsite-packages only"] P2 --> P3["COPY app code"] P3 --> P4["USER appuser"] P4 --> P5["Resultado: 130 MB\n(sem compiladores)"] end B4 -->|"COPY --from=builder\n(só as libs instaladas)"| P2 style B5 fill:#3a1010,stroke:#ef4444,color:#f87171 style P5 fill:#0a3a1a,stroke:#10b981,color:#34d399
🔥 Multi-stage para Go, Node e Java
Multi-stage brilha especialmente em linguagens compiladas. Para Go: compile no primeiro estágio, copie apenas o binário para uma imagem scratch (vazia!) ou distroless — resultado final pode ser menor que 20 MB. Para Node.js: instale devDependencies no estágio de build, copie apenas node_modules de produção. Para Java: compile com JDK completo, rode com JRE mínimo.

Health Checks — containers que se auto-monitoram

Um container "rodando" (docker ps mostra Up) não significa que a aplicação dentro está funcional. O processo pode estar travado em deadlock, com o banco de dados desconectado, ou com memória esgotada. O HEALTHCHECK ensina o Docker a distinguir um container saudável de um container vivo mas disfuncional.

ParâmetroPadrãoSignificadoRecomendação
--interval 30s Frequência das verificações 30s para produção, 10s para desenvolvimento
--timeout 30s Tempo máximo aguardado por resposta 5–10s — aplicações lentas podem mascarar problemas
--start-period 0s Janela de graça no startup Ajuste para o tempo real de boot da sua aplicação
--retries 3 Falhas consecutivas para declarar unhealthy 3 para evitar falsos positivos por picos de carga
Exemplos de HEALTHCHECK por tipo de aplicação
# API HTTP — endpoint /health dedicado (mais comum)
HEALTHCHECK --interval=30s --timeout=10s --start-period=40s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

# API HTTP — sem curl (usando Python puro, mais portável)
HEALTHCHECK --interval=30s --timeout=10s --start-period=40s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

# Worker sem HTTP — verificar se processo está respondendo
HEALTHCHECK --interval=60s --timeout=30s --start-period=120s --retries=2 \
    CMD python healthcheck.py || exit 1

# PostgreSQL — verificar se o banco aceita conexões
HEALTHCHECK --interval=10s --timeout=5s --start-period=30s --retries=5 \
    CMD pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB} || exit 1

# Redis — verificar se responde a PING
HEALTHCHECK --interval=10s --timeout=3s --retries=5 \
    CMD redis-cli ping | grep PONG || exit 1

Segurança em Dockerfiles

Containers inseguros são vetores de ataque frequentes. Um atacante que compromete um processo root dentro de um container tem chances muito maiores de escapar para o host. As práticas abaixo reduzem drasticamente a superfície de ataque.

1. Nunca rode como root

❌ Processo roda como root (padrão!)
FROM python:3.12-slim
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
# Sem USER = processo roda como root (uid 0)
CMD ["python", "app.py"]
✅ Processo roda como usuário dedicado
FROM python:3.12-slim
WORKDIR /app
RUN groupadd --gid 1001 appgroup && \
    useradd --uid 1001 --gid 1001 \
    --no-create-home --shell /bin/false appuser
COPY --chown=appuser:appgroup . .
RUN pip install -r requirements.txt
USER appuser
CMD ["python", "app.py"]

2. Nunca embutir segredos no Dockerfile

🚨 Segredos no Dockerfile ficam na imagem para sempre

Mesmo que você faça RUN rm -rf /tmp/secret em uma layer posterior, o segredo ainda está visível na layer onde foi criado. Qualquer pessoa com acesso à imagem pode extraí-lo com docker history ou inspecionando as layers.

❌ Segredo gravado na imagem
# NUNCA faça isso!
ENV DATABASE_URL=postgresql://user:senha123@db/prod
RUN pip install ... --extra-index-url https://token@pypi.company.com
✅ Segredo injetado em runtime
# Variável sem valor padrão = injetada no docker run
ENV DATABASE_URL=""
# Em runtime:
# docker run -e DATABASE_URL=$DB_URL myimage
# Ou use Docker secrets / Vault

3. Use --mount=type=secret para segredos no build

Quando você precisa de credenciais durante o build (ex: PyPI privado, npm registry privado), use BuildKit secrets — eles nunca são armazenados nas layers:

Dockerfile com BuildKit secrets
# syntax=docker/dockerfile:1
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
# O segredo é montado temporariamente apenas durante este RUN
# e nunca fica gravado na imagem
RUN --mount=type=secret,id=pip_token \
    pip install --extra-index-url https://$(cat /run/secrets/pip_token)@pypi.company.com/simple/ \
    -r requirements.txt
Build com o segredo
docker build --secret id=pip_token,env=PIP_TOKEN .
PráticaImpactoComo implementar

⚠️ Armadilhas Comuns

1. Usar ENTRYPOINT sem CMD (ou vice-versa de forma errada)

ENTRYPOINT ["uvicorn"] sem CMD significa que você sempre executa uvicorn, mas sem argumentos. A combinação correta é ENTRYPOINT para o executável fixo e CMD para argumentos padrão sobrescrevíveis. Ou simplesmente use apenas CMD para máxima flexibilidade.

2. RUN apt-get update e install em linhas separadas

Se você fizer RUN apt-get update em uma layer e RUN apt-get install em outra, o Docker pode cachear o update e usar uma versão desatualizada dos repositórios. Sempre encadeie: RUN apt-get update && apt-get install -y ...

3. Usar ADD quando COPY basta

ADD tem comportamentos implícitos — extrai archives (.tar, .gz) automaticamente e pode baixar URLs remotas. Esses efeitos colaterais tornam o build imprevisível e mais difícil de auditar. Use COPY por padrão; use ADD apenas quando você explicitamente quer a extração de tarballs.

4. HEALTHCHECK com curl — mas a imagem não tem curl

Imagens slim e alpine geralmente não incluem curl. Duas soluções: instale curl explicitamente no Dockerfile (adiciona ~5 MB), ou use a forma com Python puro (urllib.request) que já está disponível em qualquer imagem Python.

5. Copiar .env para dentro da imagem sem .dockerignore

Sem um .dockerignore, o COPY . . inclui todos os arquivos do diretório — incluindo .env com senhas, chaves de API e tokens. Sempre crie um .dockerignore antes de qualquer COPY . . e inclua .env e .env.* nele.

Termos desta Aula

Layer (Camada)
Resultado imutável de cada instrução do Dockerfile. Layers são empilhadas e compartilhadas entre imagens — se duas imagens usam a mesma base, elas compartilham aquelas layers no disco.
Layer Cache
Mecanismo que reutiliza layers já computadas quando a instrução e seus inputs são idênticos. A invalidação do cache propaga para todas as layers subsequentes.
Multi-stage Build
Técnica que usa múltiplos FROM em um Dockerfile. Permite usar ferramentas de build (compiladores, npm) apenas no estágio de build, copiando apenas os artefatos necessários para a imagem final.
HEALTHCHECK
Instrução que define como o Docker verifica periodicamente se o container está funcionando corretamente. Estados possíveis: starting, healthy, unhealthy.
Non-root container
Container cujo processo principal roda como um usuário sem privilégios (uid != 0). Prática de segurança fundamental — limita o impacto de vulnerabilidades na aplicação.
.dockerignore
Arquivo similar ao .gitignore que lista arquivos e pastas a serem excluídos do contexto de build. Reduz tamanho da imagem e evita vazar segredos acidentalmente.
BuildKit
Backend de build moderno do Docker (padrão desde Docker 23). Oferece builds paralelos, cache avançado, builds multiplataforma e suporte a segredos que não ficam gravados nas layers.
Distroless
Imagens que contêm apenas o runtime da linguagem — sem shell, sem package manager, sem utilitários Unix. Produzidas pelo Google. Superfície de ataque mínima para produção.

🏁 Checkpoint — Desafio Final

Dockerize uma aplicação Python de sua escolha (pode ser um script simples que processa dados, um servidor Flask ou a API FastAPI da aula) seguindo todas as boas práticas desta aula:

  • Dockerfile com layer caching otimizado (requirements antes do código)
  • Imagem base slim ou menor — verifique o tamanho com docker images
  • Usuário dedicado non-root com useradd e instrução USER
  • HEALTHCHECK funcional (verifique com docker inspect --format='{{.State.Health.Status}}')
  • .dockerignore excluindo .git, .env, __pycache__ e venv
  • Comprove o cache de layers fazendo dois builds e medindo o tempo do segundo
  • Bônus: Implemente multi-stage build e compare o tamanho antes/depois
💬 Dica: use docker history minha-imagem para inspecionar todas as layers geradas e ver o tamanho de cada uma. E docker scout cves minha-imagem para checar vulnerabilidades conhecidas — ótimo para entender o impacto da escolha da base image.

O que vem a seguir?

Com Dockerfiles bem escritos e imagens otimizadas, o próximo passo é orquestrar múltiplos containers juntos. O Docker Compose permite definir toda a infraestrutura da aplicação — API, banco de dados, cache, workers — em um único arquivo declarativo, com redes privadas, variáveis de ambiente e volumes configurados.
