CSS — Animações, Tipografia & Efeitos

⏱ ~35 min de leitura 🎯 Intermediário 🔧 Hands-on incluído
@keyframes transition animation transform pseudo-elementos clamp() backdrop-filter will-change
// motivação

Por que isso importa?

A diferença entre um site "funcional" e um produto que as pessoas amam usar quase sempre está no movimento. Animações bem feitas guiam o olhar, confirmam ações, revelam hierarquia e criam sensação de qualidade — tudo sem uma linha de JavaScript.

Mas há dois lados desta moeda: animações mal implementadas destroem a performance e causam náusea em usuários com sensibilidade a movimento. Dominar CSS de animação é saber quando animar, o que animar e como fazer isso com zero custo na thread principal.

💼 Por que um engenheiro sênior usa isso hoje

Times como Nubank e Notion definem design tokens de animação — variáveis CSS como --duration-fast: 150ms, --easing-spring: cubic-bezier(0.34,1.56,0.64,1) — que garantem consistência em todo o produto. Cada componente usa esses tokens, e quando o design system muda, uma única variável atualiza tudo. Além disso, toda animação obedece @media (prefers-reduced-motion) por padrão, tornando o produto acessível sem esforço extra.

// analogia

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense em um animador de cinema clássico como o Disney. Ele tem dois tipos de movimento: transições são como cenas que conectam estado A ao estado B em resposta a um evento — o personagem abre um olho quando chamado. Keyframe animations são como um loop de ação autônoma — a chama de uma vela que oscila eternamente sem precisar de gatilho externo.

O animador da Disney também sabia o segredo do movimento orgânico: objetos reais nunca se movem em linha reta com velocidade constante. Eles aceleram no início, desaceleram no fim, esticam, comprimem. Em CSS isso se chama easing (cubic-bezier). E assim como um filme tem orçamento de fotogramas, seu site tem orçamento de GPU — só as propriedades transform e opacity são "de graça".
Cena de transição entre estados → CSS transition (ativada por evento) Loop autônomo da chama → @keyframes + animation (roda sozinha) Curva de aceleração natural → cubic-bezier / easing functions Orçamento de fotogramas (60 fps) → Só animar transform & opacity
// conceito

O conceito

CSS oferece três sistemas de movimento que trabalham em conjunto: transitions para mudanças de estado reativas, @keyframes + animation para sequências autônomas, e transform como o "motor" de ambos — a única propriedade que move pixels sem disparar reflow na página.

O segredo da performance é entender o pipeline de renderização do browser: Layout → Paint → Composite. Propriedades que acionam Layout (width, height, top, margin) são as mais caras. Transform e opacity vivem apenas na etapa Composite — rodando inteiramente na GPU, sem tocar no layout da página. Por isso, um card que "voa" com transform: translateY(-8px) usa 0% da CPU; o mesmo efeito com top: -8px pode travar a animação.

// comparação

transition vs animation: guia de escolha

Característica transition @keyframes + animation
Gatilho Precisa de mudança de estado (:hover, :focus, classe JS) Roda automaticamente ao renderizar
Pontos intermediários Apenas início e fim Infinitos — 0%, 25%, 60%, 100%...
Loop Não tem loop nativo animation-iteration-count: infinite
Direção reversa Automático ao desfazer o estado animation-direction: alternate
Delay transition-delay animation-delay (aceita negativo!)
Pausa Não tem animation-play-state: paused
Uso típico Hover em botão, foco em input, abertura de menu Loading spinner, skeleton, banner, efeito de entrada

Propriedades seguras para animar (GPU-only)

Propriedade Custo Uso
transform: translate() ✅ Composite only Mover elementos, slides, parallax
transform: scale() ✅ Composite only Hover de cards, zoom, pop-in
transform: rotate() ✅ Composite only Spinners, ícones, carousséis
opacity ✅ Composite only Fade in/out, hover states
background-color ⚠️ Paint OK para transições curtas
width / height ❌ Layout + Paint + Composite Evitar — usar transform: scaleX() ou max-height hack

2
Transitions no hover do card
Elevação suave, zoom da imagem e aparecimento do badge — tudo com transition:
style.css CSS 
/* Design tokens de animação */
:root {
  --duration-fast:   150ms;
  --duration-base:   250ms;
  --duration-slow:   400ms;
  --ease-out:        cubic-bezier(0.0, 0.0, 0.2, 1);
  --ease-spring:     cubic-bezier(0.34, 1.56, 0.64, 1);
  --ease-smooth:     cubic-bezier(0.4, 0.0, 0.2, 1);
}

/* Card base */
.card {
  border-radius: 16px;
  overflow: hidden;
  background: #fff;
  box-shadow: 0 2px 8px rgba(0,0,0,0.08);
  /* Transition do card inteiro */
  transition:
    transform var(--duration-base) var(--ease-out),
    box-shadow var(--duration-base) var(--ease-out);
}

/* Hover do card: eleva com transform (GPU ✅) */
.card:hover {
  transform: translateY(-6px);
  box-shadow: 0 16px 40px rgba(0,0,0,0.15);
}

/* Imagem com zoom no hover */
.card-img-wrap {
  overflow: hidden;    /* corta o zoom */
  position: relative;
}

.card-img {
  width: 100%;
  display: block;
  /* Transition da imagem */
  transition: transform var(--duration-slow) var(--ease-smooth);
}

.card:hover .card-img {
  transform: scale(1.08);  /* zoom — GPU ✅ */
}

/* Badge que aparece no hover */
.card-badge {
  position: absolute;
  top: 12px; left: 12px;
  background: #ef4444;
  color: #fff;
  padding: 4px 10px;
  border-radius: 99px;
  font-size: 0.75rem;
  font-weight: 700;
  /* Estado inicial: invisível e levemente acima */
  opacity: 0;
  transform: translateY(-8px);
  transition:
    opacity var(--duration-fast) var(--ease-out),
    transform var(--duration-fast) var(--ease-spring);
}

.card:hover .card-badge {
  opacity: 1;
  transform: translateY(0);
}

4
Pseudo-elementos ::before/::after + prefers-reduced-motion
Efeito de brilho no hover usando apenas pseudo-elementos, e acessibilidade com prefers-reduced-motion:
style.css CSS 
/* Efeito "luz que passa" no card com ::before */
.card {
  position: relative;  /* contexto para o pseudo-elemento */
}

.card::before {
  content: '';
  position: absolute;
  inset: 0;
  border-radius: inherit;
  /* Gradiente em diagonal que vai de transparente a branco suave */
  background: linear-gradient(
    135deg,
    rgba(255,255,255,0.15) 0%,
    transparent 50%
  );
  /* Começa fora da tela à esquerda */
  transform: translateX(-100%);
  transition: transform var(--duration-slow) var(--ease-out);
  pointer-events: none;  /* não bloqueia cliques */
  z-index: 1;
}

.card:hover::before {
  transform: translateX(100%);  /* atravessa o card */
}

/* ── ACESSIBILIDADE: prefers-reduced-motion ───────── */
/* Usuários que preferem menos movimento no sistema    */
@media (prefers-reduced-motion: reduce) {
  /* Remove TODAS as animações/transições */
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

/* Ou, de forma mais cirúrgica, manter apenas feedback essencial */
@media (prefers-reduced-motion: reduce) {
  .card:hover {
    transform: none;  /* remove o lift */
    box-shadow: 0 0 0 2px #ef4444;  /* usa outline como alternativa */
  }
  .card-img { transform: none; }
}

Erros comuns

⚠️ Animar width/height em vez de transform: scale()

width: 0 → width: 100% em transition dispara reflow completo a cada frame — o browser recalcula o layout de toda a página. Resultado: janking visível, especialmente em mobile. A alternativa é transform: scaleX(0) → scaleX(1), que roda 100% na GPU. Para "revelar" altura, o truque é max-height: 0 → max-height: 500px (com cuidado) ou usar a nova transition: height allow-discrete do CSS moderno.

⚠️ Esquecer will-change em animações complexas

Para elementos com animações pesadas, will-change: transform avisa o browser para criar uma camada GPU antecipadamente. Mas use com moderação — cada will-change cria uma camada e consome memória de GPU. Aplique apenas em elementos que realmente vão animar, e remova após a animação terminar se possível. Nunca use will-change: all.

🚨 Omitir prefers-reduced-motion

Cerca de 26% dos usuários de iOS usam "Reduce Motion". Pessoas com epilepsia, vertigem, ou enxaqueca podem ser fisicamente afetadas por animações que piscam ou se movem muito. Sempre inclua @media (prefers-reduced-motion: reduce). A regra de ouro: nunca animar nada que pisque mais de 3x/segundo, e sempre oferecer alternativa estática para qualquer animação "decorativa".

⚠️ backdrop-filter sem fallback

backdrop-filter: blur() não funciona em alguns contextos (elemento pai com overflow: hidden em versões antigas, ou quando o browser não suporta). Sempre teste e ofereça um background sólido como fallback: background: rgba(0,0,0,0.8) quando backdrop-filter não estiver disponível usando @supports not (backdrop-filter: blur(1px)).

🏁 Checkpoint — valide seu aprendizado

Crie uma página de perfil animada do zero. O objetivo é demonstrar domínio das três ferramentas (transition, @keyframes e transforms) e das boas práticas de performance e acessibilidade.

  • ✓ Avatar com ring animado: borda que pulsa infinitamente usando @keyframes (escala + opacidade, não width)
  • ✓ Cards de skill com hover: ao passar o mouse, o card eleva com transform: translateY() + sombra via transition — sem tocar em margin/top
  • ✓ Animação de entrada: cada card aparece com fade-up usando animation-delay escalonado (0ms, 100ms, 200ms...) para efeito cascata
  • ✓ Botão com pseudo-elemento: efeito de "luz que passa" usando ::before + overflow: hidden + transform: translateX
  • ✓ Tipografia fluida: pelo menos o título usa clamp() — sem media query para o tamanho de fonte
  • ✓ prefers-reduced-motion: todas as animações são desabilitadas/simplificadas quando a preferência do sistema for "reduce"
💬 Dica: Comece sem nenhuma animação — HTML + CSS estático primeiro. Depois adicione as transitions, depois os @keyframes. Por fim teste no DevTools: aba Animations mostra todas as animações rodando em tempo real, e você pode simular prefers-reduced-motion em Rendering → Emulate CSS media feature.
