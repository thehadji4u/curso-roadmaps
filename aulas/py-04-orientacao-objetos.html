Por que isso importa?

Você já viu OOP antes? Provavelmente sim — toda vez que chamou lista.append(), str.upper() ou "hello".split() você estava usando objetos. A diferença agora é que vamos criar nossos próprios tipos com comportamento customizado.

Sem OOP, um sistema de e-commerce com produtos, pedidos, clientes e pagamentos rapidamente se torna um emaranhado de funções e dicionários soltos. Com OOP, cada entidade ganha sua própria representação com dados e comportamentos coesos: um Produto sabe calcular seu desconto, um Pedido sabe calcular o total, um Cliente sabe validar seu CPF. O código se torna legível, testável e extensível.

💼 Por que um engenheiro sênior domina OOP profundamente
Sêniors usam OOP com intenção arquitetural: escolhem entre herança e composição (preferindo composição), usam @property para encapsular lógica de acesso, implementam dunder methods para tornar objetos nativos ao Python (comparáveis, iteráveis, representáveis), e usam dataclasses para evitar boilerplate. Conhecem o custo de herança profunda e sabem quando uma simples função é melhor do que uma classe inteira.

Antes do código: uma analogia

🌍 Mundo Real → Tecnologia
Pense em uma fábrica de carros. A planta/blueprint do carro — os desenhos técnicos — descreve o que todo carro tem: motor, 4 rodas, tanque de combustível, e o que todo carro pode fazer: acelerar, frear, ligar. Essa planta não é um carro — é a descrição abstrata de um.

Quando a fábrica usa essa planta para montar um carro específico — com cor vermelha, motor 2.0, placa ABC-1234 — ela criou um objeto. Você pode fabricar milhares de carros com a mesma planta. Cada carro tem seus próprios dados (cor, quilometragem), mas compartilha o mesmo comportamento (acelerar funciona igual em todos).

Um carro esportivo herda tudo do carro comum, mas adiciona turbo e suspensão baixa. Ele é-um carro, mas com capacidades extras. Isso é herança.
📐 Planta/Blueprint do carro → class Carro — a definição do tipo 🚗 Carro vermelho placa ABC-1234 → meu_carro = Carro(...) — instância/objeto 📊 Cor, km, modelo → self.cor, self.km — atributos 🏎️ Carro esportivo é um carro → class Esportivo(Carro) — herança

Classe, Objeto e o __init__

Uma classe é um template que define atributos (dados) e métodos (comportamentos). Um objeto é uma instância concreta dessa classe criada em memória. O método __init__ é o construtor — executado automaticamente ao criar um objeto para inicializar seus atributos.

// conta_bancaria.py — classe completa com atributos e métodos
class ContaBancaria:
    # Atributo de CLASSE — compartilhado por TODAS as instâncias
    banco = "Banco Python S.A."
    total_contas = 0

    def __init__(self, titular: str, saldo_inicial: float = 0.0):
        # Atributos de INSTÂNCIA — únicos por objeto
        self.titular = titular
        self.saldo = saldo_inicial
        self._historico = []  # _ indica "privado por convenção"

        ContaBancaria.total_contas += 1  # atualiza atributo de classe
        self._historico.append(f"Conta criada com R${saldo_inicial:.2f}")

    def depositar(self, valor: float) -> None:
        if valor <= 0:
            raise ValueError("Valor de depósito deve ser positivo")
        self.saldo += valor
        self._historico.append(f"+ R${valor:.2f} → saldo: R${self.saldo:.2f}")

    def sacar(self, valor: float) -> float:
        if valor > self.saldo:
            raise ValueError(f"Saldo insuficiente. Disponível: R${self.saldo:.2f}")
        self.saldo -= valor
        self._historico.append(f"- R${valor:.2f} → saldo: R${self.saldo:.2f}")
        return valor

    def extrato(self) -> None:
        print(f"\n=== Extrato — {self.titular} ===")
        for linha in self._historico:
            print(f"  {linha}")
        print(f"  SALDO ATUAL: R${self.saldo:.2f}\n")

    def __repr__(self) -> str:
        return f"ContaBancaria(titular='{self.titular}', saldo={self.saldo:.2f})"

    def __str__(self) -> str:
        return f"Conta de {self.titular} — R${self.saldo:.2f}"


# Criando instâncias (objetos)
conta_alice = ContaBancaria("Alice", 1000.0)
conta_bob   = ContaBancaria("Bob")          # saldo_inicial padrão = 0.0

conta_alice.depositar(500)
conta_alice.sacar(200)
conta_alice.extrato()

print(repr(conta_alice))                    # → ContaBancaria(titular='Alice', saldo=1300.00)
print(str(conta_alice))                     # → Conta de Alice — R$1300.00
print(ContaBancaria.total_contas)           # → 2  (atributo de classe)
print(conta_alice.banco)                    # → Banco Python S.A.
🔥 Pro tip — self é uma convenção, não uma palavra reservada
self é simplesmente o primeiro parâmetro de qualquer método de instância — poderia se chamar this ou qualquer coisa, mas nunca mude isso. O Python passa a própria instância automaticamente como primeiro argumento quando você chama objeto.metodo().

Os 4 Pilares da OOP

📦

Encapsulamento

Ocultar detalhes internos. Atributos _privados por convenção, __mangling por proteção. Acesso controlado via @property.

🧬

Herança

Uma classe filha herda atributos e métodos da classe pai. Evita duplicação. Use super() para acessar o pai.

🎭

Polimorfismo

Mesmo método, comportamentos diferentes. Um Animal.falar() retorna coisas diferentes para Cachorro e Gato.

🔷

Abstração

Esconder complexidade. Você usa lista.sort() sem saber que internamente é um Timsort. Classes abstratas (ABC) formalizam contratos.

Encapsulamento e @property

Python não tem modificadores de acesso como private do Java. A convenção é: um underscore _atributo significa "interno, cuidado" e dois underscores __atributo ativam name mangling (renomeação automática para evitar colisões em herança). O decorador @property permite expor atributos com lógica de validação sem mudar a interface pública.

// property — getter, setter e deleter
class Temperatura:
    def __init__(self, celsius: float):
        self._celsius = celsius  # armazena internamente em Celsius

    @property
    def celsius(self) -> float:
        """Getter — acesso como atributo: t.celsius"""
        return self._celsius

    @celsius.setter
    def celsius(self, valor: float) -> None:
        """Setter — valida antes de atribuir: t.celsius = 25"""
        if valor < -273.15:
            raise ValueError("Temperatura abaixo do zero absoluto!")
        self._celsius = valor

    @property
    def fahrenheit(self) -> float:
        """Propriedade calculada — não armazena, calcula sempre"""
        return self._celsius * 9/5 + 32

    @property
    def kelvin(self) -> float:
        return self._celsius + 273.15


t = Temperatura(100)
print(t.celsius)      # → 100  (parece atributo, mas é método)
print(t.fahrenheit)   # → 212.0
print(t.kelvin)       # → 373.15

t.celsius = 25        # chama o setter
t.celsius = -300      # → ValueError: Temperatura abaixo do zero absoluto!

// herança.py — Animal, Cachorro, Gato com super()
class Animal:
    def __init__(self, nome: str, idade: int):
        self.nome = nome
        self.idade = idade

    def falar(self) -> str:
        return f"{self.nome} faz algum som..."

    def __str__(self) -> str:
        return f"{type(self).__name__}(nome='{self.nome}', idade={self.idade})"


class Cachorro(Animal):
    def __init__(self, nome: str, idade: int, raca: str):
        super().__init__(nome, idade)   # delega ao pai — nunca pule isso!
        self.raca = raca

    def falar(self) -> str:            # OVERRIDE — substitui o método do pai
        return f"{self.nome} diz: Au au!"

    def buscar(self, objeto: str) -> str:
        return f"{self.nome} buscou o {objeto}! 🐾"


class Gato(Animal):
    def __init__(self, nome: str, idade: int, indoor: bool = True):
        super().__init__(nome, idade)
        self.indoor = indoor

    def falar(self) -> str:
        return f"{self.nome} diz: Miau~"

    def ronronar(self) -> str:
        return f"{self.nome}: purrrrr..."


# Polimorfismo em ação
animais = [Cachorro("Rex", 3, "Labrador"), Gato("Luna", 2), Cachorro("Bolt", 5, "Husky")]

for animal in animais:
    print(animal.falar())   # cada um responde diferente ao mesmo método!

# → Rex diz: Au au!
# → Luna diz: Miau~
# → Bolt diz: Au au!

print(isinstance(animais[0], Cachorro))  # → True
print(isinstance(animais[0], Animal))    # → True (herança!)
print(type(animais[0]) is Animal)        # → False (é Cachorro, não Animal puro)

Classes Abstratas (ABC)

Uma classe abstrata define um contrato: "qualquer subclasse minha DEVE implementar esses métodos". É útil quando você quer garantir uma interface comum sem fornecer implementação. Usar abc.ABC transforma a classe em abstrata; @abstractmethod marca métodos obrigatórios.

// abstract.py — contrato com ABC
from abc import ABC, abstractmethod

class Forma(ABC):
    @abstractmethod
    def area(self) -> float:
        """Toda forma DEVE implementar area()"""
        ...

    @abstractmethod
    def perimetro(self) -> float:
        """Toda forma DEVE implementar perimetro()"""
        ...

    def descricao(self) -> str:
        # Método concreto — pode ser herdado sem override
        return f"{type(self).__name__}: área={self.area():.2f}, perímetro={self.perimetro():.2f}"


class Retangulo(Forma):
    def __init__(self, largura: float, altura: float):
        self.largura = largura
        self.altura = altura

    def area(self) -> float:
        return self.largura * self.altura

    def perimetro(self) -> float:
        return 2 * (self.largura + self.altura)


class Circulo(Forma):
    import math
    def __init__(self, raio: float):
        self.raio = raio

    def area(self) -> float:
        import math
        return math.pi * self.raio ** 2

    def perimetro(self) -> float:
        import math
        return 2 * math.pi * self.raio


formas = [Retangulo(4, 6), Circulo(5)]
for f in formas:
    print(f.descricao())

# Forma()  → TypeError: não pode instanciar classe abstrata!

Dunder Methods — Fazendo Objetos "Falarem" Python

Dunder (double underscore) methods são os "ganchos" que o Python usa para operações especiais. Implementá-los faz seus objetos se comportar como tipos nativos: comparáveis com ==, somáveis com +, iteráveis com for, e imprimíveis.

Dunder Quando é chamado Exemplo de uso
__init__ Ao criar objeto: Obj() Construtor
__repr__ repr(obj), no REPL Representação técnica para dev
__str__ str(obj), print(obj) Representação amigável
__len__ len(obj) Tamanho do objeto
__eq__ obj1 == obj2 Comparação de igualdade
__lt__ obj1 < obj2 Comparação menor que
__add__ obj1 + obj2 Adição / concatenação
__contains__ x in obj Teste de pertencimento
__iter__ for x in obj Tornar iterável
__getitem__ obj[chave] Acesso por índice/chave
__enter__ / __exit__ with obj as x Context manager
// vetor2d.py — dunder methods em ação
import math

class Vetor2D:
    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y

    def __repr__(self) -> str:
        return f"Vetor2D({self.x}, {self.y})"

    def __str__(self) -> str:
        return f"({self.x}, {self.y})"

    def __add__(self, outro: 'Vetor2D') -> 'Vetor2D':
        return Vetor2D(self.x + outro.x, self.y + outro.y)

    def __sub__(self, outro: 'Vetor2D') -> 'Vetor2D':
        return Vetor2D(self.x - outro.x, self.y - outro.y)

    def __mul__(self, escalar: float) -> 'Vetor2D':
        return Vetor2D(self.x * escalar, self.y * escalar)

    def __rmul__(self, escalar: float) -> 'Vetor2D':
        return self.__mul__(escalar)   # 3 * vetor = vetor * 3

    def __eq__(self, outro: object) -> bool:
        if not isinstance(outro, Vetor2D):
            return NotImplemented
        return self.x == outro.x and self.y == outro.y

    def __abs__(self) -> float:
        """abs(vetor) retorna magnitude"""
        return math.sqrt(self.x**2 + self.y**2)

    def __bool__(self) -> bool:
        """Falso apenas se vetor nulo"""
        return self.x != 0 or self.y != 0


v1 = Vetor2D(3, 4)
v2 = Vetor2D(1, 2)

print(v1 + v2)      # → (4, 6)
print(v1 - v2)      # → (2, 2)
print(v1 * 2)       # → (6, 8)
print(3 * v1)       # → (9, 12)  — via __rmul__
print(abs(v1))      # → 5.0      — via __abs__  (3-4-5 triângulo!)
print(v1 == v2)     # → False

@classmethod, @staticmethod e Fábricas

Além dos métodos de instância (que recebem self), existem dois outros tipos: @classmethod recebe a classe (cls) como primeiro argumento e é muito usado para criar construtores alternativos (factory methods). @staticmethod não recebe nem self nem cls — é essencialmente uma função normal que mora dentro da classe por questão de organização.

// data.py — @classmethod como factory e @staticmethod utilitário
class Data:
    def __init__(self, dia: int, mes: int, ano: int):
        self.dia = dia
        self.mes = mes
        self.ano = ano

    # ── @classmethod — factory method (construtor alternativo) ──
    @classmethod
    def de_string(cls, texto: str) -> 'Data':
        """Cria Data a partir de '25/12/2024'"""
        dia, mes, ano = map(int, texto.split('/'))
        return cls(dia, mes, ano)   # cls é a própria classe — funciona em herança!

    @classmethod
    def hoje(cls) -> 'Data':
        """Cria Data com a data atual"""
        from datetime import date
        d = date.today()
        return cls(d.day, d.month, d.year)

    # ── @staticmethod — utilitário sem acesso a self ou cls ──
    @staticmethod
    def e_bissexto(ano: int) -> bool:
        """Verifica se um ano é bissexto — não precisa de instância"""
        return (ano % 4 == 0 and ano % 100 != 0) or (ano % 400 == 0)

    def __repr__(self) -> str:
        return f"Data({self.dia:02d}/{self.mes:02d}/{self.ano})"

    def __str__(self) -> str:
        meses = ["Jan","Fev","Mar","Abr","Mai","Jun",
                 "Jul","Ago","Set","Out","Nov","Dez"]
        return f"{self.dia} de {meses[self.mes-1]} de {self.ano}"


# Usando os factory methods
natal  = Data.de_string("25/12/2024")
hoje   = Data.hoje()

print(natal)                        # → 25 de Dez de 2024
print(Data.e_bissexto(2024))        # → True  (sem criar instância!)
print(Data.e_bissexto(2023))        # → False
💡 Quando usar cada tipo de método
Use método de instância quando precisa acessar/modificar atributos do objeto (self). Use @classmethod quando quer criar construtores alternativos ou métodos que precisam da classe em si (para herança funcionar corretamente). Use @staticmethod quando a função pertence conceitualmente à classe mas não precisa de nenhum dado dela — é essencialmente uma função utilitária com namespace.

dataclasses — Python Moderno sem Boilerplate

O módulo dataclasses (Python 3.7+) elimina o código repetitivo de classes que são essencialmente containers de dados. Com um único decorador @dataclass, você ganha __init__, __repr__ e __eq__ gerados automaticamente.

❌ Classe tradicional (verbosa)
class Produto:
    def __init__(self, nome, preco, estoque):
        self.nome = nome
        self.preco = preco
        self.estoque = estoque

    def __repr__(self):
        return (f"Produto(nome={self.nome!r}, "
                f"preco={self.preco!r}, "
                f"estoque={self.estoque!r})")

    def __eq__(self, outro):
        if not isinstance(outro, Produto):
            return NotImplemented
        return (self.nome == outro.nome and
                self.preco == outro.preco)
✅ Com @dataclass (conciso)
from dataclasses import dataclass, field

@dataclass
class Produto:
    nome: str
    preco: float
    estoque: int = 0
    tags: list = field(default_factory=list)

# __init__, __repr__, __eq__
# gerados automaticamente!

# Para imutabilidade:
@dataclass(frozen=True)
class Ponto:
    x: float
    y: float
// dataclass_avancado.py — order, frozen, post_init
from dataclasses import dataclass, field
from typing import ClassVar

@dataclass(order=True)   # gera __lt__, __le__, __gt__, __ge__ automaticamente
class Funcionario:
    # ClassVar — atributo de classe, não de instância
    empresa: ClassVar[str] = "Acme Corp"

    # sort_field define qual atributo usar para ordenação
    sort_index: float = field(init=False, repr=False)

    nome: str
    cargo: str
    salario: float

    def __post_init__(self):
        """Executado após __init__ — validações e campos derivados"""
        if self.salario < 0:
            raise ValueError("Salário não pode ser negativo")
        self.sort_index = self.salario   # ordena por salário


equipe = [
    Funcionario("Alice",   "Engenheira Sênior", 12000),
    Funcionario("Bob",     "Designer",           8500),
    Funcionario("Charlie", "Engenheiro Pleno",   9500),
]

equipe_ordenada = sorted(equipe)  # usa __lt__ gerado por order=True
for f in equipe_ordenada:
    print(f"{f.nome}: R${f.salario:,.2f}")

# → Bob: R$8.500,00
# → Charlie: R$9.500,00
# → Alice: R$12.000,00

// biblioteca.py — sistema completo com herança e dunder methods
from dataclasses import dataclass, field
from datetime import date, timedelta
from abc import ABC, abstractmethod


@dataclass
class Livro:
    titulo: str
    autor: str
    isbn: str
    paginas: int
    disponivel: bool = True

    def __str__(self) -> str:
        status = "✓ disponível" if self.disponivel else "✗ emprestado"
        return f"'{self.titulo}' — {self.autor} ({status})"


class Usuario(ABC):
    """Classe base abstrata para todos os tipos de usuário"""
    def __init__(self, nome: str, email: str):
        self.nome = nome
        self.email = email
        self._emprestimos: list[Emprestimo] = []

    @property
    def emprestimos_ativos(self) -> list:
        return [e for e in self._emprestimos if not e.devolvido]

    @abstractmethod
    def limite_emprestimos(self) -> int:
        """Cada tipo de usuário tem seu próprio limite"""
        ...

    def pode_emprestar(self) -> bool:
        return len(self.emprestimos_ativos) < self.limite_emprestimos()

    def __repr__(self) -> str:
        return f"{type(self).__name__}(nome='{self.nome}')"


class UsuarioComum(Usuario):
    def limite_emprestimos(self) -> int:
        return 3


class UsuarioVIP(Usuario):
    def __init__(self, nome: str, email: str, premium_ate: date):
        super().__init__(nome, email)
        self.premium_ate = premium_ate

    def limite_emprestimos(self) -> int:
        if date.today() <= self.premium_ate:
            return 10
        return 3   # expirou — volta ao limite comum


@dataclass
class Emprestimo:
    livro: Livro
    usuario: Usuario
    data_emprestimo: date = field(default_factory=date.today)
    prazo_dias: int = 14
    devolvido: bool = False

    @property
    def data_devolucao(self) -> date:
        return self.data_emprestimo + timedelta(days=self.prazo_dias)

    @property
    def em_atraso(self) -> bool:
        return not self.devolvido and date.today() > self.data_devolucao

    def devolver(self) -> None:
        self.devolvido = True
        self.livro.disponivel = True
        print(f"  ✓ '{self.livro.titulo}' devolvido por {self.usuario.nome}")


class Biblioteca:
    def __init__(self, nome: str):
        self.nome = nome
        self._acervo: list[Livro] = []
        self._emprestimos: list[Emprestimo] = []

    def adicionar_livro(self, livro: Livro) -> None:
        self._acervo.append(livro)

    def emprestar(self, isbn: str, usuario: Usuario) -> Emprestimo | None:
        if not usuario.pode_emprestar():
            print(f"  ✗ {usuario.nome} atingiu o limite de empréstimos")
            return None

        livro = next((l for l in self._acervo if l.isbn == isbn and l.disponivel), None)
        if not livro:
            print(f"  ✗ Livro ISBN {isbn} não disponível")
            return None

        emprestimo = Emprestimo(livro, usuario)
        livro.disponivel = False
        usuario._emprestimos.append(emprestimo)
        self._emprestimos.append(emprestimo)
        print(f"  ✓ '{livro.titulo}' emprestado para {usuario.nome} até {emprestimo.data_devolucao}")
        return emprestimo

    def listar_disponiveis(self) -> None:
        print(f"\n=== Acervo disponível — {self.nome} ===")
        disponiveis = [l for l in self._acervo if l.disponivel]
        for livro in disponiveis:
            print(f"  {livro}")

    def __len__(self) -> int:
        return len(self._acervo)

    def __contains__(self, isbn: str) -> bool:
        return any(l.isbn == isbn for l in self._acervo)

    def __repr__(self) -> str:
        return f"Biblioteca(nome='{self.nome}', acervo={len(self)} livros)"

if __name__ == "__main__":
    bib = Biblioteca("Biblioteca Municipal")

Herança vs. Composição vs. Duck Typing

Abordagem Quando usar Vantagens Cuidados
Herança Relação "é-um" clara e estável. Cachorro É UM Animal. Reutilização direta Polimorfismo Hierarquias profundas criam acoplamento forte. Máximo 2-3 níveis.
Composição Relação "tem-um". Pedido TEM itens. Prefira sempre que possível. Flexível Testável Mais código inicial, mas mais fácil de mudar.
Duck Typing Qualquer objeto com o método certo serve — sem herança necessária. Desacoplado Pythônico Menos explícito; use type hints para documentar a interface esperada.
ABC / Protocol Quando precisa garantir interface explícita em sistemas maiores. Contrato formal Overhead. Use typing.Protocol para duck typing estrutural moderno.
🔥 Regra de ouro: Composição sobre Herança
A frase clássica do GoF (Gang of Four) se prova no dia a dia: prefira compor objetos a criar hierarquias profundas. Um Funcionario que tem um Cargo e um Departamento é mais flexível do que uma hierarquia Funcionario → FuncionarioSenior → GerenteDepto → Diretor.
// pitfalls

⚠️ Armadilhas Comuns

🪤 Atributo mutável como default na definição da classe

Este é o bug mais clássico de OOP em Python. Se você definir um atributo mutável (lista, dict) no corpo da classe, ele é compartilhado entre todas as instâncias.

# ❌ ERRADO — _itens é compartilhado entre TODOS os objetos!
class Carrinho:
    _itens = []   # atributo de CLASSE, não de instância!

    def adicionar(self, item):
        self._itens.append(item)

c1 = Carrinho()
c2 = Carrinho()
c1.adicionar("arroz")
print(c2._itens)   # → ['arroz']  HORROR!

# ✅ CORRETO — cada instância tem sua própria lista
class Carrinho:
    def __init__(self):
        self._itens = []   # atributo de INSTÂNCIA
🪤 Esquecer super().__init__() na herança

Se a classe filha sobrescreve __init__ sem chamar super().__init__(), os atributos do pai não são inicializados — levando a AttributeError misterioso.

# ❌ ERRADO
class Animal:
    def __init__(self, nome):
        self.nome = nome

class Cachorro(Animal):
    def __init__(self, nome, raca):
        self.raca = raca   # self.nome nunca foi inicializado!

# ✅ CORRETO
class Cachorro(Animal):
    def __init__(self, nome, raca):
        super().__init__(nome)   # inicializa self.nome
        self.raca = raca
🪤 Classe desnecessária — nem tudo precisa ser OOP

Python não é Java. Se você tem uma função utilitária isolada, uma simples função é melhor que uma classe com um único método estático. Crie classes quando há estado que precisa ser mantido e comportamentos que operam sobre esse estado. 

# ❌ DESNECESSÁRIO — classe como namespace glorificado
class MathUtils:
    @staticmethod
    def somar(a, b):
        return a + b

# ✅ SIMPLES e correto — apenas uma função
def somar(a, b):
    return a + b

Glossário

Classe
Template/blueprint que define atributos (dados) e métodos (comportamentos) de um tipo. É como o molde; os objetos são as peças fundidas com esse molde.
Instância / Objeto
Materialização concreta de uma classe em memória. meu_cachorro = Cachorro("Rex") cria uma instância. Cada instância tem seus próprios atributos.
self
Referência à instância atual dentro de um método. Passado automaticamente pelo Python; permite que métodos acessem e modifiquem os atributos do próprio objeto.
Encapsulamento
Princípio de ocultar detalhes internos e expor apenas o necessário. Em Python, convencionado com _ (interno) e __ (name mangling). Implementado com @property.
Herança
Mecanismo pelo qual uma classe filha herda atributos e métodos da classe pai. Permite reutilização e especialização. Use super() para acessar o pai.
Polimorfismo
Capacidade de objetos de diferentes classes responderem ao mesmo método de formas diferentes. Em Python, implementado via override de métodos e duck typing.
Dunder Methods (Magic Methods)
Métodos com duplo underscore (__init__, __str__, __add__). São os "ganchos" que o Python usa para operações especiais como comparação, adição, iteração e impressão.
@property
Decorador que transforma um método em um atributo acessível sem parênteses. Permite adicionar lógica de validação ao getter/setter sem mudar a interface pública.
@classmethod
Decorador para métodos que recebem a classe (cls) como primeiro argumento. Usado para factory methods (construtores alternativos) e operações que envolvem a classe, não instâncias.
Composição
Padrão onde um objeto contém outros objetos como atributos, em vez de herdar deles. "Tem-um" em vez de "é-um". Preferida sobre herança por ser mais flexível e desacoplada.
Duck Typing
"Se anda como um pato e grasna como um pato, é um pato." Em Python, o tipo de um objeto é determinado por seus métodos e atributos, não por sua classe. Qualquer objeto com o método esperado pode ser usado.
ABC (Abstract Base Class)
Classe abstrata que define um contrato. Não pode ser instanciada diretamente. Subclasses são obrigadas a implementar os métodos marcados com @abstractmethod.
@dataclass
Decorador do módulo dataclasses que gera automaticamente __init__, __repr__ e __eq__ a partir de anotações de tipo. Reduz boilerplate em classes de dados.
