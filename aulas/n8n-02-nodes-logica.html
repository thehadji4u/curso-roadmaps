n8n — Nodes Essenciais & Lógica

⏱ ~50 min de leitura 🎯 Intermediário 🔧 Hands-on incluído
HTTP Request Code Node IF / Switch Loop Merge & Set Error Trigger expressões n8n $json $items lógica condicional
// MOTIVAÇÃO

Por que isso importa?

Saber criar um workflow no n8n é o primeiro passo — mas para resolver problemas reais de automação você precisa dominar os nodes que fazem a lógica acontecer. Sem eles, seus workflows são apenas pipelines lineares sem inteligência.

Na prática, toda automação útil precisa de pelo menos um desses elementos: buscar dados de uma API externa (HTTP Request), transformar ou calcular valores (Code Node), tomar decisões (IF / Switch), iterar sobre listas (Loop), combinar dados de origens diferentes (Merge) e capturar falhas graciosamente (Error Trigger). Esta aula cobre todos eles.

💼 Por que um engenheiro sênior domina esses nodes
Engenheiros sêniors não usam o n8n apenas para "ligar A ao B". Eles constroem workflows que tratam erros, lidam com paginação de APIs, bifurcam fluxos com base em dados e processam listas arbitrariamente grandes sem travar. Esses padrões se repetem em 90% das automações de produção — dominar os nodes desta aula é dominar o núcleo da ferramenta.
// ANALOGIA

Antes do código: uma analogia

🏭 Mundo Real → Tecnologia
Imagine uma linha de produção de fábrica inteligente. Cada estação de trabalho faz uma coisa específica: uma estação busca matéria-prima do fornecedor (HTTP Request), outra realiza transformações manuais que a máquina padrão não faz (Code Node), outra inspeciona o produto e decide para qual linha enviar (IF / Switch), outra processa cada unidade de um lote individualmente (Loop), outra consolida peças de duas linhas paralelas numa só (Merge), e há uma equipe de resposta a emergências acionada quando qualquer máquina falha (Error Trigger).

No n8n, os dados são as peças que fluem entre as estações. Cada node é uma estação que recebe, processa e passa adiante.
🚚 Buscar matéria-prima do fornecedor → HTTP Request Node 👷 Operação manual especializada → Code Node (JavaScript) 🔍 Inspetor de qualidade com desvio de linha → IF / Switch Node 🔄 Processar cada peça do lote → Loop Over Items 🔀 Consolidar duas linhas em uma → Merge Node 🚨 Equipe de emergência em standby → Error Trigger Node
// FUNDAMENTO

Expressões n8n: a cola de tudo

Antes de ver cada node, precisamos entender expressões. Em n8n, expressões são pedaços de JavaScript envolvidos em {{ }} que você escreve diretamente nos campos dos nodes. Elas permitem acessar dados dos nodes anteriores dinamicamente.

{{ $json.nome }} Acessa o campo "nome" do item atual (JSON do node anterior)
{{ $json.preco * 1.1 }} Calcula 10% a mais no preço — expressões suportam operações JS
{{ $items("HTTP Request")[0].json.id }} Acessa o primeiro item da saída de um node específico pelo nome
{{ $now.toISO() }} Data/hora atual em formato ISO — útil em headers e logs
{{ $env.MINHA_API_KEY }} Acessa variável de ambiente definida nas Settings do n8n
{{ $node["Set"].json.resultado }} Referencia um node pelo nome — mais robusto que $items()
🔥 Pro tip: use o Expression Editor
Em qualquer campo do n8n, clique no ícone de engrenagem ou no botão = para abrir o Expression Editor. Ele tem autocomplete, mostra o valor atual e destaca erros em tempo real. Indispensável ao trabalhar com dados aninhados.

HTTP Request — buscar qualquer API

O node mais usado no n8n. Ele faz requisições HTTP para qualquer API REST: GET, POST, PUT, PATCH, DELETE. Com ele, você integra praticamente qualquer serviço que não tem node nativo no n8n.

Paginação automática — padrão essencial

A maioria das APIs retorna dados paginados. O n8n tem suporte nativo a paginação em Options → Pagination. Configure o campo de "next page" e o n8n itera automaticamente até acabar os resultados.

Exemplo — Buscar lista paginada de usuários
// Configuração do node HTTP Request
{
  "method": "GET",
  "url": "https://api.exemplo.com/users",
  "queryParameters": {
    "page": "={{ $pageCount + 1 }}",
    "limit": "50"
  },
  "options": {
    "pagination": {
      "paginationMode": "responseContainsNextURL",
      "nextURL": "={{ $response.body.next }}"
    }
  }
}
⚠️ Pitfall: timeout em APIs lentas
Por padrão o HTTP Request tem timeout de 300 segundos. Para APIs lentas ou downloads grandes, vá em Options → Timeout e ajuste. Em produção, prefira processar em lotes menores a fazer uma chamada gigante que pode falhar no meio.
// NODE 2

Code Node — JavaScript direto no workflow

O Code Node é a "válvula de escape" quando nenhum node nativo resolve seu problema. Você escreve JavaScript puro (ou TypeScript) e tem acesso completo aos dados de entrada. É o equivalente a um bloco .map(), .filter() ou qualquer lógica customizada.

Dois modos de operação

Run Once for All Items

Processa tudo de uma vez

Recebe $input.all() — array com todos os items. Ideal para agregações, ordenações e filtros complexos. Retorna um array de objetos { json: {...} }.

Run Once for Each Item

Processa item a item

Recebe $input.item — um item por vez. Mais simples para transformações individuais. O n8n chama a função para cada item e coleta os resultados.

Code Node — Run Once for All Items (agrupamento)
// Agrupa pedidos por status
const items = $input.all();

const grouped = {};
for (const item of items) {
  const status = item.json.status;
  if (!grouped[status]) grouped[status] = [];
  grouped[status].push(item.json);
}

// Retorna um item por grupo
return Object.entries(grouped).map(([status, pedidos]) => ({
  json: {
    status,
    total: pedidos.length,
    pedidos
  }
}));
Code Node — Run Once for Each Item (transformação)
// Normaliza nome completo e calcula desconto
const { nome, sobrenome, preco, vip } = $input.item.json;

return {
  json: {
    nomeCompleto: `${nome} ${sobrenome}`.trim(),
    precoFinal: vip ? preco * 0.8 : preco,       // 20% desconto para VIPs
    desconto: vip ? "20%" : "0%",
    processadoEm: new Date().toISOString()
  }
};
⚠️ Sem acesso a módulos externos por padrão
O Code Node roda em sandbox. Você não pode require('axios') ou importar npm packages no cloud n8n. Na versão self-hosted, é possível habilitar módulos via variável de ambiente NODE_FUNCTION_ALLOW_BUILTIN e NODE_FUNCTION_ALLOW_EXTERNAL. Para chamadas HTTP dentro do code node, use o node HTTP Request separado.
// NODE 3

IF & Switch — bifurcação de fluxo

Lógica condicional é a alma de qualquer automação inteligente. No n8n, temos dois nodes para isso:

IF Node

Condição binária (true / false)

Divide o fluxo em dois caminhos. Saída true para itens que satisfazem a condição, false para os demais. Use para triagem simples.

Switch Node

Múltiplos caminhos

Divide em N saídas com base em regras. Cada saída tem sua condição. Se nenhuma bate, pode ter uma saída "default". Use para roteamento por tipo/status/região.

// NODE 4

Loop Over Items — iteração controlada

Por padrão, o n8n já processa todos os items em paralelo entre nodes. Mas às vezes você precisa de controle explícito: processar item por item com operações dependentes de estado, ou limitar a taxa de chamadas a uma API que tem rate limit.

💡 Quando usar Loop Over Items vs processamento nativo
Não use Loop se o que você quer é apenas que cada item passe pelo mesmo node — isso já acontece automaticamente. Use Loop quando precisar de: (1) sub-workflows por item, (2) acumulação de resultados com contexto compartilhado, (3) rate limiting explícito com Wait node no meio.

  }

⚠️ Pitfall: loop infinito por conexão errada
O "done" output do Loop deve conectar a algo fora do ciclo. Se você acidentalmente reconectar o "done" de volta ao Loop ou a um node que o alimenta, o workflow fica em loop infinito e consome recursos até ser cancelado manualmente.

Merge & Set — combinação e normalização

Set Node — definir, transformar e filtrar campos

O Set Node é o canivete suíço de transformação de dados. Use-o para: renomear campos, adicionar campos calculados, remover campos sensíveis antes de enviar a uma API, ou formatar datas e strings.

Set Node em modo "Keep Only Set" — projeção de campos
// Modo: "Keep Only Set" — remove todos os campos não listados
// Útil para limpar dados antes de enviar para uma API externa

Campos a definir:
  id          → {{ $json.usuario_id }}
  nome        → {{ $json.nome_completo.split(' ')[0] }}  // só o primeiro nome
  email       → {{ $json.email.toLowerCase() }}
  timestamp   → {{ $now.toISO() }}
  fonte       → "crm_interno"   // valor fixo

Merge Node — juntar dados de fluxos paralelos

O Merge Node combina dados de dois ramos diferentes do workflow. É essencial quando você faz chamadas paralelas (ex: buscar dados de dois endpoints diferentes) e precisa consolidar os resultados.

ModoO que fazCaso de uso
Append Concatena os items dos dois inputs em sequência Unir duas listas de usuários em uma só
Merge By Index Combina item 0 do input1 com item 0 do input2, e assim por diante Juntar dados quando a ordem é garantida
Merge By Key Faz um JOIN por campo em comum (ex: id) Enriquecer usuários com dados de perfil por ID
Keep Key Matches Retorna apenas items que têm match nos dois inputs Interseção — equivalente a INNER JOIN
Remove Key Matches Retorna items do input1 que NÃO têm match no input2 Encontrar registros que existem em A mas não em B

Error Trigger — capturar falhas graciosamente

Todo workflow de produção precisa de um plano para quando as coisas dão errado. O Error Trigger é um node especial que não se conecta a outros nodes no fluxo normal — ele é acionado automaticamente quando qualquer execução de outro workflow falha.

Como funciona

1
Criar um workflow de tratamento de erros
Crie um novo workflow separado. Adicione o node Error Trigger como primeiro node. Este workflow receberá dados sobre qualquer erro.
2
Configurar nos workflows de produção
Em cada workflow que deve ser monitorado, vá em Settings → Error Workflow e selecione o workflow de tratamento criado no passo 1.
3
Usar os dados do erro no workflow de tratamento
O Error Trigger passa para o próximo node o objeto $json com detalhes: nome do workflow, node que falhou, mensagem de erro, timestamp e execution ID.
Dados disponíveis no Error Trigger ($json)
{
  "execution": {
    "id": "abc123",
    "url": "https://n8n.meusite.com/execution/abc123",
    "retryOf": null,
    "error": {
      "message": "Request failed with status code 500",
      "stack": "Error: Request failed..."
    },
    "lastNodeExecuted": "HTTP Request"
  },
  "workflow": {
    "id": "15",
    "name": "Sincronizar Pedidos"
  }
}
Workflow de erro — notificar no Slack com detalhes
// No Code Node do workflow de erro, formata a mensagem:
const { execution, workflow } = $input.item.json;

return {
  json: {
    mensagem: [
      `🚨 *Workflow com Falha*`,
      `*Workflow:* ${workflow.name} (ID: ${workflow.id})`,
      `*Node:* ${execution.lastNodeExecuted}`,
      `*Erro:* ${execution.error.message}`,
      `*Execução:* <${execution.url}|Ver no n8n>`,
      `*Hora:* ${new Date().toLocaleString('pt-BR')}`
    ].join('\n')
  }
};
🔥 Pro tip: Error Trigger + retry automático
Combine Error Trigger com o node n8n (que pode acionar execuções de outros workflows) para implementar retry automático: quando um workflow falha, o Error Trigger aciona um workflow que espera 5 minutos (Wait node) e re-executa o workflow original com o mesmo payload. Limite a 3 tentativas usando um contador em variável de workflow.

Passo a Passo: Workflow completo com todos os nodes

Vamos construir um workflow que: busca usuários de uma API pública, filtra os ativos, transforma os dados, processa em loop e notifica em caso de erro.

1
Criar novo workflow no n8n
Acesse sua instância n8n → New Workflow → nomeie como "Nodes Essenciais — Prática".
2
Adicionar Manual Trigger + HTTP Request
Manual Trigger → HTTP Request com método GET e URL https://jsonplaceholder.typicode.com/users. Execute para ver os 10 usuários retornados.
Saída esperada do HTTP Request
10 items — cada item é um usuário com id, name, email, address, phone, website, company
3
Adicionar IF Node — filtrar usuários de empresa "Romaguera"
IF → Condição: {{ $json.company.name }} contains Romaguera. Conecte a saída "true" ao próximo passo. A saída "false" pode ir para um Set node que apenas registra "ignorado".
4
Adicionar Set Node — normalizar campos
Mode: "Keep Only Set". Defina campos: id, nome ({{ $json.name }}), email ({{ $json.email.toLowerCase() }}), empresa ({{ $json.company.name }}), processadoEm ({{ $now.toISO() }}).
5
Adicionar Code Node — calcular score fictício
Mode: "Run Once for Each Item". Adicione um campo score baseado no comprimento do email (apenas para praticar o Code Node).
const { id, nome, email, empresa, processadoEm } = $input.item.json;
return {
  json: {
    id, nome, email, empresa, processadoEm,
    score: email.length * 3,  // cálculo fictício para demonstração
    tier: email.length > 20 ? "gold" : "silver"
  }
};
6
Inspecionar resultado e ativar o workflow
Execute manualmente, verifique os dados no painel de saída. Quando tudo estiver correto, ative o workflow clicando no toggle "Active" no topo. Adicione um Error Trigger separado para monitorar falhas.

Erros comuns a evitar

⚠️ Usar IF em cascata em vez de Switch
Conectar a saída "false" de um IF em outro IF, e outro, e outro… torna o workflow ilegível e difícil de manter. Sempre que tiver mais de 2 caminhos, use o Switch Node.
⚠️ Ignorar erros de HTTP Request
Por padrão, se a API retorna status 4xx ou 5xx, o node falha e para o workflow. Configure Options → Continue On Fail e trate o erro com um IF Node logo depois, checando se $json.error existe. Isso dá controle granular sobre o que fazer em cada tipo de falha.
⚠️ Esquecer que Merge precisa de dados nos dois inputs
O Merge Node aguarda dados chegarem pelos dois inputs antes de executar. Se um dos ramos estiver vazio (zero items), o Merge pode travar ou produzir resultado inesperado. Use Options → Clamp ou adicione um Set Node com um item dummy no ramo que pode ficar vazio.
⚠️ Hardcodar credenciais no Code Node ou HTTP Request
Nunca coloque API keys diretamente no campo URL ou no code. Use Credentials do n8n (para HTTP Request) ou {{ $env.MINHA_KEY }} (variáveis de ambiente) para o Code Node. Credenciais no n8n são criptografadas em repouso.
// GLOSSÁRIO

Termos desta Aula

Expressão n8n
Código JavaScript envolvido em {{ }} para acessar dados dinamicamente nos campos dos nodes. Ex: {{ $json.email }}.
$json
Objeto que representa os dados JSON do item atual sendo processado pelo node. Principal forma de acessar campos do node anterior.
$input.all()
No Code Node, retorna array com todos os items de entrada. Usado no modo "Run Once for All Items".
Rate Limiting
Restrição de uma API ao número de chamadas por segundo/minuto. Contornado no n8n com Loop + Wait node para espaçar as requisições.
Error Workflow
Workflow especial configurado nas Settings de outro workflow para ser acionado automaticamente quando aquele workflow falha.
Merge By Key
Modo do Merge Node que faz JOIN entre dois inputs usando um campo em comum. Equivalente ao JOIN de banco de dados.
Continue On Fail
Opção de nodes que permite o workflow continuar mesmo quando o node falha, passando os dados de erro como item normal para o próximo node.

🏁 Checkpoint — Desafio Final

Construa um workflow completo de enriquecimento e roteamento de leads usando os nodes desta aula:

  • Use HTTP Request para buscar dados de https://jsonplaceholder.typicode.com/posts (simula leads)
  • Filtre com IF apenas os posts cujo userId seja par ({{ $json.userId % 2 === 0 }})
  • Use Switch Node para rotear por userId: IDs 2 e 4 vão para "Fila A", IDs 6, 8 e 10 vão para "Fila B"
  • Em cada fila, use Set Node para adicionar campo fila e processadoEm com timestamp atual
  • Use Merge (Append) para reunir as duas filas em um único fluxo
  • Adicione Code Node para contar quantos items foram de cada fila e retornar um sumário
  • Crie um workflow separado com Error Trigger que loga o erro no console
💬 Dica: para testar o Error Trigger, adicione temporariamente um HTTP Request com URL inválida no workflow principal e execute — ele deve acionar o workflow de erro. Lembre-se de configurar o "Error Workflow" nas Settings do workflow principal.
