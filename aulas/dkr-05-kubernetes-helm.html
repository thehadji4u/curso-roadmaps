// motivação

Por que isso importa?

Docker resolve o problema de "funciona na minha máquina". Kubernetes resolve o problema de "como faço isso funcionar para 10 milhões de usuários?" — com auto-healing, auto-scaling, zero-downtime deploys e gerenciamento declarativo de toda a infraestrutura.

Todo cloud provider oferece Kubernetes gerenciado: GKE (Google), EKS (AWS), AKS (Azure). Conhecer k8s é hoje requisito para qualquer engenheiro que trabalhe com sistemas distribuídos em produção.

🔄

Self-healing

Pod caiu? K8s reinicia automaticamente. Node com falha? Reagenda os pods em outro nó.

📈

Auto-scaling

HPA escala horizontalmente baseado em CPU/memória/métricas customizadas sob demanda real.

🚀

Zero-downtime deploys

Rolling updates substituem pods gradualmente. Rollback em 1 comando se algo der errado.

📦

Helm Charts

Empacota toda a aplicação como um chart versionado — instala PostgreSQL, Redis, NGINX com 1 comando.

API Server
Gateway único do cluster. Todo comando kubectl passa por aqui. Valida e persiste estado no etcd.
etcd
Banco de dados chave-valor distribuído. Guarda TODO o estado do cluster. Backup crítico.
Scheduler
Decide em qual node cada pod roda, com base em recursos disponíveis, affinity rules, taints/tolerations.
kubelet
Agente em cada node. Garante que os containers definidos nos pods estejam rodando e saudáveis.

Os Objetos Fundamentais do Kubernetes

Pod — menor unidade deployável

Um pod é um ou mais containers que compartilham rede e storage. Na prática, você raramente cria pods diretamente — usa Deployments que os gerenciam.

yaml — pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: minha-api
  namespace: minha-app
  labels:
    app: minha-api
spec:
  containers:
    - name: api
      image: meuregistro/minha-api:1.2.0
      ports:
        - containerPort: 8000
      resources:
        requests:          # mínimo garantido
          memory: "64Mi"
          cpu: "100m"
        limits:            # máximo permitido
          memory: "256Mi"
          cpu: "500m"
      livenessProbe:       # k8s reinicia se falhar
        httpGet:
          path: /health
          port: 8000
        initialDelaySeconds: 10
        periodSeconds: 30
      readinessProbe:      # só recebe tráfego quando pronto
        httpGet:
          path: /ready
          port: 8000
        initialDelaySeconds: 5
        periodSeconds: 10

Deployment — gerencia ReplicaSet + rolling updates

yaml — deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: minha-api
  namespace: minha-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: minha-api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1         # pods extras durante update
      maxUnavailable: 0   # zero downtime
  template:
    metadata:
      labels:
        app: minha-api
    spec:
      containers:
        - name: api
          image: meuregistro/minha-api:1.2.0
          ports:
            - containerPort: 8000
          envFrom:
            - configMapRef:
                name: minha-api-config
            - secretRef:
                name: minha-api-secrets
          resources:
            requests:
              memory: "128Mi"
              cpu: "100m"
            limits:
              memory: "256Mi"
              cpu: "500m"
bash — rolling update e rollback
# Atualizar imagem (zero downtime)
$ kubectl set image deployment/minha-api api=meuregistro/minha-api:1.3.0
$ kubectl rollout status deployment/minha-api
Waiting for deployment "minha-api" rollout to finish: 1 out of 3 new replicas...
deployment "minha-api" successfully rolled out

# Rollback imediato se algo der errado
$ kubectl rollout undo deployment/minha-api
deployment.apps/minha-api rolled back

Service — endpoint estável para os pods

yaml — service.yaml
apiVersion: v1
kind: Service
metadata:
  name: minha-api-svc
  namespace: minha-app
spec:
  selector:
    app: minha-api       # seleciona todos os pods com este label
  ports:
    - protocol: TCP
      port: 80           # porta exposta pelo service
      targetPort: 8000   # porta do container
  type: ClusterIP        # ClusterIP (interno) | NodePort | LoadBalancer

Ingress — roteamento HTTP/HTTPS externo

yaml — ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: minha-api-ingress
  namespace: minha-app
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - api.meusite.com.br
      secretName: api-tls-cert        # gerado pelo cert-manager
  rules:
    - host: api.meusite.com.br
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: minha-api-svc
                port:
                  number: 80

ConfigMap & Secret

yaml — config-secret.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: minha-api-config
  namespace: minha-app
data:
  DATABASE_HOST: "postgres-svc"
  DATABASE_PORT: "5432"
  LOG_LEVEL: "INFO"
  APP_ENV: "production"

---
apiVersion: v1
kind: Secret
metadata:
  name: minha-api-secrets
  namespace: minha-app
type: Opaque
# valores em base64: echo -n "minha_senha" | base64
data:
  DATABASE_PASSWORD: bWluaGFfc2VuaGE=
  JWT_SECRET: bWV1X2p3dF9zZWNyZXQ=
⚠️ Secrets não são criptografia
Os valores em Secrets são apenas codificados em base64 — qualquer um com acesso ao cluster pode ler. Em produção use Sealed Secrets (Bitnami), HashiCorp Vault ou External Secrets Operator para criptografia real.

Horizontal Pod Autoscaler (HPA)

yaml — hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: minha-api-hpa
  namespace: minha-app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: minha-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   # escala quando CPU > 70%
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
// package manager

Helm — O apt-get do Kubernetes

Helm empacota aplicações Kubernetes em Charts: coleções de manifestos YAML com templating, valores configuráveis e versionamento. Um helm install pode subir PostgreSQL completo com persistência, autenticação e service account em 30 segundos.

1
Instalar Helm
Script oficial ou via package manager
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
helm version
# version.BuildInfo{Version:"v3.14.0", ...}
2
Adicionar repositórios
Repositórios são índices de charts disponíveis
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo add cert-manager https://charts.jetstack.io
helm repo update
3
Instalar aplicações
PostgreSQL, Redis, NGINX Ingress em 1 comando cada
# PostgreSQL com persistência
helm install meu-postgres bitnami/postgresql \
  --namespace minha-app \
  --set auth.postgresPassword=senha_forte \
  --set primary.persistence.size=10Gi

# Redis
helm install meu-redis bitnami/redis \
  --namespace minha-app \
  --set auth.password=redis_senha

# NGINX Ingress Controller
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace
4
Gerenciar releases
Upgrade, rollback e limpeza
# Listar releases instaladas
helm list -A

# Upgrade para nova versão
helm upgrade meu-postgres bitnami/postgresql \
  --namespace minha-app \
  --reuse-values \
  --set image.tag=16.2.0

# Rollback para revisão anterior
helm rollback meu-postgres 1 -n minha-app

# Remover release
helm uninstall meu-postgres -n minha-app

Criar seu próprio Helm Chart

bash — criar chart
helm create minha-api
# Gera a estrutura:
# minha-api/
#   Chart.yaml          ← metadados (nome, versão, descrição)
#   values.yaml         ← valores padrão (override na instalação)
#   templates/
#     deployment.yaml   ← usa {{ .Values.* }} para templating
#     service.yaml
#     ingress.yaml
#     hpa.yaml
#     _helpers.tpl      ← funções auxiliares reutilizáveis
yaml — values.yaml customizado
replicaCount: 3

image:
  repository: meuregistro/minha-api
  tag: "1.2.0"
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 80
  targetPort: 8000

ingress:
  enabled: true
  host: api.meusite.com.br
  tls: true

resources:
  limits:
    cpu: 500m
    memory: 256Mi
  requests:
    cpu: 100m
    memory: 128Mi

autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 70

env:
  LOG_LEVEL: "INFO"
  APP_ENV: "production"
bash — instalar e testar o chart
# Dry-run: ver o que seria criado sem aplicar
helm install minha-api ./minha-api --dry-run --debug -n minha-app

# Instalar com values de produção
helm install minha-api ./minha-api \
  --namespace minha-app \
  -f values-production.yaml

# Verificar
helm status minha-api -n minha-app
kubectl get all -n minha-app

Complexidade de setup Baixa Alta

// armadilhas

⚠️ Armadilhas Comuns

📊 Sem resource requests/limits
Pods sem resources.requests podem consumir todo o CPU/memória do node, derrubando outros pods. Sempre defina requests e limits — é requisito para o HPA funcionar.
🏷 Tag latest em produção
image: minha-api:latest significa que nodes diferentes podem puxar versões diferentes da imagem. Use tags semânticas: minha-api:1.2.0 ou hash do commit: minha-api:sha-abc1234.
🔒 Secrets base64 sem criptografia real
Secrets do k8s são apenas codificados em base64 no etcd, não criptografados por padrão. Use Sealed Secrets ou Vault em produção.
🏥 Sem probes de saúde
Sem livenessProbe e readinessProbe, o k8s manda tráfego para pods que ainda estão inicializando ou que travaram. Configure ambas as probes sempre.
🌐 Tudo no namespace default
Misturar todas as aplicações no namespace default dificulta RBAC, network policies, resource quotas e isolamento. Crie um namespace por aplicação/ambiente.

🏁 Checkpoint — Deploy completo no Kubernetes local

Use minikube ou kind para criar um cluster local e fazer o deploy de uma API FastAPI:

  • Criar Deployment com 2 réplicas da API FastAPI (pode usar a aula 26)
  • Criar Service do tipo NodePort para acessar localmente
  • Criar ConfigMap com variáveis de ambiente (DATABASE_URL, LOG_LEVEL)
  • Configurar HPA baseado em CPU (min=2, max=5, target=70%)
  • Empacotar tudo num Helm Chart com values.yaml
  • Fazer rolling update para uma "nova versão" e verificar zero downtime
  • Testar rollback: kubectl rollout undo deployment/minha-api
💬 Dica: Instale minikube com brew install minikube (Mac) ou baixe o binário em minikube.sigs.k8s.io. Inicie com minikube start --cpus=2 --memory=4096 para ter recursos suficientes.
